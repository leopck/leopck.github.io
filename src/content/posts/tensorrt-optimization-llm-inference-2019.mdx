TensorRT is NVIDIA's inference optimization toolkit -- it takes a trained model and produces a highly optimized engine through layer fusion, precision calibration, kernel auto-tuning, and memory optimization. For transformer-based models, TensorRT typically delivers 2-4x latency reduction over vanilla PyTorch inference. Understanding what it does (and doesn't) optimize is essential for deployment.

## Precision Calibration

TensorRT supports FP32, FP16, and INT8. FP16 is nearly free (under 0.1% accuracy loss on most models). INT8 requires calibration but delivers another 2x on Turing+:

```python
import tensorrt as trt

def build_engine(onnx_path, precision="fp16", calibrator=None):
    """Build optimized TensorRT engine from ONNX model."""
    builder = trt.Builder(trt.Logger(trt.Logger.WARNING))
    network = builder.create_network(1 << int(trt.NetworkDefinitionCreationFlag.EXPLICIT_BATCH))
    parser = trt.OnnxParser(network, trt.Logger(trt.Logger.WARNING))

    with open(onnx_path, 'rb') as f:
        parser.parse(f.read())

    config = builder.create_builder_config()
    config.set_memory_pool_limit(trt.MemoryPoolType.WORKSPACE, 1 <under 30)  # 1 GB

    if precision == "fp16":
        config.set_flag(trt.BuilderFlag.FP16)
    elif precision == "int8":
        config.set_flag(trt.BuilderFlag.INT8)
        config.int8_calibrator = calibrator  # Calibration dataset required

    engine = builder.build_serialized_network(network, config)
    return engine
```

  FP16 is essentially free accuracy-wise and gives 2x speedup. Only pursue INT8 if FP16 latency is insufficient. For INT8, try PTQ first (faster to deploy) -- if accuracy loss exceeds 1%, switch to QAT.
</Callout>

## Dynamic Shapes and Batching

LLM inference has variable sequence lengths and batch sizes. TensorRT supports optimization profiles for dynamic shapes:

```python
# Define optimization profiles for variable batch/sequence
profile = builder.create_optimization_profile()
profile.set_shape(
    "input_ids",
    min=(1, 1),      # batch=1, seq=1 (single token decode)
    opt=(16, 128),    # batch=16, seq=128 (typical serving)
    max=(64, 2048),   # batch=64, seq=2048 (max capacity)
)
config.add_optimization_profile(profile)
```

TensorRT auto-tunes kernels for the `opt` shape but must work for any shape in `[min, max]`. Setting `opt` close to your actual serving distribution improves kernel selection.

TensorRT excels at single-request latency but doesn't handle continuous batching or paged KV cache natively. TensorRT-LLM (released 2023) combines TensorRT's kernel optimization with LLM-specific features like PagedAttention and in-flight batching.

<Callout type="info" title="TensorRT-LLM vs vLLM">
  For production LLM serving in 2024+: **TensorRT-LLM** if you're on NVIDIA hardware and want maximum performance (requires more setup). **vLLM** for portability, faster iteration, and good-enough performance. Both use similar techniques (continuous batching, paged KV) -- the difference is kernel-level optimization depth.
</Callout>

## Conclusion

TensorRT delivers 2-4x inference speedup through layer fusion, precision calibration, and kernel auto-tuning. The optimization priority: start with FP16 (2x speedup, negligible accuracy loss), add INT8 if needed (additional 1.5-2x), and ensure the optimization profile matches your serving workload. For LLM-specific deployments, TensorRT-LLM extends TensorRT with continuous batching and paged KV cache. The build-time investment (minutes to hours) is negligible compared to the runtime savings across millions of inference requests.
