Tensor parallelism (TP) splits individual layers across GPUs -- each GPU computes a fraction of every layer's output. Unlike pipeline parallelism (which assigns entire layers to GPUs), TP requires an all-reduce after every layer to combine partial results. This makes TP bandwidth-hungry but pipeline-bubble-free.

## How TP Splits a Transformer Layer

The Megatron-LM strategy splits linear layers along different dimensions in a way that minimizes communication:

/>

The key insight: by splitting the first linear column-wise and the second row-wise, you need only **one all-reduce per layer** instead of two. The activation function happens on the partitioned data without communication.

<Callout type="perf" title="Why This Split Pattern?">
  Column-parallel -> activation -> row-parallel means the activation is computed on already-partitioned data (no all-reduce needed between them). The only synchronization point is after the row-parallel matmul, where partial sums from each GPU are combined via all-reduce. This halves the number of all-reduces compared to naive TP.
</Callout>

<Callout type="info" title="The Production Pattern: TP + PP + DP">
  Large-scale training uses all three: TP within a node (8 GPUs connected by NVLink), PP across nodes (connected by InfiniBand), and DP across node groups. For example, a 64-GPU setup might use TP=8 x PP=2 x DP=4. Inference serving typically uses TP only (within a single node) because pipeline parallelism adds latency.
</Callout>

The KV head count constrains maximum TP degree. Models with MQA (1 KV head) can only use TP if the implementation broadcasts the single KV head -- which adds communication but is still efficient via NVLink broadcast.

## Implementation Patterns

The actual TP implementation requires two custom all-reduce points per layer:

```python
# Simplified Megatron-LM style TP for FFN
class TensorParallelFFN(nn.Module):
    def __init__(self, d_model, d_ff, tp_rank, tp_world):
        super().__init__()
        # Column-parallel: split output dim
        self.up_proj = nn.Linear(d_model, d_ff // tp_world, bias=False)
        # Row-parallel: split input dim
        self.down_proj = nn.Linear(d_ff // tp_world, d_model, bias=False)
        self.tp_world = tp_world

    def forward(self, x):
        h = F.silu(self.up_proj(x))     # Local computation, no comm
        out = self.down_proj(h)           # Local partial result
        dist.all_reduce(out)              # Sum partials across TP group
        return out
```

## Conclusion

Tensor parallelism splits individual layers across GPUs, requiring all-reduce communication after each layer's parallel sections. The Megatron-LM column/row split minimizes this to 2 all-reduces per transformer layer. TP scales well to 4-8 GPUs on NVLink (80-92% efficiency) but degrades on lower-bandwidth interconnects like PCIe. For LLM serving, TP is the default parallelism strategy within a single node -- it has no pipeline bubble and adds minimal latency. For cross-node scaling, combine TP (intra-node) with pipeline parallelism (inter-node).
