Standard attention materializes an $N \times N$ attention matrix in GPU HBM for every head in every layer. For a 4096-token sequence with 32 attention heads across 80 layers, that is over **40 GB** of intermediate data written to and read from HBM — data that's computed, used once, and thrown away. This single bottleneck limits attention throughput to a fraction of what the GPU can deliver.

FlashAttention eliminates this waste entirely. By restructuring attention as a tiled computation that fits in on-chip SRAM, it reduces HBM traffic from $O(N^2)$ to $O(N)$ while computing the exact same mathematical result. No approximation, no quality loss — just a fundamentally better way to move data through the memory hierarchy.

## The Problem: Attention Is Memory-Bound

Standard scaled dot-product attention computes:

$$\text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right) V$$

For sequence length $N$ and head dimension $d$, this involves:

1. **Compute $S = QK^T$**: Read Q and K from HBM ($2Nd$ bytes each), write $S$ to HBM ($N^2$ elements)
2. **Compute $P = \text{softmax}(S)$**: Read $S$ from HBM, write $P$ to HBM ($N^2$ elements each direction)
3. **Compute $O = PV$**: Read $P$ and $V$ from HBM, write $O$ to HBM

Total HBM traffic: $O(Nd + N^2)$ bytes. At $N = 4096$ and $d = 128$, the $N^2$ term dominates overwhelmingly.

The arithmetic intensity of standard attention is approximately:

$$\text{AI}_{\text{standard}} = \frac{2N^2 d + 2N^2 d}{2Nd \cdot 3 + 2N^2 \cdot 3} \approx \frac{4N^2 d}{6N^2} = \frac{2d}{3} \approx 85 \text{ FLOP/byte}$$

But this is misleading — the actual bottleneck isn't the compute-to-byte ratio of the algorithm, it's the **memory traffic pattern**. The $N^2$-sized attention matrix is written to HBM and immediately read back. This round-trip through slow memory is pure waste.

<Callout type="perf" title="The Core Insight">
Standard attention performs $O(N^2 d)$ FLOPs and moves $O(N^2 + Nd)$ bytes through HBM. FlashAttention performs the **same FLOPs** but moves only $O(N^2 d^2 / M)$ bytes, where $M$ is SRAM size. Since $M \gg d^2$, this is effectively $O(Nd)$ — linear in sequence length.
</Callout>

## FlashAttention: Tiling to SRAM

The key idea: never materialize the full $N \times N$ attention matrix. Instead, compute attention in **tiles** that fit entirely in SRAM (shared memory).

The bandwidth gap between SRAM and HBM is **~10x**. If we can keep the attention computation in SRAM, we avoid the $N^2$ HBM round-trip entirely.

### The Tiling Strategy

FlashAttention divides Q into blocks of $B_r$ rows and K, V into blocks of $B_c$ rows. For each block of Q:

1. Load $Q_i$ block from HBM to SRAM (once)
2. For each K, V block:
   - Load $K_j$, $V_j$ from HBM to SRAM
   - Compute $S_{ij} = Q_i K_j^T$ in SRAM (never written to HBM)
   - Compute local softmax and accumulate $O_i$ in SRAM
3. Write final $O_i$ to HBM (once)

The critical point: the attention matrix tile $S_{ij}$ of size $B_r \times B_c$ stays entirely in SRAM. It is computed, used for softmax and the attention-value product, then discarded. **No HBM write for the attention matrix at all.**

### Tile Size Selection

SRAM on A100 is 192 KB per SM (when configured for maximum shared memory). Each tile must fit:

$$B_r \times d + B_c \times d + B_r \times B_c + B_r \times d \leq M$$

For $d = 128$ and FP16 (2 bytes), with $M = 192 \text{ KB}$:

$$B_r \times 256 + B_c \times 256 + B_r \times B_c \times 2 + B_r \times 256 \leq 196608$$

With $B_r = B_c = 128$: $128 \times 256 + 128 \times 256 + 128 \times 128 \times 2 + 128 \times 256 = 32768 + 32768 + 32768 + 32768 = 131072$ bytes = 128 KB. Fits comfortably.

### HBM Traffic Analysis

With tiling, the HBM reads become:

- **Q**: read once = $Nd$ elements = $O(Nd)$
- **K, V**: each block of K, V is read once per Q block. Total K reads = $(N/B_r) \times N \times d = N^2 d / B_r$. With $B_r = 128$: factor of 128 fewer than naive.
- **Output O**: write once = $Nd$ elements = $O(Nd)$

Total: $O(N^2 d^2 / M)$ where $M$ is SRAM size. Since $M$ is typically much larger than $d^2$, this approaches $O(Nd)$.

## The Online Softmax Trick

Tiling attention is conceptually simple — the hard part is **softmax**. Standard softmax requires seeing ALL values before computing probabilities:

$$\text{softmax}(x_i) = \frac{e^{x_i}}{\sum_j e^{x_j}}$$

The denominator requires the full row of attention scores. With tiling, we only see one block of scores at a time. The breakthrough: **online softmax** computes the result incrementally.

### Algorithm

For each row, maintain a running maximum $m$ and running sum $\ell$:

```python
# Processing tiles left-to-right for one query row
m_prev = -inf
ell_prev = 0
O_prev = zeros(d)

for j in range(num_kv_blocks):
    # Compute local scores for this tile
    S_ij = Q_i @ K_j.T / sqrt(d)

    # Update running statistics
    m_new = max(m_prev, max(S_ij))

    # Rescale previous accumulator to account for new maximum
    correction = exp(m_prev - m_new)
    ell_new = correction * ell_prev + sum(exp(S_ij - m_new))

    # Update output: rescale old output + add new contribution
    P_ij = exp(S_ij - m_new)
    O_new = correction * O_prev + P_ij @ V_j

    m_prev, ell_prev, O_prev = m_new, ell_new, O_new

# Final normalization
O = O_prev / ell_prev
```

The key insight: when we encounter a new maximum, we can **rescale** the previous accumulator by $e^{m_{\text{prev}} - m_{\text{new}}}$. This is numerically stable (the correction factor is always $\leq 1$) and mathematically exact.

<Callout type="info" title="Why Online Softmax Is Exact">
The rescaling $e^{m_{\text{old}} - m_{\text{new}}}$ applied to all previous terms is equivalent to computing softmax with the global maximum from the start. No approximation is involved — the final result is bit-identical to standard softmax (up to floating-point associativity).
</Callout>

## FlashAttention-2: Better Work Partitioning

FlashAttention-1 achieved the algorithmic breakthrough. FlashAttention-2 (2023) improved the GPU implementation:

### Key FA2 Improvements

1. **Reduced non-matmul FLOPs**: FA1 spent significant time on rescaling, max-finding, and exp computation. FA2 restructured these to minimize non-tensor-core work.

2. **Better parallelism across sequence length**: FA1 parallelized over batch and heads. FA2 additionally parallelizes over the sequence dimension — critical for long sequences where batch × heads is small.

3. **Better warp partitioning**: FA1 split work across warps within a thread block suboptimally. FA2 assigns each warp to compute a portion of the output, reducing shared memory reads/writes between warps.

4. **Forward pass optimization**: Achieved ~73% of theoretical max FLOPS on A100 (up from ~50% in FA1).

## FlashAttention-3: Exploiting Hopper

FlashAttention-3 (2024) targets NVIDIA Hopper architecture (H100) with three hardware-specific optimizations:

### 1. Asynchronous WGMMA (Warp Group Matrix Multiply-Accumulate)

Hopper introduces WGMMA — a 128-thread (4-warp) matrix multiply that executes asynchronously. While one WGMMA computes the current tile's attention scores, another can simultaneously compute the previous tile's output accumulation.

```
Timeline (FA3 on Hopper):
Tile j:   [WGMMA: Q_i @ K_j.T] [softmax] [WGMMA: P_ij @ V_j]
Tile j+1:                        [WGMMA: Q_i @ K_{j+1}.T] [softmax] [WGMMA: P @ V]
                                  ^--- overlapped with previous tile's AV compute
```

### 2. TMA (Tensor Memory Accelerator)

Hopper's TMA hardware loads data from global memory to shared memory without using any CUDA threads. This frees the SM's compute units to do actual work while data loads happen in the background.

In FA3, TMA prefetches the next K, V blocks while the current tile is being computed — fully overlapping memory loads with computation.

### 3. Warp Specialization

FA3 divides warps into **producers** (handle data loading via TMA) and **consumers** (handle computation via WGMMA). This is similar to software pipelining in CPUs but at the warp granularity.

### 4. FP8 Support

Hopper's WGMMA natively supports FP8 (E4M3) matrix multiply. FA3 can operate in FP8 with ~2x throughput over FP16, with per-tile scaling to maintain numerical accuracy.

## When FlashAttention Is Slower

FlashAttention is not universally faster. There are specific scenarios where standard attention wins:

### Very Short Sequences (N &lt; 256)

For short sequences, the $N^2$ attention matrix is small enough to fit in L2 cache. The tiling overhead (online softmax bookkeeping, tile loop, rescaling) exceeds the HBM savings. cuBLAS-based attention with GEMM fusion can be faster.

### When You Need the Attention Matrix

Some applications require the actual attention weights — interpretability tools, attention visualization, certain architectural modifications. FlashAttention never materializes this matrix. You'd need to run standard attention separately to extract it.

### Very Small Head Dimensions

When $d$ is very small (e.g., $d = 32$), the tiles become tiny, reducing the ratio of compute to overhead. This is rare in modern LLMs (most use $d = 128$).

<Callout type="warning" title="The Backward Pass Is Different">
FlashAttention's backward pass must recompute the attention matrix from stored Q, K, V and the softmax statistics (running max and sum). This recomputation costs extra FLOPs but saves the HBM traffic of storing the $N^2$ attention matrix for backward. The tradeoff is overwhelmingly positive for long sequences.
</Callout>

## Integration with GQA/MQA

FlashAttention integrates naturally with grouped-query attention. When using GQA with $G$ KV groups and $H$ query heads ($H/G$ queries share each KV group):

- The K, V blocks are loaded once per group
- $H/G$ query blocks compute attention against the same K, V
- Effective bandwidth amplification: the K, V data loaded for one query group is reused by $H/G$ queries

This makes FlashAttention even more efficient with GQA — the KV data reuse within SRAM is amplified.

### torch SDPA Backend Selection

PyTorch's `scaled_dot_product_attention` automatically selects the best backend:

```python
import torch
import torch.nn.functional as F

# PyTorch automatically selects FlashAttention when available
output = F.scaled_dot_product_attention(
    query, key, value,
    attn_mask=None,       # Causal masking handled internally
    is_causal=True,       # Enable causal FlashAttention path
    scale=1.0 / math.sqrt(d),
)

# Backend selection priority:
# 1. FlashAttention (if available and inputs are FP16/BF16 on CUDA)
# 2. Memory-efficient attention (xformers-based)
# 3. Standard math attention (fallback)
```

vLLM and SGLang both use FlashAttention for prefill and memory-efficient paged attention kernels for decode (where the paged KV cache layout requires custom kernels).

## Roofline Analysis

Let's place both standard and flash attention on the roofline for an A100:

- **A100 specs**: 312 TFLOPS FP16, 2 TB/s HBM bandwidth
- **Ridge point**: $312 / 2 = 156$ FLOP/byte

**Standard attention at N=4096, d=128:**
- FLOPs: $\sim 2 \times 4096^2 \times 128 \times 2 = 8.6 \times 10^{9}$
- HBM bytes: $\sim 67 \times 10^6 \times 2 = 1.34 \times 10^8$
- Arithmetic intensity: $\sim 64$ FLOP/byte — below ridge point, **memory-bound**

**FlashAttention at N=4096, d=128:**
- FLOPs: same $\sim 8.6 \times 10^{9}$ (identical computation)
- HBM bytes: $\sim 4 \times 10^6 \times 2 = 8 \times 10^6$
- Arithmetic intensity: $\sim 1075$ FLOP/byte — well above ridge point, **compute-bound**

## Practical Impact

FlashAttention is now the default attention implementation in virtually every LLM framework:

- **PyTorch**: Built into `F.scaled_dot_product_attention`
- **vLLM**: Uses FA for prefill, custom paged kernels for decode
- **SGLang**: FA with RadixAttention for prefix caching
- **TensorRT-LLM**: Custom FA integration with FP8 support
- **Hugging Face**: Automatic FA2 selection via `attn_implementation="flash_attention_2"`

The impact extends beyond raw speed — by making attention memory-efficient, FlashAttention enabled the jump from 2K to 128K+ context lengths that defines modern LLMs.

<Callout type="tip" title="The Lesson Beyond FlashAttention">
FlashAttention's real contribution isn't a clever algorithm — it's a demonstration that **IO-aware algorithm design** (designing algorithms around the memory hierarchy, not just FLOP counts) can yield order-of-magnitude improvements. The same principle applies to any GPU kernel: minimize HBM traffic, maximize SRAM reuse.
</Callout>

## Conclusion

FlashAttention eliminates the $O(N^2)$ HBM bottleneck in attention by tiling the computation to fit in SRAM, using online softmax to process tiles incrementally. The result: same mathematical output, but with $O(N)$ memory traffic instead of $O(N^2)$. On modern GPUs, this translates to 2-10x speedups for typical sequence lengths and enables context lengths that were previously impossible.

The evolution from FA1 (algorithmic breakthrough) to FA2 (GPU implementation optimization) to FA3 (Hopper hardware exploitation) shows how a single insight — respect the memory hierarchy — compounds across hardware generations. Every modern LLM serving system uses FlashAttention, making it arguably the most impactful single optimization in the transformer era.
