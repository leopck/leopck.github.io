Standard attention requires O(N^2) HBM reads/writes for sequence length N. FlashAttention restructures the algorithm to achieve O(N) HBM access while performing the exact same computation. This isn't magic -- it's careful exploitation of the GPU memory hierarchy.

## The Memory Bandwidth Problem

Consider attention for a single head with sequence length N=4096 and head dimension d=128:

**Standard Attention Memory Traffic:**
- Q, K, V load: 3 x N x d x 2 bytes = 3.14 MB
- S = QK^T store: N x N x 2 bytes = 33.5 MB
- S load for softmax: 33.5 MB
- P store after softmax: 33.5 MB
- P load + V load: 34.0 MB
- O store: 1.05 MB
- **Total: ~139 MB per head** -> For 32 heads: **4.4 GB** per layer

**FlashAttention Memory Traffic:**
- Q, K, V load: 3.14 MB
- O store: 1.05 MB
- **Total: ~4.2 MB per head** -> For 32 heads: **134 MB** per layer

## The Tiling Strategy

FlashAttention divides Q, K, V into blocks that fit in SRAM:

```cpp
// Conceptual tiling -- block sizes chosen to fit in shared memory
constexpr int Br = 128;  // Q block rows
constexpr int Bc = 128;  // K, V block columns
constexpr int d = 128;   // Head dimension

// SRAM usage per thread block:
// Q block:  Br x d x 2 bytes = 32 KB
// K block:  Bc x d x 2 bytes = 32 KB
// V block:  Bc x d x 2 bytes = 32 KB
// O block:  Br x d x 2 bytes = 32 KB
// S block:  Br x Bc x 2 bytes = 32 KB
// Total: ~160 KB (fits in 192KB shared memory)
```

<Callout type="perf" title="Block Size Selection">
  Optimal block sizes depend on head dimension and shared memory capacity. For A100 with d=128, Br=Bc=128 achieves ~85% shared memory utilization. FlashAttention-2 further optimizes by splitting work across warps within each thread block.
</Callout>

## The Online Softmax Trick

Standard softmax requires two passes over the NxN score matrix: one to find the max, one to normalize. FlashAttention uses **online softmax** to process tiles incrementally, maintaining running statistics:

```python
def online_softmax_attention_block(Q_block, K_block, V_block,
                                    O_prev, l_prev, m_prev):
    """
    Process one K,V block while maintaining running softmax statistics.
    Each call updates the output accumulator without needing the full score matrix.
    """
    # Compute attention scores for this block only
    S = Q_block @ K_block.T  # [Br, Bc] -- fits in SRAM

    # Update running max
    m_block = S.max(dim=-1)
    m_new = torch.maximum(m_prev, m_block)

    # Rescale previous accumulator for new max
    scale_prev = torch.exp(m_prev - m_new)
    O_prev_scaled = O_prev * scale_prev.unsqueeze(-1)
    l_prev_scaled = l_prev * scale_prev

    # Compute new block contribution
    P_block = torch.exp(S - m_new.unsqueeze(-1))
    l_block = P_block.sum(dim=-1)

    # Accumulate
    O_new = O_prev_scaled + P_block @ V_block
    l_new = l_prev_scaled + l_block

    return O_new, l_new, m_new

# After processing ALL K,V blocks:
O_final = O_new / l_new.unsqueeze(-1)
```

The critical property: at no point does an NxN matrix exist in HBM. The BrxBc tile `S` lives entirely in shared memory and is consumed immediately.

FlashAttention moves attention from **memory-bound** to **compute-bound** by increasing arithmetic intensity 30x. FlashAttention-2 pushes further by reducing non-matmul FLOPs and improving warp-level parallelism.

## FlashAttention-2 Key Improvements

FlashAttention-2 achieves 50-73% of theoretical peak FLOPS (vs ~30-50% for v1) through three changes:

**Reduced non-matmul FLOPs.** The rescaling of O_prev happens less frequently -- deferred until the end rather than every block iteration. This reduces register pressure and lets more warps execute matmul operations.

**Better parallelism.** FA-1 parallelizes over batch and heads. FA-2 additionally parallelizes over sequence length dimension, which matters for long sequences where batchxheads alone doesn't saturate all SMs.

**Improved warp scheduling.** FA-2 splits work across warps within a thread block more efficiently, reducing synchronization barriers and improving tensor core utilization.

## When FlashAttention Isn't Optimal

FlashAttention has overhead for:
- **Very short sequences** (N under 256): Tiling overhead dominates
- **Very small batch sizes**: Can't saturate SMs
- **Non-standard attention patterns**: Sliding window, cross-attention, etc. require custom kernels

<Callout type="info" title="Sequence Length Crossover">
  For N under 512, cuBLAS GEMM-based attention often outperforms FlashAttention due to lower kernel launch overhead and better SM utilization at small problem sizes. Always benchmark your specific workload.
</Callout>

## Conclusion

FlashAttention's 33x reduction in HBM traffic comes from restructuring the attention algorithm, not from approximation. The NxN intermediate score matrix -- which accounts for 97% of standard attention's memory traffic -- never materializes in HBM. Instead, it's computed and consumed in SRAM-sized tiles via the online softmax trick.

The key insight generalizes beyond attention: **any O(N^2) intermediate tensor that can be computed on-the-fly should be.** This principle applies to any algorithm where large intermediates are produced only to be immediately consumed.
