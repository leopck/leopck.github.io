## What a Transformer Layer Does

A transformer is a stack of identical layers (Llama 70B has 80 layers). Each layer:

1. **Attention**: Each token looks at all previous tokens and computes a weighted average of their representations. The weights are the "attention scores" — how relevant each previous token is to the current one.
2. **FFN**: Each token independently passes through a feed-forward network (two large matrix multiplies with a nonlinearity). This is where factual knowledge is stored.

Both operations produce an output of the same shape as the input: `[batch, seq_len, d_model]`. Residual connections add the layer's output to its input.

## KV Cache: Why It Exists

During generation, the model produces tokens one at a time. At each step, the attention mechanism needs to look at ALL previous tokens' Key and Value vectors. Without caching, you'd recompute K and V for all previous tokens every step — $O(n^2)$ total work for $n$ tokens.

**KV cache**: Store the K and V vectors from all previous tokens. Each new step only computes K, V for the new token and appends to the cache. Attention then reads the full cache.

**Cache size per token per layer**: $2 \times n_{\text{kv\_heads}} \times d_{\text{head}} \times \text{dtype\_bytes} = 2 \times 8 \times 128 \times 2 = 4{,}096$ bytes (Llama 70B, FP16).

**Across 80 layers**: $4{,}096 \times 80 = 327{,}680$ bytes = **320 KB per token**.

At batch=32 with 4096 tokens: $32 \times 4096 \times 320\text{KB} = 40$ GB of KV cache. This is why KV cache management (PagedAttention, quantization, eviction) is the central problem in inference optimization.

## Prefill vs Decode: Two Different Bottlenecks

  title="The Two Phases of LLM Inference"

**Prefill**: All prompt tokens processed simultaneously. Large batch matrix multiplies. High GPU utilization (60-80%). Bottleneck: tensor core compute throughput.

**Decode**: One token at a time. Must load all 140 GB of model weights from HBM to compute units for each single token. GPU utilization: 0.3% at batch=1. Bottleneck: HBM bandwidth (3.35 TB/s on H100).

**The arithmetic**: Decode processes 1 token. FLOPs: $2 \times 70\text{B} = 140\text{G}$ FLOPs. Bytes loaded: $70\text{B} \times 2 = 140$ GB. Arithmetic intensity: $140\text{G} / 140\text{G} = 1$ FLOP/byte. H100 ridge point: 295 FLOP/byte. Decode is **295x below** the compute ceiling — it is purely memory-bandwidth-limited.

This is the fundamental reason why every optimization in the Inference Optimization Timeline exists: batching (amortize weight loading across more tokens), quantization (reduce weight size), speculative decoding (generate multiple tokens per weight load), PagedAttention (maximize concurrent requests in limited memory).

## What You Need to Know for This Series

With this bridge knowledge, you can now understand:
- **Why KV cache is the memory bottleneck** (it grows with batch size and sequence length)
- **Why prefill and decode need different optimizations** (different bottlenecks)
- **Why batching is the most important optimization** (amortizes the weight-loading cost)
- **Why PagedAttention matters** (reduces KV cache memory waste from 68% to near-zero)
- **Why speculative decoding works** (verification of K tokens costs the same as generating 1)
