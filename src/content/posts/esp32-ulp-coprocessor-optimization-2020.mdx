The ESP32 ULP (Ultra Low Power) coprocessor is a small, independent processor core that can execute programs while the main CPU is in deep sleep. Operating at approximately 150 microamps -- three orders of magnitude less than the main CPU -- it enables continuous sensor monitoring on battery-powered devices that need to last months or years on a single charge. This post provides a complete technical treatment of the ULP architecture, its constrained instruction set, power characteristics, and practical programming patterns for real sensor applications.

## ULP Architecture Overview

The ESP32 contains a ULP coprocessor based on a Finite State Machine (FSM) design. This is distinct from the RISC-V ULP found in the ESP32-S2 and ESP32-S3. The original ESP32's ULP FSM has a deliberately minimal architecture optimized for power efficiency rather than computational capability.

<Callout type="warning" title="8 KB Is Shared Between Code and Data">
  The ULP program, ULP data, and any `RTC_DATA_ATTR` variables from the main CPU all share the same 8 KB of RTC slow memory. A complex ULP program with many instructions leaves less room for data storage. Each ULP instruction is 4 bytes, so 512 instructions consume 2 KB. Plan your memory allocation carefully -- there is no runtime error if you overwrite code with data or vice versa, just silent corruption.
</Callout>

## ULP Instruction Set Reference

The ULP FSM has a deliberately limited instruction set. It can perform basic arithmetic, memory access, ADC reads, I2C operations, GPIO control, and program flow control. It cannot perform multiplication, division, floating-point operations, or function calls.

### Duty Cycle and Average Power

The ULP typically runs a short program (10-100 instructions) then halts, waiting for the RTC timer to wake it again. The average current depends on the duty cycle:

$$
I_{avg} = I_{sleep} \cdot (1 - D) + I_{active} \cdot D
$$

Where $D$ is the duty cycle (fraction of time the ULP is executing):

$$
D = \frac{T_{execution}}{T_{period}}
$$

<Callout type="tip" title="ULP Execution Time Is Almost Irrelevant for Power">
  For typical ULP programs (ADC read + threshold check = ~50 us), the execution time is so short relative to the polling interval that the ULP's 150 uA active current barely affects average power. The dominant factor is the deep sleep base current (5-10 uA). This means you can poll at 10 Hz instead of 1 Hz with virtually no power penalty -- the difference is 10.1 uA vs 10.0 uA. Focus your power optimization efforts on minimizing the deep sleep base current (disable unused RTC peripherals, use external 32.768 kHz crystal, etc.) rather than optimizing ULP execution time.
</Callout>

## Programming the ULP: Practical Examples

### Example 1: ADC Threshold Detection with Wake-Up

The most common ULP use case: read an ADC channel, compare against a threshold, and wake the main CPU if the threshold is exceeded.

```c
#include "esp32/ulp.h"
#include "driver/rtc_io.h"
#include "driver/adc.h"
#include "soc/rtc_cntl_reg.h"

// RTC memory variables shared between ULP and main CPU
RTC_DATA_ATTR uint16_t ulp_last_reading = 0;
RTC_DATA_ATTR uint16_t ulp_threshold = 2048;  // Midscale (12-bit)
RTC_DATA_ATTR uint16_t ulp_wake_count = 0;
RTC_DATA_ATTR uint16_t ulp_sample_count = 0;

// ULP program entry point address (in 32-bit words from start of RTC_SLOW_MEM)
#define ULP_DATA_OFFSET  0       // Where data variables are stored
#define ULP_CODE_OFFSET  16      // Where program code starts (after data)

// Data layout in RTC_SLOW_MEM:
// Word 0: last ADC reading (written by ULP)
// Word 1: threshold value (written by main CPU)
// Word 2: wake count (incremented by ULP)
// Word 3: sample count (incremented by ULP)

void setup_ulp_adc_threshold(void) {
    // Configure ADC for ULP access
    adc1_config_channel_atten(ADC1_CHANNEL_6, ADC_ATTEN_DB_11);
    adc1_config_width(ADC_WIDTH_BIT_12);

    // Allow ADC to be used by ULP
    adc1_ulp_enable();

    // ULP program
    const ulp_insn_t program[] = {
        // Load threshold from RTC memory word 1
        I_LD(R2, R3, 1),              // R2 = threshold (R3=0 assumed)

        // Read ADC1 channel 6
        I_ADC(R0, 0, 6),              // R0 = ADC reading (channel 6 of ADC1)

        // Store reading to RTC memory word 0
        I_ST(R0, R3, 0),              // Store for main CPU to read

        // Increment sample counter (word 3)
        I_LD(R1, R3, 3),              // R1 = current sample count
        I_ADDI(R1, R1, 1),            // R1++
        I_ST(R1, R3, 3),              // Store back

        // Compare reading (R0) against threshold (R2)
        // JUMPR compares R0 against an immediate value
        // We need: if R0 >= threshold, wake
        // Strategy: subtract threshold from reading, check if result >= 0
        I_SUBR(R0, R0, R2),           // R0 = reading - threshold

        // If R0 >= 0 (reading >= threshold), jump to wake
        // JUMPR: jump if R0 [condition] immediate
        I_JUMPR(5, 0, JUMPR_GE),      // If R0 >= 0, jump +5 instructions (to WAKE)

        // Reading is below threshold: halt and wait for next timer wake
        I_HALT(),

        // === Wake path ===
        // Increment wake counter (word 2)
        I_LD(R1, R3, 2),              // R1 = wake count
        I_ADDI(R1, R1, 1),            // R1++
        I_ST(R1, R3, 2),              // Store back

        // Wake the main CPU
        I_WAKE(),
        I_HALT(),
    };

    // Load and start ULP program
    size_t program_size = sizeof(program) / sizeof(ulp_insn_t);
    ulp_process_macros_and_load(ULP_CODE_OFFSET, program, &program_size);

    // Initialize data words
    RTC_SLOW_MEM[0] = 0;                // Last reading
    RTC_SLOW_MEM[1] = ulp_threshold;    // Threshold
    RTC_SLOW_MEM[2] = 0;                // Wake count
    RTC_SLOW_MEM[3] = 0;                // Sample count

    // Set ULP wake-up interval: 100 ms
    // Timer counts in RTC_SLOW_CLK cycles (150 kHz)
    // 100 ms = 15000 cycles
    ulp_set_wakeup_period(0, 100 * 1000);  // Microseconds

    // Start ULP
    ulp_run(ULP_CODE_OFFSET);

    // Enable ULP as a wake-up source
    esp_sleep_enable_ulp_wakeup();
}

// Called after wake from deep sleep to read ULP data
void read_ulp_results(void) {
    uint16_t last_reading = RTC_SLOW_MEM[0] & 0xFFFF;
    uint16_t threshold = RTC_SLOW_MEM[1] & 0xFFFF;
    uint16_t wake_count = RTC_SLOW_MEM[2] & 0xFFFF;
    uint16_t sample_count = RTC_SLOW_MEM[3] & 0xFFFF;

    printf("ULP ADC Monitor Results:\n");
    printf("  Last reading: %u (threshold: %u)\n", last_reading, threshold);
    printf("  Total samples: %u\n", sample_count);
    printf("  Wake events: %u\n", wake_count);
}
```

### Example 2: Multi-Sample Averaging with Buffer

The ULP can collect multiple samples and store them in a ring buffer in RTC memory for the main CPU to process:

```c
// Data layout:
// Word 0: buffer write index (0 to BUFFER_SIZE-1)
// Word 1: buffer full flag (0 or 1)
// Words 4 to 4+BUFFER_SIZE-1: sample buffer

#define BUFFER_SIZE    64
#define BUF_START      4

const ulp_insn_t sampling_program[] = {
    // Read ADC
    I_ADC(R0, 0, 6),               // R0 = ADC reading

    // Load current write index
    I_MOVI(R3, 0),                  // R3 = 0 (base address)
    I_LD(R1, R3, 0),               // R1 = write index

    // Calculate buffer address: BUF_START + index
    I_ADDI(R2, R1, BUF_START),     // R2 = BUF_START + index

    // Store sample at buffer[index]
    I_ST(R0, R2, 0),               // RTC_SLOW_MEM[R2] = ADC reading

    // Increment index
    I_ADDI(R1, R1, 1),             // index++

    // Check if buffer is full
    I_MOVR(R0, R1),                // R0 = index (for JUMPR comparison)
    I_JUMPR(4, BUFFER_SIZE, JUMPR_GE),  // If index >= BUFFER_SIZE, jump to full

    // Buffer not full: store updated index and halt
    I_ST(R1, R3, 0),               // Store new index
    I_HALT(),

    // === Buffer full ===
    // Reset index to 0
    I_MOVI(R1, 0),
    I_ST(R1, R3, 0),               // index = 0

    // Set buffer-full flag
    I_MOVI(R1, 1),
    I_ST(R1, R3, 1),               // flag = 1

    // Wake main CPU to process the buffer
    I_WAKE(),
    I_HALT(),
};
```

### Example 3: I2C Sensor Read from ULP

The ULP has its own I2C controller (RTC_I2C) that operates independently of the main I2C peripherals. It uses different pins (RTC GPIOs) and has limited configuration options.

```c
#include "soc/rtc_i2c_reg.h"
#include "soc/sens_reg.h"

// RTC I2C pin configuration
#define RTC_I2C_SCL_PIN   2   // GPIO 2 (RTC GPIO 12)
#define RTC_I2C_SDA_PIN   4   // GPIO 4 (RTC GPIO 10)

void configure_rtc_i2c(void) {
    // Configure RTC I2C pins
    rtc_gpio_init(RTC_I2C_SCL_PIN);
    rtc_gpio_init(RTC_I2C_SDA_PIN);
    rtc_gpio_set_direction(RTC_I2C_SCL_PIN, RTC_GPIO_MODE_INPUT_OUTPUT_OD);
    rtc_gpio_set_direction(RTC_I2C_SDA_PIN, RTC_GPIO_MODE_INPUT_OUTPUT_OD);

    // Enable RTC I2C controller
    SET_PERI_REG_MASK(RTC_I2C_CTRL_REG, RTC_I2C_CTRL_CLK_EN);

    // Configure I2C clock: RTC_FAST_CLK (8 MHz) / divider
    // For 100 kHz: divider = 80 (8 MHz / 80 = 100 kHz)
    REG_SET_FIELD(RTC_I2C_CTRL_REG, RTC_I2C_CTRL_SCL_LOW_PERIOD, 40);
    REG_SET_FIELD(RTC_I2C_CTRL_REG, RTC_I2C_CTRL_SCL_HIGH_PERIOD, 40);
}

// ULP program to read temperature from an I2C sensor (e.g., TMP102 at 0x48)
const ulp_insn_t i2c_read_program[] = {
    // Read temperature register (register 0x00) from TMP102 (address 0x48)
    // I2C_RD: slave_addr, register, high_bit, low_bit, i2c_sel
    I_I2C_RD(0x48, 0x00, 7, 0, 0),   // Read byte: R0 = temp MSB

    // Store temperature MSB
    I_MOVI(R3, 0),
    I_ST(R0, R3, 0),                   // RTC_SLOW_MEM[0] = temp MSB

    // Read LSB
    I_I2C_RD(0x48, 0x01, 7, 0, 0),   // R0 = temp LSB
    I_ST(R0, R3, 1),                   // RTC_SLOW_MEM[1] = temp LSB

    // Check threshold: if temp_MSB > 30 (30 degrees C), wake
    I_LD(R0, R3, 0),                   // R0 = temp MSB
    I_JUMPR(2, 30, JUMPR_GE),          // If temp >= 30, jump to wake

    I_HALT(),

    // Wake main CPU
    I_WAKE(),
    I_HALT(),
};
```

<Callout type="warning" title="RTC I2C Has Severe Limitations">
  The RTC I2C controller used by the ULP is a simplified implementation. It supports only standard mode (100 kHz), has no clock stretching support, limited error recovery, and uses different GPIO pins than the main I2C controllers. Not all I2C devices work reliably with the RTC I2C controller -- devices that require clock stretching or multi-byte transactions with repeated STARTs may fail. Test thoroughly with your specific sensor. For complex I2C interactions, consider using ULP GPIO bit-banging instead (slower but more controllable).
</Callout>

## Data Sharing Between ULP and Main CPU

The ULP and main CPU communicate through RTC slow memory. This shared memory region requires careful synchronization to avoid data corruption.

### Memory Access Patterns

```cpp
// Main CPU reading ULP data
void read_ulp_data(void) {
    // IMPORTANT: mask to 16 bits because ULP writes are 16-bit
    uint16_t sensor_value = RTC_SLOW_MEM[0] & 0xFFFF;
    uint16_t sample_count = RTC_SLOW_MEM[3] & 0xFFFF;

    // Convert ADC reading to voltage (12-bit reading, 3.3V range at 11dB atten)
    float voltage = (sensor_value / 4095.0f) * 3.3f;

    printf("Sensor: %u (%.3f V), Samples: %u\n",
           sensor_value, voltage, sample_count);
}

// Main CPU writing threshold for ULP
void set_ulp_threshold(uint16_t threshold) {
    // ULP reads the lower 16 bits
    RTC_SLOW_MEM[1] = threshold & 0xFFFF;
}
```

### Synchronization Considerations

The ULP and main CPU do not run simultaneously in the typical deep sleep use case (ULP runs while main CPU sleeps). However, during the brief period after the main CPU wakes up but before the ULP halts (or if both are running during normal operation), race conditions are possible:

```cpp
// Safe read pattern: snapshot the data atomically
typedef struct {
    uint16_t reading;
    uint16_t sample_count;
    uint16_t wake_count;
} ulp_snapshot_t;

ulp_snapshot_t take_ulp_snapshot(void) {
    ulp_snapshot_t snap;
    // Disable ULP timer temporarily to prevent ULP from modifying data
    // during our read
    REG_CLR_BIT(RTC_CNTL_STATE0_REG, RTC_CNTL_ULP_CP_SLP_TIMER_EN);

    snap.reading = RTC_SLOW_MEM[0] & 0xFFFF;
    snap.sample_count = RTC_SLOW_MEM[3] & 0xFFFF;
    snap.wake_count = RTC_SLOW_MEM[2] & 0xFFFF;

    // Re-enable ULP timer
    REG_SET_BIT(RTC_CNTL_STATE0_REG, RTC_CNTL_ULP_CP_SLP_TIMER_EN);

    return snap;
}
```

## Use Cases: Where ULP Excels

### Use Case 1: Battery Voltage Monitor

Monitor battery voltage via ADC and wake the main CPU when voltage drops below a critical threshold:

```c
// ULP monitors battery via voltage divider on ADC1_CH7 (GPIO 35)
// Voltage divider: 100k/100k, so ADC sees half of battery voltage
// At 11dB atten, full scale = 3.3V = 4095
// Battery 3.0V -> ADC sees 1.5V -> ADC code ~1862
// Battery 3.3V -> ADC sees 1.65V -> ADC code ~2048
// Battery 4.2V -> ADC sees 2.1V -> ADC code ~2606

// Threshold for low battery (3.3V): ADC code 2048
// Threshold for critical (3.0V): ADC code 1862
```

### Use Case 2: Motion Detection

Use an analog PIR sensor with the ULP reading the comparator output via GPIO:

```c
const ulp_insn_t pir_program[] = {
    // Read RTC GPIO (PIR comparator output)
    I_RD_REG(RTC_GPIO_IN_REG, RTC_GPIO_IN_NEXT_S + 14, // GPIO 14 bit position
             RTC_GPIO_IN_NEXT_S + 14),                   // Single bit read

    // R0 = 0 (no motion) or 1 (motion detected)
    I_JUMPR(2, 1, JUMPR_GE),          // If R0 >= 1 (motion), wake

    // No motion: increment no-motion counter and halt
    I_HALT(),

    // Motion detected: wake main CPU
    I_WAKE(),
    I_HALT(),
};
```

### Use Case 3: Environmental Sensor with Hysteresis

Read temperature periodically, wake main CPU only when temperature crosses a threshold band (with hysteresis to prevent repeated wake-ups at boundary):

```c
// Data layout:
// Word 0: current reading
// Word 1: upper threshold (e.g., 35 degrees -> ADC code)
// Word 2: lower threshold (e.g., 25 degrees -> ADC code)
// Word 3: last state (0 = below lower, 1 = above upper)

const ulp_insn_t hysteresis_program[] = {
    // Read ADC (temperature sensor)
    I_ADC(R0, 0, 6),                  // R0 = ADC reading
    I_MOVI(R3, 0),
    I_ST(R0, R3, 0),                  // Store reading

    // Load last state
    I_LD(R1, R3, 3),                  // R1 = last state

    // If last state = 0 (cold), check upper threshold
    I_MOVR(R2, R1),
    I_MOVI(R1, 0),                    // Compare value
    // ... (branch based on state to check appropriate threshold)

    // This structure provides hysteresis:
    // Wake only on transitions: cold->hot or hot->cold
    // Ignore readings within the dead band

    I_HALT(),
};

## Advanced: ULP with External Comparator for Zero-Power Threshold Detection

For the absolute minimum power, combine the ULP with a hardware comparator:

```
Architecture:
                                          +-- 3.3V
                                          |
  Sensor ----+---- [comparator] ---- RTC GPIO (input)
             |         |
             +--- [voltage ref]
                       |
                      GND

The comparator output goes directly to an RTC GPIO.
ULP reads the GPIO (digital, not ADC) in ~2 instructions.
No ADC conversion needed: execution time drops from ~25 us to ~2 us.
Energy per check: ~0.3 uJ instead of ~4 uJ.
```

This approach eliminates the ADC conversion entirely. The hardware comparator (e.g., TLV3691, consuming 75 nA) performs the analog comparison continuously, and the ULP only needs to check a digital pin. The total system current is approximately 10 uA (deep sleep) + 0.075 uA (comparator) = 10.1 uA, with the ULP executing for only 2 microseconds per check.

==============

1. DEEP SLEEP (99.95% of time): 10 uA
   - ULP wakes every 100 ms via RTC timer
   - Reads temperature (ADC), humidity (I2C), checks thresholds
   - If threshold exceeded: wakes main CPU (state 2)
   - Otherwise: halts and returns to deep sleep

2. SENSOR PROCESSING (0.03% of time): ~50 mA, 20 ms
   - Main CPU wakes, reads ULP buffer data
   - Applies calibration and averaging
   - Stores timestamped readings in RTC memory
   - Returns to deep sleep

3. DATA UPLOAD (0.02% of time): ~160 mA, 3 seconds every 15 minutes
   - Main CPU wakes, connects WiFi
   - Uploads buffered sensor data to server
   - Disconnects WiFi, returns to deep sleep

Average current calculation:
  Deep sleep: 10 uA * 0.9995 = 9.995 uA
  Processing: 50000 uA * 0.0003 = 15 uA
  Upload:     160000 uA * 0.0002 = 32 uA
  TOTAL: ~57 uA average

  Battery life (2000 mAh 18650): 2000000 / 57 / 8760 = 4.0 years
```

## Conclusion

The ESP32 ULP coprocessor occupies a unique niche in the power optimization toolkit. At approximately 150 microamps active current and 5-10 microamps when halted between measurements, it enables continuous sensor monitoring at power levels three orders of magnitude below the main CPU. The architecture's limitations -- four 16-bit registers, no stack, no multiplication, 8 KB shared memory -- are deliberate design choices that keep the power budget minimal.

The ULP is most valuable for threshold detection and periodic sensor polling at rates above 1 Hz. Below 1 Hz, the energy advantage over periodic main CPU wake-up is marginal. Above 10 Hz, the ULP saves 50% or more of the total system energy compared to main CPU wake-up, because the fixed boot overhead of the main CPU (approximately 1235 microjoules per wake cycle) dominates at high polling rates.

For practical deployment, the programming model requires assembly-level thinking: structure your program as a linear sequence of read-compare-branch-halt operations, use RTC memory as your shared data store with the main CPU, and keep programs short (under 30 instructions for most applications). The ADC and I2C peripheral access capabilities cover the majority of sensor interfacing needs, though the RTC I2C controller's limitations mean that complex I2C devices may require main CPU involvement.

The optimal system architecture uses the ULP as a low-power front-end: it monitors sensors continuously and wakes the main CPU only when something interesting happens (threshold crossing, buffer full, timeout). The main CPU handles the complex processing (calibration, filtering, WiFi transmission) and then returns to deep sleep. This division of labor -- simple monitoring on the ULP, complex processing on the main CPU -- maximizes battery life while maintaining full system capability.
