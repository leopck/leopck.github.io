LLM inference looks simple from the outside — feed in a prompt, get text back. But the performance characteristics of the two phases (prefill and decode) are fundamentally different, and understanding this difference is the foundation for every inference optimization.

## Two Phases, Two Bottlenecks

Every LLM inference request has two distinct phases:

### Prefill: Compute-Bound

During prefill, all prompt tokens are processed simultaneously. A 512-token prompt creates a 512×d_model input matrix, which flows through attention and FFN layers as large matrix multiplications. These matmuls have high arithmetic intensity and keep tensor cores busy — prefill is **compute-bound**.

### Decode: Memory-Bound

During decode, you generate one token per step. Each step requires loading the entire model's weights from HBM to compute a single output token. For a 7B parameter model in FP16 (14 GB), generating one token requires reading 14 GB of weights for a few thousand FLOPs of actual computation. This is absurdly memory-bound — arithmetic intensity is near zero.

<Callout type="perf" title="The Fundamental Insight">
  Decode speed is limited by how fast you can read model weights from HBM, not by how fast you can compute. An A100 with 2 TB/s bandwidth reading a 14 GB model takes ~7 ms per token — regardless of how many FLOPS the chip can do. This is why batching matters: serving 16 requests simultaneously reads weights once for 16 tokens' worth of computation.
</Callout>

TTFT is dominated by prefill time (proportional to prompt length). ITL is dominated by decode time (proportional to model size / memory bandwidth). These two metrics often conflict: larger batch sizes improve throughput but increase both TTFT (queueing) and ITL (shared bandwidth).

At batch=16, KV cache already exceeds model weights. At batch=64, KV cache would consume 64 GB — nearly the entire GPU. This is why KV cache optimization (quantization, paging, GQA) is the most impactful area for serving efficiency.

Throughput scales nearly linearly with batch size until memory bandwidth saturates. At batch=1, the GPU reads 14 GB of weights for 1 token. At batch=32, it reads the same 14 GB for 32 tokens — 32× better utilization. Beyond batch=64, HBM bandwidth is fully utilized and adding more requests just increases latency.

<Callout type="tip" title="Where to Start">
  If you're optimizing LLM inference, start with batching (biggest single improvement), then FlashAttention (prefill), then KV cache quantization (decode throughput via larger batches). These three cover 80% of achievable gains with modest engineering effort.

## Conclusion

LLM inference has two phases with opposite performance profiles: prefill is compute-bound (optimize with FlashAttention, tensor parallelism), decode is memory-bound (optimize with batching, quantization, efficient KV management). The single most impactful optimization is increasing batch size during decode, which amortizes the cost of reading model weights across more useful output tokens. Everything else — KV quantization, continuous batching, paged allocation — ultimately serves the goal of fitting more concurrent requests into the available GPU memory.
