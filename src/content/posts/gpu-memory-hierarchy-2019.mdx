GPU memory hierarchy is fundamentally different from CPU memory. Where CPUs optimize for single-thread latency with deep cache hierarchies, GPUs optimize for aggregate bandwidth with massive parallelism. Understanding where your data lives -- and the 1000x bandwidth difference between levels -- is the single most important factor in CUDA kernel performance.

The bandwidth ratio between registers and HBM is over **10x**. Between shared memory and HBM, it's ~3.3x. This gap is the fundamental motivation behind tiling algorithms like FlashAttention.

## Registers: The Speed Limit

Registers are the fastest storage on the GPU. Every local variable, intermediate computation, and loop counter lives in registers. The V100 has 65,536 32-bit registers per SM.

The critical trade-off: **register usage controls occupancy**. More registers per thread -> fewer threads per SM -> less latency hiding.

At 32 registers per thread, you can achieve 100% occupancy (2048 threads per SM). At 64 registers, occupancy drops to 50%. This is the tension that kernel developers navigate constantly: use more registers for computation (higher ILP) or fewer registers for occupancy (more latency hiding).

<Callout type="tip" title="When to Accept Low Occupancy">
  If your kernel is compute-bound and achieves high instruction-level parallelism, 50% occupancy with more registers can outperform 100% occupancy with register spilling. Always profile -- don't chase occupancy blindly.
</Callout>

## Shared Memory: The Programmer's Cache

Shared memory is explicitly managed on-chip SRAM, shared among all threads in a block. It's the key to high-performance kernels because it enables data reuse without going back to HBM.

The canonical use case: **matrix multiplication tiling**. Instead of each thread loading its data from global memory, a block of threads cooperatively loads a tile into shared memory, synchronizes, then all threads compute from the fast local copy.

```cpp
// Tiled matrix multiply: shared memory enables data reuse
__global__ void matmul_tiled(float *A, float *B, float *C, int N) {
    __shared__ float sA[TILE][TILE];  // Tile in shared memory
    __shared__ float sB[TILE][TILE];

    float sum = 0.0f;
    for (int t = 0; t < N / TILE; t++) {
        // Cooperative load: each thread loads one element
        sA[threadIdx.y][threadIdx.x] = A[row * N + t * TILE + threadIdx.x];
        sB[threadIdx.y][threadIdx.x] = B[(t * TILE + threadIdx.y) * N + col];
        __syncthreads();

        // Compute from shared memory (TILE reads at ~3 TB/s, not 900 GB/s)
        for (int k = 0; k < TILE; k++)
            sum += sA[threadIdx.y][k] * sB[k][threadIdx.x];
        __syncthreads();
    }
    C[row * N + col] = sum;
}
```

Without tiling, each element of A and B is loaded from HBM once per output element. With 32x32 tiles, each element is loaded once per tile and reused 32 times -- a **32x reduction** in HBM traffic.

### Bank Conflicts

Shared memory is organized into 32 banks (4 bytes wide). When multiple threads in a warp access the same bank, accesses are serialized:

The fix is straightforward: add padding to break the stride pattern. `__shared__ float data[32][33]` instead of `[32][32]` eliminates column-access bank conflicts.

## Global Memory: Coalescing Is Everything

Global memory (HBM) delivers 900 GB/s peak bandwidth on V100, but only with **coalesced accesses** -- where consecutive threads in a warp access consecutive memory addresses, allowing the memory controller to merge requests into minimal transactions.

The difference between coalesced and random access is **19x**. This single factor explains more CUDA performance issues than any other. If your kernel has poor global memory coalescing, no amount of occupancy tuning or register optimization will save it.

<Callout type="warning" title="The AoS vs SoA Decision">
  Array-of-Structures (AoS) layout causes strided accesses when threads process consecutive elements. Structure-of-Arrays (SoA) gives each field a contiguous array, enabling coalesced access. For GPU code, always prefer SoA unless you have a compelling reason not to.
</Callout>

Fix global memory access patterns first -- they dominate performance in most kernels. Only after coalescing is optimal should you move to shared memory tiling, bank conflict elimination, and register tuning.

## Conclusion

The GPU memory hierarchy has extreme bandwidth ratios between levels (registers are 10x faster than HBM). High-performance CUDA kernels exploit this by:

1. **Coalescing global memory accesses** -- the single most impactful optimization
2. **Tiling data in shared memory** -- reduces HBM traffic by the tile dimension
3. **Avoiding bank conflicts** -- padding shared memory arrays
4. **Managing register pressure** -- balancing ILP vs occupancy

Understanding these fundamentals is prerequisite for understanding why algorithms like FlashAttention (tiled SRAM reuse), PagedAttention (memory-efficient KV), and tensor core GEMM (register-level tiling) achieve their performance.
