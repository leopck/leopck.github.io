Quantization reduces model weight precision from FP16 (16 bits) to INT8 (8 bits), INT4 (4 bits), or even lower. The primary benefit isn't faster math -- it's **less memory traffic**. LLM decode is memory-bandwidth-bound; halving the bytes per weight doubles the tokens you can generate per second. The trade-off is accuracy, and the engineering challenge is keeping that trade-off small.

## Why Quantization Works for LLMs

LLM decode performance is limited by how fast you can read model weights from HBM. For a 7B model in FP16 (14 GB), generating one token requires reading 14 GB of weights. At 2 TB/s (A100), that's 7 ms per token -- regardless of compute capability.

Quantize to INT4 (3.5 GB), and you read 4x less data: 1.75 ms per token. **The speedup comes from bandwidth, not compute.**

<Callout type="warning" title="INT4 Without Group Quantization Is Broken">
  Naive INT4 (single scale per tensor) degrades perplexity by 1.2+ points -- unacceptable for production. Group quantization (GPTQ, AWQ) uses per-group scale factors (typically 128 values per group), keeping quality within 0.2 points of FP16. Always use group quantization for INT4.
</Callout>

## W8A8 vs W4A16: Two Different Strategies

**W8A8 (weights 8-bit, activations 8-bit):** Both weights and activations are quantized. Requires hardware INT8 tensor core support. Delivers ~2x speedup for both compute and bandwidth. Best supported by TensorRT and SmoothQuant.

**W4A16 (weights 4-bit, activations FP16):** Only weights are quantized; activations remain FP16. Dequantize weights on-the-fly during GEMM. Delivers ~2x bandwidth reduction but no compute speedup (still FP16 math). Best supported by GPTQ, AWQ, and llama.cpp.

  **H100/Gaudi2?** -> Use FP8 (native support, best quality). **Need lowest latency?** -> SmoothQuant W8A8 with TensorRT. **Need to fit a larger model?** -> AWQ INT4 with vLLM. **CPU deployment?** -> GGUF Q4_K_M with llama.cpp. **Research/fine-tuning?** -> bitsandbytes NF4.

## Conclusion

Quantization is the most accessible LLM optimization: INT8 gives 1.8x throughput improvement with negligible quality loss, and INT4 gives 2.7x with under 0.2 perplexity degradation. The speedup comes from reduced memory bandwidth (not faster math), making it especially impactful for memory-bound decode. Always use group quantization for INT4, and prefer AWQ over GPTQ for better quality. For the latest hardware (H100, Gaudi2), FP8 is the sweet spot -- 2x compression with the best quality among 8-bit formats.
