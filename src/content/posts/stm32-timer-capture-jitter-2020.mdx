Timers on an STM32 are not just for generating PWM or triggering periodic interrupts. They are precision measurement instruments. With input capture mode, a hardware timer latches its current counter value into a capture register on the exact clock edge when an external signal transitions. No software is in the loop at the moment of capture. No interrupt latency contaminates the measurement. The counter value is frozen by hardware at the precise instant the edge occurs, giving you sub-microsecond (often sub-nanosecond) resolution limited only by the timer clock frequency.

This post covers everything needed to build a production-quality measurement system: timer clock source selection, prescaler effects on resolution and range, input capture configuration, the digital capture filter for noise rejection, a complete analysis of jitter sources (interrupt latency, DMA latency, clock jitter, quantization), DMA-based capture pipelines for burst measurement, and real jitter measurement results from an STM32F407 at 168 MHz.

### Timer Clock Source

On the STM32F4, timer input clocks derive from the APB bus clock with the doubling rule:

- TIM2-7, TIM12-14 (APB1): Timer clock = PCLK1 x 2 when APB1 prescaler is not 1. At 168 MHz SYSCLK with APB1 /4: PCLK1 = 42 MHz, timer clock = 84 MHz.
- TIM1, TIM8-11 (APB2): Timer clock = PCLK2 x 2 when APB2 prescaler is not 1. At 168 MHz SYSCLK with APB2 /2: PCLK2 = 84 MHz, timer clock = 168 MHz.

### External Clock Sources

For applications requiring synchronization to an external reference (GPS 1PPS, atomic clock reference, network PTP), timers can be clocked from external sources:

- **ETR (External Trigger)**: An external clock signal on the TIMx_ETR pin drives the counter directly or through a configurable prescaler and filter. Useful for counting external events.
- **TI1/TI2 (Timer Input)**: The timer can use its own capture input as a clock source. This is used in encoder mode or for measuring frequency by counting external edges.

```c
// Configure TIM2 to use ETR as clock source
// Useful for counting external pulses with high precision
void tim2_external_clock(void) {
    RCC->APB1ENR |= RCC_APB1ENR_TIM2EN;

    // ETR input on PA0 (alternate function 1)
    // No filter, no prescaler, rising edge
    TIM2->SMCR = (7U << TIM_SMCR_ECE_Pos);  // External clock mode 2
    TIM2->PSC = 0;
    TIM2->ARR = 0xFFFFFFFF;
    TIM2->CR1 = TIM_CR1_CEN;

    // TIM2->CNT now counts external rising edges on PA0
}
```

## Prescaler Effects on Resolution

The prescaler divides the timer clock before it reaches the counter:

$$f_{counter} = \frac{f_{timer}}{PSC + 1}$$

$$T_{tick} = \frac{PSC + 1}{f_{timer}}$$

The tick period is your measurement resolution. Every captured value has an inherent quantization uncertainty of plus or minus one tick.

<Callout type="info" title="Quantization Jitter">
  Even with a perfect input signal, the captured value has a quantization uncertainty of plus or minus 0.5 ticks because the capture event is asynchronous to the timer clock. At 84 MHz (11.9 ns tick), this introduces a measurement uncertainty of plus or minus 5.95 ns. This is the noise floor of your measurement system. You cannot measure jitter smaller than this without a faster timer clock.

```c
#include "stm32f4xx.h"

// Configure TIM2 CH1 for input capture on PA0
// Maximum resolution: 84 MHz, 11.9 ns per tick
void input_capture_init(void) {
    // Enable clocks
    RCC->APB1ENR |= RCC_APB1ENR_TIM2EN;
    RCC->AHB1ENR |= RCC_AHB1ENR_GPIOAEN;

    // PA0 as alternate function (AF1 = TIM2)
    GPIOA->MODER = (GPIOA->MODER & ~(3U << (0 * 2))) | (2U << (0 * 2));
    GPIOA->AFR[0] = (GPIOA->AFR[0] & ~(0xFU << (0 * 4))) | (1U << (0 * 4));

    // Timer base: no prescaler, free-running 32-bit counter
    TIM2->PSC = 0;              // 84 MHz timer clock
    TIM2->ARR = 0xFFFFFFFF;     // Full 32-bit range

    // Input capture on channel 1:
    // CC1S = 01 (input, IC1 mapped to TI1)
    // IC1F = 0000 (no filter)
    // IC1PSC = 00 (no prescaler, capture every edge)
    TIM2->CCMR1 = (1U << TIM_CCMR1_CC1S_Pos);

    // Capture on rising edge, enable capture
    TIM2->CCER = TIM_CCER_CC1E;  // CC1E=1, CC1P=0, CC1NP=0 -> rising edge

    // Enable capture interrupt
    TIM2->DIER = TIM_DIER_CC1IE;
    NVIC_SetPriority(TIM2_IRQn, 1);
    NVIC_EnableIRQ(TIM2_IRQn);

    // Start counter
    TIM2->CR1 = TIM_CR1_CEN;
}
```

### Capture Filter for Noise Rejection

The input capture filter (ICxF field) is a digital filter that requires the input signal to be stable for N consecutive timer clock cycles (or N consecutive samples of a divided clock) before accepting the edge. This rejects short glitches and ringing on the input.

```c
// Configure capture filter: reject glitches shorter than ~95 ns
void set_capture_filter(uint8_t filter_value) {
    // filter_value: 0-15 (IC1F field)
    TIM2->CCMR1 = (TIM2->CCMR1 & ~TIM_CCMR1_IC1F) |
                   (filter_value << TIM_CCMR1_IC1F_Pos);
}

// Example: 8-sample filter at timer clock (84 MHz)
// Minimum pulse: 8 / 84 MHz = 95.2 ns
set_capture_filter(3);  // IC1F = 0011
```

<Callout type="warning" title="Filter Adds Deterministic Latency, Not Jitter">
  The capture filter delays edge detection by a fixed number of clock cycles. This adds a constant offset to your measurements, not variable jitter. Since you typically measure period or frequency (difference between two captures), the constant offset cancels out. The filter is safe for precision measurements as long as you understand it does not affect jitter, only the absolute timestamp of each edge.
</Callout>

## Jitter Sources Analysis

Jitter in a timer capture measurement system comes from several independent sources. Understanding each one is critical for interpreting your results and knowing the noise floor of your measurement.

### Source 1: Timer Clock Jitter

The timer clock itself has jitter. If the clock source is an internal RC oscillator (HSI), it has significant cycle-to-cycle jitter due to thermal noise in the RC circuit. If it derives from a crystal oscillator (HSE) through the PLL, the jitter is much lower but still present due to PLL phase noise.

### Source 2: Quantization Jitter

The capture event is asynchronous to the timer clock. The captured value is the counter state at the clock edge *after* the input transition. This introduces a uniform random uncertainty of 0 to 1 timer tick:

$$\sigma_{quantization} = \frac{T_{tick}}{\sqrt{12}} = \frac{1}{f_{timer} \cdot \sqrt{12}}$$

At 84 MHz: $\sigma_q = 11.9 / \sqrt{12} = 3.44$ ns RMS.

At 168 MHz: $\sigma_q = 5.95 / \sqrt{12} = 1.72$ ns RMS.

### Source 3: Interrupt Latency Jitter (if using ISR-based capture)

If you read CCR in an ISR, the capture value itself is jitter-free (hardware latched it on the edge). But if you need to timestamp the event using DWT->CYCCNT or perform any time-sensitive action in the ISR, the variable interrupt entry latency adds jitter to that action.

This is a critical distinction: **the CCR value has no interrupt jitter because hardware captures it. But the ISR execution time after the capture does have jitter.**

### Source 4: DMA Latency Jitter (if using DMA-based capture)

When DMA transfers the CCR value to a memory buffer, there is a small latency between the capture event and the DMA transfer completing. This latency varies depending on bus arbitration and other DMA streams. However, since DMA copies the already-latched CCR value, the DMA latency does not affect the captured timestamp. It only affects when the CPU can read the result.

### Source 5: Signal Integrity

The input signal itself may have jitter: rise-time variation, ringing, noise coupling. The capture filter (ICxF) can mitigate some of this, but fundamentally, if the signal is noisy, the measurement will reflect that noise.

<Callout type="info" title="The Captured CCR Value Is Jitter-Free">
  This is the most important principle of input capture measurement. The hardware latches the counter value at the exact clock edge coinciding with (or immediately after) the input transition. No software latency contaminates the captured value. The CCR register contains a hardware timestamp with only quantization and clock jitter errors. All other jitter sources (interrupt, DMA, bus) only affect when you read the result, not the result itself.
</Callout>

## DMA-Based Capture Pipeline

For burst measurements (capturing many consecutive edges), DMA is the correct approach. The timer generates a DMA request on each capture event, and DMA transfers the CCR value to a buffer without CPU intervention. This allows capturing thousands of edges at the full timer rate with zero interrupt overhead.

### Architecture

```
External Signal
     |
     v
[TIM2 CH1 Input] --edge detect--> [CCR1 latched] --DMA request--> [DMA Stream] --> [SRAM Buffer]
                                                                                         |
                                                                                    [CPU processes
                                                                                     after TC IRQ]
```

### Implementation

```c
#define CAPTURE_BUF_SIZE 2048

static volatile uint32_t capture_buf[CAPTURE_BUF_SIZE]
    __attribute__((aligned(4)));
static volatile uint32_t capture_ready = 0;

void input_capture_dma_init(void) {
    // Enable clocks
    RCC->APB1ENR |= RCC_APB1ENR_TIM2EN;
    RCC->AHB1ENR |= RCC_AHB1ENR_DMA1EN | RCC_AHB1ENR_GPIOAEN;

    // PA0 as AF1 (TIM2_CH1)
    GPIOA->MODER = (GPIOA->MODER & ~(3U << 0)) | (2U << 0);
    GPIOA->AFR[0] = (GPIOA->AFR[0] & ~0xFU) | 1U;

    // TIM2: no prescaler, full 32-bit range
    TIM2->PSC = 0;
    TIM2->ARR = 0xFFFFFFFF;

    // Input capture on CH1, rising edge, no filter
    TIM2->CCMR1 = (1U << TIM_CCMR1_CC1S_Pos);
    TIM2->CCER = TIM_CCER_CC1E;

    // Enable DMA request on capture (CC1 DMA enable)
    TIM2->DIER = TIM_DIER_CC1DE;

    // DMA1 Stream 5, Channel 3 = TIM2_CH1
    DMA1_Stream5->CR &= ~DMA_SxCR_EN;
    while (DMA1_Stream5->CR & DMA_SxCR_EN) {}

    DMA1->HIFCR = DMA_HIFCR_CTCIF5 | DMA_HIFCR_CHTIF5 |
                  DMA_HIFCR_CTEIF5 | DMA_HIFCR_CDMEIF5 |
                  DMA_HIFCR_CFEIF5;

    DMA1_Stream5->PAR  = (uint32_t)&TIM2->CCR1;
    DMA1_Stream5->M0AR = (uint32_t)capture_buf;
    DMA1_Stream5->NDTR = CAPTURE_BUF_SIZE;

    DMA1_Stream5->CR = (3U << DMA_SxCR_CHSEL_Pos) |  // Channel 3
                       (2U << DMA_SxCR_MSIZE_Pos)  |  // 32-bit memory
                       (2U << DMA_SxCR_PSIZE_Pos)  |  // 32-bit peripheral
                       DMA_SxCR_MINC               |  // Memory increment
                       (0U << DMA_SxCR_DIR_Pos)    |  // Peripheral-to-memory
                       (3U << DMA_SxCR_PL_Pos)     |  // Very high priority
                       DMA_SxCR_TCIE               |  // Transfer complete IRQ
                       DMA_SxCR_EN;

    NVIC_SetPriority(DMA1_Stream5_IRQn, 2);
    NVIC_EnableIRQ(DMA1_Stream5_IRQn);

    // Start timer
    TIM2->CR1 = TIM_CR1_CEN;
}

void DMA1_Stream5_IRQHandler(void) {
    if (DMA1->HISR & DMA_HISR_TCIF5) {
        DMA1->HIFCR = DMA_HIFCR_CTCIF5;
        capture_ready = 1;
    }
}
```

### Post-Processing: Computing Period and Jitter

```c
typedef struct {
    float mean_period_ns;
    float std_dev_ns;
    float min_period_ns;
    float max_period_ns;
    float peak_to_peak_jitter_ns;
    uint32_t num_periods;
} jitter_analysis_t;

void analyze_captures(jitter_analysis_t *result) {
    if (!capture_ready) return;
    capture_ready = 0;

    const float tick_ns = 1e9f / 84e6f;  // 11.905 ns per tick at 84 MHz
    uint32_t num_periods = CAPTURE_BUF_SIZE - 1;

    // Compute periods (differences between consecutive captures)
    float sum = 0;
    float sum_sq = 0;
    float min_p = 1e12f;
    float max_p = 0;

    for (uint32_t i = 0; i < num_periods; i++) {
        // Unsigned subtraction handles counter wraparound correctly
        uint32_t delta = capture_buf[i + 1] - capture_buf[i];
        float period_ns = delta * tick_ns;

        sum += period_ns;
        sum_sq += period_ns * period_ns;

        if (period_ns < min_p) min_p = period_ns;
        if (period_ns > max_p) max_p = period_ns;
    }

    result->num_periods = num_periods;
    result->mean_period_ns = sum / num_periods;
    result->min_period_ns = min_p;
    result->max_period_ns = max_p;
    result->peak_to_peak_jitter_ns = max_p - min_p;

    float variance = (sum_sq / num_periods) -
                     (result->mean_period_ns * result->mean_period_ns);
    result->std_dev_ns = sqrtf(variance > 0 ? variance : 0);
}
```

## Advanced: Dual Capture for Pulse Width Measurement

To measure both pulse width (high time) and period simultaneously, configure two channels on the same timer input. Channel 1 captures rising edges, channel 2 captures falling edges of the same signal (using the TI1 input routed to both channels).

```c
void dual_capture_init(void) {
    // TIM2: no prescaler, 32-bit
    TIM2->PSC = 0;
    TIM2->ARR = 0xFFFFFFFF;

    // CH1: IC1 mapped to TI1, rising edge (period)
    // CH2: IC2 mapped to TI1, falling edge (pulse width)
    TIM2->CCMR1 = (1U << TIM_CCMR1_CC1S_Pos)  |  // CC1S=01: IC1->TI1
                   (2U << TIM_CCMR1_CC2S_Pos);     // CC2S=10: IC2->TI1

    TIM2->CCER = TIM_CCER_CC1E |                   // CH1 enable, rising edge
                 TIM_CCER_CC2E | TIM_CCER_CC2P;    // CH2 enable, falling edge

    // Enable both capture DMA requests
    TIM2->DIER = TIM_DIER_CC1DE | TIM_DIER_CC2DE;

    TIM2->CR1 = TIM_CR1_CEN;
    // Now CCR1 captures rising edges, CCR2 captures falling edges
    // Pulse width = CCR2 - CCR1 (for same pulse)
    // Period = CCR1[n] - CCR1[n-1]
}
```

## Advanced: Frequency Measurement with Reciprocal Counting

For low-frequency signals (less than 100 Hz), capturing individual edges is inefficient. The reciprocal counting method measures the timer count between two edges, giving frequency with resolution that improves with longer measurement times:

$$f = \frac{f_{timer}}{\Delta CCR}$$

For a 1 Hz signal: $\Delta CCR = 84{,}000{,}000$ ticks. The frequency resolution is:

$$\delta f = \frac{f_{timer}}{(\Delta CCR)^2} = \frac{84 \times 10^6}{(84 \times 10^6)^2} = 11.9 \text{ nHz per tick}$$

That is nanohertz resolution for a 1 Hz signal. This is far superior to the direct counting method (counting edges over a fixed gate time), which requires a long gate time to achieve similar resolution at low frequencies.

## Measurement Methodology Best Practices

1. **Use hardware capture (CCR), not software timestamps.** The CCR is latched by hardware with only quantization jitter. Software timestamps add interrupt and scheduling jitter.

2. **Use DMA for bulk capture.** Collect thousands of samples with zero CPU intervention, then analyze the entire dataset statistically.

3. **Use 32-bit timers (TIM2 or TIM5).** Eliminates overflow handling complexity for most signal frequencies.

4. **Run PSC = 0 for maximum resolution.** Only add prescaler if your signal period exceeds the 32-bit overflow time (51 seconds at 84 MHz -- unlikely for most applications).

5. **Apply capture filter for noisy signals.** Start with IC1F = 3 (8 samples at timer clock) for signals with ringing, increase if needed.

6. **Compute statistics on period differences, not raw CCR values.** Period = CCR[n+1] - CCR[n] cancels any constant offset from the filter delay.

7. **Account for quantization noise floor.** At 84 MHz, you cannot resolve jitter below approximately 3.4 ns RMS (11.9 ns / sqrt(12)). If your measured jitter is near this floor, the actual signal jitter may be lower.

8. **Use the same clock for generation and capture** to measure only the quantization noise floor. Use different clocks to measure actual jitter between clock domains.

## Conclusion

STM32 timer input capture is a powerful, zero-cost measurement instrument that every embedded engineer should master. The hardware captures timestamps with quantization-limited precision (11.9 ns at 84 MHz, 5.95 ns at 168 MHz), independent of software latency. Combined with DMA, you can capture thousands of consecutive edges without CPU intervention and analyze the results statistically for mean period, standard deviation, peak-to-peak jitter, and frequency stability.

The real-world measurements confirm what the theory predicts: timer-to-timer captures show only quantization jitter (4 ticks peak-to-peak), external signals add their own noise contribution, and the DMA capture path adds zero additional jitter compared to ISR-based CCR reads. The only measurement method that introduces significant jitter is software timestamping (DWT->CYCCNT in an ISR), which should be avoided for precision work. Use the hardware capture registers for what they were designed for, and reserve software timestamps for cases where hardware capture is not available.
