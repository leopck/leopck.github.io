Static batching wastes GPU cycles waiting for the longest sequence. Continuous batching fills those gaps by adding new requests as others complete. But implementing it correctly requires solving several non-trivial scheduling problems.

## The Static Batching Problem

In static batching, a batch executes together until completion. If request A generates 1000 tokens and request B generates 200, the GPU sits mostly idle during the last 800 steps -- only A is running, giving 33% utilization. With continuous batching, new requests fill those empty slots immediately, keeping utilization above 95%.

## The Scheduler Architecture

vLLM's scheduler operates at iteration granularity -- deciding before each decode step which sequences to include:

```python
class IterationLevelScheduler:
    def __init__(self, config):
        self.waiting_queue = []     # New requests
        self.running_queue = []     # Active sequences
        self.swapped_queue = []     # Preempted to CPU
        self.block_manager = BlockManager(config)

    def schedule(self):
        # Phase 1: Handle running sequences
        running_scheduled = self._schedule_running()
        # Phase 2: Try to swap in preempted sequences
        swapped_scheduled = self._schedule_swapped()
        # Phase 3: Admit new sequences if memory available
        waiting_scheduled = self._schedule_waiting()
        return SchedulerOutputs(
            scheduled_seqs=running_scheduled + swapped_scheduled + waiting_scheduled,
        )
```

## Memory-Aware Scheduling

The scheduler must check available KV cache blocks before admitting new requests:

```python
def _can_admit_sequence(self, seq_group):
    prompt_tokens = len(seq_group.prompt_token_ids)
    blocks_needed = (prompt_tokens + self.block_size - 1) // self.block_size
    min_decode_blocks = 16  # Reserve space for decode
    total_needed = blocks_needed + min_decode_blocks
    available = self.block_manager.get_available_blocks()
    return total_needed <= available
```

<Callout type="warning" title="Preemption Overhead">
  Preemption requires copying KV cache to CPU memory (swap out) and back (swap in). On A100, this costs ~2ms per GB swapped. Frequent preemption can dominate latency.
</Callout>

## Preemption Strategies

Two approaches when memory runs out:

**Recomputation:** Discard KV cache entirely. On resume, re-run prefill. Best when the sequence is short (cheap to recompute).

**Swapping:** Copy KV cache to CPU memory. On resume, copy back. Best when the sequence has generated many tokens (expensive to recompute).

```python
def choose_preemption_method(self, seq_group):
    generated = seq_group.get_num_generated_tokens()
    blocks = self.block_manager.get_blocks_for_seq(seq_group)
    recompute_cost = self._estimate_prefill_time(generated)
    swap_cost = self._estimate_swap_time(blocks * self.block_size_bytes)
    resume_probability = self._estimate_resume_probability(seq_group)
    # If recompute is cheap or sequence won't resume soon
    if recompute_cost < swap_cost or resume_probability < 0.5:
        return 'recompute'
    else:
        return 'swap'
```

## Chunked Prefill

Modern continuous batching also chunks prefill to prevent long prompts from blocking decode:

```python
def schedule_with_chunked_prefill(self):
    decode_seqs = []
    prefill_seqs = []
    prefill_budget = self.config.prefill_chunk_size  # e.g., 512 tokens

    for seq in self.running_queue:
        if seq.is_decoding():
            decode_seqs.append(seq)

    for seq in self.waiting_queue:
        if seq.is_prefilling():
            remaining = seq.get_remaining_prefill_tokens()
            chunk = min(remaining, prefill_budget)
            if chunk > 0 and self._can_schedule_prefill_chunk(seq, chunk):
                prefill_seqs.append((seq, chunk))
                prefill_budget -= chunk
                if prefill_budget <= 0:
                    break

    return SchedulerOutputs(decode_seqs=decode_seqs, prefill_seqs=prefill_seqs)
```

## Debugging Scheduler Issues

Common scheduler problems and diagnosis:

```python
class SchedulerDiagnostics:
    def diagnose_low_throughput(self):
        metrics = self.get_metrics()

        if metrics.avg_batch_size < self.config.max_batch_size * 0.5:
            if metrics.memory_utilization > 0.9:
                print("Issue: Memory pressure limiting batch size")
            elif metrics.waiting_queue_avg > 0:
                print("Issue: Scheduler not admitting waiting requests")

        if metrics.preemption_rate > 0.1:
            print(f"Issue: High preemption rate ({metrics.preemption_rate:.1%})")
            print(f"  Average swap size: {metrics.avg_swap_size_mb:.1f} MB")

        if metrics.avg_waiting_time > self.config.slo_ms:
            print(f"Issue: Queue time exceeding SLO ({metrics.avg_waiting_time:.1f}ms)")
```

## Conclusion

Continuous batching transforms inference from a batch processing problem to a real-time scheduling problem. The key challenges are memory coordination between scheduler and block manager, choosing between recomputation and swap preemption, preventing starvation of long sequences, and chunking prefill to avoid blocking decode. Properly implemented, it delivers 2-4x throughput improvement.
