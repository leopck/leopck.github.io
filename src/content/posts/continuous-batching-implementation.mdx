LLM inference at scale is a scheduling problem. You have a GPU with fixed compute and memory, a stream of requests with wildly different prompt lengths and generation lengths, and users who care about both throughput and latency. The difference between a naive implementation and a well-engineered one is often 3-4x in throughput -- and the entire difference comes down to how you batch requests together.

This post covers the full landscape: why static batching fails, how continuous batching fixes it, the internal architecture of production schedulers like vLLM, and the advanced techniques (chunked prefill, preemption, priority scheduling, multi-LoRA batching) that make it all work at scale. We will close with concrete performance data and guidance on when continuous batching is -- and is not -- the right choice.

---

## 1. The Static Batching Problem

The simplest approach to batching LLM inference is **static batching**: collect $N$ requests, pad all input sequences to the same length, run them through the model together, and return results when the entire batch finishes.

This is how most deep learning training works, and it feels natural. But for inference, it has three severe problems.

### 1.1 Padding Waste

When you batch sequences of different lengths, every sequence must be padded to the length of the longest sequence in the batch. If one request has a 128-token prompt and another has a 2,048-token prompt, you pad the short one with 1,920 useless tokens. The GPU performs attention computations over those padding tokens, producing results you immediately discard.

In production workloads -- where short chat messages coexist with long document summarization requests -- padding waste routinely consumes 50-70% of GPU cycles. You are paying for an 80GB A100 and getting the effective throughput of a much smaller card.

### 1.2 Completion Stalls

Static batching does not just waste compute on padding during prefill. During generation (decode), the batch runs until **every** request in it has finished generating. If request A generates 20 tokens and request B generates 2,000 tokens, request A's slot sits completely idle for 1,980 decode steps. The GPU is running, but one slot in the batch is doing no useful work.

Consider a batch of 8 requests with these generation lengths: 50, 120, 200, 350, 500, 800, 1200, 2000. After 50 steps, one slot is idle. After 120, two are idle. By step 1200, seven of eight slots are wasted. The average utilization across the batch lifetime is:

$$
\text{Avg utilization} = \frac{\sum_{i=1}^{N} L_i}{N \cdot L_{\max}} = \frac{50 + 120 + 200 + 350 + 500 + 800 + 1200 + 2000}{8 \times 2000} = \frac{5220}{16000} = 32.6\%
$$

That is 67.4% of GPU decode cycles wasted -- on a workload that is not even particularly skewed.

### 1.3 Queueing Delay

With static batching, new requests must wait for the current batch to fully complete before they can be served. If the current batch takes 10 seconds (because one request is generating a long response), a request that arrives at second 1 waits 9 seconds before it even begins processing. This is unacceptable for interactive applications.

<Callout type="warning" title="The Real Cost of Static Batching">
  Measured across production workloads with mixed sequence lengths, static batching wastes 50-70% of GPU cycles. The waste comes from two sources: padding tokens during prefill and idle slots during decode. Both are eliminated by continuous batching.
</Callout>

### 1.4 Why Not Just Sort by Length?

A natural improvement is **dynamic batching**: sort incoming requests by length and batch similarly-sized sequences together. This reduces padding waste significantly:

```python
def form_batch_by_length(queue, max_batch_size=16, length_tolerance=1.2):
    """Group requests with similar lengths to minimize padding."""
    queue.sort(key=lambda r: len(r.tokens))

    batch = [queue[0]]
    base_length = len(queue[0].tokens)

    for req in queue[1:]:
        if len(batch) >= max_batch_size:
            break
        if len(req.tokens) <= base_length * length_tolerance:
            batch.append(req)

    return batch
```

With a 20% length tolerance, padding waste drops from ~42% to under 8% for typical workloads. But dynamic batching does **not** solve the completion stall problem. You still wait for the longest generation in the batch, and generation lengths are unpredictable (you do not know how many tokens a request will generate until it emits an EOS token). Dynamic batching is a half-measure. The real solution is continuous batching.

## 2. The Continuous Batching Insight

The core insight behind continuous batching is simple: **do not wait for all requests to finish. As one request completes, immediately add a new one. The batch evolves every iteration.**

In static batching, the batch is fixed for its entire lifetime. In continuous batching, the batch is re-evaluated at every single decode step. This is why continuous batching is also called **iteration-level scheduling** -- the scheduling decision happens at the granularity of individual token generation steps.

### 2.1 How It Works

At each decode iteration, the scheduler:

1. **Removes** any requests that have finished (emitted EOS or hit max length)
2. **Adds** new requests from the waiting queue, if memory is available
3. **Runs** one decode step on the updated batch

This means a request that finishes after 50 tokens immediately frees its slot for a new request. There are no idle slots, no waiting for the longest sequence, no batch-level synchronization.

### 2.2 Why It Works So Well

Continuous batching eliminates both sources of waste in static batching:

- **No padding waste**: each request processes exactly its own tokens, at its own length. There is no concept of "padding to the longest sequence" because the batch does not have a fixed shape.
- **No idle slots**: the moment a request finishes, its memory and compute slot are reclaimed and given to the next request in queue.
- **No batch completion stalls**: new requests do not wait for the current batch to finish. They join the running batch at the next iteration.

## 3. How the Scheduler Works: vLLM Internals

Understanding the theory is useful, but the engineering details are where things get interesting. Let us walk through how vLLM's scheduler actually operates, step by step.

### 3.1 The Three Queues

vLLM's scheduler maintains three queues that track the lifecycle of every request:

```python
class IterationLevelScheduler:
    def __init__(self, config):
        self.waiting_queue = []     # New requests awaiting prefill
        self.running_queue = []     # Active sequences being decoded
        self.swapped_queue = []     # Preempted sequences (KV cache on CPU)
        self.block_manager = BlockManager(config)
```

- **Waiting queue**: incoming requests that have not yet been prefilled. Sorted by arrival time (FCFS) or priority. These requests need prompt processing before they can begin generating.
- **Running queue**: requests currently being decoded. Each iteration, every request in this queue generates one token. This is the "active batch."
- **Swapped queue**: requests that were running but got preempted due to memory pressure. Their KV cache has been copied to CPU memory. They will be swapped back in when memory becomes available.

### 3.2 The Schedule Loop

Every iteration (roughly every 15-30ms on a modern GPU), the scheduler runs this logic:

```python
def schedule(self):
    # Phase 1: Handle running sequences
    # - Check if any have finished (EOS or max length)
    # - Remove finished sequences, free their KV cache blocks
    running_scheduled = self._schedule_running()

    # Phase 2: Try to swap in preempted sequences
    # - If memory is available and swapped queue is non-empty
    # - Copy KV cache back from CPU, move to running queue
    swapped_scheduled = self._schedule_swapped()

    # Phase 3: Admit new sequences from waiting queue
    # - Only if memory is available for their KV cache
    # - Run prefill for newly admitted sequences
    waiting_scheduled = self._schedule_waiting()

    return SchedulerOutputs(
        scheduled_seqs=running_scheduled + swapped_scheduled + waiting_scheduled,
    )
```

The order of phases matters. Running sequences have highest priority (they already have allocated KV cache and are mid-generation). Swapped sequences come next (they have partial results worth preserving). New sequences from the waiting queue are admitted last, only if there is remaining capacity.

### 3.3 Memory-Aware Admission

The scheduler cannot blindly admit new requests. Each request needs GPU memory for its KV cache, and running out of memory mid-generation is catastrophic (you would have to kill a request and lose all its generated tokens). The admission check looks like this:

```python
def _can_admit_sequence(self, seq_group):
    prompt_tokens = len(seq_group.prompt_token_ids)
    blocks_needed = (prompt_tokens + self.block_size - 1) // self.block_size
    min_decode_blocks = 16  # Reserve space for future decode tokens
    total_needed = blocks_needed + min_decode_blocks
    available = self.block_manager.get_available_blocks()
    return total_needed <= available
```

The `min_decode_blocks` reservation is critical. If you allocate memory only for the prompt and nothing for generation, the request will immediately need more blocks on its first decode step. The scheduler must predict how much decode memory a request will need. In practice, reserving 16-32 blocks (256-512 tokens) provides a reasonable buffer, with preemption as a fallback if the request generates more than expected.

This typically allows 3-5x more concurrent requests compared to naive pre-allocation, because most requests do not actually reach the maximum sequence length.

### 3.4 The Full Iteration Timeline

Here is what a single iteration looks like in detail:

1. **Check completions** (~0.01ms): scan running queue for requests that emitted EOS or hit max tokens. Remove them, free their KV blocks, return results to clients.
2. **Admission decisions** (~0.05ms): check waiting and swapped queues. For each candidate, compute memory requirements and compare against available blocks. Admit as many as possible within budget.
3. **Prefill new requests** (variable, 1-50ms): for newly admitted requests, run the prefill pass to populate their KV cache with prompt information.
4. **Decode step** (~15-25ms for typical batch sizes): run one forward pass for all active requests. Each generates one token.
5. **Block allocation** (~0.01ms): allocate new KV blocks for the tokens just generated, if needed.

At high load, the scheduler is making dozens of admission and eviction decisions per second, maintaining a continuously evolving batch that maximizes GPU utilization.

## 4. Prefill-Decode Interleaving

One of the subtlest problems in continuous batching is the interaction between prefill and decode. Prefill (processing the input prompt) and decode (generating output tokens) have fundamentally different compute profiles, and mixing them naively creates performance problems.

### 4.1 The Prefill Stall Problem

Prefill is compute-intensive: processing a 4,096-token prompt requires the same compute as generating 4,096 decode tokens, but it happens in a single forward pass. If a request with a 32K-token prompt arrives, its prefill takes hundreds of milliseconds -- and during that time, **all decode requests in the batch are stalled**.

Why? Because the GPU is running one forward pass at a time. If that forward pass includes a massive prefill, the decode tokens in the same batch experience a huge spike in per-token latency. Users see their streaming output freeze for several seconds while the system processes someone else's long prompt.

### 4.2 Chunked Prefill

The solution is **chunked prefill**: break long prompts into chunks (typically 512-2048 tokens) and interleave them with decode steps. Instead of processing the entire 32K prompt in one pass, the scheduler processes it in 64 chunks of 512 tokens each, running decode steps for other requests between chunks.

```python
def schedule_with_chunked_prefill(self):
    token_budget = self.config.max_tokens_per_iteration  # e.g., 8192

    decode_seqs = []
    prefill_chunks = []

    # First, account for all decode requests (1 token each)
    for seq in self.running_queue:
        if seq.is_decoding():
            decode_seqs.append(seq)
            token_budget -= 1  # Each decode uses 1 token of budget

    # Then, fill remaining budget with prefill chunks
    for seq in self.waiting_queue:
        if token_budget <= 0:
            break
        remaining = seq.get_remaining_prefill_tokens()
        chunk_size = min(remaining, self.config.prefill_chunk_size, token_budget)
        if chunk_size > 0 and self._can_schedule_prefill_chunk(seq, chunk_size):
            prefill_chunks.append((seq, chunk_size))
            token_budget -= chunk_size

    return SchedulerOutputs(
        decode_seqs=decode_seqs,
        prefill_chunks=prefill_chunks,
    )
```

### 4.3 The Token Budget

Each iteration has a **token budget** -- the maximum number of tokens the GPU should process in a single forward pass. This budget is shared between decode tokens and prefill chunks.

For example, with a budget of 8,192 tokens and 64 active decode requests:
- Decode consumes 64 tokens (one per request)
- Remaining budget: 8,128 tokens available for prefill chunks
- A 32K prompt is processed over $\lceil 32000 / 8128 \rceil = 4$ iterations instead of one massive pass

The token budget is a tuning parameter that directly controls the tradeoff between two latency metrics:

- **TTFT (Time to First Token)**: how long a new request waits before it starts generating. Smaller prefill chunks mean more iterations to complete prefill, increasing TTFT.
- **TBT (Time Between Tokens)**: how much time passes between consecutive tokens for an active request. Smaller prefill chunks mean less interference with decode, reducing TBT variability.

<Callout type="tip" title="Think in Iterations, Not Requests">
  The GPU runs per-iteration batches of active sequences. This iteration-centric view is the key mental model for understanding continuous batching performance.
</Callout>

### 4.4 Prefill-Decode Separation vs Interleaving

An alternative to chunked prefill is **prefill-decode separation**: dedicate some GPUs entirely to prefill and others to decode, passing KV cache between them. This eliminates interference entirely but requires:

- Network bandwidth to transfer KV cache between GPUs
- Load balancing between prefill and decode pools
- More complex deployment topology

Systems like Splitwise and DistServe explore this direction. For single-GPU or small-cluster deployments, chunked prefill is the more practical approach.

## 5. Preemption Strategies

Memory is finite. When the scheduler has admitted too many requests and the KV cache is full, it must **preempt** a running request to free memory for others. This is the most operationally sensitive part of the scheduler.

### 5.1 When Preemption Happens

Preemption is triggered when:
- A new request cannot be admitted because there are no free KV blocks
- A running request needs to allocate a new block (it generated enough tokens to fill its current block) but none are available
- A higher-priority request arrives and the system needs to make room

### 5.2 Swap: Copy KV Cache to CPU

The first strategy is **swapping**: copy the preempted request's KV cache from GPU memory to CPU memory. When the request is resumed later, its KV cache is copied back.

**Cost**: PCIe transfer time. On an A100 with PCIe Gen4 x16, bandwidth is ~32 GB/s. A Llama-70B request with 2,000 generated tokens has a KV cache of roughly 1.25 GB, so swapping takes ~40ms each way (80ms round-trip).

**Advantage**: the request resumes exactly where it left off. No computation is wasted.

**Disadvantage**: PCIe bandwidth is shared with other operations. During a swap, the GPU may stall if it needs data from CPU. Also, CPU memory is limited -- you can only swap out so many requests before CPU memory is also full.

### 5.3 Recompute: Discard and Redo

The second strategy is **recomputation**: simply discard the preempted request's KV cache entirely. When the request is resumed, it re-runs prefill on the original prompt plus all previously generated tokens to reconstruct the KV cache.

**Cost**: prefill compute time. For a request with a 1,000-token prompt and 500 generated tokens, you must re-run prefill on 1,500 tokens.

**Advantage**: no PCIe transfer, no CPU memory usage. The GPU just does compute, which it is good at.

**Disadvantage**: all the compute spent generating those 500 tokens is wasted -- you have to redo it.

### 5.4 Choosing Between Them

The decision is straightforward in principle:

```python
def choose_preemption_method(self, seq_group):
    generated = seq_group.get_num_generated_tokens()
    prompt_len = len(seq_group.prompt_token_ids)
    total_tokens = prompt_len + generated
    blocks = self.block_manager.get_blocks_for_seq(seq_group)

    # Estimate costs
    recompute_cost = self._estimate_prefill_time(total_tokens)
    swap_cost = self._estimate_swap_time(blocks * self.block_size_bytes)
    resume_probability = self._estimate_resume_probability(seq_group)

    # Swap is better for long sequences (lots of KV to recompute)
    # Recompute is better for short sequences (cheap to redo)
    if recompute_cost < swap_cost or resume_probability < 0.5:
        return 'recompute'
    else:
        return 'swap'
```

- The request is unlikely to resume soon (why burn PCIe bandwidth for something that may be preempted again?)

<Callout type="warning" title="Preemption Overhead">
  Preemption requires copying KV cache to CPU memory (swap out) and back (swap in). On A100, this costs ~2ms per GB swapped. Frequent preemption can dominate latency -- if your system is preempting more than 5% of iterations, you are overloaded and need to either reduce the admission rate or add GPUs.
</Callout>

### 5.5 Which Request to Preempt?

When you must preempt, which running request do you choose? Common policies:

- **Last-come-first-served (LCFS)**: preempt the most recently admitted request. It has the least generated tokens, so the least work is lost.
- **Shortest-remaining**: preempt the request estimated to have the most tokens remaining. This frees memory for the longest time.
- **Lowest-priority**: in priority-based scheduling, preempt the lowest-priority request.

vLLM defaults to LCFS, which is a reasonable heuristic: the most recently admitted request has the smallest KV cache (least costly to swap/recompute) and has consumed the least total compute (least work wasted).

## 6. Priority Scheduling

Not all requests are equal. In production, you often need to differentiate between premium users and free-tier users, between latency-sensitive chat requests and bulk processing jobs, or between requests with different SLO targets. Priority scheduling adds this capability to the continuous batching scheduler.

### 6.1 First-Come-First-Served (FCFS)

The simplest policy: serve requests in arrival order.

**Pros**: simple, predictable, perceived as "fair."

**Cons**: no SLO awareness. A bulk job that arrives one second before a latency-sensitive chat request will be served first, even though the chat request has a 200ms SLO and the bulk job has a 30-second SLO.

### 6.2 Priority Queues

Assign each request a priority level and always serve higher-priority requests first.

```python
class PriorityScheduler:
    def __init__(self, config):
        self.waiting_queue = []  # Priority queue (min-heap by priority)
        self.running_queue = []
        self.block_manager = BlockManager(config)

    def enqueue(self, request):
        priority = self._compute_priority(request)
        heapq.heappush(self.waiting_queue, (priority, request))

    def _compute_priority(self, request):
        # Lower number = higher priority
        base = request.priority_class  # 0=premium, 1=standard, 2=bulk
        return base

    def _schedule_waiting(self):
        admitted = []
        while self.waiting_queue:
            priority, req = self.waiting_queue[0]
            if self._can_admit_sequence(req):
                heapq.heappop(self.waiting_queue)
                admitted.append(req)
            else:
                break
        return admitted
```

**Pros**: direct control over which requests are served first. Premium users get consistently better latency.

**Cons**: **starvation**. If high-priority requests keep arriving, low-priority requests may never be served. In the worst case, a bulk job waits indefinitely.

### 6.3 Fair-Share Scheduling

To prevent starvation, fair-share scheduling ensures that every priority class gets a minimum share of GPU time:

```python
def _compute_priority(self, request):
    base = request.priority_class
    # Age boost: increase priority as request waits longer
    age_seconds = time.time() - request.arrival_time
    age_boost = age_seconds / self.config.max_wait_seconds  # 0 to 1
    # After max_wait_seconds, any request reaches top priority
    return base - age_boost
```

The age boost ensures that even the lowest-priority request eventually rises to the top of the queue. The `max_wait_seconds` parameter controls how long a low-priority request can be starved before it is promoted.

### 6.4 SLO-Aware Scheduling

The most sophisticated approach: schedule to maximize the number of requests meeting their latency targets.

Each request arrives with an SLO (e.g., "first token within 500ms" or "complete within 5 seconds"). The scheduler computes each request's **slack** -- how much time remains before its SLO is violated -- and prioritizes requests with the least slack.

```python
def _compute_priority(self, request):
    now = time.time()
    elapsed = now - request.arrival_time
    remaining_slo = request.slo_seconds - elapsed
    # Negative slack means SLO already violated
    return remaining_slo  # Lower value = more urgent
```

This naturally handles mixed workloads: a chat request with a 200ms SLO and 180ms elapsed has a slack of 20ms (urgent), while a batch job with a 30s SLO and 180ms elapsed has a slack of 29.82s (not urgent at all).

<Callout type="info" title="Scheduling Is a First-Class Optimization Problem">
  You do not control request arrivals, but you do control how you group and order work on the GPU. In multi-tenant environments, the scheduling policy often matters more for user satisfaction than raw throughput. A system running at 90% of peak throughput but meeting 99% of SLOs is better than one running at 100% throughput but missing 20% of SLOs.
</Callout>

### 6.5 Per-Tenant Fairness

In multi-tenant deployments, you also need per-tenant isolation:

- **Token budget per tenant**: cap how many tokens each tenant can have in-flight at once
- **Request rate limiting**: prevent one tenant from flooding the queue
- **Separate SLO tracking**: monitor p50/p90/p99 per tenant, not just globally

Without these controls, a single tenant submitting large batch jobs can degrade the experience for all other tenants, even with priority scheduling.

## 7. Multi-LoRA Batching

LoRA (Low-Rank Adaptation) is the dominant approach for serving personalized or fine-tuned models. Instead of deploying a separate copy of the full model for each fine-tune, you deploy one base model and swap in small LoRA adapter weights per request. But this creates a batching challenge.

### 7.1 The Problem

In standard continuous batching, all requests in a batch share the same model weights. The GPU loads weights once and applies them to all requests in the batch. With LoRA, different requests may use different adapters, meaning the weight matrices are different for each request.

Concretely, for a linear layer $y = Wx$, LoRA modifies it to:

$$
y = Wx + BAx
$$

where $B \in \mathbb{R}^{d \times r}$ and $A \in \mathbb{R}^{r \times d}$ are the LoRA adapter matrices, and $r$ is the LoRA rank (typically 8-64, much smaller than $d$). Different requests may have different $A$ and $B$ matrices.

You cannot simply batch requests with different LoRA adapters using standard matrix multiplication, because the adapter weights differ per request. Naive approaches either:
- Process each LoRA adapter separately (destroying batch efficiency)
- Or apply the same adapter to all requests (incorrect results)

### 7.2 The S-LoRA Approach

S-LoRA (Serving LoRA) solves this by:

1. **Grouping requests by adapter**: requests using the same LoRA adapter are batched together within a sub-batch
2. **Custom CUDA kernels**: that can apply different adapters to different requests within the same GPU kernel launch, avoiding the overhead of separate launches
3. **Adapter memory management**: using a pool of GPU memory to store frequently-used adapters, swapping cold adapters to CPU

```python
class MultiLoRAScheduler:
    def __init__(self, config):
        self.adapter_pool = AdapterPool(max_gpu_adapters=config.max_lora_adapters)
        self.base_scheduler = IterationLevelScheduler(config)

    def schedule(self):
        # Get base scheduling decisions
        scheduled = self.base_scheduler.schedule()

        # Group by adapter
        adapter_groups = defaultdict(list)
        for seq in scheduled.scheduled_seqs:
            adapter_id = seq.lora_adapter_id or "base"
            adapter_groups[adapter_id].append(seq)

        # Ensure all needed adapters are in GPU memory
        needed_adapters = set(adapter_groups.keys()) - {"base"}
        self.adapter_pool.ensure_loaded(needed_adapters)

        return MultiLoRASchedulerOutputs(
            adapter_groups=adapter_groups,
            adapter_pool=self.adapter_pool,
        )
```

## 8. When NOT to Use Continuous Batching

Continuous batching is not a universal improvement. There are scenarios where it adds complexity without benefit, or where simpler approaches are better.

### 8.1 Single-User Scenarios

If you are running a model for a single user on a single GPU (e.g., local inference with llama.cpp), there is no batching opportunity. There is only one request at a time, and the overhead of a scheduler provides no value. Just run the model directly.

### 8.2 Real-Time Applications Where ANY Batching Adds Latency

In some applications, the only metric that matters is minimum possible latency for a single request. Any form of batching -- even continuous -- introduces overhead:

- Scheduler decision time (~0.05ms per iteration, negligible)
- Sharing GPU compute with other requests (the main cost: each request gets a smaller share of memory bandwidth)
- Memory management overhead for block allocation

If your application demands absolute minimum latency and you are willing to sacrifice throughput for it, running at batch size 1 is optimal. This applies to some robotics control loops, real-time safety systems, and other scenarios where microseconds matter and throughput is irrelevant.

### 8.3 Very Homogeneous Workloads

If all your requests have nearly identical prompt lengths and generation lengths (e.g., a classification pipeline where every input is exactly 512 tokens and the output is always 1 token), static batching works fine. The waste from padding and completion stalls is near-zero because there is no length variation.

In this case, static batching is simpler to implement, easier to debug, and has marginally lower overhead than continuous batching. There is no benefit to the added complexity.

### 8.4 When You Are Compute-Bound, Not Memory-Bound

During prefill (processing the input prompt), the GPU is usually compute-bound, not memory-bandwidth-bound. Batching helps most during decode (which is memory-bandwidth-bound). If your workload is almost entirely prefill (e.g., embedding generation, classification), the benefit of continuous batching is smaller.

## 10. Metrics You Should Track

Operating a continuous batching system requires monitoring the right metrics. Here are the essential ones:

### 10.1 Throughput Metrics
- **Tokens per second per GPU**: the primary throughput metric. Track both prompt tokens/s and generation tokens/s separately.
- **Requests per second**: useful for capacity planning, but less informative than tokens/s because request sizes vary.

### 10.2 Latency Metrics
- **TTFT (Time to First Token)**: measures queueing + prefill time. Track p50, p90, p99.
- **TBT (Time Between Tokens)**: measures decode latency stability. Track p50, p90, p99.
- **End-to-end latency**: total time from request arrival to completion. Track p50, p90, p99.

### 10.3 System Metrics
- **GPU utilization**: SM active percentage and memory bandwidth utilization.
- **KV cache utilization**: percentage of KV blocks in use. If consistently above 90%, you are at risk of frequent preemption.
- **Batch size distribution**: histogram of actual batch sizes per iteration. Low average batch size at high load indicates a scheduling problem.
- **Preemption rate**: fraction of iterations involving a preemption. Above 5% is a red flag.
- **Queue depth**: number of requests waiting. Growing queue depth means you are overloaded.

### 10.4 Debugging Common Issues

```python
class SchedulerDiagnostics:
    def diagnose_low_throughput(self):
        metrics = self.get_metrics()

        if metrics.avg_batch_size < self.config.max_batch_size * 0.5:
            if metrics.memory_utilization > 0.9:
                print("Issue: Memory pressure limiting batch size")
                print("  -> Consider KV cache quantization or smaller max_seq_len")
            elif metrics.waiting_queue_avg > 0:
                print("Issue: Scheduler not admitting waiting requests")
                print("  -> Check admission policy and block reservation")

        if metrics.preemption_rate > 0.05:
            print(f"Issue: High preemption rate ({metrics.preemption_rate:.1%})")
            print(f"  Average swap size: {metrics.avg_swap_size_mb:.1f} MB")
            print("  -> Reduce max concurrent requests or add GPU memory")

        if metrics.avg_waiting_time > self.config.slo_ms:
            print(f"Issue: Queue time exceeding SLO ({metrics.avg_waiting_time:.1f}ms)")
            print("  -> System is overloaded, need more GPUs or request throttling")

        if metrics.prefill_stall_rate > 0.1:
            print(f"Issue: Prefill causing decode stalls ({metrics.prefill_stall_rate:.1%})")
            print("  -> Enable chunked prefill or reduce chunk size")
```

## 11. Queueing Theory Perspective

For those who want a more formal understanding, continuous batching can be modeled as a queueing system.

### 11.1 Basic Model

Requests arrive at rate $\lambda$ (requests per second). Each token decode step costs time $T$ for a single request. Batching $B$ requests together costs $T_B \leq B \cdot T$ (sublinear because weight loading is amortized). Throughput with batch size $B$:

$$
R_B \approx \frac{B}{T_B}
$$

Latency has two components:
- **Queueing delay**: time waiting to enter the batch (depends on arrival rate and current batch occupancy)
- **Service time**: time from entering the batch to completion (depends on generation length and per-token decode time)

### 11.2 Static vs Continuous Queueing

In static batching, the system behaves like an M/G/1 queue with batch service: requests accumulate until a batch forms, the entire batch is served, then the next batch starts. The queueing delay scales with batch service time (the time for the longest request in the batch to complete).

In continuous batching, the system behaves more like an M/G/c queue where the number of servers $c$ equals the maximum batch size: each request enters service (almost) immediately if there is capacity, and leaves independently when it finishes. The queueing delay is determined by system utilization, not batch completion time.

The key insight: in static batching, queueing delay is $O(L_{\max})$ where $L_{\max}$ is the maximum generation length in the batch. In continuous batching, queueing delay is $O(1/\mu)$ where $\mu$ is the rate at which slots become available. Under high load with diverse generation lengths, this difference is enormous.

### 11.3 Practical Guidance by Load Level

- **Low traffic** ($\lambda$ &lt; 4 req/s): prioritize latency over throughput. Small batches, tight timeouts. Continuous batching helps little because the GPU is rarely processing multiple requests.
- **Medium traffic** (4-32 req/s): the sweet spot for continuous batching. Significant throughput gains with good latency.
- **High traffic** (&gt; 32 req/s): continuous batching is essential. Use age-based fairness to prevent starvation. Monitor preemption rate closely.
- **Multi-tenant**: enforce per-tenant token budgets. Monitor per-tenant p99 separately. Use SLO-aware scheduling.

## 12. Implementation Considerations

### 12.1 Handling Streaming Responses

Most LLM applications use streaming: tokens are sent to the client as they are generated, rather than waiting for the entire response. Continuous batching integrates naturally with streaming because each decode step produces one token per active request. The scheduler simply sends each token to the appropriate client's response stream after each iteration.

### 12.2 Speculative Decoding Interaction

Speculative decoding (using a smaller draft model to generate candidate tokens verified by the target model) interacts non-trivially with continuous batching. The draft phase generates multiple candidate tokens per request, which temporarily increases memory consumption. The verification phase may reject some candidates, leading to variable-length decode steps. Modern schedulers handle this by reserving extra KV blocks for speculative candidates and adjusting the token budget to account for the variable work per iteration.

### 12.3 Quantized KV Cache

Quantizing the KV cache (e.g., from FP16 to INT8 or FP8) reduces memory consumption per request by 2x, allowing more concurrent requests in the batch. This amplifies the benefit of continuous batching: more concurrent requests means higher effective batch size, which means higher throughput. On A100 with Llama-70B, KV cache quantization to FP8 increases maximum concurrent requests from ~64 to ~128, improving peak throughput by an additional 40-60%.

### 12.4 Request Cancellation

Users cancel requests (close browser tab, timeout on client side). The scheduler must handle this gracefully: immediately free the cancelled request's KV cache blocks and remove it from the running queue. In continuous batching, this happens naturally at the next scheduling iteration. The freed blocks are immediately available for new requests.

## 13. Conclusion

Continuous batching is the single most important system-level optimization for LLM inference at scale. It transforms the problem from batch processing (with its inherent waste from padding and completion stalls) into real-time scheduling (where every GPU cycle does useful work).

The key ideas, in summary:

**Static batching wastes 50-70% of GPU cycles** on padding and idle slots. The waste increases with workload heterogeneity -- exactly the conditions found in production.

**Continuous batching eliminates this waste** by re-evaluating the batch at every decode iteration. Requests enter and exit independently, no padding is needed, and no slots sit idle. The result is 2-4x throughput improvement.

**The scheduler is the brain** of the system, maintaining three queues (waiting, running, swapped) and making memory-aware admission decisions every iteration. It must balance throughput against latency, fairness against priority, and prefill against decode.

**Chunked prefill** prevents long prompts from stalling decode requests by breaking prefill into manageable chunks that interleave with decode work. The chunk size controls the TTFT-vs-TBT tradeoff.

**Preemption** is the safety valve: when memory runs out, the scheduler evicts running requests via swap (copy KV to CPU) or recompute (discard KV and redo later). Swap is usually better for long sequences; recompute for short ones.

**Priority and fairness** policies determine which requests are served when. FCFS is simple but SLO-blind. Priority queues give control but risk starvation. Fair-share and SLO-aware scheduling balance competing objectives.

**Multi-LoRA batching** extends continuous batching to serve multiple fine-tuned models from a single base model, with measurable but manageable overhead from adapter switching and reduced batch efficiency.

The field continues to evolve. Prefill-decode separation, speculative decoding integration, and ever-larger context windows create new scheduling challenges. But the fundamental insight -- that inference scheduling should be iteration-level, memory-aware, and continuously adaptive -- is now firmly established as the foundation of production LLM serving.
