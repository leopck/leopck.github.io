I2C is the workhorse bus for sensor interfacing on microcontrollers. At 100 kHz, it works with almost anything -- plug in the pull-ups, call the driver, read your data. But when you need to pull IMU data at 1-4 kHz, stream multiple sensors simultaneously, or push toward the 1 MHz Fast Mode Plus limit, I2C becomes a serious signal integrity and bus engineering problem. This post provides a complete treatment of I2C performance on the ESP32, from bit-level protocol timing through pull-up resistor calculations to measured bus analyzer captures showing where time is actually spent.

## I2C Protocol Deep Dive: What Happens on Every Transaction

Understanding where time is spent at the bit level is essential for optimizing I2C throughput. Every I2C read transaction involves a specific sequence of conditions and data transfers.

### START and STOP Conditions

The I2C START condition is defined by SDA transitioning from HIGH to LOW while SCL is HIGH. The STOP condition is SDA transitioning from LOW to HIGH while SCL is HIGH. These transitions require specific timing:

The bus free time ($t_{BUF}$) between a STOP condition and the next START condition is often overlooked. At 400 kHz, it is 1.3 microseconds -- which means back-to-back transactions have a mandatory 1.3-microsecond gap even if software is ready instantly.

## Clock Stretching and Its Impact on Bus Throughput

Clock stretching is a mechanism where a slave device holds SCL LOW to signal that it is not ready for the next bit. The master must wait until SCL is released before continuing. This is the single most unpredictable factor in I2C timing.

<Callout type="warning" title="Clock Stretching Blocks the Entire Bus">
  When any device on the I2C bus stretches the clock, ALL devices on that bus are stalled. A single slow EEPROM doing a 5 ms page write blocks your IMU from being read for that entire 5 ms. If you have a device that stretches significantly, either put it on a separate I2C bus (the ESP32 has two I2C controllers) or ensure your transaction scheduling accounts for the stretch duration.
</Callout>

### ESP32 Clock Stretching Timeout

The ESP32 I2C hardware includes a configurable clock-stretching timeout to prevent a malfunctioning slave from locking the bus permanently:

```cpp
#include "driver/i2c.h"

void configure_i2c_with_timeout(void) {
    i2c_config_t conf = {
        .mode = I2C_MODE_MASTER,
        .sda_io_num = GPIO_NUM_21,
        .scl_io_num = GPIO_NUM_22,
        .sda_pullup_en = GPIO_PULLUP_ENABLE,
        .scl_pullup_en = GPIO_PULLUP_ENABLE,
        .master = {
            .clk_speed = 400000,
        },
    };

    i2c_param_config(I2C_NUM_0, &conf);
    i2c_driver_install(I2C_NUM_0, I2C_MODE_MASTER, 0, 0, 0);

    // Set clock stretching timeout
    // Timeout = (timeout_value * APB_CLK_period)
    // At 80 MHz APB: timeout = value * 12.5 ns
    // 0xFFFFF (~1M) * 12.5 ns = ~13 ms
    i2c_set_timeout(I2C_NUM_0, 0xFFFFF);  // ~13 ms timeout
}
```

If the timeout fires, the ESP32 I2C hardware generates a bus error, and the driver returns `ESP_ERR_TIMEOUT`. Your application must then issue a bus reset (9 clock pulses on SCL) to recover.

### Bus Reset After Clock Stretch Timeout

```cpp
#define I2C_SCL_PIN  22
#define I2C_SDA_PIN  21

esp_err_t i2c_bus_reset(void) {
    // Deinit I2C to take control of pins
    i2c_driver_delete(I2C_NUM_0);

    // Configure SCL as open-drain output, SDA as input
    gpio_set_direction(I2C_SCL_PIN, GPIO_MODE_OUTPUT_OD);
    gpio_set_direction(I2C_SDA_PIN, GPIO_MODE_INPUT);
    gpio_set_level(I2C_SCL_PIN, 1);

    // Generate 9 clock pulses to free any stuck slave
    for (int i = 0; i < 9; i++) {
        gpio_set_level(I2C_SCL_PIN, 0);
        ets_delay_us(5);
        gpio_set_level(I2C_SCL_PIN, 1);
        ets_delay_us(5);

        // Check if SDA has been released
        if (gpio_get_level(I2C_SDA_PIN) == 1) {
            break;
        }
    }

    // Generate STOP condition: SDA LOW-to-HIGH while SCL HIGH
    gpio_set_direction(I2C_SDA_PIN, GPIO_MODE_OUTPUT_OD);
    gpio_set_level(I2C_SDA_PIN, 0);
    ets_delay_us(5);
    gpio_set_level(I2C_SCL_PIN, 1);
    ets_delay_us(5);
    gpio_set_level(I2C_SDA_PIN, 1);
    ets_delay_us(5);

    // Reinitialize I2C
    configure_i2c_with_timeout();

    return ESP_OK;
}
```

## Multi-Device Bus Sharing and Arbitration

The I2C specification supports multiple masters and up to 128 devices (using 7-bit addressing) on a single bus. The ESP32 supports both multi-master arbitration and multi-device bus sharing.

### Bus Arbitration Mechanism

When two masters attempt to transmit simultaneously, I2C uses a wired-AND arbitration scheme: both masters monitor SDA while transmitting. If a master sends a HIGH but reads SDA as LOW, another master is driving the bus LOW -- the first master loses arbitration and must stop transmitting immediately.

On the ESP32, arbitration loss is reported as `ESP_ERR_TIMEOUT` or through the I2C status register. The ESP-IDF driver handles arbitration loss automatically by releasing the bus, but it does not automatically retry -- your application code must implement retry logic:

```cpp
esp_err_t i2c_read_with_retry(uint8_t dev_addr, uint8_t reg_addr,
                               uint8_t* data, size_t len,
                               int max_retries) {
    esp_err_t ret;
    int retry_count = 0;

    do {
        i2c_cmd_handle_t cmd = i2c_cmd_link_create();
        i2c_master_start(cmd);
        i2c_master_write_byte(cmd, (dev_addr << 1) | I2C_MASTER_WRITE, true);
        i2c_master_write_byte(cmd, reg_addr, true);
        i2c_master_start(cmd);  // Repeated START
        i2c_master_write_byte(cmd, (dev_addr << 1) | I2C_MASTER_READ, true);
        i2c_master_read(cmd, data, len, I2C_MASTER_LAST_NACK);
        i2c_master_stop(cmd);

        ret = i2c_master_cmd_begin(I2C_NUM_0, cmd, pdMS_TO_TICKS(50));
        i2c_cmd_link_delete(cmd);

        if (ret == ESP_OK) break;

        retry_count++;
        vTaskDelay(pdMS_TO_TICKS(1));  // Brief delay before retry
    } while (retry_count < max_retries);

    return ret;
}
```

## Fast Mode Plus (1 MHz): Pushing the Electrical Limits

I2C Fast Mode Plus (FM+) operates at 1 MHz, providing 2.5x the throughput of standard Fast Mode (400 kHz). However, the electrical requirements are significantly more demanding.

The critical constraint at 1 MHz is rise time. The I2C specification requires $t_r \leq 120$ ns for FM+, and the rise time is determined by the pull-up resistor and bus capacitance:

$$
t_r \approx -R_{pu} \cdot C_{bus} \cdot \ln\left(\frac{V_{DD} - V_{IH}}{V_{DD} - V_{OL}}\right)
$$

For a 3.3V system with $V_{IH} = 0.7 \times V_{DD} = 2.31\text{V}$ and $V_{OL} = 0.4\text{V}$:

$$
t_r \approx 0.8473 \cdot R_{pu} \cdot C_{bus}
$$

## Pull-Up Resistor Selection: The Calculation

Choosing the correct pull-up resistor is the most critical electrical design decision for I2C performance. Too large and the rise time exceeds specification, causing bit errors. Too small and the sink current exceeds the device IOL rating, preventing proper LOW generation.

### Calculating Bus Capacitance

Total bus capacitance is the sum of:

$$
C_{bus} = C_{wire} + \sum_{i=1}^{N} C_{pin,i} + C_{stray}
$$

Where:
- $C_{wire}$: Trace/wire capacitance (~1 pF/cm for PCB trace, ~50 pF/m for cable)
- $C_{pin,i}$: Input capacitance of each device (typically 3-10 pF per device)
- $C_{stray}$: Board-level stray capacitance (10-30 pF typical)

### Measuring Bus Capacitance In Situ

You can measure actual bus capacitance using the ESP32 and a known pull-up resistor:

```cpp
#include "driver/gpio.h"
#include "esp_timer.h"

#define SDA_PIN    21
#define KNOWN_PULLUP_OHM  4700.0f  // Known external pull-up value

float measure_bus_capacitance_pf(void) {
    // Step 1: Drive SDA LOW to discharge the bus
    gpio_set_direction(SDA_PIN, GPIO_MODE_OUTPUT_OD);
    gpio_set_level(SDA_PIN, 0);
    ets_delay_us(100);  // Allow full discharge

    // Step 2: Release SDA (go high-impedance) and time the rise
    int64_t t_start = esp_timer_get_time();
    gpio_set_direction(SDA_PIN, GPIO_MODE_INPUT);

    // Wait for SDA to reach HIGH (VIH threshold)
    while (gpio_get_level(SDA_PIN) == 0) {
        if ((esp_timer_get_time() - t_start) > 1000) {
            return -1.0f;  // Timeout: no pull-up or bus stuck
        }
    }
    int64_t t_rise_us = esp_timer_get_time() - t_start;

    // Step 3: Calculate capacitance
    // t_r = 0.8473 * R * C  =>  C = t_r / (0.8473 * R)
    float t_rise_s = t_rise_us * 1e-6f;
    float c_bus_f = t_rise_s / (0.8473f * KNOWN_PULLUP_OHM);
    float c_bus_pf = c_bus_f * 1e12f;

    printf("Rise time: %lld us\n", t_rise_us);
    printf("Bus capacitance: %.0f pF\n", c_bus_pf);

    return c_bus_pf;
}
```

<Callout type="tip" title="Practical Pull-Up Resistor Selection">
  For most ESP32 projects with 2-4 I2C devices on a short PCB (under 10 cm traces), bus capacitance is typically 50-100 pF. Use 4.7 kOhm pull-ups for 400 kHz (safe, universally compatible), or 2.2 kOhm for 400 kHz on longer buses. For FM+ at 1 MHz, use 1 kOhm with verified FM+-rated devices and bus capacitance under 150 pF. When in doubt, measure rise time with an oscilloscope -- if the rising edges are rounded and the rise time exceeds 70% of the spec, your pull-ups are too weak.
</Callout>

## ESP32 I2C Hardware Configuration

The ESP32 has two independent I2C controllers. Distributing devices across both buses doubles total bandwidth and isolates slow or stretching devices.

### Basic I2C Configuration

```cpp
#include "driver/i2c.h"

// Initialize I2C port 0: high-speed sensor bus
void i2c_sensor_bus_init(void) {
    i2c_config_t conf = {
        .mode = I2C_MODE_MASTER,
        .sda_io_num = GPIO_NUM_21,
        .scl_io_num = GPIO_NUM_22,
        .sda_pullup_en = GPIO_PULLUP_DISABLE,  // External pull-ups used
        .scl_pullup_en = GPIO_PULLUP_DISABLE,   // External pull-ups used
        .master = {
            .clk_speed = 400000,  // 400 kHz Fast Mode
        },
    };
    i2c_param_config(I2C_NUM_0, &conf);
    i2c_driver_install(I2C_NUM_0, I2C_MODE_MASTER, 0, 0, 0);
}

// Initialize I2C port 1: slow device bus (EEPROM, display)
void i2c_slow_bus_init(void) {
    i2c_config_t conf = {
        .mode = I2C_MODE_MASTER,
        .sda_io_num = GPIO_NUM_18,
        .scl_io_num = GPIO_NUM_19,
        .sda_pullup_en = GPIO_PULLUP_DISABLE,
        .scl_pullup_en = GPIO_PULLUP_DISABLE,
        .master = {
            .clk_speed = 100000,  // 100 kHz Standard Mode
        },
    };
    i2c_param_config(I2C_NUM_1, &conf);
    i2c_driver_install(I2C_NUM_1, I2C_MODE_MASTER, 0, 0, 0);
}
```

<Callout type="tip" title="External Pull-Ups Always Beat Internal">
  The ESP32's internal pull-ups are approximately 45 kOhm -- far too weak for 400 kHz I2C. At 100 pF bus capacitance, 45 kOhm gives a rise time of approximately 3.8 microseconds, which exceeds even the 100 kHz specification (1.0 us max). Always use external pull-up resistors. Disable the internal pull-ups to avoid the slight parallel resistance effect, which is negligible anyway at 45 kOhm.
</Callout>

### Optimized Burst Read Implementation

```cpp
// High-performance burst read with minimal overhead
esp_err_t i2c_burst_read(i2c_port_t port, uint8_t dev_addr,
                          uint8_t reg_addr, uint8_t* data, size_t len) {
    // Use static command link buffer to avoid malloc per transaction
    static uint8_t cmd_buf[I2C_LINK_RECOMMENDED_SIZE(8)] = {0};

    i2c_cmd_handle_t cmd = i2c_cmd_link_create_static(cmd_buf, sizeof(cmd_buf));

    // Write phase: send register address
    i2c_master_start(cmd);
    i2c_master_write_byte(cmd, (dev_addr << 1) | I2C_MASTER_WRITE, true);
    i2c_master_write_byte(cmd, reg_addr, true);

    // Read phase: repeated start + read data
    i2c_master_start(cmd);
    i2c_master_write_byte(cmd, (dev_addr << 1) | I2C_MASTER_READ, true);

    if (len > 1) {
        i2c_master_read(cmd, data, len - 1, I2C_MASTER_ACK);
    }
    i2c_master_read_byte(cmd, data + len - 1, I2C_MASTER_NACK);

    i2c_master_stop(cmd);

    esp_err_t ret = i2c_master_cmd_begin(port, cmd, pdMS_TO_TICKS(50));
    i2c_cmd_link_delete_static(cmd);

    return ret;
}
```

The `i2c_cmd_link_create_static()` function avoids dynamic memory allocation on every transaction. For a sensor read at 1000 Hz, this eliminates 1000 malloc/free pairs per second, reducing both CPU overhead and heap fragmentation.

### Normal 14-Byte Burst Read at 400 kHz

```
SCL  __|^^|__|^^|__|^^|__|^^|__|^^|__|^^|__|^^|__|^^|__|^^|__|^^|...
SDA  __|  S  |A6|A5|A4|A3|A2|A1|A0| W|ACK|R7|R6|R5|R4|R3|R2|R1|R0|ACK|Sr|...

     |------ 10 us ------|  per bit at 400 kHz

Timeline:
  0 us:     START condition
  2.5 us:   First address bit
  22.5 us:  ACK from slave
  25 us:    Register address byte begins
  47.5 us:  ACK from slave
  50 us:    Repeated START
  72.5 us:  ACK from slave (read mode)
  75 us:    First data byte begins
  ...
  390 us:   14th data byte NACK
  395 us:   STOP condition

Total: ~400 us for 14 bytes = 28.6 KB/s effective
Overhead: 75 us (address phases) + 5 us (START/STOP) = 80 us
Payload: 14 * 22.5 us = 315 us
Efficiency: 315/395 = 79.7%
```

### Clock Stretching Visible on Analyzer

```
Normal:     SCL  |^^|__|^^|__|^^|__|^^|__|
                  bit7  bit6  bit5  bit4

Stretched:  SCL  |^^|__|^^|__|^^|__________...________|^^|__|
                  bit7  bit6  bit5  <-- slave holds SCL LOW -->  bit4

The stretch can last from microseconds to milliseconds.
During this time, all bus activity is frozen.
```

### Bus Error: SDA Stuck LOW

```
SCL  |^^|__|^^|__|^^|__|^^|__|^^|__|^^|__|^^|__|^^|__|^^|__|^^|__
SDA  _________________________________________________________________

SDA is stuck LOW -- slave did not release after ACK.
The master cannot generate a START or STOP condition.
Recovery: generate 9 SCL pulses until SDA goes HIGH, then STOP.
```

## Software Optimization Strategies

### 1. Pack Reads into Burst Transactions

This is the single highest-impact optimization. Instead of multiple small reads:

```cpp
// BAD: 7 separate transactions for IMU data
uint8_t ax_l, ax_h, ay_l, ay_h, az_l, az_h, temp;
i2c_read_reg(IMU_ADDR, REG_ACCEL_X_L, &ax_l, 1);
i2c_read_reg(IMU_ADDR, REG_ACCEL_X_H, &ax_h, 1);
i2c_read_reg(IMU_ADDR, REG_ACCEL_Y_L, &ay_l, 1);
// ... 4 more reads
// Total: 7 * 390 us = 2730 us

// GOOD: single burst read
uint8_t imu_data[14];
i2c_burst_read(I2C_NUM_0, IMU_ADDR, REG_ACCEL_X_L, imu_data, 14);
// Total: ~400 us (6.8x faster)
```

### 2. Cache Configuration Registers

Many I2C sensor operations involve read-modify-write sequences on configuration registers. Cache the register values in RAM:

```cpp
typedef struct {
    uint8_t ctrl_reg1;
    uint8_t ctrl_reg2;
    uint8_t ctrl_reg3;
    uint8_t ctrl_reg4;
    bool dirty;
} sensor_config_cache_t;

static sensor_config_cache_t config_cache;

void sensor_set_output_rate(uint8_t rate) {
    // Modify cached value (no bus transaction)
    config_cache.ctrl_reg1 = (config_cache.ctrl_reg1 & 0x0F) | (rate << 4);
    config_cache.dirty = true;
}

void sensor_flush_config(void) {
    if (config_cache.dirty) {
        // Single write to update changed registers
        i2c_write_reg(SENSOR_ADDR, REG_CTRL1, config_cache.ctrl_reg1);
        config_cache.dirty = false;
    }
}
```

### 5. Schedule I2C Around WiFi and BLE

On the ESP32, WiFi and BLE task execution can introduce latency jitter into I2C transactions. If you need deterministic sensor timing:

```cpp
// Pin I2C sensor reading to Core 1, WiFi runs on Core 0
void sensor_task(void* param) {
    TickType_t last_wake = xTaskGetTickCount();

    while (true) {
        uint8_t data[14];
        int64_t t0 = esp_timer_get_time();

        // Read IMU data
        esp_err_t ret = i2c_burst_read(I2C_NUM_0, IMU_ADDR, 0x3B, data, 14);

        int64_t dt = esp_timer_get_time() - t0;

        if (ret == ESP_OK) {
            process_imu_data(data, dt);
        }

        // Fixed-period scheduling
        vTaskDelayUntil(&last_wake, pdMS_TO_TICKS(1));  // 1 kHz
    }
}

void setup_sensor_task(void) {
    // Pin to Core 1 (WiFi uses Core 0)
    xTaskCreatePinnedToCore(sensor_task, "sensor", 4096, NULL,
                            configMAX_PRIORITIES - 1,  // Highest priority
                            NULL, 1);  // Core 1
}
```

## Measuring Bus Utilization

Define bus utilization as:

$$
U = \frac{\text{total bits transferred}}{\text{clock rate} \times \text{elapsed time}}
$$

```cpp
typedef struct {
    uint32_t total_bytes;
    uint32_t total_overhead_bits;
    uint32_t transaction_count;
    int64_t start_time;
} i2c_bus_stats_t;

static i2c_bus_stats_t bus_stats = {0};

void bus_stats_record_transaction(uint32_t payload_bytes) {
    bus_stats.total_bytes += payload_bytes;
    bus_stats.total_overhead_bits += 31;  // Fixed overhead per transaction
    bus_stats.transaction_count++;
}

void bus_stats_print(uint32_t clock_hz) {
    int64_t elapsed_us = esp_timer_get_time() - bus_stats.start_time;
    double elapsed_s = elapsed_us / 1e6;

    uint32_t total_bits = bus_stats.total_bytes * 9 + bus_stats.total_overhead_bits;
    double utilization = (double)total_bits / (clock_hz * elapsed_s);

    printf("I2C Bus Statistics (%.1f seconds):\n", elapsed_s);
    printf("  Transactions: %lu\n", bus_stats.transaction_count);
    printf("  Payload bytes: %lu\n", bus_stats.total_bytes);
    printf("  Total bits on wire: %lu\n", total_bits);
    printf("  Bus utilization: %.1f%%\n", utilization * 100.0);
    printf("  Effective data rate: %.1f KB/s\n",
           bus_stats.total_bytes / elapsed_s / 1024.0);
}
```

## Conclusion

I2C performance optimization on the ESP32 comes down to three categories: electrical design, protocol efficiency, and software architecture.

On the electrical side, pull-up resistor selection based on measured bus capacitance determines the maximum reliable clock rate. The formula $t_r \approx 0.8473 \cdot R_{pu} \cdot C_{bus}$ must yield a rise time within specification for your target speed mode. For 400 kHz, use 2.2-4.7 kOhm with typical bus capacitances of 50-100 pF. For 1 MHz FM+, use 680-1.5 kOhm and verify all devices on the bus are FM+-rated.

On the protocol side, burst reads are the single highest-impact optimization. A 14-byte burst read is 6.8x more efficient than 14 individual single-byte reads due to address and start/stop overhead amortization. Beyond that, distributing devices across both I2C controllers isolates slow or clock-stretching devices from time-critical sensors.

On the software side, static command link buffers, core pinning for deterministic timing, and register caching eliminate unnecessary overhead. Combined with proper bus scheduling around WiFi activity, these techniques allow the ESP32 to reliably maintain 1-4 kHz sensor read rates on a 400 kHz I2C bus with margin for other devices.
