Most CUDA kernels are memory-bandwidth-bound — the GPU can compute far faster than it can feed data from HBM. On an A100, the peak compute-to-bandwidth ratio is 156 FLOP/byte (FP16 tensor cores). Any operation with arithmetic intensity below this threshold is limited by how fast data moves, not how fast it's processed. This post covers the practical techniques for maximizing memory throughput.

## Measuring Your Bandwidth

Before optimizing, measure achieved bandwidth:

```bash
ncu --metrics \
  dram__bytes_read.sum,dram__bytes_write.sum,\
  l2__read_throughput.avg.pct_of_peak_sustained_active,\
  sm__throughput.avg.pct_of_peak_sustained_active \
  ./my_kernel
```

## Technique 1: Vectorized Loads (float4, int4)

Loading 4 floats at once (float4 = 128 bits) instead of 1 float (32 bits) reduces the number of memory transactions and improves coalescing:

```cpp
// Scalar: 32 threads × 4 bytes = 128 bytes per transaction
__global__ void kernel_scalar(float *in, float *out, int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    out[idx] = in[idx] * 2.0f;  // 1 float per thread
}

// Vectorized: 32 threads × 16 bytes = 512 bytes per transaction
__global__ void kernel_vec4(float4 *in, float4 *out, int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    float4 val = in[idx];  // 4 floats per thread
    val.x *= 2.0f; val.y *= 2.0f; val.z *= 2.0f; val.w *= 2.0f;
    out[idx] = val;
}
```

<Callout type="tip" title="When to Use float4">
  Use `float4` loads whenever your data is 16-byte aligned and you process 4 contiguous elements per thread. This matches the GPU's 128-bit memory bus width and minimizes transaction count. Most element-wise kernels (activation functions, normalization, residual add) benefit from vectorization.
</Callout>

```cpp
// Hint to L2 cache: persist this data
// Ampere+ supports L2 cache persistence control
cudaAccessPolicyWindow window;
window.base_ptr = kv_cache_ptr;
window.num_bytes = kv_cache_size;
window.hitRatio = 1.0f;  // Try to keep in L2
window.hitProp = cudaAccessPropertyPersisting;
window.missProp = cudaAccessPropertyStreaming;
cudaCtxSetAccessPolicyWindow(&window);
```

## Technique 3: Async Copy (cp.async)

Ampere introduced `cp.async` for direct global→shared memory copy without going through registers, enabling overlap of copy and compute:

```cpp
// Traditional: Global → Registers → Shared (serialized)
__shared__ float smem[BLOCK_SIZE];
smem[threadIdx.x] = global[idx];  // Load to register, then store to smem
__syncthreads();

// Async: Global → Shared directly (can overlap with compute)
#include <cuda/pipeline>
__shared__ float smem[BLOCK_SIZE];
cuda::memcpy_async(&smem[threadIdx.x], &global[idx], sizeof(float), pipeline);
pipeline.commit();
// ... do other work while copy happens ...
pipeline.wait_priorunder 0>();  // Wait for copy to complete
```

<Callout type="warning" title="AoS Is the Silent Killer">
  Array-of-Structures (AoS) layout where adjacent threads access different fields of the same struct causes strided access. Structure-of-Arrays (SoA) where each field is a contiguous array enables perfect coalescing. For GPU code, always default to SoA. The 4× bandwidth difference is real.
</Callout>

## Conclusion

GPU memory bandwidth optimization follows a clear priority: fix coalescing first (4× impact), then use vectorized loads (1.2×), then ensure alignment (1.05×). For tiled kernels, async copy and L2 cache exploitation provide additional gains. The target is >85% of peak HBM bandwidth for streaming kernels and over 60% for complex access patterns. Always measure with `ncu` before and after — the achieved bandwidth metric directly tells you how close you are to the hardware limit.
