When a model is too large to fit in a single GPU's memory, pipeline parallelism splits the model across devices — each GPU holds a subset of layers. The challenge is keeping all GPUs busy simultaneously. Both GPipe (Google, 2019) and PipeDream (Microsoft, 2019) solve this with microbatching, but their scheduling strategies create fundamentally different trade-offs.

## Why Pipeline Parallelism?

Pipeline parallelism is attractive because inter-stage communication is minimal: only activations at the boundary between stages need to cross GPUs. This makes it work well across nodes connected by InfiniBand, where tensor parallelism's per-layer all-reduce would be too expensive.

## The Pipeline Bubble Problem

With naive pipeline execution (one microbatch, sequential stages), most GPUs sit idle most of the time. With P pipeline stages, GPU utilization is only 1/P:

<PerfChart

**Pipeline efficiency = M / (M + P - 1)** where M = microbatches and P = pipeline stages. With 4 stages and 32 microbatches, efficiency reaches 91%. The remaining 9% is the "pipeline bubble" — time when stages are filling or draining.

## GPipe: Synchronous Pipeline

GPipe processes all microbatches through the forward pass first, then all through the backward pass. Gradients are accumulated and applied synchronously:

**GPipe advantages:** Deterministic training (same result regardless of timing), simple gradient accumulation (just sum microbatch gradients), identical to standard SGD semantics.

**GPipe disadvantages:** Must store all intermediate activations for all microbatches during the forward pass (memory grows as O(M × activations_per_stage)). The pipeline bubble is maximal because forward and backward are strictly separated.

<Callout type="warning" title="GPipe Memory Overhead">
  GPipe stores activations for ALL microbatches simultaneously during the forward pass. With 32 microbatches and 4 stages, each GPU stores 32× the activations of a single forward pass. Activation checkpointing (recomputing activations during backward) reduces this to ~1× but adds ~33% compute overhead.
</Callout>

## PipeDream: Asynchronous Pipeline (1F1B)

PipeDream interleaves forward and backward passes in a "one forward, one backward" (1F1B) steady-state schedule. As soon as a stage completes a forward pass, it immediately starts the backward pass for an earlier microbatch:

<Benchmark

  notes="1F1B steady state: each GPU alternates one forward and one backward step. Much less memory than GPipe."

**PipeDream advantages:** Each GPU stores activations for at most P microbatches (not M), dramatically reducing memory. The 1F1B schedule fills the pipeline faster.

**PipeDream disadvantages:** Weight versions can differ between stages (weight staleness). The original PipeDream used asynchronous weight updates, which changes optimization semantics.

## PipeDream-2BW and PipeDream-Flush

The original PipeDream's weight staleness issue led to two refined variants:

**PipeDream-Flush** (used in Megatron-LM): Runs the 1F1B schedule within a minibatch but synchronously flushes the pipeline between minibatches. This eliminates weight staleness at the cost of a small flush bubble.

**PipeDream-2BW**: Maintains two weight versions and uses the correct version for each microbatch. Eliminates staleness with only 2× weight memory overhead (minor compared to activation savings).

<Callout type="info" title="What's Used in Production (2024)">
  Modern frameworks (Megatron-LM, DeepSpeed) use **PipeDream-Flush** (also called "interleaved 1F1B") as the default pipeline schedule. It combines PipeDream's memory efficiency with GPipe's deterministic semantics. The pipeline bubble is minimized by interleaving virtual pipeline stages.
</Callout>

## Conclusion

GPipe and PipeDream represent the two fundamental approaches to pipeline scheduling: synchronous (fill-then-drain) vs interleaved (1F1B). PipeDream-Flush has emerged as the practical winner — combining PipeDream's memory efficiency and smaller bubble with GPipe's deterministic training semantics. The key tuning knobs are microbatch count (higher = less bubble), stage balance (equalize compute time per stage), and whether to use activation checkpointing (trades 33% compute for O(1) activation memory). For production training of 100B+ parameter models, pipeline parallelism is combined with tensor parallelism (intra-node) and data parallelism (inter-group) in a 3D parallelism strategy.
