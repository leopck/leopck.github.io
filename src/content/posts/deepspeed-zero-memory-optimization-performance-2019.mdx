## Introduction

When training a 1B parameter model with Adam optimizer, each GPU must store: the model parameters (4GB in FP32), a full copy of the gradients (4GB), and two optimizer states — momentum and variance (8GB). That's **16GB per GPU**, and every GPU in data-parallel training stores the *exact same thing*. With 8 GPUs, you're using 128GB total to store 8 identical copies of 16GB of state.

Microsoft's ZeRO (Zero Redundancy Optimizer), released with DeepSpeed in late 2019, attacks this redundancy directly. Instead of replicating everything, it partitions optimizer state, gradients, and even parameters across GPUs — trading communication for memory.

### Stage 1: The Free Win

ZeRO-1 partitions only optimizer states. Each GPU owns 1/N of the Adam momentum and variance tensors. After computing gradients locally (via the standard all-reduce), each GPU updates only its assigned slice of parameters using its local optimizer state, then broadcasts the updated parameter shard to all other GPUs.

The critical insight: **ZeRO-1 has the same communication volume as standard data parallelism.** The all-reduce is replaced with a reduce-scatter (each GPU gets its gradient shard) followed by an all-gather (broadcasting updated parameters). The total bytes transferred per GPU is identical — you get memory savings for free.

### Stage 2: Gradient Partitioning

ZeRO-2 extends the partitioning to gradients. Instead of each GPU storing all gradients after the all-reduce, each GPU only receives and stores the gradient shard corresponding to its assigned parameters. This is implemented by replacing all-reduce with reduce-scatter — a communication primitive that's already well-optimized in NCCL.

<Callout type="perf" title="Why ZeRO-2 Is the Production Sweet Spot">
  ZeRO-2 delivers ~3× memory reduction over baseline with zero additional communication overhead (reduce-scatter has the same bandwidth cost as all-reduce). It's enabled by default in most DeepSpeed deployments and is the most widely-used stage.
</Callout>

### Stage 3: Full Partitioning

ZeRO-3 partitions even the model parameters. Each GPU stores only 1/N of the full parameter tensor. During the forward pass, parameters are gathered on-demand via all-gather before each layer's computation, then discarded. The backward pass does the same.

This comes at a cost: **1.5× the communication volume of baseline.** For each layer, you need an all-gather during forward and another during backward, plus the reduce-scatter for gradients. In practice, DeepSpeed overlaps these communications with computation, hiding much of the latency.

ZeRO-1 and ZeRO-2 are communication-neutral — they reorganize the same bytes differently but don't add traffic. ZeRO-3 adds 50% more communication, which is the price for fully eliminating memory redundancy.

<Callout type="info" title="When to Use Each Stage">
  **ZeRO-1:** Your model fits in GPU memory but optimizer states don't. Basically free — always enable it. **ZeRO-2:** Model + gradients are tight. Still zero overhead. Use as default. **ZeRO-3:** Model parameters themselves don't fit on a single GPU. Accept 1.5× communication cost for the ability to train at all.
</Callout>

## Conclusion

ZeRO's key insight is that data-parallel training stores N redundant copies of state that can be partitioned instead. Stages 1 and 2 achieve this with no communication penalty — they're strict improvements over baseline. Stage 3 pushes further at the cost of 1.5× bandwidth, enabling training of models that would otherwise require model parallelism.

Combined with mixed-precision training (which halves the base memory) and activation checkpointing, ZeRO makes it possible to train 100B+ parameter models on commodity GPU clusters — a capability that was previously limited to specialized infrastructure.
