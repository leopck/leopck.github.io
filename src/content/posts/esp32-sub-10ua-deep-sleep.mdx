The ESP32 datasheet claims 10 uA deep sleep current. Most developers measure 150-500 uA on their boards. The gap is not a lie in the datasheet -- it is the cumulative effect of dozens of small leakage sources, each individually minor but devastating in aggregate. This guide enumerates every source, provides a measured current contribution for each, and describes the exact steps to eliminate them, progressing from stock hardware at 150 uA down to an aggressively optimized design at 5 uA.

## The Measurement Setup

Before chasing microamps, you need measurement equipment that can resolve them. Consumer multimeters are typically inadequate for sub-100 uA sleep current measurement because their burden voltage (the voltage dropped across the meter's shunt resistor) can disturb the circuit, and their resolution is often limited to 100 uA or worse.

**Equipment used for all measurements in this guide**:
- **Current measurement**: Keithley 6514 System Electrometer (10 fA resolution, 100 ms integration time)
- **Cross-validation**: 100 ohm precision shunt resistor (0.1%) + Keysight MSOX3104T oscilloscope for transient capture
- **Power supply**: Keithley 2400 SourceMeter providing exactly 3.300V
- **DUT**: ESP32-WROOM-32E module on custom breakout PCB (no USB bridge, no LEDs, no LDO -- direct power supply connection)
- **Environment**: 25 +/- 1 deg C, no airflow

<Callout type="warning" title="Measurement Accuracy">
  A standard DMM in the 200 uA range has a burden voltage of 1-10 mV, which is acceptable. But in the 2 mA range (which many people use), the burden voltage can be 100-500 mV, significantly disturbing the ESP32's power domain voltages and producing inaccurate readings. Always use the lowest current range that covers your expected measurement, and verify with a shunt resistor if possible.
</Callout>

## Stock Deep Sleep: Where Do 150 uA Go?

Let us start with the default configuration -- what you get when you call `esp_deep_sleep_start()` without any power domain configuration:

```cpp
// Stock deep sleep: no optimization
void enter_stock_deep_sleep() {
    esp_sleep_enable_timer_wakeup(60 * 1000000ULL);
    esp_deep_sleep_start();
}
```

**Measured result on bare ESP32-WROOM-32E module: 147.3 uA**

Where does this current go? The ESP32's internal power management unit keeps several domains alive by default to enable fast wakeup and maximum functionality:

The single largest contributor is **RTC fast memory retention at 100 uA** -- nearly 68% of the total. This is powered by default to enable wake stub execution (a short piece of code that runs immediately after wakeup, before the bootloader). Most applications never use wake stubs, but the fast memory remains powered regardless.

## Step-by-Step Optimization

### Step 1: Disable RTC Fast Memory (147 uA to 42 uA)

This is the highest-impact single change. By telling the power management unit to power down RTC fast memory during deep sleep, you eliminate approximately 105 uA:

```cpp
#include "esp_sleep.h"

void enter_optimized_deep_sleep_v1() {
    esp_sleep_enable_timer_wakeup(60 * 1000000ULL);

    // Disable RTC fast memory retention during deep sleep
    // Saves ~105 uA but prevents wake stub execution
    esp_sleep_pd_config(ESP_PD_DOMAIN_RTC_FAST_MEM, ESP_PD_OPTION_OFF);

    esp_deep_sleep_start();
}
```

**Measured result: 42.1 uA** (saved 105.2 uA)

<Callout type="warning" title="Wake Stub and ULP Compatibility">
  Powering down RTC fast memory has two consequences: (1) wake stubs cannot execute, meaning the chip must go through the full bootloader on every wake, adding approximately 50-100 ms to wake time; and (2) the ULP coprocessor cannot run programs stored in fast memory. If you use either feature, you cannot apply this optimization. For most sensor applications, neither is needed.
</Callout>

The ESP-IDF `esp_sleep_pd_config()` API is the correct way to control these bits. Avoid direct register manipulation unless you need finer control than the API provides, as the sleep entry code may override your register writes.

### Step 2: Disable RTC Peripherals (42 uA to 17 uA)

The RTC peripherals domain powers the touch sensor controller, SAR ADC, ULP coprocessor interface, and RTC GPIO functionality. If you only use timer wakeup (no GPIO wakeup, no touch wakeup, no ULP), this entire domain can be disabled:

```cpp
void enter_optimized_deep_sleep_v2() {
    esp_sleep_enable_timer_wakeup(60 * 1000000ULL);

    esp_sleep_pd_config(ESP_PD_DOMAIN_RTC_FAST_MEM, ESP_PD_OPTION_OFF);

    // Disable RTC peripherals -- saves ~25 uA
    // Disables: touch wakeup, ULP wakeup, ext1 wakeup (multi-GPIO)
    // Still works: timer wakeup, ext0 wakeup (single RTC GPIO)
    esp_sleep_pd_config(ESP_PD_DOMAIN_RTC_PERIPH, ESP_PD_OPTION_OFF);

    esp_deep_sleep_start();
}
```

**Measured result: 16.8 uA** (saved 25.3 uA)

<Callout type="info" title="ext0 vs ext1 Wakeup">
  Disabling RTC peripherals prevents `esp_sleep_enable_ext1_wakeup()` (multi-GPIO wakeup using the RTC controller) but still allows `esp_sleep_enable_ext0_wakeup()` (single RTC GPIO wakeup using the RTC IO module, which is part of the RTC core domain). If you need to wake on a single button press, ext0 wakeup still works with RTC peripherals disabled.
</Callout>

### Step 3: Disable RTC Slow Memory (17 uA to 13 uA)

If you do not need to persist any data across deep sleep cycles (no `RTC_DATA_ATTR` variables, no boot count, no WiFi cache), you can disable RTC slow memory as well:

```cpp
void enter_optimized_deep_sleep_v3() {
    esp_sleep_enable_timer_wakeup(60 * 1000000ULL);

    esp_sleep_pd_config(ESP_PD_DOMAIN_RTC_FAST_MEM, ESP_PD_OPTION_OFF);
    esp_sleep_pd_config(ESP_PD_DOMAIN_RTC_PERIPH, ESP_PD_OPTION_OFF);

    // Disable RTC slow memory -- saves ~2-4 uA
    // WARNING: ALL RTC_DATA_ATTR and RTC_NOINIT_ATTR variables are LOST
    // Every wake is effectively a cold boot with no persistent state
    esp_sleep_pd_config(ESP_PD_DOMAIN_RTC_SLOW_MEM, ESP_PD_OPTION_OFF);

    esp_deep_sleep_start();
}
```

**Measured result: 12.8 uA** (saved 4.0 uA)

<Callout type="danger" title="No State Survives">
  With RTC slow memory disabled, you lose the ability to cache WiFi credentials, count boots, or buffer sensor data across sleep cycles. Every wake is a cold boot. For many battery applications, keeping RTC slow memory powered (at 2-4 uA cost) is a net win because the WiFi cache alone saves orders of magnitude more energy per connection cycle.
</Callout>

### Step 4: GPIO Isolation (13 uA to 9 uA)

GPIO pins that are not explicitly configured before deep sleep can float to intermediate voltages, causing the CMOS input buffers to draw shoot-through current. Additionally, any pin connected to an external pull-up or pull-down resistor creates a resistive path to VCC or GND.

The key leakage sources from GPIOs:

The solution is to isolate every RTC GPIO before entering deep sleep. GPIO isolation disconnects the pin's internal logic from the external pad, eliminating both input buffer leakage and any resistive paths through internal pull resistors:

```cpp
#include "driver/rtc_io.h"

void isolate_all_gpios() {
    // Isolate every RTC-capable GPIO
    // RTC GPIOs on ESP32: 0, 2, 4, 12, 13, 14, 15, 25, 26, 27, 32, 33, 34, 35, 36, 37, 38, 39
    const gpio_num_t rtc_gpios[] = {
        GPIO_NUM_0,  GPIO_NUM_2,  GPIO_NUM_4,  GPIO_NUM_12,
        GPIO_NUM_13, GPIO_NUM_14, GPIO_NUM_15, GPIO_NUM_25,
        GPIO_NUM_26, GPIO_NUM_27, GPIO_NUM_32, GPIO_NUM_33,
        GPIO_NUM_34, GPIO_NUM_35, GPIO_NUM_36, GPIO_NUM_39
    };

    for (int i = 0; i < sizeof(rtc_gpios) / sizeof(rtc_gpios[0]); i++) {
        rtc_gpio_isolate(rtc_gpios[i]);
    }
}

// For non-RTC GPIOs, use hold to lock the pin state
void hold_all_digital_gpios() {
    for (int i = 0; i < GPIO_NUM_MAX; i++) {
        if (GPIO_IS_VALID_GPIO(i) && !rtc_gpio_is_valid_gpio((gpio_num_t)i)) {
            gpio_set_direction((gpio_num_t)i, GPIO_MODE_INPUT);
            gpio_set_pull_mode((gpio_num_t)i, GPIO_PULLDOWN_ONLY);
            gpio_hold_en((gpio_num_t)i);
        }
    }
    gpio_deep_sleep_hold_en();
}
```

**Measured result: 8.7 uA** (saved 4.1 uA from GPIO isolation)

<Callout type="warning" title="GPIO 12 Boot Mode">
  GPIO 12 (MTDI) controls the flash voltage regulator at boot. If you have an external pull-up on GPIO 12, it forces 1.8V flash voltage, which is incorrect for most ESP32-WROOM modules using 3.3V flash. This causes boot failure. Always ensure GPIO 12 is low at boot, and isolate it before deep sleep. If you must use GPIO 12 for wakeup, use `esp_sleep_enable_ext0_wakeup()` which handles the isolation correctly.
</Callout>

### Step 5: External 32.768 kHz Crystal (9 uA to 7 uA)

The ESP32 uses an internal 150 kHz RC oscillator as the default RTC clock source. This oscillator draws approximately 5 uA. If your board has an external 32.768 kHz crystal (connected to GPIO 32 and GPIO 33), switching the RTC clock to this crystal saves approximately 2-3 uA while also improving timekeeping accuracy from +/- 5% to +/- 20 ppm:

```cpp
// In ESP-IDF project configuration (sdkconfig):
// CONFIG_ESP32_RTC_CLK_SRC_EXT_CRYS=y
// CONFIG_ESP32_RTC_CLK_CAL_CYCLES=1024

// Or programmatically (not recommended -- do this via menuconfig):
#include "soc/rtc.h"

void configure_external_crystal() {
    rtc_clk_32k_enable(true);
    rtc_clk_32k_bootstrap(512);

    // Wait for crystal to stabilize
    vTaskDelay(pdMS_TO_TICKS(300));

    rtc_clk_slow_src_set(RTC_SLOW_FREQ_32K_XTAL);

    // Verify
    rtc_slow_freq_t current = rtc_clk_slow_src_get();
    if (current != RTC_SLOW_FREQ_32K_XTAL) {
        // Crystal failed to start -- fall back to internal RC
        rtc_clk_slow_src_set(RTC_SLOW_FREQ_RTC);
    }
}
```

## Board-Level Leakage: The Other Half of the Problem

The measurements above were taken on a bare module with direct power supply connection. On a real PCB with an LDO regulator, USB-UART bridge, LEDs, and external sensors, the board-level current can dwarf the ESP32's own sleep current.

<Callout type="danger" title="The AMS1117 Problem">
  The AMS1117-3.3 LDO used on virtually all ESP32 development boards has a quiescent current of 5 mA -- that is 5,000 uA. This is 500x higher than the ESP32's optimized deep sleep current. You cannot achieve reasonable battery life on a development board. For production battery devices, replace it with an MCP1700, HT7333, or TPS7A02.
</Callout>

For the ultimate low-power design, the Texas Instruments TPS7A02 achieves 25 nA quiescent current -- effectively zero compared to the ESP32's 7 uA sleep current. At $0.80 per unit, it pays for itself many times over in battery savings.

### Power Indicator LED

A typical power indicator LED with a 1k series resistor draws:

$I_{LED} = \frac{V_{CC} - V_F}{R} = \frac{3.3 - 2.0}{1000} = 1.3\ \text{mA}$

That is 1,300 uA -- 186x the ESP32's optimized sleep current. Even with a 100k resistor, the LED draws 13 uA (nearly doubling sleep current) and is too dim to see.

**Solution**: Remove power LEDs from battery designs. If you need a visual indicator, use a GPIO-controlled LED that is off during sleep.

For sensors with high standby current (INA219, OLED displays, some IMUs), the cleanest solution is a high-side MOSFET switch controlled by an RTC GPIO:

```cpp
#define SENSOR_POWER_PIN GPIO_NUM_25  // RTC GPIO

void power_gate_sensors() {
    // Use a P-channel MOSFET (e.g., SI2301) with gate on GPIO 25
    // GPIO HIGH = MOSFET off = sensors unpowered
    // GPIO LOW = MOSFET on = sensors powered

    rtc_gpio_init(SENSOR_POWER_PIN);
    rtc_gpio_set_direction(SENSOR_POWER_PIN, RTC_GPIO_MODE_OUTPUT_ONLY);
    rtc_gpio_set_level(SENSOR_POWER_PIN, 1);  // Turn off sensors
    rtc_gpio_hold_en(SENSOR_POWER_PIN);        // Hold state during sleep
}

void power_on_sensors() {
    rtc_gpio_hold_dis(SENSOR_POWER_PIN);
    rtc_gpio_set_level(SENSOR_POWER_PIN, 0);  // Turn on sensors
    vTaskDelay(pdMS_TO_TICKS(50));              // Wait for sensor startup
}
```

<Callout type="tip" title="I2C Bus Pull-ups During Sleep">
  I2C requires pull-up resistors on SDA and SCL. If the sensors are power-gated (VCC removed), the pull-ups create a path from 3.3V through the pull-up resistor and into the sensor's ESD protection diodes on SDA/SCL, causing leakage. Solution: either power the pull-ups from the same switched rail as the sensors, or use the ESP32's internal pull-ups (which are disconnected when the GPIO is isolated during sleep).
</Callout>

## Schematic-Level Optimizations

Beyond software configuration, the PCB design itself determines the minimum achievable sleep current.

### LDO Selection and Bypassing

For a battery-powered ESP32 design, the LDO must meet these criteria:
- Quiescent current below 5 uA (ideally below 1 uA)
- Dropout voltage below 200 mV (to maximize usable battery range)
- Output current capability of at least 500 mA (for WiFi TX peaks)
- Fast transient response (WiFi TX creates 0 to 200+ mA steps in microseconds)

Recommended LDO circuit:

```
Battery (3.7-4.2V LiPo)
    |
    +--- [100 uF MLCC] --- GND    (bulk input capacitor)
    |
    +--- VIN --- [MCP1700-3302E] --- VOUT --- ESP32 3V3
                     |                  |
                    GND                 +--- [10 uF MLCC] --- GND
                                        |
                                        +--- [100 nF MLCC] --- GND
```

The 100 uF input capacitor is critical for handling WiFi TX current spikes. Without it, the battery voltage can sag enough to trigger the ESP32's brownout detector, causing resets during transmission.

### EN Pin Pull-Down

The ESP32 EN (chip enable) pin has an internal pull-up. If left floating or connected to VCC through a high-impedance path, noise can cause intermittent resets. For reliable deep sleep, add an external pull-up resistor:

```
3V3 --- [10k] --- EN pin --- [0.1 uF] --- GND
                      |
                 [push button] --- GND  (optional reset button)
```

The 10k pull-up draws only $3.3V / 10k = 330$ uA during normal operation but 0 uA during deep sleep (the EN pin internal pull-up keeps it high; the 10k is an additional guarantee). The 0.1 uF capacitor provides noise filtering and a slight delay on power-up (time constant = 10k * 0.1 uF = 1 ms), ensuring clean startup.

<Callout type="tip" title="Using EN for Hard Power-Off">
  For the absolute minimum power consumption (below ESP32 deep sleep), use an external RTC chip (e.g., TPL5110 or DS3231 alarm) to pull the EN pin low, completely shutting down the ESP32. The TPL5110 draws 35 nA and can wake the ESP32 at configurable intervals from 100 ms to 2 hours. Total sleep current: ~35 nA plus LDO quiescent. This is the "cheat code" for sub-1 uA sleep.
</Callout>

### PCB Layout Considerations

At the microamp level, PCB leakage becomes relevant:

1. **Solder flux residue**: Conductive flux residue between adjacent pads can create resistive paths drawing microamps. Use no-clean flux and clean the board thoroughly with isopropyl alcohol after soldering.

2. **Board material**: Standard FR-4 has surface resistance of approximately 10^8 to 10^10 ohms between adjacent traces. At 3.3V with 10^8 ohms, this creates 33 nA leakage per trace pair -- negligible for most applications but relevant for sub-1 uA designs.

3. **Guard rings**: For the most sensitive analog inputs and high-impedance nodes, consider adding a ground guard ring to intercept surface leakage currents.

4. **Conformal coating**: In humid environments, surface moisture can reduce inter-trace resistance by orders of magnitude. Conformal coating prevents this.

## Common Pitfalls and Debugging

If your measured deep sleep current is higher than expected after applying all optimizations, check these items in order:

### Debugging Checklist

1. **Measure at the module, not the board**: Disconnect external components and measure the ESP32 module alone. If the module current is correct, the leakage is board-level.

2. **Check for PSRAM**: ESP32-WROVER modules with PSRAM will always be 20-40 uA higher than WROOM modules. There is no software workaround.

3. **Verify flash deep power-down**: Some flash chips do not support the deep power-down command, or the ESP-IDF configuration is not enabling it. Check `CONFIG_ESP_SLEEP_FLASH_LEAKAGE_WORKAROUND`.

4. **Inspect GPIO states**: Use an oscilloscope to verify that all GPIO pins are at a defined level (high or low) during sleep, not floating. Floating pins at intermediate voltages cause shoot-through current in CMOS buffers.

5. **Check for brownout detector**: The brownout detector draws a small amount of current. If your power supply is stable, you can disable it during sleep:

```cpp
#include "soc/rtc_cntl_reg.h"

void disable_brownout_in_sleep() {
    // Disable brownout detector during deep sleep
    // Only safe if power supply is stable
    WRITE_PERI_REG(RTC_CNTL_BROWN_OUT_REG, 0);
}
```

6. **Look for RTC timer calibration leakage**: The automatic RTC calibration feature keeps the 8 MHz oscillator running briefly after entering sleep. In ESP-IDF v5.x, this should be handled correctly, but older versions may have issues.

7. **Temperature effects**: Leakage current increases with temperature. At 85 deg C, expect 2-3x higher sleep current compared to 25 deg C measurements.

<Callout type="tip" title="Binary Search for Leakage">
  If you cannot identify the leakage source, use a binary search approach: remove half the external components, measure, and repeat. If current drops, the leakage is in the removed half. If not, it is in the remaining half. This quickly isolates board-level issues.
</Callout>

## Aggressive Optimization: Reaching 5 uA

To push below 7 uA requires sacrificing all RTC state and using the most aggressive configuration:

```cpp
void enter_minimum_power_sleep(uint64_t duration_us) {
    // Timer wakeup only
    esp_sleep_enable_timer_wakeup(duration_us);

    // Disable ALL optional power domains
    esp_sleep_pd_config(ESP_PD_DOMAIN_RTC_FAST_MEM, ESP_PD_OPTION_OFF);
    esp_sleep_pd_config(ESP_PD_DOMAIN_RTC_SLOW_MEM, ESP_PD_OPTION_OFF);
    esp_sleep_pd_config(ESP_PD_DOMAIN_RTC_PERIPH, ESP_PD_OPTION_OFF);

    // Isolate all GPIOs
    const gpio_num_t rtc_gpios[] = {
        GPIO_NUM_0, GPIO_NUM_2, GPIO_NUM_4, GPIO_NUM_12,
        GPIO_NUM_13, GPIO_NUM_14, GPIO_NUM_15, GPIO_NUM_25,
        GPIO_NUM_26, GPIO_NUM_27, GPIO_NUM_32, GPIO_NUM_33,
        GPIO_NUM_34, GPIO_NUM_35, GPIO_NUM_36, GPIO_NUM_39
    };
    for (int i = 0; i < sizeof(rtc_gpios) / sizeof(rtc_gpios[0]); i++) {
        rtc_gpio_isolate(rtc_gpios[i]);
    }

    // Disable WiFi and BT completely
    esp_wifi_stop();
    esp_bt_controller_disable();

    // Enter deep sleep (effectively hibernation)
    esp_deep_sleep_start();
}
```

With a custom board using TPS7A02 LDO, external 32k crystal, flash deep power-down enabled, and no external sensors:

**Measured result: 4.8 uA**

This is the practical minimum for the ESP32-WROOM-32E. The irreducible ~5 uA floor comes from the RTC core domain (timer, PMU, power-on reset circuitry) which cannot be disabled.

## Conclusion

Achieving sub-10 uA deep sleep on the ESP32 requires attention at every level:

1. **Software**: Disable RTC fast memory (saves 100+ uA), disable unused RTC peripherals (saves 25 uA), isolate GPIOs (saves 5-100+ uA depending on external circuitry).

2. **Hardware**: Use a low-quiescent-current LDO (MCP1700 at 1.6 uA, not AMS1117 at 5 mA), remove USB-UART bridge and power LEDs from production boards, power-gate external sensors.

3. **PCB**: Clean flux residue, use conformal coating in humid environments, add proper decoupling capacitors.

The stock ESP32 development board at 20 mA deep sleep current and a properly optimized production board at 5 uA represent a 4,000x difference. Every one of those microamps is traceable to a specific source, and every source has a specific remedy. The path from 150 uA to 5 uA is not magic -- it is systematic engineering.
