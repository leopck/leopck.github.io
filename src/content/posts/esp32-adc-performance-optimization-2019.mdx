The ESP32's ADC is versatile but has well-documented performance limitations -- notably nonlinearity at the extremes and significant noise floor. Understanding where the real bottlenecks are (and aren't) is essential for high-performance sensor applications.

<Callout type="warning" title="ADC2 and WiFi">
  ADC2 shares hardware resources with the WiFi radio. When WiFi is active, ADC2 reads will fail silently or return garbage values. For applications requiring both WiFi and analog sensing, use ADC1 exclusively.

The 2.4x gap between software polling (85 KS/s) and DMA (200 KS/s) comes from function call overhead, context switching, and the lack of hardware double-buffering in the polling path. For any application needing more than ~50 KS/s, DMA is mandatory.

### DMA Setup

The critical code for achieving maximum throughput:

```cpp
#include "driver/adc.h"

#define ADC_SAMPLES 4096
#define ADC_SAMPLE_RATE 200000  // 200 KS/s target

void configure_adc_dma() {
    adc1_config_width(ADC_WIDTH_BIT_12);
    adc1_config_channel_atten(ADC1_CHANNEL_0, ADC_ATTEN_DB_11);

    adc_dma_config_t dma_config = {
        .dma_chan = ADC_DMA_CHANNEL,
        .dma_buf_size = ADC_SAMPLES,
        .dma_buf_count = 2,  // Double buffering -- process one while filling other
    };
    adc_dma_config(&dma_config);
    adc_dma_start();
}
```

Double buffering is key: while the DMA controller fills one buffer, your processing code works on the other. This eliminates dead time between batches.

The effective number of bits (ENOB) is always lower than the configured resolution due to the ESP32's relatively high noise floor. At 12-bit configured resolution, you get approximately 10.5 effective bits -- meaning the bottom 1.5 bits are noise. This is important: switching from 12-bit to 10-bit mode loses almost no actual information for most applications while gaining 20% more sampling speed.

### Software Averaging

The most effective noise reduction technique is oversampling with averaging. Taking N samples and averaging reduces noise by sqrtN:

```cpp
uint32_t adc_read_averaged(adc1_channel_t channel, int num_samples) {
    uint32_t sum = 0;
    for (int i = 0; i < num_samples; i++) {
        sum += adc1_get_raw(channel);
    }
    return sum / num_samples;
}

// 16x oversampling: noise reduced by 4x, ENOB improves by ~2 bits
// Effective rate: 85 KS/s / 16 = ~5.3 KS/s -- still fast enough for most sensors
```

<Callout type="tip" title="The 16x Sweet Spot">
  16x oversampling (averaging 16 samples) improves ENOB from ~10.5 to ~12.5 effective bits while maintaining ~5 KS/s throughput. For most sensor applications (temperature, pressure, light), 5 KS/s is more than sufficient. Beyond 16x, the law of diminishing returns applies -- you need 64x to gain one more bit.
</Callout>

### Calibration

Espressif provides factory-programmed calibration data that corrects the ADC's nonlinearity:

```cpp
#include "esp_adc_cal.h"

esp_adc_cal_characteristics_t adc_chars;

void calibrate_adc() {
    // Check if eFuse calibration data is available
    esp_adc_cal_value_t val_type = esp_adc_cal_characterize(
        ADC_UNIT_1, ADC_ATTEN_DB_11, ADC_WIDTH_BIT_12,
        1100,  // Default VREF (mV)
        &adc_chars
    );

    if (val_type == ESP_ADC_CAL_VAL_EFUSE_TP) {
        printf("Calibrated with Two Point eFuse data\n");
    } else if (val_type == ESP_ADC_CAL_VAL_EFUSE_VREF) {
        printf("Calibrated with VREF eFuse data\n");
    }
}

uint32_t read_calibrated_mv(adc1_channel_t channel) {
    int raw = adc1_get_raw(channel);
    return esp_adc_cal_raw_to_voltage(raw, &adc_chars);
}
```

Calibration improves accuracy by 5-10x in the nonlinear regions (below 100mV and above 3.1V). Always use it.

## Conclusion

The ESP32 ADC is capable of ~12.5 effective bits at 5 KS/s with software optimization (calibration + 16x oversampling), or ~10.5 effective bits at 200 KS/s in DMA mode. The key decisions are: use ADC1 if WiFi is active, use DMA for anything above 50 KS/s, always enable eFuse calibration, and apply oversampling appropriate to your throughput requirements.
