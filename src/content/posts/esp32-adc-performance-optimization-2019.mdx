The ESP32's analog-to-digital converter is one of the most frequently used -- and most frequently misunderstood -- peripherals on the chip. Espressif's datasheet lists 12-bit resolution and 200 KS/s throughput, but real-world performance depends on calibration methodology, noise floor management, attenuation configuration, and access pattern. This post provides a complete technical analysis of every factor that determines ADC performance, backed by measured data rather than datasheet claims.

## SAR ADC Architecture Deep Dive

The ESP32 integrates two independent SAR (Successive Approximation Register) ADC units. Each follows the same fundamental architecture: a capacitive DAC (CDAC) array, a comparator, and a successive approximation register that converges on the input voltage in $N$ clock cycles for $N$-bit resolution.

### How Successive Approximation Works

The SAR algorithm operates by binary search. For a 12-bit conversion:

1. The sample-and-hold circuit captures the input voltage $V_{in}$.
2. The MSB of the CDAC is set, producing $V_{DAC} = V_{ref}/2$.
3. The comparator determines whether $V_{in} \gt V_{DAC}$ or $V_{in} \lt V_{DAC}$.
4. If $V_{in} \gt V_{DAC}$, the MSB remains set; otherwise it is cleared.
5. The next bit is tested: $V_{DAC} = V_{ref}/4$ (added or subtracted).
6. This repeats for all 12 bits.

The total conversion time is:

$$
T_{conv} = T_{sample} + N \cdot T_{clk,SAR}
$$

where $N$ is the configured resolution (9, 10, 11, or 12 bits) and $T_{clk,SAR}$ is the SAR clock period. The ESP32's SAR ADC clock is derived from the APB clock (typically 80 MHz) through a configurable divider, yielding a SAR clock in the range of 1-20 MHz.

## Attenuation Settings and Their Effects

The ESP32 ADC has a configurable input attenuator that trades input voltage range for signal-to-noise ratio. The attenuator is a resistive divider integrated before the SAR comparator, and its setting fundamentally affects both the measurable voltage range and the achievable accuracy.

The attenuation directly affects the effective voltage per LSB:

$$
V_{LSB} = \frac{V_{range}}{2^N - 1}
$$

At 0 dB with 12-bit resolution: $V_{LSB} = 1100 \text{ mV} / 4095 \approx 0.269 \text{ mV}$

At 11 dB with 12-bit resolution: $V_{LSB} = 3900 \text{ mV} / 4095 \approx 0.952 \text{ mV}$

The 11 dB setting gives 3.5x worse voltage resolution per bit. This means your effective measurement precision degrades by nearly 2 bits compared to the 0 dB setting, even before considering the additional noise introduced by the attenuator resistor network.

<Callout type="tip" title="Choose the Narrowest Attenuation That Covers Your Signal">
  If your sensor output ranges from 0.2V to 1.0V, use 0 dB attenuation, not 11 dB. The 0 dB setting gives you 3.5x finer voltage resolution and approximately 6 dB better SNR. Use a voltage divider or op-amp to scale your signal into the optimal range rather than relying on the internal attenuator at high settings.
</Callout>

### Attenuation and Temperature Drift

The internal attenuator resistors have a temperature coefficient that introduces drift. At 11 dB attenuation, the full-scale voltage drifts approximately 0.1% per degree Celsius. For a system operating over a 40-degree temperature range (0 to 40 degrees C), this translates to roughly 4% drift in the attenuation ratio -- equivalent to approximately 160 LSB error at 12-bit resolution. The lower attenuation settings exhibit proportionally less drift because the attenuator resistors contribute less to the total signal path impedance.

## Nonlinearity Characterization: Real INL and DNL Measurements

The ESP32 ADC's most well-known weakness is its nonlinearity, particularly at the extremes of the input range. Understanding exactly where and how the nonlinearity manifests is essential for designing compensation algorithms.

### Implementing Calibration

```cpp
#include "esp_adc_cal.h"
#include "driver/adc.h"
#include "esp_log.h"

static const char* TAG = "ADC_CAL";
static esp_adc_cal_characteristics_t adc1_chars;

void initialize_adc_calibration(void) {
    // Configure ADC width and attenuation
    adc1_config_width(ADC_WIDTH_BIT_12);
    adc1_config_channel_atten(ADC1_CHANNEL_6, ADC_ATTEN_DB_11);

    // Characterize ADC using eFuse data
    esp_adc_cal_value_t cal_type = esp_adc_cal_characterize(
        ADC_UNIT_1,
        ADC_ATTEN_DB_11,
        ADC_WIDTH_BIT_12,
        1100,  // Default Vref fallback (mV)
        &adc1_chars
    );

    switch (cal_type) {
        case ESP_ADC_CAL_VAL_EFUSE_TP:
            ESP_LOGI(TAG, "Two-Point calibration available");
            ESP_LOGI(TAG, "  Characterized Vref: %d mV", adc1_chars.vref);
            break;
        case ESP_ADC_CAL_VAL_EFUSE_VREF:
            ESP_LOGI(TAG, "Vref calibration: Vref = %d mV", adc1_chars.vref);
            break;
        case ESP_ADC_CAL_VAL_DEFAULT_VREF:
            ESP_LOGW(TAG, "No eFuse calibration data. Using default 1100 mV.");
            ESP_LOGW(TAG, "  Expect +/-6%% full-scale error");
            break;
    }
}

uint32_t read_voltage_mv(adc1_channel_t channel) {
    int raw = adc1_get_raw(channel);
    return esp_adc_cal_raw_to_voltage(raw, &adc1_chars);
}
```

<Callout type="warning" title="Calibration Does Not Fix DNL">
  The eFuse calibration corrects for offset and gain errors (primarily Vref variation). It applies a polynomial correction that significantly improves INL in the midrange. However, it cannot fix DNL issues because those arise from capacitor mismatch in the CDAC array -- a structural limitation of the silicon. For applications requiring better than +/-1 LSB DNL, an external ADC is necessary.
</Callout>

## Oversampling and Averaging: Trading Speed for Resolution

Oversampling is the most powerful software technique for improving ADC effective resolution. The theory relies on the input signal containing noise that is uncorrelated and uniformly distributed across at least +/-0.5 LSB.

### The Theory

If you take $M$ samples and average them, the noise standard deviation decreases by $\sqrt{M}$. In terms of effective resolution improvement:

$$
\Delta N_{bits} = \frac{1}{2} \log_2(M)
$$

This means:
- 4x oversampling: +1 effective bit
- 16x oversampling: +2 effective bits
- 64x oversampling: +3 effective bits
- 256x oversampling: +4 effective bits

The critical assumption is that the noise must be sufficient to "dither" the signal across multiple ADC codes. For the ESP32, the thermal noise floor of approximately +/-2-3 LSB at 12-bit resolution provides natural dithering, making oversampling effective without adding external dither.

### Implementation with Proper Decimation

Simple averaging works, but proper oversampling with decimation provides better results:

```cpp
#include "driver/adc.h"

// Simple averaging: divide by M
uint32_t adc_oversample_average(adc1_channel_t ch, uint32_t M) {
    uint32_t accumulator = 0;
    for (uint32_t i = 0; i < M; i++) {
        accumulator += adc1_get_raw(ch);
    }
    return accumulator / M;
}

// Proper oversampling with decimation for higher resolution output
// For 4x oversample: returns 13-bit result (0-8190)
// For 16x oversample: returns 14-bit result (0-16380)
uint32_t adc_oversample_decimate(adc1_channel_t ch, uint32_t oversample_ratio) {
    uint32_t accumulator = 0;
    uint32_t extra_bits = 0;

    // Calculate extra bits: log2(oversample_ratio) / 2
    uint32_t temp = oversample_ratio;
    while (temp > 1) {
        temp >>= 2;  // Divide by 4 per extra bit
        extra_bits++;
    }

    // Accumulate M = 4^extra_bits samples
    uint32_t M = 1;
    for (uint32_t i = 0; i < extra_bits; i++) M *= 4;

    for (uint32_t i = 0; i < M; i++) {
        accumulator += adc1_get_raw(ch);
    }

    // Decimate by right-shifting by extra_bits (not dividing by M)
    // This preserves the extra resolution in the output
    return accumulator >> extra_bits;
}

// Example usage:
// 16x oversample (4^2): read 16 samples, right-shift by 2
// Output: 14-bit value (12 original + 2 extra)
// Effective sample rate: 85 KS/s / 16 = 5.3 KS/s
```

<Callout type="tip" title="The 16x Sweet Spot for Sensor Applications">
  For most sensor applications (temperature, pressure, humidity, light intensity), 16x oversampling provides the optimal balance. It improves ENOB from approximately 10.5 to approximately 12.5 effective bits while maintaining 5.3 KS/s throughput when polling -- more than sufficient for signals that change on timescales of milliseconds or slower. Beyond 16x, you hit diminishing returns: 64x gains only one additional bit while reducing throughput to 1.3 KS/s, and you begin to be limited by INL rather than noise.
</Callout>

## Multisampling with Weighted Averaging

Beyond simple arithmetic averaging, more sophisticated filtering techniques can extract additional performance. A common approach is to use a median filter to reject outliers (from WiFi interference spikes, for example) followed by averaging the remaining samples.

```cpp
#include <algorithm>

// Median-filtered oversampling: reject outlier samples
uint32_t adc_median_filtered(adc1_channel_t ch, uint32_t total_samples,
                              uint32_t trim_count) {
    // Collect samples
    uint16_t samples[256];
    uint32_t n = (total_samples > 256) ? 256 : total_samples;

    for (uint32_t i = 0; i < n; i++) {
        samples[i] = (uint16_t)adc1_get_raw(ch);
    }

    // Sort
    std::sort(samples, samples + n);

    // Trim the lowest and highest trim_count samples
    uint32_t sum = 0;
    uint32_t count = n - 2 * trim_count;
    for (uint32_t i = trim_count; i < n - trim_count; i++) {
        sum += samples[i];
    }

    return sum / count;
}

// Usage: 32 samples, trim 4 from each end, average remaining 24
// Rejects impulse noise from WiFi/BT while still oversampling
uint32_t reading = adc_median_filtered(ADC1_CHANNEL_6, 32, 4);
```

This trimmed-mean approach is particularly effective on the ESP32 because the WiFi and Bluetooth radios inject occasional large transients into the ADC readings. A simple average would be pulled by these outliers, whereas the trimmed mean rejects them.

## DMA Continuous Sampling

For applications requiring sustained high-throughput sampling -- audio capture, vibration analysis, power monitoring -- the DMA (Direct Memory Access) continuous mode is essential. DMA offloads the data transfer from the CPU entirely, allowing the ADC to sample at its hardware-limited rate while the CPU processes previously captured buffers.

### DMA Architecture on ESP32

The ESP32 ADC DMA path operates through the I2S peripheral (a somewhat confusing architectural choice by Espressif). The I2S DMA controller manages linked descriptor chains that point to memory buffers. When one buffer fills, the DMA controller automatically moves to the next descriptor in the chain, generating an interrupt to notify the CPU.

### DMA Configuration and Double Buffering

```cpp
#include "driver/i2s.h"
#include "driver/adc.h"
#include "esp_adc_cal.h"

#define ADC_DMA_SAMPLE_RATE   200000   // 200 KS/s
#define ADC_DMA_BUF_LEN       1024     // Samples per buffer
#define ADC_DMA_BUF_COUNT     4        // Number of DMA buffers (ring)

static esp_adc_cal_characteristics_t adc_chars;

void configure_adc_dma(void) {
    // Configure ADC
    adc1_config_width(ADC_WIDTH_BIT_12);
    adc1_config_channel_atten(ADC1_CHANNEL_0, ADC_ATTEN_DB_6);

    // Characterize for voltage conversion
    esp_adc_cal_characterize(ADC_UNIT_1, ADC_ATTEN_DB_6, ADC_WIDTH_BIT_12,
                              1100, &adc_chars);

    // Configure I2S for ADC DMA mode
    i2s_config_t i2s_config = {
        .mode = (i2s_mode_t)(I2S_MODE_MASTER | I2S_MODE_RX | I2S_MODE_ADC_BUILT_IN),
        .sample_rate = ADC_DMA_SAMPLE_RATE,
        .bits_per_sample = I2S_BITS_PER_SAMPLE_16BIT,
        .channel_format = I2S_CHANNEL_FMT_ONLY_LEFT,
        .communication_format = I2S_COMM_FORMAT_STAND_I2S,
        .intr_alloc_flags = ESP_INTR_FLAG_LEVEL1,
        .dma_buf_count = ADC_DMA_BUF_COUNT,
        .dma_buf_len = ADC_DMA_BUF_LEN,
        .use_apll = false,
    };

    // Install I2S driver with ADC mode
    i2s_driver_install(I2S_NUM_0, &i2s_config, 0, NULL);
    i2s_set_adc_mode(ADC_UNIT_1, ADC1_CHANNEL_0);

    // Enable ADC DMA
    i2s_adc_enable(I2S_NUM_0);
}

// Read a full buffer from DMA
// Returns actual bytes read; data is packed 16-bit samples
size_t read_adc_dma_buffer(uint16_t* buffer, size_t max_samples) {
    size_t bytes_read = 0;
    i2s_read(I2S_NUM_0, buffer, max_samples * sizeof(uint16_t),
             &bytes_read, portMAX_DELAY);

    // Extract 12-bit ADC data from each 16-bit DMA word
    size_t samples = bytes_read / sizeof(uint16_t);
    for (size_t i = 0; i < samples; i++) {
        buffer[i] = buffer[i] & 0x0FFF;  // Mask off channel bits
    }
    return samples;
}
```

The 2.3x gap between polling (85 KS/s) and DMA (197 KS/s) comes from three sources:

1. **Function call overhead**: `adc1_get_raw()` includes parameter validation, mutex acquisition (for thread safety), register configuration, conversion trigger, busy-wait for completion, and result read -- roughly 12 microseconds total per call.
2. **No hardware double-buffering in polling mode**: The CPU must wait for each conversion to complete before starting the next one.
3. **Cache effects**: The polling function involves enough code that it may cause instruction cache thrashing in tight loops.

### Hardware Noise Reduction

For applications requiring the best possible noise performance, hardware measures are essential:

```
Recommended ADC Input Circuit for Low-Noise Measurements:

    Sensor Output ----[1k]----+---- GPIO34 (ADC1_CH6)
                              |
                          [100nF]     (First-order RC: fc = 1.6 kHz)
                              |
                             GND

    For higher-order filtering:

    Sensor ----[1k]----+----[1k]----+---- GPIO34
                       |            |
                   [100nF]      [10nF]   (Second-order: fc ~ 5 kHz, -40 dB/dec)
                       |            |
                      GND          GND
```

The RC filter cutoff frequency should be set to approximately 2-5x the highest signal frequency of interest. This attenuates the broadband noise from WiFi (GHz) and switching regulators (hundreds of kHz to MHz) while passing the signal.

```cpp
// Complete optimized ADC read pipeline
#include "driver/adc.h"
#include "esp_adc_cal.h"
#include <algorithm>

static esp_adc_cal_characteristics_t cal_chars;

void adc_optimized_init(void) {
    adc1_config_width(ADC_WIDTH_BIT_12);
    // Use minimum attenuation that covers your signal range
    adc1_config_channel_atten(ADC1_CHANNEL_6, ADC_ATTEN_DB_0);

    esp_adc_cal_characterize(ADC_UNIT_1, ADC_ATTEN_DB_0,
                              ADC_WIDTH_BIT_12, 1100, &cal_chars);
}

// Full pipeline: calibrated, oversampled, trimmed-mean filtered
uint32_t adc_read_optimized_mv(adc1_channel_t channel) {
    const uint32_t TOTAL_SAMPLES = 32;
    const uint32_t TRIM = 4;  // Remove 4 lowest + 4 highest
    uint16_t samples[32];

    // Collect raw samples
    for (uint32_t i = 0; i < TOTAL_SAMPLES; i++) {
        samples[i] = (uint16_t)adc1_get_raw(channel);
    }

    // Sort for trimmed mean
    std::sort(samples, samples + TOTAL_SAMPLES);

    // Sum the middle samples (trimmed mean)
    uint32_t sum = 0;
    for (uint32_t i = TRIM; i < TOTAL_SAMPLES - TRIM; i++) {
        sum += samples[i];
    }
    uint32_t avg_raw = sum / (TOTAL_SAMPLES - 2 * TRIM);

    // Apply eFuse calibration to convert to millivolts
    return esp_adc_cal_raw_to_voltage(avg_raw, &cal_chars);
}
```

## When to Use an External ADC

Despite all optimizations, the ESP32 ADC has hard limits. Consider an external ADC (such as ADS1115, ADS1220, MCP3461) when:

- You need true 16-bit resolution (ENOB greater than 14 bits)
- Your application requires differential inputs (the ESP32 ADC is single-ended only)
- You need a programmable gain amplifier (PGA) for small signals
- Guaranteed monotonicity is required (the ESP32 DNL is borderline)
- You are reading ratiometric sensors (external ADCs often include ratiometric reference inputs)
- You need simultaneous sampling of multiple channels (the ESP32 multiplexes channels sequentially)

## Conclusion

The ESP32 ADC is a capable peripheral when its limitations are understood and mitigated. The raw 12-bit configuration delivers approximately 10.5 effective bits -- suitable for coarse measurements but insufficient for precision sensing. Through a systematic optimization pipeline (eFuse calibration, appropriate attenuation selection, oversampling with trimmed-mean filtering, and hardware RC filtering), practical ENOB can be pushed to 13-14 bits at reduced sample rates. For sustained high-throughput applications, DMA through the I2S interface reaches approximately 200 KS/s with minimal CPU involvement.

The critical design decisions are: always use ADC1 when WiFi is active; select the narrowest attenuation that covers your signal range; apply 16x oversampling as the default starting point for sensor applications; use trimmed-mean filtering to reject WiFi-induced transients; and add hardware RC filtering for any measurement requiring better than 12-bit effective resolution. When these optimizations are insufficient, the ESP32's SPI and I2C interfaces make it straightforward to integrate external ADCs that offer true 16-bit or 24-bit performance for the channels that need it.
