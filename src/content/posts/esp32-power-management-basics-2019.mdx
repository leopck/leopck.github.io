Every milliamp matters when your ESP32 runs on a coin cell or a small LiPo. The difference between a device that lasts two weeks and one that lasts two years comes down to understanding what the chip is doing with its power budget at every moment, and controlling it precisely. This guide covers the ESP32 power architecture from silicon to software, with real measurements taken on production hardware, practical code for every sleep mode, and the calculations you need to size your battery correctly.

## The ESP32 Power Domain Architecture

The ESP32 is not a monolithic power consumer. It is organized into multiple independently controllable power domains, each serving a distinct set of peripherals and memory regions. Understanding these domains is the foundation for every optimization that follows.

The digital core domain is the largest consumer. It powers both Xtensa LX6 CPU cores (running at up to 240 MHz), the instruction and data caches, all digital peripherals (SPI, I2C, UART, timers), and the entire 520 KB of main SRAM. When you enter any sleep mode, this is the first domain that gets shut down or clock-gated.

The WiFi and Bluetooth radio domain is the second major consumer. The 2.4 GHz radio draws approximately 100-180 mA during transmission (depending on output power level) and 80-100 mA during reception. Even in idle-connected state, the radio consumes roughly 50 mA to maintain beacon tracking and association.

The RTC domains survive deep sleep. The RTC core domain cannot be turned off -- it houses the wakeup logic and the power management unit itself. But the RTC fast memory, RTC slow memory, and RTC peripherals domains can each be independently powered down to reduce deep sleep current.

### Active Mode

Active mode is where the ESP32 performs useful work. Current consumption varies enormously depending on the CPU frequency, which peripherals are enabled, and especially whether the radio is active.

The key insight is that CPU frequency scaling alone provides roughly a 2x current reduction (240 MHz to 80 MHz), but disabling the radio provides another 2-3x reduction on top of that. For compute-bound tasks that do not require connectivity, dropping to 80 MHz with the radio off is the most effective active-mode optimization.

```cpp
#include "esp_pm.h"
#include "esp_wifi.h"

// Configure dynamic frequency scaling (DFS)
// The CPU will automatically clock down when idle
void configure_power_management() {
    esp_pm_config_esp32_t pm_config = {
        .max_freq_mhz = 240,   // Max when active
        .min_freq_mhz = 80,    // Min when idle (saves ~40 mA)
        .light_sleep_enable = false  // Set true for automatic light sleep
    };
    ESP_ERROR_CHECK(esp_pm_configure(&pm_config));
}

// Manual frequency control for known workloads
void run_sensor_reading() {
    // Drop to 80 MHz for sensor I2C communication
    setCpuFrequencyMhz(80);
    float temp = read_temperature_sensor();
    float humidity = read_humidity_sensor();

    // Boost to 240 MHz for data compression
    setCpuFrequencyMhz(240);
    compress_and_encode(temp, humidity);

    // Back to 80 MHz while waiting
    setCpuFrequencyMhz(80);
}
```

<Callout type="tip" title="Dynamic Frequency Scaling">
  ESP-IDF supports automatic dynamic frequency scaling (DFS) through the power management framework. When enabled, the CPU frequency drops to `min_freq_mhz` whenever no locks are held, and boosts to `max_freq_mhz` when a peripheral or task acquires a power management lock. This is the recommended approach for applications with mixed workloads.
</Callout>

### Modem Sleep

Modem sleep is the first power-saving mode and is specifically designed for WiFi-connected applications. The CPU continues running at full speed, but the WiFi radio is clock-gated between DTIM beacon intervals. The radio wakes briefly to check for buffered data at the access point, then returns to sleep.

```cpp
#include "esp_wifi.h"

void configure_modem_sleep() {
    // Connect to WiFi first
    wifi_init_sta();

    // Enable modem sleep -- radio sleeps between DTIM intervals
    esp_wifi_set_ps(WIFI_PS_MIN_MODEM);

    // WIFI_PS_MIN_MODEM: wake at every DTIM beacon (~100 ms)
    //   Current: ~20 mA average
    //   Latency: depends on DTIM interval (typically 100-300 ms)

    // WIFI_PS_MAX_MODEM: wake at listen interval (configurable)
    //   Current: ~15 mA average
    //   Latency: higher, depends on listen interval setting
    // esp_wifi_set_ps(WIFI_PS_MAX_MODEM);
}
```

The distinction between `WIFI_PS_MIN_MODEM` and `WIFI_PS_MAX_MODEM` is important. With `MIN_MODEM`, the radio wakes at every DTIM beacon (every 1-3 beacon intervals, typically 100-300 ms). With `MAX_MODEM`, the radio can skip multiple DTIM intervals using the listen interval, waking less frequently but potentially missing time-sensitive data.

### Light Sleep

Light sleep suspends both CPU cores and clock-gates most digital peripherals, but retains the full 520 KB of SRAM contents. Execution resumes from the exact instruction where it was suspended, with all variables, stack frames, and peripheral registers intact. From the application's perspective, it looks like a brief pause.

```cpp
#include "esp_sleep.h"
#include "esp_wifi.h"
#include "esp_pm.h"

// Method 1: Explicit light sleep with timer wakeup
void enter_light_sleep_timed(uint64_t duration_us) {
    esp_sleep_enable_timer_wakeup(duration_us);
    esp_light_sleep_start();
    // Execution resumes here immediately after wakeup
    // All RAM, registers, and peripherals preserved
}

// Method 2: Explicit light sleep with GPIO wakeup
void enter_light_sleep_gpio() {
    // Wake on GPIO 0 going LOW (e.g., button press)
    esp_sleep_enable_gpio_wakeup();
    gpio_wakeup_enable(GPIO_NUM_0, GPIO_INTR_LOW_LEVEL);

    esp_light_sleep_start();

    esp_sleep_wakeup_cause_t cause = esp_sleep_get_wakeup_cause();
    if (cause == ESP_SLEEP_WAKEUP_GPIO) {
        // Handle button press
    }
}

// Method 3: Automatic light sleep (recommended for WiFi applications)
void configure_auto_light_sleep() {
    // Enable WiFi modem sleep first
    esp_wifi_set_ps(WIFI_PS_MIN_MODEM);

    // Configure power management with light sleep enabled
    esp_pm_config_esp32_t pm_config = {
        .max_freq_mhz = 240,
        .min_freq_mhz = 80,
        .light_sleep_enable = true  // CPU enters light sleep when idle
    };
    esp_pm_configure(&pm_config);

    // Now the CPU automatically enters light sleep whenever
    // all tasks are blocked and no PM locks are held.
    // WiFi wakes the CPU when data arrives.
}
```

<Callout type="warning" title="Light Sleep and Peripherals">
  UART reception is lost during light sleep because the UART peripheral is clock-gated. If you need to receive serial data, either use the UART wakeup feature (wake on start bit detection) or avoid light sleep during reception windows. SPI and I2C transactions in progress will also be interrupted.
</Callout>

The automatic light sleep mode (method 3 above) is particularly powerful for WiFi-connected applications. The CPU automatically enters light sleep whenever the RTOS has no ready tasks, and the WiFi stack wakes the CPU when a beacon or data frame arrives. This can reduce average current from ~20 mA (modem sleep only) to approximately 1-3 mA, depending on network traffic.

### Deep Sleep

Deep sleep is the workhorse mode for battery-powered sensors. The entire digital core is powered off -- both CPUs, all main SRAM, and all digital peripherals are gone. Only the RTC domains survive. When the chip wakes from deep sleep, it goes through a full reset sequence, re-executing the bootloader and `app_main()` from scratch.

```cpp
#include "esp_sleep.h"
#include "driver/rtc_io.h"

// Data that persists across deep sleep cycles
RTC_DATA_ATTR static int boot_count = 0;
RTC_DATA_ATTR static float accumulated_temperature = 0.0f;
RTC_DATA_ATTR static uint8_t wifi_channel = 0;
RTC_DATA_ATTR static uint8_t wifi_bssid[6] = {0};

void app_main() {
    boot_count++;

    // Determine wake cause
    esp_sleep_wakeup_cause_t cause = esp_sleep_get_wakeup_cause();

    switch (cause) {
        case ESP_SLEEP_WAKEUP_TIMER:
            printf("Timer wakeup, boot #%d\n", boot_count);
            break;
        case ESP_SLEEP_WAKEUP_EXT0:
            printf("External wakeup (GPIO), boot #%d\n", boot_count);
            break;
        default:
            printf("Initial power-on boot\n");
            boot_count = 1;
            break;
    }

    // Do work: read sensor, accumulate
    float temp = read_temperature();
    accumulated_temperature += temp;

    // Every 10 readings, transmit
    if (boot_count % 10 == 0) {
        float avg = accumulated_temperature / 10.0f;
        wifi_transmit(avg);
        accumulated_temperature = 0.0f;
    }

    // Configure wakeup sources
    // Timer: wake every 60 seconds
    esp_sleep_enable_timer_wakeup(60 * 1000000ULL);

    // GPIO: wake on button press (GPIO 0, active low)
    esp_sleep_enable_ext0_wakeup(GPIO_NUM_0, 0);

    // Enter deep sleep
    esp_deep_sleep_start();
    // Execution never reaches this point
}
```

```cpp
#include "esp_sleep.h"

void enter_optimized_deep_sleep(uint64_t sleep_us) {
    // Configure wakeup
    esp_sleep_enable_timer_wakeup(sleep_us);

    // Disable RTC fast memory (saves ~100 uA)
    // Prevents ULP execution and wake stub usage
    esp_sleep_pd_config(ESP_PD_DOMAIN_RTC_FAST_MEM, ESP_PD_OPTION_OFF);

    // Optionally disable RTC slow memory (saves ~15-20 uA)
    // WARNING: all RTC_DATA_ATTR variables will be lost
    // esp_sleep_pd_config(ESP_PD_DOMAIN_RTC_SLOW_MEM, ESP_PD_OPTION_OFF);

    // Disable RTC peripherals if not using GPIO wakeup or ULP
    // esp_sleep_pd_config(ESP_PD_DOMAIN_RTC_PERIPH, ESP_PD_OPTION_OFF);

    // Isolate GPIOs to prevent leakage
    for (int i = 0; i < GPIO_NUM_MAX; i++) {
        if (rtc_gpio_is_valid_gpio((gpio_num_t)i)) {
            rtc_gpio_isolate((gpio_num_t)i);
        }
    }

    esp_deep_sleep_start();
}
```

<Callout type="danger" title="Deep Sleep Resets Everything">
  Deep sleep is a full chip reset. WiFi connections, TCP sockets, peripheral configurations, and all main SRAM contents are gone. The only surviving state is in RTC memory (if powered) and flash storage. Design your application as a state machine that reconstructs its context from RTC variables on each boot.
</Callout>

### Hibernation

Hibernation is deep sleep with all optional RTC domains disabled. Only the RTC timer and the minimal wakeup logic remain powered. No RTC memory survives, no ULP program can run, and no GPIO wakeup is possible (only RTC timer wakeup works). This is the absolute minimum power state.

```cpp
void enter_hibernation(uint64_t sleep_us) {
    // Configure timer wakeup (the ONLY wakeup source in hibernation)
    esp_sleep_enable_timer_wakeup(sleep_us);

    // Disable all optional power domains
    esp_sleep_pd_config(ESP_PD_DOMAIN_RTC_FAST_MEM, ESP_PD_OPTION_OFF);
    esp_sleep_pd_config(ESP_PD_DOMAIN_RTC_SLOW_MEM, ESP_PD_OPTION_OFF);
    esp_sleep_pd_config(ESP_PD_DOMAIN_RTC_PERIPH, ESP_PD_OPTION_OFF);

    // With all RTC domains off, only timer wakeup works.
    // GPIO wakeup, touch wakeup, and ULP wakeup are NOT available.
    // All RTC_DATA_ATTR variables are LOST.

    esp_deep_sleep_start();
    // On wake: full cold boot, no state preserved
}
```

## Battery Life Calculations with Realistic Duty Cycles

Battery life estimation requires modeling the actual duty cycle of your application, not just the sleep current. The formula for average current consumption is:

$I_{avg} = \frac{I_{active} \times t_{active} + I_{sleep} \times t_{sleep}}{t_{active} + t_{sleep}}$

And battery life in hours:

$T_{battery} = \frac{C_{battery}}{I_{avg}}$

where $C_{battery}$ is the battery capacity in mAh and $I_{avg}$ is in mA.

### Scenario 1: Environmental Sensor (Wake, Read, Sleep)

A temperature and humidity sensor that wakes every 60 seconds, reads the sensor over I2C, stores the reading in RTC memory, and sleeps. Every 10th wake, it transmits the batch over WiFi.

**Sensor-only wake cycle** (9 out of 10 wakes):
- Boot and initialize I2C: 80 mA for 50 ms = 4.0 mAs
- Read sensor: 30 mA for 20 ms = 0.6 mAs
- Enter deep sleep: 30 mA for 5 ms = 0.15 mAs
- Deep sleep (optimized): 0.010 mA for 59,925 ms = 0.60 mAs
- **Total per cycle: 5.35 mAs over 60 seconds**

**WiFi transmission wake cycle** (1 out of 10 wakes):
- Boot and initialize WiFi: 100 mA for 200 ms = 20 mAs
- WiFi connect (cached BSSID): 150 mA for 400 ms = 60 mAs
- Transmit batch (10 readings): 180 mA for 200 ms = 36 mAs
- Disconnect and enter sleep: 50 mA for 50 ms = 2.5 mAs
- Deep sleep: 0.010 mA for 59,150 ms = 0.59 mAs
- **Total per cycle: 119 mAs over 60 seconds**

**Weighted average**: $(9 \times 5.35 + 1 \times 119) / 10 = 16.7$ mAs per 60-second cycle

**Average current**: $16.7 / 60 = 0.278$ mA = 278 uA

### Scenario 2: Door Sensor (Sleep, Wake on GPIO)

A security sensor that sleeps indefinitely until a magnetic reed switch triggers a GPIO interrupt. When triggered, it connects to WiFi and sends an alert.

- Deep sleep (optimized, GPIO wakeup enabled): 10 uA
- Alert cycle: 150 mA average for 1.5 seconds = 225 mAs

With a 2500 mAh battery and one alert per day:

$I_{avg} = \frac{0.150 \times 1.5 + 0.010 \times 86398.5}{86400} = 0.01003$ mA = 10.03 uA

**Battery life: ~28 years** (exceeds battery self-discharge lifetime of approximately 5-10 years)

### Scenario 3: Connected Display (Frequent Updates)

An e-ink display that updates every 5 minutes, requiring WiFi data fetch:

- WiFi connect + fetch: 120 mA average for 2 seconds = 240 mAs
- Display update: 40 mA for 1 second = 40 mAs
- Light sleep between updates: 0.8 mA for 297 seconds = 237.6 mAs
- **Average current**: $(240 + 40 + 237.6) / 300 = 1.73$ mA

Choosing the right sleep mode depends on your application's wake frequency, latency requirements, and state complexity.

### Decision Flowchart Logic

The decision process follows this logic:

1. **Do you need continuous WiFi connectivity?** If yes, use modem sleep or active mode with dynamic frequency scaling. You cannot drop below approximately 15-20 mA average while maintaining WiFi association.

2. **Do you wake more than once per second?** If yes, use light sleep. Deep sleep's 200-400 ms boot time creates excessive overhead at high wake frequencies. The breakeven point is approximately once every 2-5 seconds.

3. **Do you need to preserve state across sleep?** If yes, use deep sleep with RTC memory. If no, use hibernation for the lowest current.

4. **Do you need to monitor a sensor during sleep?** If yes, use deep sleep with the ULP coprocessor. The ULP can read ADC channels and RTC GPIOs while consuming only approximately 150 uA, waking the main CPU only when a threshold is exceeded.

```cpp
// Example: ULP-based threshold monitoring
// The ULP reads a soil moisture sensor every 10 seconds.
// It only wakes the main CPU when the soil is too dry.

#include "esp32/ulp.h"
#include "ulp_main.h"

RTC_DATA_ATTR static int ulp_wake_count = 0;

void setup_ulp_monitoring() {
    // Load ULP program
    ulp_load_binary(0, ulp_main_bin_start,
        (ulp_main_bin_end - ulp_main_bin_start) / sizeof(uint32_t));

    // Set ULP wakeup period (10 seconds)
    ulp_set_wakeup_period(0, 10 * 1000 * 1000);

    // Configure ADC channel for soil moisture sensor
    // (ULP program reads ADC1 channel 6 = GPIO 34)

    // Set threshold: wake main CPU if reading below 1000
    ulp_threshold = 1000;

    // Start ULP program
    ulp_run(&ulp_entry - RTC_SLOW_MEM);

    // Enable ULP wakeup
    esp_sleep_enable_ulp_wakeup();

    // Enter deep sleep -- ULP continues running
    esp_deep_sleep_start();
}
```

## Putting It All Together: A Complete Power-Optimized Sensor Node

Here is a complete example that ties together all the concepts. This firmware implements a temperature sensor that logs every 60 seconds, batches 10 readings, and transmits over WiFi with fast reconnection.

```cpp
#include <stdio.h>
#include <string.h>
#include "esp_sleep.h"
#include "esp_wifi.h"
#include "esp_event.h"
#include "esp_log.h"
#include "nvs_flash.h"
#include "driver/rtc_io.h"
#include "driver/gpio.h"

#define BATCH_SIZE 10
#define SLEEP_DURATION_US (60ULL * 1000000ULL)  // 60 seconds

// Persistent state in RTC slow memory
RTC_DATA_ATTR static int boot_count = 0;
RTC_DATA_ATTR static uint16_t sensor_buffer[BATCH_SIZE];
RTC_DATA_ATTR static uint8_t buffer_index = 0;
RTC_DATA_ATTR static uint8_t cached_wifi_channel = 0;
RTC_DATA_ATTR static uint8_t cached_wifi_bssid[6] = {0};
RTC_DATA_ATTR static bool wifi_cache_valid = false;

static const char *TAG = "sensor_node";

// Convert float temperature to fixed-point uint16_t
// Range: -40.00 to 125.00 deg C with 0.01 resolution
static uint16_t temp_to_fixed(float temp) {
    return (uint16_t)((temp + 40.0f) * 100.0f);
}

static float fixed_to_temp(uint16_t fixed) {
    return ((float)fixed / 100.0f) - 40.0f;
}

void isolate_gpios() {
    for (int i = 0; i < GPIO_NUM_MAX; i++) {
        if (rtc_gpio_is_valid_gpio((gpio_num_t)i)) {
            rtc_gpio_isolate((gpio_num_t)i);
        }
    }
}

void enter_deep_sleep() {
    esp_sleep_enable_timer_wakeup(SLEEP_DURATION_US);

    // Keep RTC slow memory (for our variables) but disable fast memory
    esp_sleep_pd_config(ESP_PD_DOMAIN_RTC_FAST_MEM, ESP_PD_OPTION_OFF);

    // Isolate GPIOs to prevent leakage through external components
    isolate_gpios();

    ESP_LOGI(TAG, "Entering deep sleep for 60 seconds...");
    esp_deep_sleep_start();
}

void transmit_batch() {
    ESP_LOGI(TAG, "Transmitting batch of %d readings", BATCH_SIZE);

    // Initialize WiFi with cached parameters for fast reconnect
    wifi_config_t wifi_config = {
        .sta = {
            .ssid = "MyNetwork",
            .password = "MyPassword",
        },
    };

    if (wifi_cache_valid) {
        memcpy(wifi_config.sta.bssid, cached_wifi_bssid, 6);
        wifi_config.sta.bssid_set = true;
        wifi_config.sta.channel = cached_wifi_channel;
        ESP_LOGI(TAG, "Using cached WiFi: ch %d", cached_wifi_channel);
    }

    // ... WiFi init, connect, HTTP POST, disconnect ...
    // On successful connection, update cache:
    // cached_wifi_channel = connected_channel;
    // memcpy(cached_wifi_bssid, connected_bssid, 6);
    // wifi_cache_valid = true;

    // Reset buffer
    buffer_index = 0;
}

void app_main() {
    boot_count++;

    esp_sleep_wakeup_cause_t cause = esp_sleep_get_wakeup_cause();
    if (cause != ESP_SLEEP_WAKEUP_TIMER && cause != ESP_SLEEP_WAKEUP_EXT0) {
        // Cold boot: reset state
        boot_count = 1;
        buffer_index = 0;
        wifi_cache_valid = false;
    }

    ESP_LOGI(TAG, "Boot #%d, buffer: %d/%d", boot_count, buffer_index, BATCH_SIZE);

    // Read sensor (fast, no WiFi needed)
    float temperature = read_temperature();  // Your sensor driver
    sensor_buffer[buffer_index++] = temp_to_fixed(temperature);

    if (buffer_index >= BATCH_SIZE) {
        // Buffer full: transmit
        transmit_batch();
    }

    enter_deep_sleep();
}
```

This pattern achieves approximately 278 uA average current, yielding over one year of battery life on a 2500 mAh 18650 cell.

## Summary

ESP32 power management is a layered discipline. At the top level, you choose the right sleep mode for your application's wake frequency. At the middle level, you optimize the active-mode duty cycle by caching WiFi credentials, batching data, and minimizing connection time. At the lowest level, you control individual power domains and isolate GPIOs to squeeze the last microamps out of deep sleep.

The original ESP32 can achieve excellent battery life with careful optimization, but the newer ESP32-C3 and ESP32-S3 make many of these optimizations unnecessary by providing low deep sleep current out of the box. For new designs, start with the ESP32-C3 unless you specifically need dual cores, Bluetooth Classic, or the ULP coprocessor.
