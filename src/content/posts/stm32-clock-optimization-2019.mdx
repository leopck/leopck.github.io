The STM32's clock tree is one of the most configurable aspects of the platform — and one of the most impactful for both performance and power consumption. A poorly configured clock tree can waste 50% of your power budget on peripherals running faster than needed, or leave performance on the table by under-clocking the core.

## PLL Configuration

The PLL multiplies a low-frequency input to the target system clock. Getting it right matters for both frequency accuracy and jitter:

```c
// STM32F4: Configure PLL for 168 MHz from 8 MHz HSE
void configure_pll_168mhz(void) {
    // PLL formula: SYSCLK = (HSE / PLLM) * PLLN / PLLP
    // Target: (8 MHz / 8) * 336 / 2 = 168 MHz
    RCC->PLLCFGR = RCC_PLLCFGR_PLLSRC_HSE  // HSE as PLL source
        | (8   << RCC_PLLCFGR_PLLM_Pos)     // PLLM = 8 (VCO input = 1 MHz)
        | (336 << RCC_PLLCFGR_PLLN_Pos)      // PLLN = 336 (VCO = 336 MHz)
        | (0   << RCC_PLLCFGR_PLLP_Pos);     // PLLP = 2 (SYSCLK = 168 MHz)

    // Configure flash wait states for 168 MHz (5 wait states)
    FLASH->ACR = FLASH_ACR_LATENCY_5WS | FLASH_ACR_PRFTEN | FLASH_ACR_ICEN | FLASH_ACR_DCEN;

    // Configure bus prescalers
    RCC->CFGR = RCC_CFGR_HPRE_DIV1    // AHB = 168 MHz (no division)
        | RCC_CFGR_PPRE1_DIV4          // APB1 = 42 MHz (max for F4)
        | RCC_CFGR_PPRE2_DIV2;         // APB2 = 84 MHz (max for F4)

    RCC->CR |= RCC_CR_PLLON;
    while (!(RCC->CR & RCC_CR_PLLRDY));  // Wait for PLL lock

    RCC->CFGR |= RCC_CFGR_SW_PLL;        // Switch to PLL
    while ((RCC->CFGR & RCC_CFGR_SWS) != RCC_CFGR_SWS_PLL);
}
```

<Callout type="warning" title="Flash Wait States">
  At 168 MHz, the STM32F4 needs 5 flash wait states. Forgetting to set these before switching to PLL causes hard faults. Always configure flash latency BEFORE switching the clock source. Enable prefetch, instruction cache, and data cache to mitigate the wait state penalty.
</Callout>

Counterintuitively, **running faster is more energy-efficient per operation** (298 µA/MHz at 168 MHz vs 531 µA/MHz at 16 MHz). This means for burst workloads, it's more efficient to run at maximum speed and enter sleep mode quickly ("race to sleep") than to run slowly and stay awake longer.

## Peripheral Clock Gating

Peripherals consume current even when idle. Disabling unused peripheral clocks saves 5-15 mA:

```c
void optimize_peripheral_clocks(void) {
    // Enable only what you need
    RCC->AHB1ENR = RCC_AHB1ENR_GPIOAEN   // GPIO Port A (for UART, SPI pins)
                 | RCC_AHB1ENR_DMA2EN;     // DMA2 (for SPI transfers)

    RCC->APB1ENR = RCC_APB1ENR_USART2EN   // UART for debug
                 | RCC_APB1ENR_TIM3EN;      // Timer for PWM

    RCC->APB2ENR = RCC_APB2ENR_SPI1EN     // SPI for sensors
                 | RCC_APB2ENR_ADC1EN;      // ADC for analog readings

    // CRITICAL: Disable everything else (they default ON after HAL_Init!)
    // GPIO B-K, I2C, CAN, DAC, USB, etc. — all consuming current if enabled
}
```

<Callout type="tip" title="HAL Trap">
  `HAL_Init()` enables all GPIO clocks by default. If you're optimizing power, explicitly disable unused peripheral clocks AFTER HAL initialization. Better yet, use register-level initialization and only enable what you need.
</Callout>

## Dynamic Frequency Scaling

For workloads that alternate between high-compute and idle, dynamically adjusting clock speed saves power:

```c
void switch_to_low_power_mode(void) {
    // Drop to 16 MHz HSI (no PLL)
    RCC->CFGR &= ~RCC_CFGR_SW;           // Switch to HSI
    while ((RCC->CFGR & RCC_CFGR_SWS) != 0);
    RCC->CR &= ~RCC_CR_PLLON;            // Disable PLL (saves ~2 mA)
    FLASH->ACR = FLASH_ACR_LATENCY_0WS;  // 0 wait states at 16 MHz
    // Current: ~8.5 mA (vs 50 mA at 168 MHz)
}

void switch_to_high_performance(void) {
    FLASH->ACR = FLASH_ACR_LATENCY_5WS | FLASH_ACR_PRFTEN | FLASH_ACR_ICEN;
    RCC->CR |= RCC_CR_PLLON;
    while (!(RCC->CR & RCC_CR_PLLRDY));
    RCC->CFGR |= RCC_CFGR_SW_PLL;
    while ((RCC->CFGR & RCC_CFGR_SWS) != RCC_CFGR_SWS_PLL);
    // Current: ~50 mA but 10.5× more compute throughput
}
```

The switching time is ~2 µs for HSI→PLL (dominated by PLL lock time). For tasks that take >100 µs, the switching overhead is negligible.

## Conclusion

STM32 clock optimization has three levers: PLL configuration (choose the right system frequency), peripheral clock gating (disable unused clocks for 20-30% savings), and dynamic frequency scaling (match clock speed to workload). The counterintuitive insight is that running faster and sleeping sooner ("race to sleep") is more energy-efficient than running slowly — the per-DMIPS energy cost decreases at higher frequencies. Always configure flash wait states before clock switching, and always gate unused peripheral clocks.
