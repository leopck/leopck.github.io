The clock tree is the nervous system of an STM32 microcontroller. Every peripheral, every timer, every bus transaction, and every CPU instruction is clocked by a signal that ultimately derives from a handful of oscillator sources routed through a configurable network of multiplexers, PLLs, and dividers. Understanding this network -- and understanding it precisely, not approximately -- is the prerequisite for correct peripheral configuration, accurate timing, optimal power consumption, and reliable operation.

This post is a register-level deep dive into the STM32F4 clock tree. We derive PLL configurations from first principles, explain why the APB prescaler creates a non-obvious timer clock doubling effect, measure real power consumption at different operating points, and implement a practical dynamic voltage and frequency scaling (DVFS) system.

After reset, the STM32F4 runs from HSI at 16 MHz with no PLL, no prescalers, and 0 flash wait states. This is intentionally conservative: it works regardless of whether an external crystal is populated, and it avoids the flash wait state configuration that higher frequencies require.

### HSI vs. HSE: When Does It Matter?

The HSI is adequate for many applications. Its 1% accuracy at room temperature is sufficient for UART communication (UART tolerates up to 3-5% baud rate error when both sides use the same oversampling), GPIO toggling, ADC conversion (ADC clock accuracy affects conversion time, not conversion accuracy), and most timer-based PWM.

You need HSE when:
- UART must work reliably across temperature extremes (automotive, industrial)
- USB requires 48 MHz with less than 0.25% accuracy (USB spec mandates this)
- CAN bus requires precise bit timing
- You need an accurate 1 PPS (pulse per second) or timestamp clock
- Your PLL output frequency must be exactly 168.000000 MHz, not approximately 168 MHz

## The PLL: Deriving System Clock

The main PLL takes a clock source (HSI or HSE), divides it by M, multiplies by N, and divides the result by P to produce the system clock, or by Q to produce the 48 MHz USB/SDIO clock.

$$f_{VCO} = f_{input} \times \frac{N}{M}$$

$$f_{SYSCLK} = \frac{f_{VCO}}{P} = f_{input} \times \frac{N}{M \times P}$$

$$f_{USB} = \frac{f_{VCO}}{Q} = f_{input} \times \frac{N}{M \times Q}$$

  ST recommends a VCO input frequency of 2 MHz for lower PLL jitter, but 1 MHz gives more flexibility in hitting exact target frequencies. ### Deriving 168 MHz from 8 MHz HSE

Target: SYSCLK = 168 MHz, USB = 48 MHz.

**Step 1:** Choose M to get VCO input in range. With HSE = 8 MHz and M = 8: VCO_in = 8/8 = 1 MHz.

**Step 2:** Choose P = 2 (smallest divisor for maximum SYSCLK). Need VCO = 168 * 2 = 336 MHz.

**Step 3:** N = VCO / VCO_in = 336 / 1 = 336. Check: 50 &lt;= 336 &lt;= 432. Valid.

**Step 4:** Q = VCO / 48 = 336 / 48 = 7. Check: 2 &lt;= 7 &lt;= 15. Valid. And 336/7 = 48.000 MHz exactly.

Final: M = 8, N = 336, P = 2, Q = 7.

### Deriving 84 MHz (Medium Speed)

Target: SYSCLK = 84 MHz, USB = 48 MHz.

Using the same VCO (336 MHz) with P = 4: SYSCLK = 336/4 = 84 MHz. Q remains 7 for 48 MHz USB.

Alternatively, reduce VCO: M = 8, N = 168, P = 2 gives SYSCLK = 168/2 = 84 MHz. But VCO = 168 MHz, and Q = 168/48 = 3.5 -- not an integer. So we must keep VCO = 336 MHz and use P = 4.

### Complete PLL Configuration Code

```c
#include "stm32f4xx.h"

typedef struct {
    uint32_t pllm;
    uint32_t plln;
    uint32_t pllp;  // 2, 4, 6, or 8
    uint32_t pllq;
    uint32_t flash_latency;  // Wait states
    uint32_t ahb_prescaler;
    uint32_t apb1_prescaler;
    uint32_t apb2_prescaler;
    uint32_t sysclk_mhz;     // For reference
} clock_config_t;

static const clock_config_t clk_168mhz = {
    .pllm = 8, .plln = 336, .pllp = 2, .pllq = 7,
    .flash_latency = FLASH_ACR_LATENCY_5WS,
    .ahb_prescaler = RCC_CFGR_HPRE_DIV1,    // AHB = 168 MHz
    .apb1_prescaler = RCC_CFGR_PPRE1_DIV4,  // APB1 = 42 MHz
    .apb2_prescaler = RCC_CFGR_PPRE2_DIV2,  // APB2 = 84 MHz
    .sysclk_mhz = 168
};

static const clock_config_t clk_84mhz = {
    .pllm = 8, .plln = 336, .pllp = 4, .pllq = 7,
    .flash_latency = FLASH_ACR_LATENCY_2WS,
    .ahb_prescaler = RCC_CFGR_HPRE_DIV1,    // AHB = 84 MHz
    .apb1_prescaler = RCC_CFGR_PPRE1_DIV2,  // APB1 = 42 MHz
    .apb2_prescaler = RCC_CFGR_PPRE2_DIV1,  // APB2 = 84 MHz
    .sysclk_mhz = 84
};

void configure_system_clock(const clock_config_t *cfg) {
    // 1. Enable HSE and wait for ready
    RCC->CR |= RCC_CR_HSEON;
    while (!(RCC->CR & RCC_CR_HSERDY)) {}

    // 2. Configure power regulator voltage scale
    //    Scale 1 (default) for up to 168 MHz
    RCC->APB1ENR |= RCC_APB1ENR_PWREN;
    PWR->CR |= PWR_CR_VOS;  // Scale 1

    // 3. Configure flash latency BEFORE switching to higher frequency
    FLASH->ACR = FLASH_ACR_ICEN | FLASH_ACR_DCEN |
                 FLASH_ACR_PRFTEN | cfg->flash_latency;

    // 4. Configure PLL
    //    Must disable PLL before changing configuration
    RCC->CR &= ~RCC_CR_PLLON;
    while (RCC->CR & RCC_CR_PLLRDY) {}

    uint32_t pllp_bits = ((cfg->pllp / 2) - 1) << 16;  // Encode P
    RCC->PLLCFGR = (cfg->pllm << 0)  |
                   (cfg->plln << 6)  |
                   pllp_bits          |
                   RCC_PLLCFGR_PLLSRC_HSE |
                   (cfg->pllq << 24);

    // 5. Enable PLL and wait for lock
    RCC->CR |= RCC_CR_PLLON;
    while (!(RCC->CR & RCC_CR_PLLRDY)) {}

    // 6. Configure bus prescalers
    RCC->CFGR = (RCC->CFGR & ~(RCC_CFGR_HPRE | RCC_CFGR_PPRE1 | RCC_CFGR_PPRE2)) |
                cfg->ahb_prescaler | cfg->apb1_prescaler | cfg->apb2_prescaler;

    // 7. Switch system clock to PLL
    RCC->CFGR = (RCC->CFGR & ~RCC_CFGR_SW) | RCC_CFGR_SW_PLL;
    while ((RCC->CFGR & RCC_CFGR_SWS) != RCC_CFGR_SWS_PLL) {}

    // 8. Update SystemCoreClock variable
    SystemCoreClock = cfg->sysclk_mhz * 1000000;
}
```

<Callout type="danger" title="Flash Latency Must Be Set Before Increasing Frequency">
  If you switch to 168 MHz with 0 wait states (the reset default), the CPU will attempt to read flash faster than the flash can respond. The result is not a clean error -- the CPU reads corrupted instructions and executes them. This typically manifests as a HardFault within microseconds of the clock switch, but it can also cause silent data corruption. Always set the flash latency BEFORE enabling the PLL.
</Callout>

## Flash Wait States and the ART Accelerator

The STM32F4 internal flash has a maximum access speed of approximately 30 MHz. At 168 MHz SYSCLK, the CPU needs 5 wait states per flash access, meaning each instruction fetch would take 6 clock cycles instead of 1. This would destroy performance.

The solution is the **ART Accelerator** (Adaptive Real-Time accelerator), which consists of:

1. **Instruction prefetch buffer**: A 128-bit wide prefetch buffer that reads an entire flash line (4 Thumb-2 instructions) per access. While the CPU executes the first instruction, the next three are already in the buffer.

2. **Instruction cache (I-cache)**: 64 lines of 128 bits each (1 KB total). Caches recently fetched flash lines so loops execute at full speed without flash wait states.

3. **Data cache (D-cache)**: 8 lines of 128 bits each (128 bytes total). Caches recently read constant data.

## AHB and APB Prescaler Effects

After the PLL generates SYSCLK, it feeds through a hierarchy of prescalers to reach the peripheral buses:

$$HCLK = \frac{SYSCLK}{AHB\_prescaler}$$

$$PCLK1 = \frac{HCLK}{APB1\_prescaler}$$

$$PCLK2 = \frac{HCLK}{APB2\_prescaler}$$

### The Timer Clock Doubling Rule

This is one of the most frequently misunderstood aspects of STM32 clock configuration. The rule is:

**If the APB prescaler for a timer's bus is 1 (no division), the timer clock equals the APB clock. If the APB prescaler is anything other than 1, the timer clock is TWICE the APB clock.**

This is specified in the RCC section of the reference manual and applies to all general-purpose and advanced timers.

At 168 MHz with our standard configuration:

- APB1 prescaler = /4, so PCLK1 = 42 MHz. APB1 timers (TIM2-7, TIM12-14) run at $42 \times 2 = 84$ MHz.
- APB2 prescaler = /2, so PCLK2 = 84 MHz. APB2 timers (TIM1, TIM8-11) run at $84 \times 2 = 168$ MHz.

<Callout type="warning" title="Getting Timer Frequency Wrong Causes Subtle Bugs">
  If you assume TIM2 runs at 42 MHz (PCLK1) instead of 84 MHz (PCLK1 x 2), all your timer periods will be off by a factor of 2. Your 1 kHz PWM becomes 500 Hz. Your 1-second delay becomes 2 seconds. Your baud rate generator produces half the intended rate. This bug is insidious because the system "kind of works" -- just at the wrong speed.
</Callout>

```c
// Correct timer frequency calculation
uint32_t get_timer_clock(TIM_TypeDef *tim) {
    // Determine which APB bus this timer is on
    uint32_t apb_prescaler;
    uint32_t pclk;

    if (tim == TIM1 || tim == TIM8 || tim == TIM9 ||
        tim == TIM10 || tim == TIM11) {
        // APB2 timers
        uint32_t ppre2 = (RCC->CFGR & RCC_CFGR_PPRE2) >> 13;
        apb_prescaler = (ppre2 & 0x4) ? (2 << (ppre2 & 0x3)) : 1;
        pclk = SystemCoreClock / apb_prescaler;
    } else {
        // APB1 timers (TIM2-7, TIM12-14)
        uint32_t ppre1 = (RCC->CFGR & RCC_CFGR_PPRE1) >> 10;
        apb_prescaler = (ppre1 & 0x4) ? (2 << (ppre1 & 0x3)) : 1;
        pclk = SystemCoreClock / apb_prescaler;
    }

    // Timer clock doubling rule
    if (apb_prescaler > 1) {
        return pclk * 2;
    }
    return pclk;
}

// Example: Configure TIM2 for 1 kHz interrupt
void tim2_1khz_init(void) {
    RCC->APB1ENR |= RCC_APB1ENR_TIM2EN;

    uint32_t tim_clk = get_timer_clock(TIM2);  // 84 MHz

    // PSC + 1 divides the timer clock
    // ARR + 1 is the period in prescaled ticks
    // Frequency = tim_clk / ((PSC+1) * (ARR+1))
    // 1 kHz = 84 MHz / ((83+1) * (999+1)) = 84e6 / 84000 = 1000 Hz

    TIM2->PSC = 83;     // Divide by 84: 84 MHz / 84 = 1 MHz tick
    TIM2->ARR = 999;    // Count 1000 ticks: 1 MHz / 1000 = 1 kHz
    TIM2->DIER = TIM_DIER_UIE;
    TIM2->CR1 = TIM_CR1_CEN;

    NVIC_SetPriority(TIM2_IRQn, 5);
    NVIC_EnableIRQ(TIM2_IRQn);
}
```

### Sleep Mode

In Sleep mode, the CPU clock is stopped but all peripheral clocks continue running. The CPU state (registers, stack) is preserved. Any interrupt wakes the CPU in approximately 1 microsecond (the time to restore the pipeline).

```c
void enter_sleep(void) {
    // Clear SLEEPDEEP bit (ensure we enter Sleep, not Stop)
    SCB->SCR &= ~SCB_SCR_SLEEPDEEP_Msk;

    // WFI (Wait For Interrupt) enters Sleep mode
    __WFI();

    // Execution resumes here after any enabled interrupt
}
```

Sleep mode is useful in a polling architecture where the main loop processes events triggered by interrupts:

```c
while (1) {
    if (data_ready) {
        data_ready = 0;
        process_data();
    }
    enter_sleep();  // Sleep until next interrupt
}
```

### Stop Mode

Stop mode stops all clocks including the PLL and HSE. Only the LSI/LSE can remain running (for RTC). SRAM and register contents are preserved by the voltage regulator, which can operate in normal or low-power mode.

Wake-up sources are limited to EXTI lines (including RTC alarm/wakeup as EXTI17).

```c
void enter_stop_mode(void) {
    // Enable PWR clock
    RCC->APB1ENR |= RCC_APB1ENR_PWREN;

    // Select low-power regulator in Stop mode
    PWR->CR |= PWR_CR_LPDS;

    // Set SLEEPDEEP bit
    SCB->SCR |= SCB_SCR_SLEEPDEEP_Msk;

    // Clear WUF (wake-up flag)
    PWR->CR |= PWR_CR_CWUF;

    // Enter Stop mode with WFI
    __WFI();

    // --- After wake-up: system clock is HSI (16 MHz) ---
    // Must reconfigure PLL if needed
    SCB->SCR &= ~SCB_SCR_SLEEPDEEP_Msk;

    // Reconfigure to 168 MHz
    configure_system_clock(&clk_168mhz);
}
```

<Callout type="warning" title="After Stop Mode, Clock Reverts to HSI">
  When waking from Stop mode, the system clock source is always HSI (16 MHz), regardless of what it was before entering Stop. The PLL is off and must be reconfigured and re-locked. This takes approximately 200-300 us (HSE startup + PLL lock time). Factor this into your wake-up latency budget.
</Callout>

## DVFS: Dynamic Voltage and Frequency Scaling

DVFS means changing the system clock frequency at runtime to match the current workload. Run fast when you need performance, run slow (or sleep) when you do not. The energy saved can be substantial for battery-powered devices.

### Implementation Strategy

The key challenge in DVFS is safely switching between clock configurations while peripherals are active. The procedure is:

1. **Notify peripherals** that a clock change is coming (e.g., disable UART TX, pause DMA).
2. **Increase flash wait states** if switching to a higher frequency. (Decrease after switching if going to lower frequency.)
3. **Switch clock source** (e.g., from PLL to HSI, reconfigure PLL, switch back).
4. **Update prescalers** to keep APB clocks within their maximum ratings.
5. **Update SystemCoreClock** and recalculate peripheral baud rates.
6. **Re-enable peripherals**.

```c
typedef enum {
    SPEED_LOW    = 0,   // 16 MHz (HSI, no PLL)
    SPEED_MEDIUM = 1,   // 84 MHz (PLL)
    SPEED_HIGH   = 2    // 168 MHz (PLL)
} system_speed_t;

static volatile system_speed_t current_speed = SPEED_HIGH;

void dvfs_set_speed(system_speed_t speed) {
    if (speed == current_speed) return;

    // Disable interrupts during clock transition
    __disable_irq();

    switch (speed) {
        case SPEED_LOW:
            // Switch to HSI (16 MHz)
            // First, increase flash latency if coming from lower WS
            // (not needed when going DOWN in frequency)

            // Switch SYSCLK to HSI
            RCC->CFGR = (RCC->CFGR & ~RCC_CFGR_SW) | RCC_CFGR_SW_HSI;
            while ((RCC->CFGR & RCC_CFGR_SWS) != RCC_CFGR_SWS_HSI) {}

            // Now safe to reduce flash latency
            FLASH->ACR = (FLASH->ACR & ~FLASH_ACR_LATENCY) |
                         FLASH_ACR_LATENCY_0WS;

            // Set prescalers: all /1 at 16 MHz
            RCC->CFGR = (RCC->CFGR & ~(RCC_CFGR_HPRE | RCC_CFGR_PPRE1 |
                         RCC_CFGR_PPRE2)) |
                         RCC_CFGR_HPRE_DIV1 |
                         RCC_CFGR_PPRE1_DIV1 |
                         RCC_CFGR_PPRE2_DIV1;

            // Disable PLL to save power
            RCC->CR &= ~RCC_CR_PLLON;

            SystemCoreClock = 16000000;
            break;

        case SPEED_MEDIUM:
            configure_system_clock(&clk_84mhz);
            break;

        case SPEED_HIGH:
            configure_system_clock(&clk_168mhz);
            break;
    }

    current_speed = speed;
    __enable_irq();

    // Reconfigure baud rates for peripherals that depend on PCLK
    reconfigure_uart_baudrate();
    reconfigure_spi_clock();
}
```

  Running at 168 MHz for 100 ms and sleeping for 400 ms uses less total energy than running at 84 MHz for 200 ms and sleeping for 300 ms, even though 168 MHz draws more current. The key metric is energy per task (current x time), not current alone. Finish the work as fast as possible, then enter the deepest sleep mode available. This is called "race to idle."

## Peripheral Clock Gating

Every peripheral on the STM32F4 has a clock enable bit in the RCC registers (AHB1ENR, APB1ENR, APB2ENR). Disabling unused peripheral clocks reduces current consumption by 1-5 mA per peripheral group.

```c
void optimize_peripheral_clocks(void) {
    // Disable all GPIO port clocks except what we actually use
    RCC->AHB1ENR &= ~(RCC_AHB1ENR_GPIOBEN | RCC_AHB1ENR_GPIOCEN |
                       RCC_AHB1ENR_GPIODEN | RCC_AHB1ENR_GPIOEEN |
                       RCC_AHB1ENR_GPIOFEN | RCC_AHB1ENR_GPIOGEN |
                       RCC_AHB1ENR_GPIOHEN | RCC_AHB1ENR_GPIOIEN);

    // Only enable GPIOA (for ADC) and GPIOD (for UART TX)
    RCC->AHB1ENR |= RCC_AHB1ENR_GPIOAEN | RCC_AHB1ENR_GPIODEN;

    // Disable unused APB1 peripherals
    RCC->APB1ENR &= ~(RCC_APB1ENR_TIM3EN | RCC_APB1ENR_TIM4EN |
                       RCC_APB1ENR_SPI2EN | RCC_APB1ENR_SPI3EN |
                       RCC_APB1ENR_I2C1EN | RCC_APB1ENR_I2C2EN |
                       RCC_APB1ENR_I2C3EN | RCC_APB1ENR_CAN1EN |
                       RCC_APB1ENR_CAN2EN | RCC_APB1ENR_DACEN);

    // Disable unused APB2 peripherals
    RCC->APB2ENR &= ~(RCC_APB2ENR_TIM1EN | RCC_APB2ENR_TIM8EN |
                       RCC_APB2ENR_SPI1EN | RCC_APB2ENR_SDIOEN);
}
```

## Real-World Clock Configuration Recipes

### Recipe 1: Maximum Performance (168 MHz, USB)

HSE = 8 MHz, M = 8, N = 336, P = 2, Q = 7.
SYSCLK = 168 MHz, APB1 = 42 MHz, APB2 = 84 MHz.
Flash: 5 WS, ART fully enabled.
Power: ~82 mA at 3.3V.

### Recipe 2: Balanced Performance/Power (84 MHz, USB)

HSE = 8 MHz, M = 8, N = 336, P = 4, Q = 7.
SYSCLK = 84 MHz, APB1 = 42 MHz, APB2 = 84 MHz.
Flash: 2 WS, ART fully enabled.
Power: ~48 mA at 3.3V.

### Recipe 3: Low Power with UART

HSI = 16 MHz, no PLL.
SYSCLK = HCLK = PCLK1 = PCLK2 = 16 MHz.
Flash: 0 WS.
Power: ~18 mA at 3.3V.
UART baud rate limited by 16 MHz clock (max reliable: 1 Mbps).

### Recipe 4: Ultra-Low Power Sensor Sampling

Stop mode with periodic RTC wake-up.
Wake every 1 second, sample ADC, store result, go back to Stop.
Average current: ~0.5 mA (dominated by 200 us wake + sample time at 168 MHz).
Battery life on 220 mAh CR2032: ~18 days.

## Conclusion

The STM32F4 clock tree is a flexible but demanding subsystem. Getting PLL configuration right requires understanding the VCO constraints and the M/N/P/Q relationship. The timer clock doubling rule is an essential detail that trips up experienced engineers. Flash wait states and the ART accelerator determine whether your 168 MHz CPU actually runs at 168 MHz effective speed or is throttled by flash access latency. And DVFS -- dynamically matching clock speed to workload -- can reduce energy consumption by 60-80% in duty-cycled applications.

The core principle is simple: run only as fast as you need to, clock only the peripherals you use, and sleep as deeply as your wake-up latency budget allows. Every milliamp saved extends battery life, reduces thermal stress, and improves system reliability.
