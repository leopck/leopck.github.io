Dynamic Voltage and Frequency Scaling (DVFS) is the single most effective power optimization technique available on the ESP32. The chip can operate its CPU at frequencies ranging from 240 MHz down to 10 MHz, with power consumption scaling from approximately 100 mA to 15 mA. But simply lowering the clock is not always optimal -- the relationship between frequency, power, execution time, and total energy is nuanced. This post provides a rigorous analysis of the ESP32 clock architecture, measured power at every frequency step, and energy-per-task calculations that reveal the optimal operating point for different workload profiles.

## ESP32 Clock Architecture

The ESP32 has a sophisticated clock tree with multiple sources and configurable paths. Understanding this architecture is essential for predicting how frequency changes affect different peripherals.

The clock multiplexing hierarchy:

```
                    +---> CPU_CLK (up to 240 MHz)
PLL (320 MHz) -----|
                    +---> APB_CLK (always 80 MHz when PLL active)

                    +---> CPU_CLK (40 or 20 MHz)
XTAL (40 MHz) -----|
                    +---> APB_CLK (equals CPU_CLK in XTAL mode)

                    +---> CPU_CLK (8 MHz)
RTC8M (~8 MHz) ----|
                    +---> APB_CLK (8 MHz -- most peripherals fail)
```

The CPU frequency derivation:

$$
f_{CPU} = \begin{cases}
\frac{f_{PLL}}{2} = 240 \text{ MHz} & \text{if PLL=480, SEL=2} \\
\frac{f_{PLL}}{2} = 160 \text{ MHz} & \text{if PLL=320, SEL=1} \\
\frac{f_{PLL}}{4} = 80 \text{ MHz} & \text{if PLL=320, SEL=0} \\
f_{XTAL} = 40 \text{ MHz} & \text{if XTAL source} \\
\frac{f_{XTAL}}{2} = 20 \text{ MHz} & \text{if XTAL/2} \\
f_{RTC8M} \approx 8 \text{ MHz} & \text{if RTC8M source}
\end{cases}
$$

### Power Scaling Analysis

The power consumption does not scale linearly with frequency. This is because the ESP32's power has two components:

$$
P_{total} = P_{static} + P_{dynamic} = P_{leakage} + C_{eff} \cdot V_{DD}^2 \cdot f
$$

Where:
- $P_{static}$ (leakage): approximately 10-12 mA, independent of frequency
- $P_{dynamic}$: proportional to frequency
- $C_{eff}$: effective switching capacitance (~0.35 nF estimated from measurements)
- $V_{DD}$: constant at 3.3V (ESP32 does not support voltage scaling)

The lack of voltage scaling is a key limitation. True DVFS (as on application processors) reduces both voltage and frequency, giving a cubic power reduction. The ESP32 only scales frequency, giving a linear dynamic power reduction while the static component remains constant. This limits the power savings achievable through frequency scaling alone.

For most battery applications, 40 MHz represents the knee of the power curve -- below it, the power savings per unit of lost performance become poor. If you need power below the 40 MHz level, consider deep sleep with periodic wake-up instead.
</Callout>

## Energy-Per-Task Analysis: Which Frequency Minimizes Total Energy

The most important insight in frequency scaling is that the lowest-power frequency is not always the most energy-efficient. Energy is power multiplied by time:

$$
E_{task} = P(f) \times T_{task}(f)
$$

If the task execution time scales inversely with frequency (CPU-bound):

$$
T_{task}(f) = \frac{T_{task,max}}{f / f_{min}}
$$

Then:

$$
E_{task}(f) = P(f) \times \frac{T_{task,max} \cdot f_{min}}{f}
$$

Because $P(f) = P_{static} + k \cdot f$:

$$
E_{task}(f) = \left(P_{static} + k \cdot f\right) \times \frac{T_{task,max} \cdot f_{min}}{f} = T_{task,max} \cdot f_{min} \cdot \left(\frac{P_{static}}{f} + k\right)
$$

This function decreases as $f$ increases (because $P_{static}/f$ decreases). In other words, for CPU-bound tasks, running at maximum frequency and then sleeping is more energy-efficient than running slowly. This is the "race to sleep" principle.

## Implementing Dynamic Frequency Scaling in FreeRTOS

The ESP-IDF provides a power management framework that integrates with FreeRTOS. It uses "power management locks" that tasks acquire and release to signal their frequency requirements.

### ESP-IDF Power Management Configuration

```cpp
#include "esp_pm.h"
#include "esp_log.h"

static const char* TAG = "DVFS";

void configure_power_management(void) {
    // Configure the DVFS range
    esp_pm_config_esp32_t pm_config = {
        .max_freq_mhz = 240,       // Maximum CPU frequency
        .min_freq_mhz = 10,        // Minimum CPU frequency when idle
        .light_sleep_enable = true, // Allow automatic light sleep
    };

    esp_err_t ret = esp_pm_configure(&pm_config);
    if (ret == ESP_OK) {
        ESP_LOGI(TAG, "DVFS configured: %d - %d MHz, light sleep %s",
                 pm_config.min_freq_mhz, pm_config.max_freq_mhz,
                 pm_config.light_sleep_enable ? "ON" : "OFF");
    } else {
        ESP_LOGE(TAG, "PM config failed: %s", esp_err_to_name(ret));
    }
}
```

### Power Management Locks

Tasks use locks to request specific frequency levels:

```cpp
#include "esp_pm.h"

static esp_pm_lock_handle_t cpu_freq_lock;
static esp_pm_lock_handle_t apb_freq_lock;

void init_pm_locks(void) {
    // Lock for CPU-intensive work: requests 240 MHz
    esp_pm_lock_create(ESP_PM_CPU_FREQ_MAX, 0, "compute_lock", &cpu_freq_lock);

    // Lock for peripheral work: requests APB >= 80 MHz
    // This prevents APB from dropping below 80 MHz (keeps PLL active)
    esp_pm_lock_create(ESP_PM_APB_FREQ_MAX, 0, "periph_lock", &apb_freq_lock);
}

// CPU-bound task: acquire max-frequency lock during computation
void compute_task(void* param) {
    while (true) {
        // Wait for data (CPU can idle at minimum frequency)
        sensor_data_t data;
        xQueueReceive(data_queue, &data, portMAX_DELAY);

        // Acquire lock: CPU goes to 240 MHz
        esp_pm_lock_acquire(cpu_freq_lock);

        // Do heavy computation
        process_data(&data);

        // Release lock: CPU can drop back to minimum
        esp_pm_lock_release(cpu_freq_lock);

        // Send result (I/O-bound: no lock needed)
        send_result(&data);
    }
}

// Peripheral task: acquire APB lock during SPI/UART/I2C transactions
void sensor_task(void* param) {
    while (true) {
        // Acquire APB lock to keep peripheral clocks stable
        esp_pm_lock_acquire(apb_freq_lock);

        // Read sensors (APB stays at 80 MHz, peripherals work correctly)
        uint8_t data[14];
        i2c_burst_read(I2C_NUM_0, IMU_ADDR, 0x3B, data, 14);

        // Release APB lock
        esp_pm_lock_release(apb_freq_lock);

        // Process data at whatever frequency PM chooses
        process_imu(data);

        vTaskDelay(pdMS_TO_TICKS(1));
    }
}
```

<Callout type="warning" title="Always Acquire APB Lock Before Peripheral Access">
  If the power management system drops CPU frequency below 80 MHz (to 40, 20, or 10 MHz), the APB clock drops proportionally. This changes the baud rate of every UART, the clock speed of every SPI bus, and the frequency of every I2C bus. Peripheral transactions initiated without an APB lock will silently run at the wrong speed, causing protocol errors, data corruption, or timeout failures. The ESP-IDF UART, SPI, and I2C drivers automatically acquire APB locks internally, but if you access peripheral registers directly, you must manage locks yourself.
</Callout>

### Automatic Light Sleep Integration

When all power management locks are released and no tasks are ready, the ESP-IDF power management framework can automatically enter light sleep:

```cpp
// The idle task hooks into power management
// When light_sleep_enable = true:
//
// 1. All tasks blocked (waiting on queue, semaphore, or delay)
// 2. No PM locks held
// 3. System enters light sleep:
//    - CPU clock gated (0 MHz effective)
//    - PLL powered off
//    - APB clock gated
//    - RTC still running (maintains FreeRTOS tick via RTC timer)
//    - Current: ~0.8 mA (vs ~15 mA at 10 MHz active)
//
// 4. Wake sources: timer (next FreeRTOS tick), GPIO, UART, touch, ULP
// 5. Wake latency: ~1 ms (PLL relock) to ~0.3 ms (if PLL stays on)
```

### Frequency Change Callback Registration

You can register callbacks to be notified of frequency changes:

```cpp
#include "esp_pm.h"

static void IRAM_ATTR freq_change_callback(void* arg,
                                            esp_pm_cb_event_t event,
                                            void* data) {
    switch (event) {
        case ESP_PM_CB_FREQ_BEFORE_CHANGE:
            // About to change frequency
            // Pause any timing-sensitive operations
            break;
        case ESP_PM_CB_FREQ_AFTER_CHANGE:
            // Frequency has changed
            // Reconfigure any manually-managed peripherals
            {
                uint32_t new_apb = *(uint32_t*)data;
                reconfigure_custom_peripheral(new_apb);
            }
            break;
    }
}

void register_freq_callback(void) {
    esp_pm_register_inform_out_of_light_sleep_cb(freq_change_callback, NULL);
}
```

### When Is Frequency Switching Worth It?

The energy cost of a switch must be recovered through power savings during the low-frequency period. The break-even time:

$$
T_{break-even} = \frac{E_{switch}}{P_{high} - P_{low}}
$$

For a 80 MHz to 40 MHz transition:

$$
T_{break-even} = \frac{5 \text{ uJ}}{(158 - 99) \text{ mW}} = \frac{5 \text{ uJ}}{59 \text{ mW}} \approx 85 \text{ us}
$$

If the low-frequency period will be shorter than 85 microseconds, the switch wastes energy. For a 240 MHz to 40 MHz transition with PLL shutdown:

$$
T_{break-even} = \frac{25 \text{ uJ}}{(314 - 99) \text{ mW}} = \frac{25 \text{ uJ}}{215 \text{ mW}} \approx 116 \text{ us}
$$

In practice, frequency changes should correspond to workload phases lasting at least several hundred microseconds.

## Practical DVFS Strategies

### Strategy 1: Race to Sleep (CPU-Bound Tasks)

For applications dominated by periodic CPU-intensive tasks followed by long idle periods:

```cpp
void race_to_sleep_task(void* param) {
    // PM config: max=240, min=10, light_sleep=true
    esp_pm_lock_handle_t lock;
    esp_pm_lock_create(ESP_PM_CPU_FREQ_MAX, 0, "race", &lock);

    TickType_t last_wake = xTaskGetTickCount();

    while (true) {
        // Acquire lock: CPU goes to 240 MHz
        esp_pm_lock_acquire(lock);

        // Do all computation as fast as possible
        acquire_sensor_data();
        process_data();
        prepare_output();

        // Release lock: CPU drops to minimum, may enter light sleep
        esp_pm_lock_release(lock);

        // Sleep until next period (light sleep during this time)
        vTaskDelayUntil(&last_wake, pdMS_TO_TICKS(100));  // 10 Hz
    }
}
```

### Strategy 2: Workload-Adaptive Frequency

For applications with varying computational demands:

```cpp
void adaptive_task(void* param) {
    esp_pm_lock_handle_t cpu_lock;
    esp_pm_lock_handle_t apb_lock;
    esp_pm_lock_create(ESP_PM_CPU_FREQ_MAX, 0, "cpu", &cpu_lock);
    esp_pm_lock_create(ESP_PM_APB_FREQ_MAX, 0, "apb", &apb_lock);

    while (true) {
        // Phase 1: I/O (sensor read) -- only need APB lock
        esp_pm_lock_acquire(apb_lock);
        sensor_data_t raw = read_sensors();
        esp_pm_lock_release(apb_lock);

        // Phase 2: Light processing -- no lock needed (runs at min freq)
        if (!needs_heavy_processing(&raw)) {
            apply_simple_filter(&raw);
            store_result(&raw);
            continue;
        }

        // Phase 3: Heavy processing -- need max CPU
        esp_pm_lock_acquire(cpu_lock);
        run_fft(&raw);
        run_peak_detection(&raw);
        esp_pm_lock_release(cpu_lock);

        // Phase 4: WiFi transmit -- only need APB lock
        esp_pm_lock_acquire(apb_lock);
        transmit_result(&raw);
        esp_pm_lock_release(apb_lock);
    }
}
```

## Real-Time Constraints and DVFS

When tasks have hard deadlines, DVFS must be constrained to ensure deadlines are met. The relationship between frequency and worst-case execution time:

$$
WCET(f) = WCET_{max} \times \frac{f_{max}}{f}
$$

To meet a deadline $D$:

$$
f_{min} \geq \frac{WCET_{max} \times f_{max}}{D}
$$

```cpp
// Calculate minimum frequency to meet a deadline
uint32_t min_freq_for_deadline(uint32_t wcet_at_240mhz_us,
                                uint32_t deadline_us) {
    // wcet at freq f: wcet_240 * 240 / f
    // Require: wcet_240 * 240 / f <= deadline
    // Therefore: f >= wcet_240 * 240 / deadline

    uint32_t min_freq_mhz = (wcet_at_240mhz_us * 240 + deadline_us - 1)
                             / deadline_us;

    // Round up to nearest available frequency
    if (min_freq_mhz <= 10) return 10;
    if (min_freq_mhz <= 20) return 20;
    if (min_freq_mhz <= 40) return 40;
    if (min_freq_mhz <= 80) return 80;
    if (min_freq_mhz <= 160) return 160;
    return 240;
}

// Example: task WCET is 500 us at 240 MHz, deadline is 2000 us
// min_freq = 500 * 240 / 2000 = 60 MHz -> round to 80 MHz
// At 80 MHz: WCET = 500 * 240/80 = 1500 us < 2000 us (meets deadline)
// Power saving: 158 mW vs 314 mW = 50% reduction while meeting deadline
```

## Conclusion

ESP32 CPU frequency scaling is a powerful but nuanced optimization technique. The six available frequency steps (240, 160, 80, 40, 20, 10 MHz) span a 6x performance range and a 6.3x power range (95 mA to 15 mA). However, the relationship between frequency and energy efficiency is workload-dependent.

For CPU-bound tasks, the "race to sleep" strategy is optimal: run at 240 MHz, complete the computation as fast as possible, then drop to minimum frequency or enter light sleep. This works because the static power component (approximately 33-40 mW) is amortized over a shorter execution time.

For I/O-bound tasks, minimum frequency is optimal because execution time is determined by the peripheral (I2C bus, UART baud rate, WiFi radio), not the CPU. Running the CPU at 240 MHz during a 400-microsecond I2C transaction wastes 5.8x more energy than running at 10 MHz.

The ESP-IDF power management framework with lock-based frequency control integrates cleanly with FreeRTOS and handles the complexities of APB frequency changes, PLL management, and peripheral reconfiguration automatically. The critical implementation detail is to acquire `ESP_PM_CPU_FREQ_MAX` locks during computation and `ESP_PM_APB_FREQ_MAX` locks during peripheral access, releasing them promptly when the work is complete.

The often-overlooked factor is frequency switching overhead: PLL-to-XTAL transitions cost approximately 200 microseconds, so frequency changes should correspond to workload phases lasting at least several hundred microseconds to provide a net energy benefit. For workloads that alternate rapidly between CPU-intensive and I/O phases within the same millisecond, staying at 80 MHz (the lowest PLL-based frequency) avoids PLL relock overhead while still saving significant power compared to 240 MHz.
