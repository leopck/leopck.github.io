Interrupts are the mechanism by which an STM32 responds to real-world events in real time. A sensor completes a conversion, a communication peripheral receives a byte, a timer overflows, or an external pin changes state -- in each case, the hardware signals the CPU to suspend its current task and execute a specific handler function. How quickly this happens, how the CPU decides which interrupt to service first when multiple are pending, and how you protect shared data between interrupt handlers and the main thread: these are the topics that separate a system that "works on the bench" from one that works reliably in the field.

This post covers the Cortex-M4 NVIC (Nested Vectored Interrupt Controller) in architectural detail, explains the priority grouping system, dives into the tail-chaining and late-arrival optimizations that ARM built into the hardware, quantifies interrupt latency with cycle-accurate analysis, compares DMA vs. interrupts vs. polling for different use cases, and provides production-quality critical section implementations using BASEPRI and PRIMASK.

## NVIC Architecture

The NVIC is tightly integrated into the Cortex-M4 processor core -- it is not a separate peripheral on the bus but part of the core itself. This is why ARM calls it "nested" and "vectored": it supports preemption (nesting) natively and uses a hardware vector table for zero-overhead dispatch.

### The Vector Table

The vector table is an array of 32-bit function pointers starting at address 0x00000000 (or wherever VTOR points). The first two entries are special: entry 0 is the initial stack pointer, and entry 1 is the reset handler address. Entries 2 through 15 are system exceptions (NMI, HardFault, MemManage, BusFault, UsageFault, SVCall, PendSV, SysTick). Entries 16 and above are external interrupt handlers, numbered IRQ0 through IRQn.

When an interrupt fires, the hardware reads the vector table entry for that interrupt number, loads the address into the program counter, and begins executing the ISR. There is no software dispatch table lookup, no function pointer indirection by the compiler -- it is pure hardware.

## Priority Grouping: Preemption vs. Sub-Priority

The Cortex-M4 has an 8-bit priority register per interrupt, but STM32F4 only implements the top 4 bits (bits 7:4), giving 16 effective priority levels (0-15, where 0 is highest priority).

These 4 bits are split into two fields by the **priority grouping** configuration:

- **Preemption priority** (group priority): Determines whether one ISR can interrupt (preempt) another ISR that is already executing. A lower preemption number can preempt a higher preemption number.

- **Sub-priority**: Used as a tie-breaker when two interrupts with the same preemption priority are pending simultaneously. The lower sub-priority number is serviced first, but it cannot preempt the other.

  With PRIGROUP = 3 (all 4 bits for preemption), you get 16 distinct preemption levels and no sub-priority. This is the simplest model: every interrupt either can or cannot preempt every other interrupt, with no ambiguity. Sub-priority is only useful when you have many interrupts at the same preemption level and want deterministic ordering among them -- a rare requirement.
</Callout>

### Priority Assignment Strategy

A well-designed priority scheme follows this pattern:

```c
void configure_interrupt_priorities(void) {
    // Set priority grouping: all 4 bits for preemption
    NVIC_SetPriorityGrouping(3);  // NVIC_PRIORITYGROUP_4

    // Priority 0: Time-critical hardware (highest)
    // Motor control PWM fault
    NVIC_SetPriority(TIM1_BRK_TIM9_IRQn, 0);

    // Priority 1: Real-time sampling
    // DMA transfer complete for ADC/audio
    NVIC_SetPriority(DMA2_Stream0_IRQn, 1);

    // Priority 2: Communication (high-rate)
    // SPI DMA completion
    NVIC_SetPriority(DMA2_Stream3_IRQn, 2);

    // Priority 3-4: Communication (lower-rate)
    NVIC_SetPriority(USART1_IRQn, 3);
    NVIC_SetPriority(USART2_IRQn, 4);

    // Priority 5-7: Timers, general purpose
    NVIC_SetPriority(TIM2_IRQn, 5);
    NVIC_SetPriority(TIM3_IRQn, 6);

    // Priority 8-10: Low-priority I/O
    NVIC_SetPriority(I2C1_EV_IRQn, 8);
    NVIC_SetPriority(EXTI0_IRQn, 9);

    // Priority 15: Lowest (background tasks)
    // SysTick for RTOS or delay counter
    NVIC_SetPriority(SysTick_IRQn, 15);
}
```

<Callout type="warning" title="FreeRTOS Priority Constraint">
  If you use FreeRTOS, all interrupts that call FreeRTOS API functions (xQueueSendFromISR, xSemaphoreGiveFromISR, etc.) must have a priority number equal to or greater than `configMAX_SYSCALL_INTERRUPT_PRIORITY`. Interrupts with priority numbers lower than this value (i.e., higher actual priority) must NOT call any FreeRTOS API. This is how FreeRTOS protects its internal data structures without disabling all interrupts. Typically, `configMAX_SYSCALL_INTERRUPT_PRIORITY` is set to 5, meaning priorities 0-4 are "above FreeRTOS" and cannot use the API.
</Callout>

## Interrupt Entry: The 12-Cycle Hardware Stacking

When an interrupt is accepted, the Cortex-M4 hardware automatically saves 8 registers to the current stack (MSP or PSP) in exactly 12 clock cycles:

  notes="The stacking and vector fetch happen in parallel on the bus matrix. If the vector table is in a different memory than the stack, they truly overlap and the total is 12 cycles. If both are in the same SRAM, there may be bus contention adding 1-2 cycles."
/>

After 12 cycles, the first instruction of the ISR begins executing. This is the **minimum interrupt latency**: 12 cycles from the interrupt being asserted to the first ISR instruction running. At 168 MHz, this is 71.4 nanoseconds.

But the actual latency you observe in practice is longer, because:

1. **Multi-cycle instructions**: If the CPU is executing a multi-cycle instruction (e.g., a divide that takes 2-12 cycles), the interrupt waits until the instruction completes.

2. **Critical sections**: If interrupts are masked via PRIMASK or BASEPRI, the interrupt pends but is not accepted until the mask is cleared.

3. **Higher-priority ISR**: If a higher-priority ISR is already running, the new interrupt pends until the higher-priority ISR returns (unless the new interrupt has even higher priority and can preempt).

4. **Bus contention**: If the AHB bus is busy with a DMA burst, the stacking sequence may stall for a few cycles.

## Tail-Chaining: Zero-Overhead Back-to-Back Interrupts

When an ISR completes and another interrupt of equal or lower priority is already pending, the Cortex-M4 does not perform a full unstack-then-restack sequence. Instead, it performs **tail-chaining**: it skips the unstacking, keeps the stacked context on the stack, fetches the new ISR vector, and jumps to it in only 6 cycles.

Without tail-chaining, returning from ISR A and entering ISR B would take:
- 12 cycles to unstack (restore 8 registers)
- 12 cycles to stack again (save 8 registers for new context)
- Total: 24 cycles

With tail-chaining: 6 cycles. This is a 4x improvement in back-to-back interrupt throughput.

```
Interrupt A fires    Interrupt B pending
    |                     |
    v                     v
[12 cycles: stack] -> [ISR A executes] -> [6 cycles: tail-chain] -> [ISR B executes] -> [12 cycles: unstack]
                                          ^^^^^^^^^^^^^^^^^^^^^
                                          No unstack + restack!
```

Tail-chaining is completely automatic. You do not need to enable it or configure anything. The NVIC does it whenever the conditions are met (pending interrupt at exit of current ISR). It is one of the key advantages of the Cortex-M architecture over older ARM7/ARM9 designs where interrupt context switching required software intervention.

## Late Arrival: Higher-Priority Interrupt During Stacking

**Late arrival** handles the case where a higher-priority interrupt arrives while the CPU is in the process of stacking registers for a lower-priority interrupt. Instead of completing the stacking, entering the lower-priority ISR, and then immediately preempting it with the higher-priority one (which would require another full stacking), the NVIC redirects the vector fetch to the higher-priority ISR.

The stacking continues (it has already started and must complete to save the pre-interrupt context), but the vector fetched is for the higher-priority interrupt. After the higher-priority ISR completes, the lower-priority ISR runs via tail-chaining, and then the original context is restored.

The result: the higher-priority ISR runs with zero additional stacking overhead. It effectively "steals" the stacking that was already happening for the lower-priority interrupt.

```
Low-priority IRQ fires -> [stacking starts] -> High-priority IRQ arrives (late arrival)
                              |                          |
                              v                          v
                     [stacking completes] -> [High-priority ISR runs] -> [tail-chain to Low-priority ISR] -> [unstack]
```

<Callout type="info" title="Late Arrival Has a Window">
  Late arrival only works if the higher-priority interrupt arrives during the 12-cycle stacking phase. If it arrives after the lower-priority ISR has already started executing, normal preemption occurs (full stacking for a nested interrupt). The practical benefit is most visible in systems with many interrupts firing in rapid succession.
</Callout>

### Measuring Interrupt Latency

The most accurate measurement uses the DWT cycle counter:

```c
// Globals for latency measurement
volatile uint32_t trigger_time = 0;
volatile uint32_t isr_entry_time = 0;
volatile uint32_t latency_cycles = 0;
volatile uint32_t latency_samples[1024];
volatile uint32_t sample_index = 0;

// Enable DWT cycle counter
void dwt_init(void) {
    CoreDebug->DEMCR |= CoreDebug_DEMCR_TRCENA_Msk;
    DWT->CYCCNT = 0;
    DWT->CTRL |= DWT_CTRL_CYCCNTENA_Msk;
}

// EXTI0 ISR: record entry time as first instruction
void EXTI0_IRQHandler(void) {
    isr_entry_time = DWT->CYCCNT;  // MUST be first instruction
    EXTI->PR = EXTI_PR_PR0;        // Clear pending

    latency_cycles = isr_entry_time - trigger_time;
    if (sample_index < 1024) {
        latency_samples[sample_index++] = latency_cycles;
    }
}

// Trigger from main loop
void measure_latency(void) {
    // Configure PA0 as EXTI input with rising edge
    // ... (GPIO, SYSCFG, EXTI configuration) ...

    // Software trigger via EXTI->SWIER
    for (uint32_t i = 0; i < 1024; i++) {
        trigger_time = DWT->CYCCNT;
        EXTI->SWIER = EXTI_SWIER_SWIER0;

        // Wait for ISR to complete
        while (sample_index <= i) {}
    }

    // Analyze results
    uint32_t min_lat = UINT32_MAX, max_lat = 0, sum = 0;
    for (uint32_t i = 0; i < 1024; i++) {
        if (latency_samples[i] < min_lat) min_lat = latency_samples[i];
        if (latency_samples[i] > max_lat) max_lat = latency_samples[i];
        sum += latency_samples[i];
    }
    uint32_t avg_lat = sum / 1024;
    // Results (typical): min=15, avg=17, max=45
}
```

### Decision Framework

**Use polling when:**
- The wait is expected to be very short (less than 20-30 cycles)
- You are in a time-critical section where interrupt jitter is unacceptable
- Example: waiting for SPI TX buffer empty when sending a single byte in a bit-bang protocol

**Use interrupts when:**
- Events are infrequent or unpredictable (button presses, UART RX)
- Each event requires small, immediate action (less than 100 cycles)
- The CPU should be free between events
- Example: UART character reception, timer overflow, external pin change

**Use DMA when:**
- Transferring blocks of data (greater than 4-8 bytes)
- The peripheral generates data continuously (ADC, I2S, SPI)
- You need the CPU completely free during the transfer
- Example: ADC sampling, SPI flash reads, UART bulk transmission

**Use DMA + interrupts together when:**
- DMA handles the bulk transfer
- An interrupt signals transfer completion
- The CPU processes completed blocks
- Example: the double-buffered DMA pattern from the companion post

```c
// Example: SPI single-byte transfer (polling is correct)
static inline uint8_t spi_xfer_byte(uint8_t tx) {
    while (!(SPI1->SR & SPI_SR_TXE)) {}  // Poll: typically 1-2 iterations
    SPI1->DR = tx;
    while (!(SPI1->SR & SPI_SR_RXNE)) {} // Poll: typically 1-2 iterations
    return SPI1->DR;
}

// Example: SPI 256-byte transfer (DMA is correct)
void spi_xfer_block(uint8_t *tx, uint8_t *rx, uint32_t len) {
    // Configure DMA TX and RX streams
    // Enable DMA, SPI generates requests, CPU is free
    // TC interrupt signals completion
    spi_dma_start(tx, rx, len);
}
```

## Critical Section Management

A critical section is a code region where shared data is accessed and must not be interrupted by code that also accesses the same data. On a bare-metal Cortex-M4, the two primary mechanisms are PRIMASK and BASEPRI.

### PRIMASK: Disable All Interrupts

PRIMASK is a 1-bit register. When set to 1, it blocks all interrupts except NMI and HardFault. This is the sledgehammer approach: simple and effective, but it blocks everything, including time-critical interrupts that have nothing to do with the shared data you are protecting.

```c
// Method 1: GCC intrinsics
static inline uint32_t critical_enter(void) {
    uint32_t primask;
    __asm volatile (
        "MRS %0, PRIMASK\n"
        "CPSID I\n"
        : "=r" (primask)
        :
        : "memory"
    );
    return primask;
}

static inline void critical_exit(uint32_t primask) {
    __asm volatile (
        "MSR PRIMASK, %0\n"
        :
        : "r" (primask)
        : "memory"
    );
}

// Usage
void update_shared_counter(void) {
    uint32_t saved = critical_enter();
    shared_counter++;
    critical_exit(saved);
}
```

<Callout type="warning" title="Always Save and Restore PRIMASK">
  Never blindly use `__disable_irq()` / `__enable_irq()` because they do not nest. If function A disables interrupts and calls function B, which also disables then enables interrupts, B's `__enable_irq()` will re-enable interrupts while A still expects them to be disabled. Always save the previous PRIMASK value and restore it.
</Callout>

### BASEPRI: Selective Interrupt Masking

BASEPRI is an 8-bit register (only top 4 bits matter on STM32F4) that masks all interrupts with a priority number equal to or greater than the BASEPRI value. Interrupts with a lower priority number (higher actual priority) remain enabled.

This is the precision scalpel: you can protect a critical section from a specific set of interrupts while still allowing higher-priority interrupts to fire.

```c
// Mask interrupts at priority >= threshold
static inline uint32_t basepri_enter(uint32_t priority) {
    uint32_t old_basepri;
    // STM32F4: priority must be shifted left by 4 (only top 4 bits used)
    uint32_t basepri_val = priority << 4;
    __asm volatile (
        "MRS %0, BASEPRI\n"
        "MSR BASEPRI, %1\n"
        : "=r" (old_basepri)
        : "r" (basepri_val)
        : "memory"
    );
    return old_basepri;
}

static inline void basepri_exit(uint32_t old_basepri) {
    __asm volatile (
        "MSR BASEPRI, %0\n"
        :
        : "r" (old_basepri)
        : "memory"
    );
}

// Example: protect data shared with UART ISR (priority 3)
// while allowing DMA ISR (priority 1) and motor fault (priority 0) to fire
void update_uart_buffer(void) {
    // Mask priority 3 and below (3, 4, 5, ..., 15)
    // Priority 0, 1, 2 remain enabled
    uint32_t saved = basepri_enter(3);

    // Safe to modify uart_buffer here
    uart_buffer[uart_head] = new_data;
    uart_head = (uart_head + 1) % UART_BUF_SIZE;

    basepri_exit(saved);
}
```

### Critical Section Duration Guidelines

The duration of a critical section directly impacts the worst-case interrupt latency for all masked interrupts. Every cycle spent in a critical section is a cycle of potential delay for a pending interrupt.

```c
// BAD: Long critical section
void bad_critical_section(void) {
    __disable_irq();
    for (int i = 0; i < 1000; i++) {
        buffer[i] = compute_value(i);  // Could be thousands of cycles
    }
    __enable_irq();
    // Impact: ALL interrupts delayed by thousands of cycles
}

// GOOD: Minimize critical section
void good_approach(void) {
    // Compute outside critical section
    uint32_t temp[1000];
    for (int i = 0; i < 1000; i++) {
        temp[i] = compute_value(i);
    }

    // Only protect the actual shared data update
    uint32_t saved = critical_enter();
    memcpy(buffer, temp, sizeof(temp));
    critical_exit(saved);
    // Impact: Interrupts delayed only by memcpy duration
}

// BEST: Use DMA or atomic operations to avoid critical section entirely
void best_approach(void) {
    // Single-word writes are atomic on Cortex-M4
    // No critical section needed if the shared variable is a single uint32_t
    shared_flag = 1;  // Atomic (single STR instruction)
}
```

<Callout type="perf" title="Atomic Operations Avoid Critical Sections">
  On Cortex-M4, any aligned 32-bit load or store is atomic. If your shared data is a single `uint32_t` flag or counter, you may not need a critical section at all. For compound operations (read-modify-write), use the LDREX/STREX exclusive access instructions or the `__LDREXW`/`__STREXW` intrinsics. These provide lock-free atomic operations without disabling interrupts.
</Callout>

### LDREX/STREX for Lock-Free Atomics

```c
// Atomic increment without disabling interrupts
static inline uint32_t atomic_increment(volatile uint32_t *ptr) {
    uint32_t old_val, new_val, status;
    do {
        old_val = __LDREXW(ptr);
        new_val = old_val + 1;
        status = __STREXW(new_val, ptr);
    } while (status != 0);  // Retry if interrupted between LDREX and STREX
    return old_val;
}

// Atomic compare-and-swap
static inline uint32_t atomic_cas(volatile uint32_t *ptr,
                                   uint32_t expected,
                                   uint32_t desired) {
    uint32_t old_val, status;
    do {
        old_val = __LDREXW(ptr);
        if (old_val != expected) {
            __CLREX();
            return 0;  // CAS failed
        }
        status = __STREXW(desired, ptr);
    } while (status != 0);
    return 1;  // CAS succeeded
}
```

## Optimized ISR Design Patterns

### Pattern 1: Flag-Based Deferred Processing

The ISR sets a flag; the main loop does the work. This minimizes ISR duration and interrupt latency for other interrupts.

```c
volatile uint32_t event_flags = 0;

#define EVT_UART_RX    (1U << 0)
#define EVT_ADC_DONE   (1U << 1)
#define EVT_TIMER_TICK (1U << 2)
#define EVT_BUTTON     (1U << 3)

void USART1_IRQHandler(void) {
    if (USART1->SR & USART_SR_RXNE) {
        rx_fifo_push(USART1->DR);  // ~10 cycles
        event_flags |= EVT_UART_RX;
    }
}

void DMA2_Stream0_IRQHandler(void) {
    DMA2->LIFCR = DMA_LIFCR_CTCIF0;
    event_flags |= EVT_ADC_DONE;
}

// Main loop
while (1) {
    uint32_t flags = event_flags;
    if (flags) {
        if (flags & EVT_UART_RX) {
            event_flags &= ~EVT_UART_RX;
            process_uart_data();
        }
        if (flags & EVT_ADC_DONE) {
            event_flags &= ~EVT_ADC_DONE;
            process_adc_block();
        }
        // ... other events ...
    } else {
        __WFI();  // Sleep until next interrupt
    }
}
```

### Pattern 2: Ring Buffer for High-Rate Data

For peripherals that produce data faster than the main loop can process, use a ring buffer in the ISR:

```c
#define RX_BUF_SIZE 256  // Must be power of 2 for fast modulo

static volatile uint8_t rx_buf[RX_BUF_SIZE];
static volatile uint32_t rx_head = 0;  // Written by ISR
static volatile uint32_t rx_tail = 0;  // Read by main loop

void USART1_IRQHandler(void) {
    if (USART1->SR & USART_SR_RXNE) {
        uint32_t next_head = (rx_head + 1) & (RX_BUF_SIZE - 1);
        if (next_head != rx_tail) {  // Not full
            rx_buf[rx_head] = USART1->DR;
            rx_head = next_head;
        }
        // If full, byte is dropped (intentional)
    }
}

// Main loop reader (no critical section needed!)
uint32_t uart_read(uint8_t *dst, uint32_t max_len) {
    uint32_t count = 0;
    while (rx_tail != rx_head && count < max_len) {
        dst[count++] = rx_buf[rx_tail];
        rx_tail = (rx_tail + 1) & (RX_BUF_SIZE - 1);
    }
    return count;
}
```

<Callout type="tip" title="Single Producer / Single Consumer Ring Buffer Needs No Lock">
  When exactly one writer (ISR) and one reader (main loop) access the ring buffer, and the head and tail indices are `volatile uint32_t`, no critical section or atomic operation is needed. The ISR only writes `rx_head` and reads `rx_tail`. The main loop only writes `rx_tail` and reads `rx_head`. Since 32-bit aligned writes are atomic on Cortex-M4, there is no race condition.
</Callout>

## Practical ISR Timing Measurements

To validate ISR design decisions, measure actual ISR duration using GPIO toggling and DWT:

```c
// Instrumented ISR
void USART1_IRQHandler(void) {
    GPIOA->BSRR = GPIO_BSRR_BS1;  // Set PA1 high (ISR entry marker)
    uint32_t t_start = DWT->CYCCNT;

    if (USART1->SR & USART_SR_RXNE) {
        uint32_t next = (rx_head + 1) & (RX_BUF_SIZE - 1);
        if (next != rx_tail) {
            rx_buf[rx_head] = USART1->DR;
            rx_head = next;
        }
    }

    uint32_t t_end = DWT->CYCCNT;
    isr_duration = t_end - t_start;  // Typically 15-25 cycles

    GPIOA->BSRR = GPIO_BSRR_BR1;  // Clear PA1 (ISR exit marker)
}
```

## Conclusion

The Cortex-M4 NVIC is a sophisticated piece of hardware that, when properly understood and configured, delivers deterministic interrupt response with minimal overhead. The priority grouping system gives you fine-grained control over preemption behavior. Tail-chaining and late-arrival optimizations are automatic and ensure back-to-back interrupts are handled efficiently. The 12-cycle hardware stacking provides a hard lower bound on interrupt latency that no software overhead can improve upon -- but software can easily make it worse through poorly designed critical sections.

The practical principles are straightforward: use NVIC_PRIORITYGROUP_4 for maximum preemption granularity, keep ISR bodies under 50 cycles (defer heavy work to the main loop), use BASEPRI instead of PRIMASK when you need selective masking, minimize critical section duration, and prefer lock-free patterns (single-producer/single-consumer ring buffers, atomic flag variables) over interrupt disabling wherever possible. Measure actual latency with DWT cycle counters to validate your design decisions. And always consider whether DMA can replace an interrupt-driven transfer entirely.
