Deep sleep on the ESP32 reduces current from approximately 240 mA (active with WiFi) to 10 uA or less -- a 24,000x reduction. The cost is total loss of main SRAM: all 520 KB of variables, stack frames, heap allocations, and peripheral registers are gone when you wake up. The chip reboots from scratch. The only bridge between sleep cycles is the RTC memory: 16 KB of SRAM that stays powered during deep sleep, allowing you to carry state, buffer sensor data, cache WiFi credentials, and even execute code immediately after wakeup before the main bootloader runs. This guide covers every aspect of RTC memory on the ESP32 -- its physical architecture, the software APIs for using it, practical patterns for real-world applications, wake stub programming, ULP coprocessor data sharing, and measured timing data for each phase of the wake-process-sleep cycle.

The asymmetry between the two banks is critical for power optimization. RTC FAST memory draws approximately 100 uA during deep sleep retention -- a dominant contributor to sleep current. Unless you are using wake stubs or storing time-critical data, you should disable RTC FAST memory retention using `esp_sleep_pd_config(ESP_PD_DOMAIN_RTC_FAST_MEM, ESP_PD_OPTION_OFF)`, saving 100+ uA.

RTC SLOW memory at approximately 2 uA is a bargain. Keeping it powered provides 8 KB of persistent storage across deep sleep for roughly the same current as a single floating GPIO.

## RTC_DATA_ATTR vs RTC_NOINIT_ATTR

ESP-IDF provides two attributes for placing variables in RTC memory:

### RTC_DATA_ATTR

Variables marked with `RTC_DATA_ATTR` are placed in the `.rtc.data` section of RTC SLOW memory. They are initialized to their declared value on the first boot (cold boot after power cycle or firmware flash), and they retain their value across deep sleep cycles.

```c
#include "esp_attr.h"

// Initialized on first boot, preserved across deep sleep
RTC_DATA_ATTR static int boot_count = 0;
RTC_DATA_ATTR static float last_temperature = 0.0f;
RTC_DATA_ATTR static uint32_t total_readings = 0;

// Arrays work too -- this allocates 400 bytes in RTC SLOW memory
RTC_DATA_ATTR static uint16_t sensor_buffer[200];
```

The initialization behavior: on a true cold boot (power cycle or firmware flash), `boot_count` is set to 0. After the first deep sleep cycle, when the chip wakes and re-executes `app_main()`, the initialization code does NOT run again -- `boot_count` retains whatever value it had when the chip entered deep sleep. This is because the linker places the initial values in flash, and the startup code only copies them to RTC memory when the RTC memory contents are invalid (first boot).

<Callout type="warning" title="Declaration Location Matters">
  `RTC_DATA_ATTR` variables must be declared at file scope (global or static). They cannot be declared inside a function as local variables. The compiler needs to place them in a specific linker section, which requires a fixed address determined at link time.
</Callout>

### RTC_NOINIT_ATTR

Variables marked with `RTC_NOINIT_ATTR` are placed in the `.rtc.noinit` section. Unlike `RTC_DATA_ATTR`, they are NEVER initialized by the startup code -- not even on cold boot. Their contents are whatever random values the RTC memory happens to contain after power-on.

```c
// NEVER initialized by startup code -- random value after power-on
RTC_NOINIT_ATTR static uint32_t magic_marker;

void app_main() {
    if (magic_marker != 0xDEADBEEF) {
        // First boot (or power loss) -- initialize everything
        magic_marker = 0xDEADBEEF;
        boot_count = 0;
        memset(sensor_buffer, 0, sizeof(sensor_buffer));
    } else {
        // Wake from deep sleep -- state is intact
        boot_count++;
    }
}
```

The key difference is behavior after a **software reset** (e.g., `esp_restart()`). `RTC_DATA_ATTR` variables are re-initialized because the startup code treats software reset as a fresh boot. `RTC_NOINIT_ATTR` variables survive software resets because the noinit section is never written by startup code. This makes `RTC_NOINIT_ATTR` ideal for magic markers that distinguish cold boot from warm wake.

<Callout type="tip" title="Use Both Together">
  The best practice is to use `RTC_NOINIT_ATTR` for a magic marker and `RTC_DATA_ATTR` for all other persistent data. The magic marker tells you whether the `RTC_DATA_ATTR` contents are valid (deep sleep wake) or stale (cold boot). On cold boot, reinitialize everything. On deep sleep wake, trust the preserved values.
</Callout>

### Data Compression Strategies

When 8 KB is not enough, compression techniques can double or triple your effective capacity:

**1. Fixed-point representation**: Use `uint16_t` instead of `float` for sensor data. A temperature range of -40.00 to 125.00 deg C with 0.01 resolution fits perfectly in 16 bits:

```c
// float: 4 bytes per reading, range unlimited
// uint16_t: 2 bytes per reading, range -40.00 to 615.35 deg C
static inline uint16_t float_to_fixed(float value) {
    return (uint16_t)((value + 40.0f) * 100.0f);
}

static inline float fixed_to_float(uint16_t fixed) {
    return ((float)fixed / 100.0f) - 40.0f;
}

// Doubles buffer capacity: 2000 readings instead of 1000
RTC_DATA_ATTR static uint16_t temp_buffer[2000];  // 4000 bytes
```

**2. Delta encoding**: If consecutive readings are similar (temperature changes slowly), store deltas instead of absolute values:

```c
RTC_DATA_ATTR static uint16_t baseline_temp;       // First reading (absolute)
RTC_DATA_ATTR static int8_t temp_deltas[4000];     // 4000 bytes = 4000 readings
RTC_DATA_ATTR static uint16_t delta_index;

void store_reading(float temperature) {
    uint16_t fixed = float_to_fixed(temperature);

    if (delta_index == 0) {
        baseline_temp = fixed;
        temp_deltas[0] = 0;
    } else {
        int diff = (int)fixed - (int)float_to_fixed(
            fixed_to_float(baseline_temp) +
            accumulated_delta(temp_deltas, delta_index));

        // Clamp to int8_t range (-1.28 to +1.27 deg C change per sample)
        if (diff > 127) diff = 127;
        if (diff < -128) diff = -128;
        temp_deltas[delta_index] = (int8_t)diff;
    }
    delta_index++;
}
```

With delta encoding using `int8_t`, you get 4x the capacity compared to `float` storage (4000 readings in the same 4000 bytes that would hold only 1000 floats).

**3. Bit-packing**: For sensors with limited resolution (e.g., 12-bit ADC), pack readings tightly:

```c
// 12-bit ADC readings: 3 readings per 4 bytes (instead of 2 bytes each)
// Capacity: 8 KB / 1.33 bytes per reading = ~6000 readings
RTC_DATA_ATTR static uint8_t packed_buffer[6000];
RTC_DATA_ATTR static uint16_t packed_count;

void store_12bit(uint16_t reading) {
    uint16_t byte_pos = (packed_count * 12) / 8;
    uint8_t bit_offset = (packed_count * 12) % 8;

    if (bit_offset == 0) {
        packed_buffer[byte_pos] = (reading >> 4) & 0xFF;
        packed_buffer[byte_pos + 1] = (packed_buffer[byte_pos + 1] & 0x0F)
                                     | ((reading & 0x0F) << 4);
    } else {
        packed_buffer[byte_pos] = (packed_buffer[byte_pos] & 0xF0)
                                 | ((reading >> 8) & 0x0F);
        packed_buffer[byte_pos + 1] = reading & 0xFF;
    }
    packed_count++;
}
```

The pattern: check RTC variables at the very beginning of `app_main()` to determine what work is actually needed on this wake cycle.

```c
#define BATCH_SIZE 60
#define SLEEP_US (60ULL * 1000000ULL)

RTC_NOINIT_ATTR static uint32_t magic;
RTC_DATA_ATTR static uint16_t readings[BATCH_SIZE];
RTC_DATA_ATTR static uint8_t idx;
RTC_DATA_ATTR static uint8_t wifi_ch;
RTC_DATA_ATTR static uint8_t wifi_bssid[6];
RTC_DATA_ATTR static bool wifi_cached;

void app_main() {
    // Phase 1: Determine boot context (0 ms -- just reading RTC memory)
    bool cold_boot = (magic != 0xCAFEBABE);
    if (cold_boot) {
        magic = 0xCAFEBABE;
        idx = 0;
        wifi_cached = false;
    }

    // Phase 2: Read sensor (50+20 ms -- always needed)
    sensor_init();
    readings[idx] = read_sensor_fixed_point();
    idx++;

    // Phase 3: Transmit ONLY when buffer is full (skip on 59 of 60 wakes)
    if (idx >= BATCH_SIZE) {
        // This wake needs WiFi -- initialize it
        wifi_init();
        if (wifi_cached) {
            connect_cached(wifi_ch, wifi_bssid);  // ~200 ms
        } else {
            connect_full_scan();  // ~3000 ms (first time only)
        }
        transmit_batch(readings, BATCH_SIZE);

        // Update cache for next WiFi wake
        wifi_ch = get_connected_channel();
        get_connected_bssid(wifi_bssid);
        wifi_cached = true;

        idx = 0;  // Reset buffer
    }
    // Phase 3 is SKIPPED on 59 of 60 wakes -- saving 200-3000 ms each time

    // Phase 4: Sleep
    esp_sleep_enable_timer_wakeup(SLEEP_US);
    esp_sleep_pd_config(ESP_PD_DOMAIN_RTC_FAST_MEM, ESP_PD_OPTION_OFF);
    esp_deep_sleep_start();
}
```

## Wake Stub Programming

A wake stub is a small piece of code that executes immediately after the ESP32 wakes from deep sleep, before the bootloader and main application run. It executes from RTC FAST memory at the RTC clock speed, providing an ultra-fast path for simple decisions like "should I actually wake up fully, or go right back to sleep?"

  Wake stubs run in a severely restricted environment. ### Wake Stub Example: Conditional Wake

A common pattern is a wake stub that checks an RTC counter and only allows a full wake every N cycles. The other cycles immediately return to deep sleep, never paying the cost of the full boot:

```c
#include "esp_sleep.h"
#include "esp_attr.h"
#include "soc/rtc_cntl_reg.h"
#include "soc/timer_group_reg.h"

// Place counter in RTC SLOW memory (accessible by both stub and app)
RTC_DATA_ATTR static int stub_wake_count = 0;
RTC_DATA_ATTR static int stub_full_boot_interval = 10;

// The wake stub itself -- placed in RTC FAST memory
void RTC_IRAM_ATTR esp_wake_deep_sleep(void) {
    // This function runs BEFORE the bootloader
    // Increment counter
    stub_wake_count++;

    if (stub_wake_count < stub_full_boot_interval) {
        // Go back to sleep immediately -- never boot the main app
        // This saves ~200 ms of boot time and ~15 mAs of energy

        // Re-arm the timer wakeup
        // (Direct register writes -- no driver API available)
        SET_PERI_REG_MASK(RTC_CNTL_STATE0_REG, RTC_CNTL_SLEEP_EN);

        // Feed the watchdog to prevent reset
        WRITE_PERI_REG(TIMG_WDTFEED_REG(0), 1);

        // Enter deep sleep again
        while (true) {
            // Wait for sleep to take effect
        }
    }

    // stub_wake_count >= stub_full_boot_interval
    // Allow normal boot -- the default stub behavior is to
    // continue to the bootloader, which eventually calls app_main()
    stub_wake_count = 0;  // Reset for next cycle
}
```

The wake stub is exceptionally useful for high-frequency sampling with the ULP coprocessor. The ULP collects ADC readings during deep sleep, and the wake stub checks whether a threshold was exceeded. If not, it returns to sleep in under 1 ms. If yes, it allows a full boot to process and transmit the data.

### Advanced Wake Stub: Reading ULP Data

```c
#include "esp_attr.h"
#include "soc/rtc_cntl_reg.h"
#include "soc/sens_reg.h"

// Shared variable: ULP writes, wake stub reads
RTC_DATA_ATTR static volatile uint16_t ulp_last_reading = 0;
RTC_DATA_ATTR static uint16_t threshold_high = 3000;
RTC_DATA_ATTR static uint16_t threshold_low = 500;

void RTC_IRAM_ATTR esp_wake_deep_sleep(void) {
    // Check if ULP triggered the wake
    uint32_t cause = REG_GET_FIELD(RTC_CNTL_WAKEUP_STATE_REG,
                                    RTC_CNTL_WAKEUP_CAUSE);

    if (cause & RTC_CNTL_ULP_WAKEUP) {
        // ULP detected something -- check the reading
        uint16_t reading = ulp_last_reading & 0x0FFF;  // 12-bit ADC value

        if (reading > threshold_high || reading < threshold_low) {
            // Threshold exceeded -- allow full boot for processing
            return;  // Continue to bootloader
        }
    }

    // False alarm or periodic timer wake -- go back to sleep
    // Re-enable timer wakeup
    SET_PERI_REG_MASK(RTC_CNTL_STATE0_REG, RTC_CNTL_SLEEP_EN);

    while (true) {
        // Sleep takes effect
    }
}
```

## ULP Coprocessor Data Sharing via RTC Memory

The ULP (Ultra-Low-Power) coprocessor is a simple processor that runs during deep sleep, consuming approximately 150 uA. It can read ADC channels, GPIOs, and temperature sensor values, storing results in RTC SLOW memory where the main CPU can read them after wakeup.

The ULP shares RTC SLOW memory with the main application. Variables are accessed using a specific naming convention:

```c
// .bss section variables are placed in RTC SLOW memory
// and accessible from the main CPU as ulp_variable_name

    .bss
    .global sample_count
sample_count:
    .long 0

    .global adc_reading
adc_reading:
    .long 0

    .global threshold
threshold:
    .long 0

    .text
    .global entry
entry:
    // Read ADC1 channel 6 (GPIO 34)
    adc r0, 0, 6       // r0 = ADC reading

    // Store in shared memory
    move r1, adc_reading
    st r0, r1, 0

    // Increment counter
    move r1, sample_count
    ld r2, r1, 0
    add r2, r2, 1
    st r2, r1, 0

    // Check threshold
    move r1, threshold
    ld r1, r1, 0
    sub r0, r0, r1
    jump wake_cpu, ov   // Wake main CPU if reading > threshold

    // Go back to sleep
    halt

wake_cpu:
    wake                // Trigger main CPU wake from deep sleep
    halt
```

```c
// In the main application (app_main.c):
#include "esp32/ulp.h"
#include "ulp_main.h"  // Generated header from ULP assembly

// Access ULP shared variables from main CPU
void read_ulp_data() {
    // ULP variables are accessed through the ulp_ prefix
    // They are 32-bit values but only the lower 16 bits are valid
    uint16_t adc_value = (uint16_t)(ulp_adc_reading & 0xFFFF);
    uint32_t count = ulp_sample_count & 0xFFFF;

    printf("ULP collected %lu samples, last ADC: %u\n", count, adc_value);
}

void setup_ulp() {
    // Load ULP program into RTC SLOW memory
    ESP_ERROR_CHECK(ulp_load_binary(
        0,                              // Load offset
        ulp_main_bin_start,             // Program binary start
        (ulp_main_bin_end - ulp_main_bin_start) / sizeof(uint32_t)
    ));

    // Set threshold (shared variable)
    ulp_threshold = 2500;  // Wake CPU if ADC > 2500

    // Configure ULP wakeup period (how often ULP runs)
    ulp_set_wakeup_period(0, 100 * 1000);  // Every 100 ms

    // Configure ADC for ULP
    adc1_config_channel_atten(ADC1_CHANNEL_6, ADC_ATTEN_DB_11);
    adc1_ulp_enable();

    // Start ULP program
    ESP_ERROR_CHECK(ulp_run(&ulp_entry - RTC_SLOW_MEM));

    // Enable ULP as wakeup source
    esp_sleep_enable_ulp_wakeup();

    // Also enable timer wakeup as a fallback
    esp_sleep_enable_timer_wakeup(3600 * 1000000ULL);  // 1 hour max

    // Enter deep sleep -- ULP continues running
    esp_deep_sleep_start();
}
```

<Callout type="info" title="ULP Memory Budget">
  The ULP program binary and shared variables consume RTC SLOW memory. A typical ULP program is 200-500 bytes. Shared variables add another 50-200 bytes. This leaves approximately 7.5 KB for `RTC_DATA_ATTR` application variables. Monitor your RTC memory usage with `idf.py size` to avoid overflows.
</Callout>

The ULP approach breaks even with main CPU wake-per-sample at approximately 1 sample per 100 seconds. Below that frequency (e.g., once per 5 minutes), it is cheaper to wake the main CPU. Above that frequency (e.g., once per second), the ULP saves enormous energy:

$I_{main\_wake} = \frac{E_{boot} + E_{sample}}{T_{sample}} + I_{sleep}$

$I_{ULP} = I_{sleep} + I_{ULP\_active}$

For $T_{sample} = 1$ second:

$I_{main\_wake} = \frac{15.7\ \text{mAs} + 0.3\ \text{mAs}}{1\ \text{s}} + 0.010\ \text{mA} = 16.01\ \text{mA}$

$I_{ULP} = 0.010 + 0.150 = 0.160\ \text{mA}$

The ULP uses 100x less current for 1 Hz sampling.

## Flash vs RTC Memory: When to Use Which

<Benchmark
  title="Persistence Strategies Comparison"
  columns={["Method", "Size", "Write Speed", "Write Endurance", "Sleep Cost", "Survives Power Loss"]}
  rows={[
    { values: ["RTC SLOW Memory", "8 KB", "Instant (~10 ns)", "Unlimited", "+2 uA", "No"], highlight: true },
    { values: ["RTC FAST Memory", "8 KB", "Instant (~10 ns)", "Unlimited", "+100 uA", "No"], highlight: false },
    { values: ["NVS (flash)", "Configurable (4-64 KB)", "~5-15 ms per key", "~100K writes/sector", "+0 uA", "Yes"], highlight: false },
    { values: ["SPIFFS/LittleFS", "Up to MB", "~10-100 ms per file", "~100K writes/sector", "+0 uA", "Yes"], highlight: false },
    { values: ["External EEPROM (I2C)", "4-256 KB", "~5-10 ms per page", "~1M writes", "Varies", "Yes"], highlight: false }
  ]}
  notes="Write endurance for flash is per sector, with wear leveling extending effective life. RTC memory has zero wear concern."
/>

**Use RTC memory for**:
- Data that changes every wake cycle (sensor readings, counters, timestamps)
- WiFi connection cache (BSSID, channel)
- Application state machine variables
- Anything that would cause flash wear if written every 60 seconds

**Use NVS flash for**:
- Configuration parameters (changed rarely, e.g., by user or OTA)
- Calibration data (written once at factory, read every boot)
- Data that must survive power loss (battery removal, brownout)

**Use filesystem (SPIFFS/LittleFS) for**:
- Large data that exceeds 8 KB
- Log files that must survive power loss
- Data that needs complex structure (JSON, CSV)

<Callout type="danger" title="Flash Wear Warning">
  NVS writes to flash have a finite endurance of approximately 100,000 writes per sector. At one write per minute, a single sector would fail in approximately 69 days. ESP-IDF NVS includes wear leveling that distributes writes across multiple sectors, but for data changing every wake cycle, RTC memory is the only appropriate choice. Never use NVS for boot counters or sensor buffers that update every minute.
</Callout>

### Hybrid Pattern: RTC Buffer with Flash Backup

For maximum reliability, use RTC memory as a fast write buffer and periodically flush to flash:

```c
#define RTC_BUFFER_SIZE 200
#define FLASH_FLUSH_INTERVAL 50  // Flush every 50 readings

RTC_NOINIT_ATTR static uint32_t rtc_magic;
RTC_DATA_ATTR static uint16_t rtc_buffer[RTC_BUFFER_SIZE];
RTC_DATA_ATTR static uint16_t rtc_index;
RTC_DATA_ATTR static uint32_t flash_total_readings;

void app_main() {
    bool cold_boot = (rtc_magic != 0xBEEFCAFE);
    if (cold_boot) {
        rtc_magic = 0xBEEFCAFE;
        rtc_index = 0;
        // Read flash_total_readings from NVS
        nvs_get_u32(nvs_handle, "total", &flash_total_readings);
    }

    // Store reading in RTC (instant, no wear)
    rtc_buffer[rtc_index++] = read_sensor_fixed_point();

    // Periodically flush to flash for power-loss protection
    if (rtc_index % FLASH_FLUSH_INTERVAL == 0) {
        // Write batch to flash filesystem (takes ~50 ms)
        append_to_log_file(rtc_buffer + rtc_index - FLASH_FLUSH_INTERVAL,
                          FLASH_FLUSH_INTERVAL);
        flash_total_readings += FLASH_FLUSH_INTERVAL;
        nvs_set_u32(nvs_handle, "total", flash_total_readings);
        nvs_commit(nvs_handle);
    }

    // Transmit and reset when buffer is full
    if (rtc_index >= RTC_BUFFER_SIZE) {
        wifi_transmit(rtc_buffer, RTC_BUFFER_SIZE);
        rtc_index = 0;
    }

    esp_sleep_enable_timer_wakeup(60 * 1000000ULL);
    esp_deep_sleep_start();
}
```

This pattern gives you the speed and zero-wear of RTC memory for normal operations, while ensuring that at most 50 readings (less than 1 hour of data) are lost if the battery dies unexpectedly.

## Monitoring RTC Memory Usage

To verify your RTC memory allocation fits within 8 KB, use the ESP-IDF size analysis tool:

```bash
idf.py size-components

# Look for these sections in the output:
# .rtc.data      -- RTC_DATA_ATTR variables
# .rtc.bss       -- RTC zero-initialized variables
# .rtc.noinit    -- RTC_NOINIT_ATTR variables
# .rtc.text      -- Wake stub code (RTC FAST memory)
# .rtc.force_fast -- Explicitly placed in RTC FAST memory
```

You can also check at runtime:

```c
extern uint32_t _rtc_slow_length;
extern uint32_t _rtc_fast_length;

void check_rtc_usage() {
    printf("RTC SLOW used: %lu / 8192 bytes\n",
           (uint32_t)&_rtc_slow_length);
    printf("RTC FAST used: %lu / 8192 bytes\n",
           (uint32_t)&_rtc_fast_length);
}
```

<Callout type="warning" title="Silent Overflow">
  If your RTC variables exceed 8 KB, the linker will typically produce a warning but may still generate a binary. The overflow corrupts adjacent memory regions and causes unpredictable behavior -- often manifesting as random reboots or corrupted sensor data after deep sleep. Always check the linker map file for RTC section sizes.
</Callout>

The 16 KB of RTC memory (8 KB fast + 8 KB slow) is a scarce resource that demands careful budgeting. With fixed-point encoding and delta compression, 8 KB can hold thousands of sensor readings -- enough for hours or days of data between WiFi transmissions. The combination of RTC buffering, cached WiFi, and conditional boot optimization routinely achieves 10-100x power savings compared to naive implementations that connect WiFi on every wake.
