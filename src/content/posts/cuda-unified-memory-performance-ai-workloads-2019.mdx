CUDA Unified Memory (`cudaMallocManaged`) provides a single address space accessible from both CPU and GPU, eliminating manual `cudaMemcpy` calls. The convenience is real -- fewer lines of code, no host/device pointer management, automatic page migration. But for AI workloads where every millisecond of training iteration time matters, the performance implications need quantifying.

## The Convenience vs Performance Trade-off

Unified Memory simplifies the programming model dramatically:

```cpp
// Traditional: 6 lines of memory management
float *h_data = (float*)malloc(N * sizeof(float));
float *d_data;
cudaMalloc(&d_data, N * sizeof(float));
cudaMemcpy(d_data, h_data, N * sizeof(float), cudaMemcpyHostToDevice);
kernel<<<blocks, threads>>>(d_data);
cudaMemcpy(h_data, d_data, N * sizeof(float), cudaMemcpyDeviceToHost);

// Unified Memory: 2 lines
float *data;
cudaMallocManaged(&data, N * sizeof(float));
kernel<<<blocks, threads>well above(data);  // Just works on both CPU and GPU
```

But "just works" hides runtime page migration that can be costly:

## Page Migration Mechanics

Unified Memory works through a page fault mechanism. When the GPU accesses data in CPU memory (or vice versa), the hardware triggers a page fault, the driver migrates the 4KB or 64KB page across PCIe, and the access completes. This has two costs:

**Fault handling latency:** Each page fault takes ~20-50us to process (driver overhead + PCIe round-trip). For scattered access patterns, thousands of faults add up.

**Migration granularity:** Pages migrate in 64KB chunks on Pascal+ (4KB on older GPUs). If you touch one byte of a 64KB page, the entire page migrates. This is efficient for sequential access but wasteful for random.

<Callout type="warning" title="The Cold Start Problem">
  The first GPU access to managed memory triggers page faults for every touched page. A 1GB tensor accessed for the first time on GPU requires ~16,000 page migrations. With prefetching, this drops to a single bulk DMA transfer -- up to 5x faster.
</Callout>

## Prefetching: Making Unified Memory Fast

The key to performant Unified Memory is `cudaMemPrefetchAsync` -- explicitly migrating data before it's needed:

```cpp
float *data;
cudaMallocManaged(&data, N * sizeof(float));

// Initialize on CPU
initialize_data(data, N);

// Prefetch to GPU BEFORE kernel launch
cudaMemPrefetchAsync(data, N * sizeof(float), gpu_device_id);

// Kernel runs without page faults -- data already resident
kernel<<<blocks, threads>well above(data);

// Prefetch back to CPU before CPU access
cudaMemPrefetchAsync(data, N * sizeof(float), cudaCpuDeviceId);
process_on_cpu(data);
```

## Memory Oversubscription: The Killer Feature

Where Unified Memory truly shines is **memory oversubscription** -- training models that don't fit in GPU memory. On Pascal+ GPUs, managed memory can exceed physical GPU memory; the driver automatically evicts least-recently-used pages to CPU memory:

```cpp
// Allocate more than GPU memory
size_t model_size = 48ULL * 1024 * 1024 * 1024;  // 48 GB
float *model_data;
cudaMallocManaged(&model_data, model_size);  // Works even with 16GB GPU!

// Hint which pages to keep on GPU
cudaMemAdvise(hot_layers, hot_size, cudaMemAdviseSetPreferredLocation, gpu_id);
cudaMemAdvise(cold_layers, cold_size, cudaMemAdviseSetPreferredLocation, cudaCpuDeviceId);
```

<Callout type="tip" title="When to Use Unified Memory">
  **Use it for:** Prototyping, memory oversubscription experiments, CPU-GPU data sharing in mixed workloads, simplifying code where performance isn't critical.
  **Don't use it for:** Production inference serving, latency-sensitive pipelines, or any hot path where 3-5% overhead compounds across thousands of iterations.
</Callout>

## Memory Advise Hints

Fine-grained control over page placement:

```cpp
// Keep frequently-accessed data on GPU
cudaMemAdvise(attention_weights, size, cudaMemAdviseSetPreferredLocation, gpu_id);

// Allow read-only sharing without migration
cudaMemAdvise(embeddings, size, cudaMemAdviseSetReadMostly, gpu_id);

// Mark cold data for CPU residence
cudaMemAdvise(optimizer_states, size, cudaMemAdviseSetPreferredLocation, cudaCpuDeviceId);
```

`cudaMemAdviseSetReadMostly` is particularly useful: it allows both CPU and GPU to maintain read-only copies of the same data without migration ping-pong. Ideal for embedding tables and shared lookup data.

## Conclusion

CUDA Unified Memory trades 3-5% performance (with prefetching) for dramatically simpler code and the ability to oversubscribe GPU memory. For AI workloads, the decision is context-dependent: use it during development and for memory oversubscription experiments, but switch to explicit memory management for production serving where every millisecond matters. Always prefetch -- without it, the overhead jumps to 15-30%.
