The CPU cache hierarchy is the most important hardware feature that software engineers don't understand well enough. A cache miss to main memory costs 100-300 cycles -- enough time for the CPU to have completed hundreds of useful operations. Understanding cache behavior is the difference between code that runs at memory speed and code that runs at compute speed.

## Access Patterns: Sequential vs Random

The cache hierarchy rewards **spatial locality** (accessing nearby addresses) and **temporal locality** (accessing the same address repeatedly). Sequential access is fast because hardware prefetchers detect the pattern and load data before you need it:

Random DRAM access achieves only 1.5 GB/s -- **17x slower** than sequential DRAM access and **133x slower** than L1 hits. This single measurement explains why hash tables are slow for large datasets, why linked lists are terrible on modern hardware, and why column-oriented databases outperform row-oriented ones for analytics.

<Callout type="perf" title="The Most Important Number">
  If your hot loop's working set fits in L1 (32KB), you get ~200 GB/s bandwidth. If it spills to DRAM, you get ~25 GB/s. That's an 8x performance difference from the same code -- the only change is data size. Always know your working set size.
</Callout>

## Cache Line Effects

Caches operate on 64-byte cache lines, not individual bytes. Two consequences matter:

**Spatial prefetch benefit:** Accessing one byte of a struct loads the entire 64-byte line. If you access adjacent fields, they're already cached -- effectively free.

**False sharing penalty:** If two threads modify different variables that share a cache line, the hardware coherency protocol bounces the line between cores, serializing access. This can cause 10-50x slowdown for what looks like independent operations.

The fix is straightforward: pad shared-but-independent data to cache line boundaries.

```cpp
// Bad: counters share cache lines
struct Counters {
    uint64_t thread_0;  // offset 0
    uint64_t thread_1;  // offset 8 -- same cache line!
    uint64_t thread_2;  // offset 16
    uint64_t thread_3;  // offset 24
};

// Good: each counter on its own cache line
struct alignas(64) PaddedCounter {
    uint64_t value;
};
PaddedCounter counters[4];  // Each 64-byte aligned
```

## Practical Optimization Checklist

**Know your working set size.** If it's under 32KB, you're in L1 territory. Under 256KB, L2. Under ~20MB, L3. Use `perf stat -e cache-misses` to measure.

**Access data sequentially.** Arrays iterated in order are ~10x faster than pointer chasing. Prefer `std::vector` over `std::list`, SoA over AoS.

**Avoid false sharing.** Align per-thread data to 64-byte boundaries. Use `alignas(64)` or padding.

**Block/tile your algorithms.** Process data in cache-sized chunks. Matrix multiplication, for example, gets 10x faster with L1-sized tiles.

**Prefetch judiciously.** `__builtin_prefetch` helps for predictable but non-sequential patterns (like strided access). Don't prefetch for sequential access -- the hardware already does it.

<Callout type="tip" title="Measuring Cache Performance">
  Use `perf stat -e L1-dcache-load-misses,LLC-load-misses` to measure cache miss rates. An L1 miss rate above 5% or L3 miss rate above 1% usually indicates optimization opportunity. `perf c2c` is invaluable for detecting false sharing.
</Callout>

## Conclusion

The CPU cache hierarchy is a 200x performance multiplier hiding in plain sight. Code that respects cache behavior (sequential access, small working sets, no false sharing) runs at L1 speed. Code that ignores it runs at DRAM speed -- on the same hardware, with the same instructions. The first step to optimization is always: measure your cache miss rate.
