Loading a 70B parameter model in FP16 means reading 140 GB from disk — 70 seconds on a fast NVMe SSD (2 GB/s), 23 minutes on a spinning disk. Memory mapping (`mmap`) and modern model formats (safetensors) can reduce perceived loading time to near-zero by lazily loading weights only when they're first accessed, and by eliminating the unsafe `pickle` deserialization that dominates standard PyTorch loading.

## How mmap Works for Model Loading

Memory mapping creates a virtual address mapping from the model file directly into the process's address space. No data is read from disk until a page is actually accessed:

```python
import mmap
import numpy as np

def load_model_mmap(filepath):
    """Load model weights via memory mapping — near-instant startup."""
    with open(filepath, 'rb') as f:
        # Create memory mapping — doesn't read any data yet
        mm = mmap.mmap(f.fileno(), 0, access=mmap.ACCESS_READ)
        # Create numpy array backed by the mmap — still no disk reads
        weights = np.frombuffer(mm, dtype=np.float16)
        # First access to weights[i] triggers a page fault → disk read
    return weights  # "Loaded" instantly, data read lazily on demand
```

The page fault mechanism is key: when the model's forward pass first accesses a weight tensor, the OS kernel reads the corresponding 4 KB page from disk into physical memory. This happens transparently — the application sees a normal memory access, but the OS handles the I/O in the background.

<Callout type="perf" title="Why mmap Is Fast for First Token">
  A 70B model has ~35,000 parameter tensors across 80 layers. The first inference step only touches layers 1-80 sequentially. With mmap, only the pages needed for the current layer are loaded — the OS prefetcher detects the sequential access pattern and reads ahead. Layer 1's weights are loaded from disk while layer 0's computation runs on GPU.
</Callout>

  Once mmap eliminates the upfront disk read and safetensors eliminates pickle, the bottleneck shifts to PCIe transfer (CPU→GPU). At PCIe Gen4 x16 (32 GB/s), transferring 140 GB takes ~4.4 seconds. ## Advanced: Tensor Parallelism Sharded Loading

For multi-GPU deployment with tensor parallelism, each GPU only needs 1/N of each weight tensor:

```python
from safetensors.torch import load_file

def load_sharded_tp(filepath, tp_rank, tp_world):
    """Load only this GPU's shard of each weight tensor."""
    state_dict = load_file(filepath, device="cpu")  # mmap

    sharded = {}
    for name, tensor in state_dict.items():
        if "q_proj" in name or "k_proj" in name or "v_proj" in name:
            # Column-parallel: split along output dim
            chunk_size = tensor.shape[0] // tp_world
            sharded[name] = tensor[tp_rank * chunk_size:(tp_rank + 1) * chunk_size]
        elif "down_proj" in name or "o_proj" in name:
            # Row-parallel: split along input dim
            chunk_size = tensor.shape[1] // tp_world
            sharded[name] = tensor[:, tp_rank * chunk_size:(tp_rank + 1) * chunk_size]
        else:
            sharded[name] = tensor  # Replicated (norms, embeddings)

    return sharded
```

With mmap, each GPU only faults in the pages it actually slices — 1/N of the parameter data. This means TP=8 loading reads only 1/8 of the file per GPU, and reads happen in parallel across GPUs.

## Conclusion

Memory mapping transforms model loading from a sequential read-everything-first operation to a lazy, on-demand process. Combined with safetensors (eliminating pickle deserialization), time-to-first-token drops from 85 seconds to under 1 second for a 70B model. The remaining bottleneck is CPU→GPU PCIe transfer (~4.4 seconds for 140 GB). For production serving, pre-load models to GPU memory at startup; for development/debugging, mmap provides near-instant model switching. Always use safetensors format — it's safer (no code execution), faster (mmap-compatible), and has become the standard across HuggingFace, vLLM, and llama.cpp.
