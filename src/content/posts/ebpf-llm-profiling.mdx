GPU profilers like Nsight Compute and Nsight Systems tell you what happens on the GPU: kernel execution times, memory bandwidth utilization, occupancy, warp stalls. They are excellent tools. But LLM inference is a full-system problem. The path from an HTTP request to a generated token touches the network stack, the Python runtime, the CUDA driver, PCIe bus, GPU memory, and often multiple GPUs communicating over NVLink or InfiniBand. GPU profilers are blind to everything except the GPU. eBPF sees everything else.

eBPF (extended Berkeley Packet Filter) is a technology built into the Linux kernel that allows you to attach small, verified programs to nearly any kernel or user-space function. These programs execute in a sandboxed virtual machine inside the kernel, with strict safety guarantees: they cannot crash the system, cannot loop infinitely, and add overhead measured in nanoseconds per event. This makes eBPF uniquely suited for production profiling -- the kind of profiling you leave running continuously, not the kind you enable for five minutes during a maintenance window.

This post covers how to use eBPF to profile every non-GPU aspect of LLM inference: CUDA driver interactions, memory allocation patterns, CPU-side scheduling overhead, NCCL communication, and Python runtime behavior. Combined with GPU profiling tools, eBPF provides complete end-to-end observability from user request to generated token.

## Why eBPF for LLM Serving

Traditional profiling instruments the application. You modify source code to add timing calls, link against profiling libraries, or run under a profiler that intercepts function calls. This approach has three problems for LLM serving:

**Application modification is risky in production.** LLM serving stacks are complex -- vLLM, SGLang, TGI, and Triton Inference Server involve millions of lines of Python and C++ code, third-party CUDA kernels, and deep framework integrations. Adding instrumentation risks introducing bugs, changing timing behavior (probe effect), or breaking the carefully tuned interaction between components.

**Overhead is unpredictable.** Application-level profilers like `cProfile` or `py-spy` add 5-20% overhead depending on the workload. For a serving system targeting P99 latency SLAs, this overhead is unacceptable. Worse, the overhead is non-uniform: it disproportionately affects the fast path, making profiling data misleading.

**Visibility is limited to one layer.** The PyTorch profiler sees PyTorch operations. Nsight sees CUDA operations. Neither sees the operating system: page faults that stall CUDA allocations, context switches that delay kernel launches, TCP retransmissions that slow NCCL, or filesystem I/O that blocks model loading.

eBPF solves all three problems. It instruments the kernel, not the application. It adds under 1% overhead in practice. And it sees everything the kernel sees: system calls, scheduler decisions, network packets, memory management, and file I/O.

## How eBPF Works: A Quick Primer

Before diving into LLM-specific profiling, a brief overview of eBPF for readers unfamiliar with the technology.

eBPF programs are small pieces of code (typically 10-100 lines of restricted C) that attach to events in the kernel or user space. When the event fires, the eBPF program executes, collects data, and stores it in shared maps (hash tables, arrays, ring buffers) that user-space tools can read.

The key components:

- **Tracepoints**: Static instrumentation points compiled into the kernel. Examples: `tracepoint:syscalls:sys_enter_ioctl` fires on every `ioctl` system call entry. These are stable across kernel versions.
- **kprobes/kretprobes**: Dynamic instrumentation of any kernel function. Example: `kprobe:tcp_sendmsg` fires when the kernel sends a TCP message. These depend on kernel symbol names and may break across versions.
- **uprobes/uretprobes**: Dynamic instrumentation of user-space functions. Example: `uprobe:/usr/lib/libcuda.so:cuLaunchKernel` fires when CUDA launches a kernel. These are the primary mechanism for tracing GPU driver calls.
- **USDT (User Statically-Defined Tracing)**: Static tracepoints compiled into user-space applications. Some CUDA libraries include USDT probes for key operations.
- **Maps**: Shared data structures between eBPF programs and user space. Hash maps for per-key aggregation, histograms for latency distributions, ring buffers for event streaming.

Two main tools for writing and running eBPF programs:

- **bpftrace**: A high-level scripting language for one-liners and short scripts. Best for ad-hoc investigation.
- **BCC (BPF Compiler Collection)**: A Python/C framework for more complex programs. Best for production monitoring tools.

```bash
# Verify eBPF is available
uname -r  # Need kernel 4.15+ (5.x recommended)
bpftrace --version
# Requires root or CAP_BPF + CAP_PERFMON capabilities
```

## Tracing GPU Kernel Launches

Every CUDA kernel launch from PyTorch or a serving framework goes through the CUDA driver. The driver exposes functions like `cuLaunchKernel` that eBPF can intercept via uprobes. This lets you measure kernel launch latency from the CPU side -- the time between the application requesting a kernel launch and the driver completing the submission.

### Intercepting CUDA Driver Calls

```bash
#!/usr/bin/env bpftrace
// trace_cuda_launches.bt
// Trace CUDA kernel launch latency through the driver
// Usage: sudo bpftrace trace_cuda_launches.bt

uprobe:/usr/lib/x86_64-linux-gnu/libcuda.so:cuLaunchKernel
{
    @launch_start[tid] = nsecs;
    @launch_count = count();
}

uretprobe:/usr/lib/x86_64-linux-gnu/libcuda.so:cuLaunchKernel
/@launch_start[tid]/
{
    $latency_us = (nsecs - @launch_start[tid]) / 1000;
    @launch_latency_us = hist($latency_us);

    // Flag slow launches (> 50us suggests driver contention)
    if ($latency_us > 50) {
        printf("SLOW launch: pid=%d tid=%d latency=%dus\n",
               pid, tid, $latency_us);
    }

    delete(@launch_start[tid]);
}

// Also trace synchronization (performance killer)
uprobe:/usr/lib/x86_64-linux-gnu/libcuda.so:cuCtxSynchronize
{
    @sync_start[tid] = nsecs;
}

uretprobe:/usr/lib/x86_64-linux-gnu/libcuda.so:cuCtxSynchronize
/@sync_start[tid]/
{
    $sync_us = (nsecs - @sync_start[tid]) / 1000;
    @sync_latency_us = hist($sync_us);
    @sync_count = count();

    if ($sync_us > 1000) {
        printf("LONG sync: pid=%d latency=%dus\n", pid, $sync_us);
    }

    delete(@sync_start[tid]);
}

interval:s:10
{
    printf("\n--- 10s summary ---\n");
    printf("Kernel launches: %lld\n", @launch_count);
    printf("Syncs: %lld\n", @sync_count);
    print(@launch_latency_us);
}
```

### What Kernel Launch Latency Reveals

Kernel launch latency is the time the CPU spends in the CUDA driver submitting work to the GPU. In a healthy system, this is 5-15 microseconds per launch. Elevated latency indicates:

- **Driver lock contention**: Multiple threads launching kernels simultaneously. The CUDA driver uses internal locks that serialize launches. eBPF reveals this as bimodal latency -- most launches are fast, but some wait 100+ us for the lock.
- **GPU command queue full**: The GPU's hardware command queue has a fixed depth. When full, the driver blocks until a slot opens. This manifests as launch latency that correlates with GPU utilization.
- **Page table operations**: If the launch involves memory that needs page table updates (newly allocated memory, or memory that was freed and reallocated), the driver performs extra work.

  The `cuCtxSynchronize` function forces the CPU to wait until all pending GPU work completes. In a serving system, this should never be called during normal inference. Each sync wastes 100-5000 us of CPU time (and prevents kernel launch overlap). Find and eliminate every one.

### Detecting Scheduling Issues

The gap between kernel launches is just as important as the launch itself. eBPF can measure inter-launch intervals to detect CPU-side scheduling problems:

```bash
#!/usr/bin/env bpftrace
// inter_launch_gap.bt
// Measure time between consecutive kernel launches per thread
// Large gaps indicate CPU bottlenecks between GPU work

uprobe:/usr/lib/x86_64-linux-gnu/libcuda.so:cuLaunchKernel
{
    if (@prev_launch[tid]) {
        $gap_us = (nsecs - @prev_launch[tid]) / 1000;
        @inter_launch_gap_us = hist($gap_us);

        // Gaps > 100us between launches = CPU is the bottleneck
        if ($gap_us > 100) {
            @slow_gaps = count();
            printf("CPU gap: %dus between launches (tid=%d pid=%d)\n",
                   $gap_us, tid, pid);
        }
    }
    @prev_launch[tid] = nsecs;
}

interval:s:5
{
    printf("\n--- Inter-launch gap distribution ---\n");
    print(@inter_launch_gap_us);
    printf("Slow gaps (>100us): %lld\n", @slow_gaps);
}
```

The inter-launch gap histogram tells a clear story:

- **Tight distribution at 5-20 us**: The CPU is keeping the GPU fed. Kernel launch overhead dominates, not CPU work. This is the healthy state for decode steps using CUDA Graphs.
- **Bimodal distribution with peaks at 10 us and 200+ us**: Some launches are back-to-back (within a CUDA Graph or fused operation) while others have CPU work between them (scheduler, sampling, memory management). Normal for serving without CUDA Graphs.
- **Wide distribution or long tail above 500 us**: The CPU is struggling. Python GIL contention, garbage collection pauses, or complex scheduler logic is starving the GPU. Optimization opportunity.

## Memory Allocation Tracking

CUDA memory allocation (`cudaMalloc`, `cudaFree`) goes through the NVIDIA kernel driver via `ioctl` system calls. eBPF can trace these at the system call level to detect leaks, fragmentation, and allocation hotspots without any application modification.

### Tracing cudaMalloc and cudaFree

```bash
#!/usr/bin/env bpftrace
// trace_cuda_memory.bt
// Trace CUDA memory allocation and deallocation via driver ioctl
// Detects leaks, large allocations, and allocation frequency

tracepoint:syscalls:sys_enter_ioctl
/comm == "python" || comm == "python3" || comm == "pt_main_thread"/
{
    @ioctl_start[tid] = nsecs;
    @ioctl_count[comm] = count();
}

tracepoint:syscalls:sys_exit_ioctl
/@ioctl_start[tid]/
{
    $latency_us = (nsecs - @ioctl_start[tid]) / 1000;
    @ioctl_latency_us = hist($latency_us);

    // Flag slow ioctls (> 1ms suggests driver doing heavy work)
    if ($latency_us > 1000) {
        printf("SLOW ioctl: %s pid=%d tid=%d latency=%dus\n",
               comm, pid, tid, $latency_us);
    }

    delete(@ioctl_start[tid]);
}

// Track mmap calls (GPU memory mapping)
tracepoint:syscalls:sys_enter_mmap
/comm == "python" || comm == "python3"/
{
    @mmap_start[tid] = nsecs;
    @mmap_size[tid] = args.len;
}

tracepoint:syscalls:sys_exit_mmap
/@mmap_start[tid]/
{
    $latency_us = (nsecs - @mmap_start[tid]) / 1000;
    $size_mb = @mmap_size[tid] / (1024 * 1024);

    @mmap_latency_us = hist($latency_us);

    if ($size_mb > 100) {
        printf("LARGE mmap: %s pid=%d size=%dMB latency=%dus\n",
               comm, pid, $size_mb, $latency_us);
    }

    delete(@mmap_start[tid]);
    delete(@mmap_size[tid]);
}
```

### Detecting Memory Leaks with eBPF

A memory leak in CUDA manifests as `ioctl` calls for allocation that are not balanced by corresponding deallocation calls. While eBPF cannot directly distinguish `cudaMalloc` ioctls from other driver ioctls without parsing the ioctl command numbers, we can use a higher-level approach with uprobes:

```bash
#!/usr/bin/env bpftrace
// cuda_leak_detector.bt
// Track cudaMalloc/cudaFree balance to detect leaks
// Requires libcudart.so symbols

uprobe:/usr/local/cuda/lib64/libcudart.so:cudaMalloc
{
    @malloc_start[tid] = nsecs;
    @malloc_count = count();
}

uretprobe:/usr/local/cuda/lib64/libcudart.so:cudaMalloc
/@malloc_start[tid]/
{
    $latency_us = (nsecs - @malloc_start[tid]) / 1000;
    @malloc_latency_us = hist($latency_us);
    @total_mallocs = count();
    delete(@malloc_start[tid]);
}

uprobe:/usr/local/cuda/lib64/libcudart.so:cudaFree
{
    @free_start[tid] = nsecs;
    @total_frees = count();
}

uretprobe:/usr/local/cuda/lib64/libcudart.so:cudaFree
/@free_start[tid]/
{
    $latency_us = (nsecs - @free_start[tid]) / 1000;
    @free_latency_us = hist($latency_us);
    delete(@free_start[tid]);
}

interval:s:10
{
    printf("\n=== CUDA Memory Balance (10s) ===\n");
    printf("cudaMalloc calls: %lld\n", @total_mallocs);
    printf("cudaFree calls:   %lld\n", @total_frees);
    $delta = @total_mallocs - @total_frees;
    printf("Delta (leak indicator): %lld\n", $delta);
    if ($delta > 100) {
        printf("WARNING: malloc/free imbalance growing!\n");
    }
}
```

### Fragmentation Pattern Detection

By tracking allocation sizes over time, eBPF can reveal fragmentation-inducing patterns:

```bash
#!/usr/bin/env bpftrace
// alloc_pattern.bt
// Histogram of allocation sizes to detect fragmentation risk
// Many different sizes = high fragmentation risk

uprobe:/usr/local/cuda/lib64/libcudart.so:cudaMalloc
{
    @alloc_start[tid] = nsecs;
}

uretprobe:/usr/local/cuda/lib64/libcudart.so:cudaMalloc
/@alloc_start[tid]/
{
    // The size parameter is the first argument to cudaMalloc
    // Note: accessing function arguments in uretprobe requires
    // saving them in the entry probe. Simplified here.
    $latency = (nsecs - @alloc_start[tid]) / 1000;

    // Bucket by latency as proxy for size
    // (larger allocations take longer)
    if ($latency < 20) {
        @alloc_bucket["tiny (<20us)"] = count();
    } else if ($latency < 100) {
        @alloc_bucket["small (20-100us)"] = count();
    } else if ($latency < 500) {
        @alloc_bucket["medium (100-500us)"] = count();
    } else {
        @alloc_bucket["large (>500us)"] = count();
    }

    delete(@alloc_start[tid]);
}

interval:s:30
{
    printf("\n=== Allocation Pattern (30s window) ===\n");
    print(@alloc_bucket);
    clear(@alloc_bucket);
}
```

A healthy serving system shows a narrow distribution of allocation sizes -- vLLM's PagedAttention allocates fixed-size KV cache blocks, and model weights are allocated once at startup. A widening distribution indicates dynamic allocation patterns that fragment the allocator.

<Callout type="tip" title="The PagedAttention Signal in eBPF">
  When profiling vLLM with eBPF, you should see very few `cudaMalloc` calls during steady-state serving because PagedAttention pre-allocates the KV cache pool at startup. If you see frequent `cudaMalloc`/`cudaFree` calls during inference, something is bypassing the pool -- possibly a custom operation, a logging function allocating tensors, or a bug in cache management. eBPF makes this immediately visible without modifying vLLM's code.
</Callout>

## bpftrace One-Liners for LLM Profiling

Here are practical one-liners for common LLM profiling tasks. Each can be run immediately with `sudo bpftrace -e '...'`.

### Histogram of ioctl Durations (GPU Driver Calls)

```bash
# Distribution of GPU driver call latencies
sudo bpftrace -e '
tracepoint:syscalls:sys_enter_ioctl /comm == "python3"/ { @start[tid] = nsecs; }
tracepoint:syscalls:sys_exit_ioctl /@start[tid]/ {
    @us = hist((nsecs - @start[tid]) / 1000);
    delete(@start[tid]);
}'
```

This produces a power-of-2 histogram of all ioctl latencies. For LLM serving, you want to see the distribution concentrated below 100 us. A long tail above 1 ms indicates driver contention or memory management overhead.

### Count System Calls by Type

```bash
# What system calls does the serving process make?
sudo bpftrace -e '
tracepoint:raw_syscalls:sys_enter /comm == "python3"/ {
    @syscalls[args.id] = count();
}'
```

This reveals the syscall profile of the serving process. Healthy LLM serving is dominated by `ioctl` (GPU driver), `futex` (thread synchronization), `epoll_wait` (network), and `write` (logging). Unexpected high counts of `mmap`/`munmap` (memory allocation), `clock_gettime` (excessive timing), or `read` (unexpected I/O) indicate optimization opportunities.

### Track Context Switches (GIL Contention Indicator)

```bash
# How often is the serving process descheduled?
sudo bpftrace -e '
tracepoint:sched:sched_switch /args.prev_comm == "python3"/ {
    @switches[args.prev_state] = count();
    @switch_rate = count();
}
interval:s:1 {
    printf("Context switches/sec: %lld\n", @switch_rate);
    clear(@switch_rate);
}'
```

For Python-based LLM serving, high context switch rates (more than 10,000/sec) typically indicate GIL contention. The `prev_state` field reveals whether switches are voluntary (thread yielding, waiting for I/O) or involuntary (preempted by scheduler). Involuntary switches suggest the process is CPU-bound and competing for cores.

### Monitor Page Faults

```bash
# Track page faults -- these stall CUDA operations
sudo bpftrace -e '
tracepoint:exceptions:page_fault_user /comm == "python3"/ {
    @faults = count();
    @fault_addr = hist(args.address);
}
interval:s:5 {
    printf("Page faults in 5s: %lld\n", @faults);
    clear(@faults);
}'
```

Page faults during inference indicate memory that was not pre-faulted. When CUDA allocates memory and the host touches it for the first time, a page fault stalls the operation. vLLM and similar frameworks should pre-fault all memory during startup. Page faults during serving are a bug.

### Trace File I/O (Unexpected Disk Access)

```bash
# Is the serving process hitting disk during inference?
sudo bpftrace -e '
tracepoint:syscalls:sys_enter_read /comm == "python3"/ {
    @read_start[tid] = nsecs;
    @read_fd[tid] = args.fd;
}
tracepoint:syscalls:sys_exit_read /@read_start[tid]/ {
    $latency_us = (nsecs - @read_start[tid]) / 1000;
    if ($latency_us > 100) {
        printf("SLOW read: fd=%d latency=%dus\n", @read_fd[tid], $latency_us);
    }
    delete(@read_start[tid]);
    delete(@read_fd[tid]);
}'
```

Disk I/O during inference is a red flag. The model should be fully loaded into GPU memory. If you see reads during inference, something is loading weights lazily, reading configuration files per-request, or logging synchronously to disk.

## CPU-Side Profiling: Where Time Goes

In LLM serving, the CPU is responsible for tokenization, scheduling, sampling, memory management, and orchestrating GPU kernel launches. Understanding CPU time distribution is critical because the CPU is often the bottleneck during decode steps -- especially at small batch sizes where GPU kernels are memory-bound and fast.

### Decomposing CPU Time with eBPF

```bash
#!/usr/bin/env bpftrace
// cpu_breakdown.bt
// Measure CPU time spent in different categories for LLM serving
// Uses uprobe on key Python/C++ functions

// Tokenizer time (HuggingFace tokenizers library)
uprobe:/path/to/tokenizers.cpython-310-x86_64-linux-gnu.so:*encode*
{
    @tokenizer_start[tid] = nsecs;
}

uretprobe:/path/to/tokenizers.cpython-310-x86_64-linux-gnu.so:*encode*
/@tokenizer_start[tid]/
{
    $us = (nsecs - @tokenizer_start[tid]) / 1000;
    @tokenizer_us = hist($us);
    @tokenizer_total_us = sum($us);
    delete(@tokenizer_start[tid]);
}

// CUDA kernel launch time
uprobe:/usr/local/cuda/lib64/libcudart.so:cudaLaunchKernel
{
    @cuda_launch_start[tid] = nsecs;
}

uretprobe:/usr/local/cuda/lib64/libcudart.so:cudaLaunchKernel
/@cuda_launch_start[tid]/
{
    $us = (nsecs - @cuda_launch_start[tid]) / 1000;
    @cuda_launch_us = hist($us);
    @cuda_launch_total_us = sum($us);
    delete(@cuda_launch_start[tid]);
}

interval:s:10
{
    printf("\n=== CPU Time Breakdown (10s) ===\n");
    printf("Tokenizer total:     %lld us\n", @tokenizer_total_us);
    printf("CUDA launch total:   %lld us\n", @cuda_launch_total_us);
}
```

### Python GIL Profiling

The Global Interpreter Lock (GIL) is a well-known bottleneck in Python-based serving. eBPF can measure GIL contention by tracing the futex system calls that Python uses for GIL acquisition:

```bash
#!/usr/bin/env bpftrace
// gil_contention.bt
// Measure Python GIL contention via futex tracing
// High futex wait times indicate GIL is a bottleneck

tracepoint:syscalls:sys_enter_futex
/comm == "python3" && args.op == 0/  // FUTEX_WAIT
{
    @futex_start[tid] = nsecs;
    @futex_waits = count();
}

tracepoint:syscalls:sys_exit_futex
/@futex_start[tid]/
{
    $wait_us = (nsecs - @futex_start[tid]) / 1000;
    @futex_wait_us = hist($wait_us);

    if ($wait_us > 1000) {
        printf("LONG futex wait: tid=%d %dus (likely GIL)\n", tid, $wait_us);
    }

    delete(@futex_start[tid]);
}

interval:s:10
{
    printf("\n=== GIL Contention (10s) ===\n");
    printf("Total futex waits: %lld\n", @futex_waits);
    print(@futex_wait_us);
    clear(@futex_waits);
}
```

<Callout type="info" title="GIL Impact on LLM Serving">
  In multi-threaded Python serving (vLLM uses multiple threads for tokenization, scheduling, and GPU management), GIL contention can add 100-500 us per decode step. eBPF's futex tracing shows the actual wait time. If futex waits dominate the off-CPU histogram, consider: (1) moving tokenization to a separate process (subprocess or Rust tokenizer with released GIL); (2) using CUDA Graphs to batch kernel launches (fewer Python-to-CUDA transitions); (3) upgrading to Python 3.13+ with free-threaded mode for experimental GIL removal.
</Callout>

Flamegraphs are the single most effective visualization for understanding CPU time in LLM serving. A flamegraph shows the call stack on the x-axis (width proportional to time) and stack depth on the y-axis, making it immediately obvious where CPU time is concentrated.

### Reading the Flamegraph for LLM Serving

A typical vLLM flamegraph during steady-state serving shows these regions:

**Tokenizer stack (5-15% of CPU time)**: Look for frames containing `tokenizers`, `encode`, `decode`, or `HuggingFace`. The Rust-based HuggingFace tokenizer is fast but still consumes measurable CPU. If this region is wide (more than 15%), batch tokenization may help.

**Scheduler stack (5-10% of CPU time)**: Frames containing `schedule`, `_schedule`, `SchedulerOutputs`, `_get_num_new_tokens`. The scheduler selects which requests to include in the next batch, manages preemption, and allocates KV cache blocks. Complex scheduling policies (e.g., priority-based, fairness-aware) widen this region.

**CUDA launch stack (3-8% of CPU time)**: Frames containing `cudaLaunchKernel`, `cuLaunchKernel`, `THCudaInternal`. Each kernel launch traverses the CUDA runtime and driver. CUDA Graphs collapse hundreds of individual launches into a single replay, dramatically narrowing this region.

**Sampling stack (3-5% of CPU time)**: Frames containing `sample`, `_sample`, `top_k`, `top_p`, `temperature`. Sampling logic runs on CPU (or partially on GPU) after logits computation. Complex sampling (beam search, top-p with temperature) is more expensive than greedy.

**Python interpreter overhead (5-10% of CPU time)**: Frames containing `PyObject`, `_PyEval`, `_PyFunction_Vectorcall`. This is pure Python overhead -- the interpreter evaluating bytecode, managing reference counts, checking signals. Wider than 10% suggests hot paths that should be moved to C++ or Cython.

**NCCL communication stack (0-15% of CPU time, multi-GPU only)**: Frames containing `ncclAllReduce`, `ncclSend`, `ncclRecv`, `tcp_sendmsg`, `tcp_recvmsg`. For tensor-parallel serving, NCCL communication shows up in the flamegraph as CPU time spent setting up and waiting for collective operations.

### Off-CPU Flamegraphs

On-CPU flamegraphs show where the process is running. Off-CPU flamegraphs show where the process is waiting. For LLM serving, off-CPU analysis often reveals more actionable insights:

```bash
# Off-CPU flamegraph using eBPF
# Captures stack traces when the process is descheduled
sudo offcputime-bpfcc -df -p $(pgrep -f "vllm") 30 \
    > vllm_offcpu.folded
flamegraph.pl --title "vLLM Off-CPU" \
    --colors io \
    --countname "us" \
    vllm_offcpu.folded > vllm_offcpu_flame.svg
```

The off-CPU flamegraph shows the call stacks at the moment the process was blocked. Common findings:

- **futex_wait in Python internals**: GIL contention. Threads are waiting to acquire the GIL.
- **ep_poll (epoll_wait)**: Waiting for network I/O. Normal for idle periods, concerning during active serving.
- **ioctl to nvidia driver**: Waiting for GPU memory operations. If this dominates off-CPU time, the CUDA allocator is a bottleneck.
- **nanosleep / clock_nanosleep**: Deliberate sleeps in polling loops. Some serving frameworks poll for request completion. These can often be replaced with event-driven approaches.

The on-CPU flamegraph immediately reveals if CPU time is dominated by expected work (tokenization, scheduling) or unexpected overhead (Python GC, memory allocation, lock contention). The off-CPU flamegraph reveals what the process is waiting for -- GIL, I/O, or GPU operations. Together, they provide a complete picture in under a minute of profiling.

## Tracing NCCL Communication

For multi-GPU LLM serving with tensor parallelism (TP) or pipeline parallelism (PP), NCCL (NVIDIA Collective Communications Library) handles inter-GPU communication. NCCL operations can consume 30-50% of end-to-end latency for small-batch decode with TP greater than 2. eBPF traces the socket-level and system-level operations that NCCL performs.

### Socket-Level NCCL Tracing

```bash
#!/usr/bin/env bpftrace
// nccl_socket_trace.bt
// Trace NCCL socket send/recv latency and throughput
// NCCL uses TCP sockets for inter-node, shared memory for intra-node

tracepoint:syscalls:sys_enter_sendmsg
/comm == "nccl-net" || comm == "nccl-socket"/
{
    @send_start[tid] = nsecs;
    @send_count = count();
}

tracepoint:syscalls:sys_exit_sendmsg
/@send_start[tid]/
{
    $latency_us = (nsecs - @send_start[tid]) / 1000;
    @send_latency_us = hist($latency_us);

    if ($latency_us > 500) {
        printf("SLOW NCCL send: tid=%d latency=%dus\n", tid, $latency_us);
    }

    delete(@send_start[tid]);
}

tracepoint:syscalls:sys_enter_recvmsg
/comm == "nccl-net" || comm == "nccl-socket"/
{
    @recv_start[tid] = nsecs;
    @recv_count = count();
}

tracepoint:syscalls:sys_exit_recvmsg
/@recv_start[tid]/
{
    $latency_us = (nsecs - @recv_start[tid]) / 1000;
    @recv_latency_us = hist($latency_us);
    delete(@recv_start[tid]);
}

interval:s:10
{
    printf("\n=== NCCL Communication (10s) ===\n");
    printf("Sends: %lld, Recvs: %lld\n", @send_count, @recv_count);
    printf("Send latency:\n");
    print(@send_latency_us);
    printf("Recv latency:\n");
    print(@recv_latency_us);
}
```

### NVLink vs PCIe Detection

eBPF can help determine whether NCCL is using NVLink (fast) or PCIe (slow) for a particular GPU pair:

```bash
# Check which transport NCCL is using
# NCCL logs this at startup, but eBPF can trace it dynamically

# Monitor shared memory operations (NVLink intra-node path)
sudo bpftrace -e '
tracepoint:syscalls:sys_enter_mmap
/comm == "nccl-sharp" || comm == "nccl-net"/ {
    @nccl_mmap = count();
    printf("NCCL mmap: pid=%d tid=%d len=%lu\n", pid, tid, args.len);
}'

# Compare send/recv volume vs shared memory volume
# NVLink path uses CUDA IPC (shared memory), not sockets
# If socket traffic is high, NCCL is falling back to PCIe/TCP
```

When NCCL falls back from NVLink to PCIe, communication latency increases 3-5x. When it falls back to TCP (inter-node), latency increases 10-50x. eBPF socket tracing immediately reveals these fallbacks through the latency distribution -- a jump from 40 us to 200 us per collective indicates PCIe fallback, while a jump to 1+ ms indicates TCP.

### Detecting Performance Regressions

eBPF monitoring enables automated regression detection by establishing baselines and alerting on deviations:

class EBPFRegressionDetector:
    def __init__(self, baseline_file="ebpf_baseline.json"):

            return None

        metrics = {}

                @csw = count();
            }

## Advanced: Building a Custom eBPF Profiler for LLM Serving

For teams with specific profiling needs, building a custom eBPF tool using BCC (BPF Compiler Collection) provides more power than bpftrace one-liners:

```python
#!/usr/bin/env python3
"""
llm_profiler.py - Custom eBPF profiler for LLM inference serving
Uses BCC to trace CUDA driver, memory, and scheduling events
"""

from bcc import BPF
import time
import json

bpf_program = """
#include <uapi/linux/ptrace.h>

// Hash maps for tracking latency
BPF_HASH(ioctl_start, u32, u64);
BPF_HASH(launch_start, u32, u64);

// Histograms for latency distribution
BPF_HISTOGRAM(ioctl_latency, int);
BPF_HISTOGRAM(launch_latency, int);

// Counters
BPF_ARRAY(counters, u64, 4);  // 0=ioctl, 1=launch, 2=sync, 3=malloc

// Trace ioctl entry (GPU driver calls)
TRACEPOINT_PROBE(syscalls, sys_enter_ioctl) {
    u32 tid = bpf_get_current_pid_tgid();
    u64 ts = bpf_ktime_get_ns();
    ioctl_start.update(&tid, &ts);

    u32 key = 0;
    u64 *val = counters.lookup(&key);
    if (val) (*val)++;
    return 0;
}

// Trace ioctl exit
TRACEPOINT_PROBE(syscalls, sys_exit_ioctl) {
    u32 tid = bpf_get_current_pid_tgid();
    u64 *tsp = ioctl_start.lookup(&tid);
    if (!tsp) return 0;

    u64 delta = (bpf_ktime_get_ns() - *tsp) / 1000;  // us
    int key = bpf_log2l(delta);
    ioctl_latency.atomic_increment(key);

    ioctl_start.delete(&tid);
    return 0;
}

// Trace context switches for the target process
TRACEPOINT_PROBE(sched, sched_switch) {
    // Count voluntary vs involuntary switches
    u32 key = 2;  // reuse counter slot
    u64 *val = counters.lookup(&key);
    if (val) (*val)++;
    return 0;
}
"""

def main():
    b = BPF(text=bpf_program)

    print("Tracing LLM serving process... Ctrl+C to stop")

    try:
        while True:
            time.sleep(10)

            print("\n=== ioctl latency (us) ===")
            b["ioctl_latency"].print_log2_hist("us")
            b["ioctl_latency"].clear()

            counters = b["counters"]
            print(f"ioctl calls: {counters[0].value}")
            print(f"context switches: {counters[2].value}")

    except KeyboardInterrupt:
        print("\nDone.")

if __name__ == "__main__":
    main()
```

This BCC program can be extended with:

- **Per-request tracing**: Correlate eBPF events with individual inference requests by tracing NVTX range push/pop calls.
- **Adaptive sampling**: Increase tracing frequency when latency exceeds thresholds, reduce during normal operation.
- **Prometheus export**: Push metrics to Prometheus for dashboarding alongside application metrics.
- **Ring buffer streaming**: Stream individual events (not just aggregates) for detailed post-hoc analysis.

## Conclusion

eBPF fills the critical observability gap for LLM inference: the CPU and system layer that GPU profilers cannot see. In production, 10-20% of end-to-end latency comes from non-GPU sources -- CUDA driver overhead, memory allocation, scheduler logic, NCCL socket operations, Python GIL contention, and OS scheduling delays. GPU profilers report these periods as "GPU idle" but offer no explanation. eBPF explains exactly what the CPU and kernel are doing during every microsecond of that idle time.

The practical toolkit is straightforward: `bpftrace` one-liners for ad-hoc investigation, `profile-bpfcc` and `offcputime-bpfcc` for flamegraphs, custom BCC scripts for continuous monitoring, and correlation with Nsight Systems for the complete picture. The overhead -- consistently under 1% for targeted tracing -- makes eBPF safe for production use, not just laboratory investigation.

The combination of Nsight Systems (GPU timeline) and eBPF (system-level tracing) provides complete end-to-end observability from HTTP request to generated token. Every microsecond is accounted for. Every bottleneck is visible. For teams operating LLM serving at scale, this combination is not optional -- it is the difference between guessing where latency comes from and knowing.
