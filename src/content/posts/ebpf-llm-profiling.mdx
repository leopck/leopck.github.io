GPU profilers (Nsight Compute, Nsight Systems) show you what happens *on* the GPU. But LLM inference is a full-system problem — CPU scheduling, memory allocation, PCIe transfers, NCCL socket operations, and Python runtime overhead all contribute to end-to-end latency. eBPF (extended Berkeley Packet Filter) provides kernel-level observability for everything the GPU profiler can't see.

## Tracing GPU Memory Allocation

CUDA memory allocation (`cudaMalloc`) goes through the kernel via `ioctl` calls to the NVIDIA driver. eBPF can trace these:

```bash
#!/usr/bin/env bpftrace
// Trace CUDA memory allocations via NVIDIA driver ioctl
// Shows allocation size, latency, and calling process

tracepoint:syscalls:sys_enter_ioctl
/comm == "python" || comm == "python3"/
{
    @start[tid] = nsecs;
    @ioctl_count[comm] = count();
}

tracepoint:syscalls:sys_exit_ioctl
/@start[tid]/
{
    $latency_us = (nsecs - @start[tid]) / 1000;
    @alloc_latency_us = hist($latency_us);
    delete(@start[tid]);

    // Flag slow allocations (> 1ms suggests fragmentation)
    if ($latency_us > 1000) {
        printf("SLOW ioctl: %s pid=%d latency=%dµs\n", comm, pid, $latency_us);
    }
}
```

## Tracing CPU-Side Overhead

LLM serving has significant CPU work between GPU kernel launches:

```bash
#!/usr/bin/env bpftrace
// Measure time between GPU kernel launches (CPU gap)
// High gaps indicate CPU-side bottlenecks

usdt:libtorch_cuda.so:cuda_kernel_launch
{
    if (@last_launch[tid]) {
        $gap_us = (nsecs - @last_launch[tid]) / 1000;
        @cpu_gap_us = hist($gap_us);

        if ($gap_us > 100) {
            printf("CPU gap: %dµs between kernel launches (tid=%d)\n", $gap_us, tid);
        }
    }
    @last_launch[tid] = nsecs;
}
```

## Tracing NCCL Communication

For multi-GPU serving with tensor parallelism, NCCL communication adds latency. eBPF can trace the socket-level operations:

```bash
#!/usr/bin/env bpftrace
// Trace NCCL socket send/recv latency
tracepoint:syscalls:sys_enter_sendmsg
/comm == "nccl-net"/
{
    @send_start[tid] = nsecs;
}

tracepoint:syscalls:sys_exit_sendmsg
/@send_start[tid]/
{
    $latency = (nsecs - @send_start[tid]) / 1000;
    @nccl_send_us = hist($latency);
    delete(@send_start[tid]);
}
```

<Callout type="info" title="The TP Overhead Surprise">
  Each NCCL all-reduce is fast (42-45 µs) but there are 160 of them per decode step across 80 layers. eBPF reveals that the aggregate overhead (7 ms) is comparable to the GPU compute time itself. This is why TP=2 doesn't give 2× throughput — it gives ~1.6× at best due to communication overhead that's invisible to GPU profilers.
</Callout>

## Production Monitoring with eBPF

For ongoing production profiling, use eBPF-based tools:

```bash
# Flame graph of CPU usage in vLLM process
# Reveals where Python/C++ CPU time goes
profile-bpfcc -F 99 -p $(pgrep -f "vllm") 30 > vllm.folded
flamegraph.pl vllm.folded > vllm_flame.svg

# Monitor GPU driver ioctl latency continuously
funclatency-bpfcc -p $(pgrep -f "vllm") 'c:ioctl' -m 5

# Trace context switches (GIL contention indicator)
cpudist-bpfcc -p $(pgrep -f "vllm") 10
```

## Combining GPU + eBPF Profiling

The power comes from combining both perspectives:

1. **Nsight Systems** gives you the GPU timeline — kernel durations, CUDA API calls, memory copies
2. **eBPF** gives you the CPU/system timeline — what happens between GPU events
3. **Correlate by timestamp** — match CPU gaps in Nsight Systems with eBPF traces to explain mysterious latency

```bash
# Synchronized profiling session
# Terminal 1: Nsight Systems (GPU timeline)
nsys profile -t cuda,nvtx -o gpu_trace python -m vllm.entrypoints.api_server

# Terminal 2: eBPF (CPU/system timeline)
bpftrace -e '
  tracepoint:syscalls:sys_enter_ioctl /comm == "python3"/ { @ioctl = count(); }
  tracepoint:sched:sched_switch /comm == "python3"/ { @ctxsw = count(); }
' -c 'sleep 30'
```

<Callout type="tip" title="Start Here">
  For your first eBPF profiling session on an LLM serving system: run `profile-bpfcc` for 30 seconds and generate a flame graph. The flame graph immediately reveals if CPU time is dominated by expected work (tokenization, scheduling) or unexpected overhead (Python GC, memory allocation, lock contention).
</Callout>

## Conclusion

eBPF fills the critical observability gap for LLM inference: the CPU/system layer that GPU profilers don't see. In production, 10-20% of end-to-end latency comes from non-GPU sources — memory allocation, scheduler logic, NCCL socket operations, and Python runtime overhead. eBPF traces these at kernel level with <1% overhead, making it safe for production use. The combination of Nsight Systems (GPU) + eBPF (system) provides complete end-to-end observability from user request to token generation.
