GEMM (General Matrix Multiplication) is the single most important operation in neural networks -- it dominates compute in linear layers, attention, and convolutions (via im2col). On a V100, peak FP16 GEMM throughput is 125 TFLOPS. In practice, most neural network GEMMs achieve 60-80% of peak. Understanding the gap -- and when to close it -- is essential for inference optimization.

## cuBLAS: The Baseline

NVIDIA's cuBLAS is highly optimized and auto-selects kernel implementations based on matrix dimensions:

The pattern is clear: cuBLAS shines on large, square-ish matrices (prefill, large batch). It struggles with skinny matrices (small batch decode, where M=1-32) because these are memory-bound -- there isn't enough compute per byte loaded.

<Callout type="perf" title="The Batch-1 Problem">
  At batch=1, the FFN down-projection becomes [1, 11008] x [11008, 4096] -- a matrix-vector multiply. Arithmetic intensity is ~0.5 FLOP/byte. On a V100 with 900 GB/s bandwidth, the theoretical peak is only ~0.9 TFLOPS -- 0.7% of FP16 peak. No GEMM kernel can fix this; the operation is fundamentally memory-bound.
</Callout>

For standard large GEMMs, cuBLAS is within 5% of custom kernels -- the engineering cost of custom kernels isn't justified. The wins come from **fusion** (eliminating intermediate memory traffic) and **non-standard precisions** (FP8/INT4 that cuBLAS doesn't support).

## GEMM Fusion: The Real Optimization

The biggest gains come not from faster GEMM kernels but from fusing GEMM with adjacent operations:

```cpp
// Unfused: 3 kernel launches, 2 intermediate tensors in HBM
// 1. Y = X @ W (GEMM -- writes Y to HBM)
// 2. Y = Y + bias (reads Y from HBM, writes back)
// 3. Y = gelu(Y) (reads Y from HBM, writes back)

// Fused: 1 kernel, 0 intermediate tensors in HBM
// Compute Y = gelu(X @ W + bias) in one pass
// Bias and activation applied in registers before writing to HBM
```

Fusion eliminates 2 HBM round-trips (bias add + activation), saving ~3.2 GB of memory traffic per FFN layer. Across 32 layers, that's 102 GB saved -- significant at 900 GB/s bandwidth.

## The Tiling Hierarchy

High-performance GEMM kernels use a multi-level tiling strategy that maps to the GPU memory hierarchy:

<Benchmark
  title="GEMM Tiling Levels (CUTLASS-style)"
  columns={["Level", "Tile Size", "Storage", "Bandwidth", "Purpose"]}
  rows={[
    { values: ["Thread Block", "128x128", "Shared Memory", "~3 TB/s", "Cooperative loading from HBM"], highlight: false },
    { values: ["Warp", "64x32", "Registers", "~10 TB/s", "Tensor core MMA operations"], highlight: true },
    { values: ["Instruction", "16x8x16", "Registers", "Instruction throughput", "Single MMA instruction"], highlight: true },
  ]}
  notes="CUTLASS defines these levels as ThreadblockShape, WarpShape, and InstructionShape in its templates."
/>

The tile sizes are chosen so that each level's data fits in the corresponding memory level. A 128x128 block tile in FP16 requires 128x128x2 = 32 KB for each of A and B -- fitting in the V100's 96 KB shared memory.

<Callout type="info" title="When to Use CUTLASS vs Triton">
  **CUTLASS** (C++ templates): Maximum control and performance. Use when you need custom precision, custom fusion, or are building a production inference engine. Steep learning curve. **Triton** (Python): 80-90% of CUTLASS performance with 10x less code. Use for rapid prototyping, research kernels, and when you need fusion but not maximum throughput. Both generate optimized PTX targeting tensor cores.
</Callout>

## Conclusion

GEMM optimization for neural networks follows a clear priority: (1) ensure your shapes are large enough to saturate the GPU (batch during decode), (2) use cuBLAS for standard large GEMMs (85-90% of peak), (3) fuse GEMM with adjacent operations (elementwise, activation) to eliminate HBM traffic (30-40% speedup on fused layers), (4) use CUTLASS/Triton for non-standard precisions or complex fusion patterns. The single biggest insight: batch-1 decode is fundamentally memory-bound regardless of GEMM kernel quality -- increasing batch size is the only way to improve arithmetic intensity.
