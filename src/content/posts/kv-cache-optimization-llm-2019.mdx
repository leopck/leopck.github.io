KV caching is what makes autoregressive LLM inference tractable. Without it, generating token N requires recomputing attention over all N-1 previous tokens -- O(N^2) work per token, or O(N^3) total for a sequence. With KV caching, each new token only computes its own K and V projections and attends to the cached history -- O(N) per token. The cost: memory that grows linearly with sequence length, batch size, and model depth.

## The Memory Formula

KV cache memory per request, per layer:

**KV_per_layer = 2 x num_kv_heads x head_dim x seq_len x dtype_bytes**

For the full model:

**KV_total = KV_per_layer x num_layers x batch_size**

## Why KV Cache Exists: The Compute Savings

Without caching, generating 100 tokens from a 200-token prompt requires:
- Token 1: attend to 200 previous positions
- Token 2: attend to 201 positions
- ...
- Token 100: attend to 299 positions
- **Total attention operations: ~25,000 positions**

With caching, the prompt is processed once (prefill), then each decode step attends to only 1 new query against the cached KV:
- Prefill: 200 positions (once)
- Decode: 100 steps x ~250 avg KV length
- **Total: ~25,200 positions** -- same math, but no redundant computation

The real win is that decode becomes memory-bound (loading cached KV) rather than compute-bound (recomputing all attention). On modern GPUs, memory bandwidth is the bottleneck anyway during decode, so caching converts wasted compute into a memory cost we're already paying.

## Optimization Techniques

### 1. KV Cache Quantization

Quantizing cached KV values from FP16 to INT8 or FP8 halves memory with minimal quality loss:

INT8 KV quantization is nearly free in quality and doubles serving capacity. It's the single easiest optimization to deploy.

### 2. Paged Allocation (PagedAttention)

Traditional allocation reserves `max_seq_len` worth of KV memory per request upfront. PagedAttention allocates in 16-token pages on demand, eliminating waste from over-allocation:

## Conclusion

KV cache memory grows as O(batch x seq_len x layers x kv_heads x head_dim) and dominates GPU memory at production batch sizes. The optimization stack is clear: use GQA models (architectural), apply INT8 KV quantization (easy win), deploy PagedAttention for paged allocation (system-level), and consider sliding window for short-context workloads. Together, these techniques reduce KV memory by 10-16x, directly translating to higher concurrent request capacity and throughput.
