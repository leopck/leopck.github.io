KV cache is the dominant memory consumer in LLM serving at production batch sizes. For Llama-7B serving 32 requests at 2K context, KV cache alone consumes 32 GB — more than the model weights (14 GB). Quantizing KV from FP16 to INT8 halves this to 16 GB, enabling 64 concurrent requests on the same GPU. The quality trade-off? Less than 0.1% perplexity increase.

## Why KV Cache Quantization

Weight quantization reduces model size. KV cache quantization reduces *serving* memory — the memory that scales with batch size and sequence length:

The granularity of scale factors determines the accuracy-memory trade-off:

/>

<Callout type="perf" title="Why Per-Channel Works">
  Within a single attention head, K and V values have relatively uniform magnitude distributions. Different heads may have very different scales (some heads attend broadly with small values, others attend sharply with large values). Per-channel (per-head) quantization captures this inter-head variance while keeping intra-head quantization simple and efficient.
</Callout>

## Integration with vLLM

vLLM supports KV cache quantization via the `--kv-cache-dtype` flag:

```bash
# FP8 KV cache (H100, L40, Ada GPUs)
python -m vllm.entrypoints.openai.api_server \
    --model meta-llama/Llama-2-7b-hf \
    --kv-cache-dtype fp8_e4m3 \
    --max-model-len 4096

# INT8 KV cache (any GPU with Turing+ tensor cores)
# Requires quantization calibration or auto-scaling
python -m vllm.entrypoints.openai.api_server \
    --model meta-llama/Llama-2-7b-hf \
    --kv-cache-dtype int8 \
    --quantization-param-path kv_scales.json
```

<Callout type="tip" title="FP8 vs INT8 for KV Cache">
  On H100/Gaudi2/L40 (which have native FP8 support): use FP8. It has slightly better quality than INT8 at the same bit width because FP8's floating-point representation handles the varying magnitude of attention values more gracefully than fixed-point INT8. On older GPUs: use INT8 with per-channel scaling.
</Callout>

## Conclusion

KV cache quantization is one of the highest-impact, lowest-effort optimizations for LLM serving: a single configuration flag doubles concurrent request capacity with <0.02 perplexity increase (INT8 per-channel). Use FP8 on H100/Gaudi2 hardware, INT8 everywhere else. Combined with INT4 weight quantization, total serving capacity increases 2.5-3× — transforming single-GPU deployments from handling 30 to 100+ concurrent requests.
