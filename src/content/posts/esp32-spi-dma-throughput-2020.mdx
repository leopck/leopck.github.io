SPI is the fastest general-purpose serial interface on the ESP32, capable of clock rates up to 80 MHz in master mode. However, the gap between theoretical bandwidth and measured throughput can be enormous -- exceeding 50% at high clock speeds -- if bus configuration, DMA sizing, and peripheral routing are not handled correctly. This post provides a rigorous treatment of every factor that determines ESP32 SPI throughput, with logic-analyzer-verified measurements at each standard clock frequency.

  SPI0 and SPI1 share the bus with the external flash that holds your program code. ## IOMUX vs GPIO Matrix: The Critical Routing Decision

The ESP32 has two ways to connect internal peripherals to GPIO pins: the IOMUX (direct IO multiplexer) and the GPIO matrix (a crossbar switch). This choice has a profound impact on maximum achievable SPI clock.

### IOMUX Direct Routing

IOMUX connects each peripheral signal to a specific, fixed GPIO pin with minimal propagation delay (approximately 2 ns). For SPI2 (HSPI):

### GPIO Matrix Routing

The GPIO matrix allows any peripheral signal to be routed to any GPIO pin, providing maximum PCB layout flexibility. However, it adds approximately 25 ns of propagation delay per signal, which at 80 MHz (12.5 ns period) is nearly two full clock cycles of delay. This delay causes setup/hold time violations at high clock rates.

  If your application requires SPI clock rates above 26 MHz, plan your PCB layout around the IOMUX pin assignments from the start. Reworking a PCB to move SPI signals to different pins is expensive. At 10-20 MHz, the GPIO matrix works fine, but at 40 MHz you are at its margin, and at 80 MHz it simply does not work. The 80 MHz IOMUX path is the single largest throughput gain available on the ESP32 SPI peripheral.

## SPI Master Mode: Configuration and DMA

### Basic Master Configuration

```cpp
#include "driver/spi_master.h"
#include "esp_log.h"

static const char* TAG = "SPI_MASTER";
static spi_device_handle_t spi_dev;

void spi_master_init(void) {
    // Bus configuration -- use IOMUX pins for maximum speed
    spi_bus_config_t bus_cfg = {
        .mosi_io_num = 13,          // HSPI IOMUX MOSI
        .miso_io_num = 12,          // HSPI IOMUX MISO
        .sclk_io_num = 14,          // HSPI IOMUX SCLK
        .quadwp_io_num = -1,        // Not used
        .quadhd_io_num = -1,        // Not used
        .max_transfer_sz = 4096,    // Maximum DMA transfer size
        .flags = SPICOMMON_BUSFLAG_MASTER | SPICOMMON_BUSFLAG_IOMUX,
    };

    // Initialize bus with DMA channel auto-selection
    esp_err_t ret = spi_bus_initialize(HSPI_HOST, &bus_cfg, SPI_DMA_CH_AUTO);
    if (ret != ESP_OK) {
        ESP_LOGE(TAG, "SPI bus init failed: %s", esp_err_to_name(ret));
        return;
    }

    // Device configuration
    spi_device_interface_config_t dev_cfg = {
        .clock_speed_hz = 40 * 1000 * 1000,  // 40 MHz
        .mode = 0,                             // CPOL=0, CPHA=0
        .spics_io_num = 15,                    // HSPI IOMUX CS0
        .queue_size = 7,                       // Transaction queue depth
        .flags = 0,                            // Full-duplex by default
        .pre_cb = NULL,
        .post_cb = NULL,
    };

    ret = spi_bus_add_device(HSPI_HOST, &dev_cfg, &spi_dev);
    if (ret != ESP_OK) {
        ESP_LOGE(TAG, "SPI device add failed: %s", esp_err_to_name(ret));
    }
}
```

## DMA Descriptor Chains for Zero-Copy Transfers

The ESP32 SPI DMA engine uses linked-list descriptor chains. Each descriptor points to a memory buffer and contains a link to the next descriptor, allowing the DMA controller to process multiple buffers sequentially without CPU intervention.

### Descriptor Structure

Each DMA descriptor is a 12-byte structure in memory:

The second and third words of the descriptor contain the buffer pointer and the next-descriptor pointer respectively. The ESP-IDF driver manages descriptor allocation automatically, but understanding the structure is important for optimizing throughput.

### Zero-Copy Transfer Pipeline

For maximum throughput, you want the DMA engine to always have work queued. The ESP-IDF provides an asynchronous API for this:

```cpp
#include "driver/spi_master.h"

#define NUM_BUFFERS 4
#define BUFFER_SIZE 4096

// DMA-capable buffers (must be in DMA-accessible memory)
static DRAM_ATTR uint8_t tx_buffers[NUM_BUFFERS][BUFFER_SIZE];
static spi_transaction_t transactions[NUM_BUFFERS];

void spi_pipeline_init(void) {
    for (int i = 0; i < NUM_BUFFERS; i++) {
        memset(&transactions[i], 0, sizeof(spi_transaction_t));
        transactions[i].length = BUFFER_SIZE * 8;  // Length in bits
        transactions[i].tx_buffer = tx_buffers[i];
        transactions[i].rx_buffer = NULL;  // TX-only
    }
}

// Streaming pipeline: queue transactions faster than they complete
void spi_stream_data(const uint8_t* source, size_t total_bytes) {
    size_t offset = 0;
    int queued = 0;
    int completed = 0;

    while (completed * BUFFER_SIZE < total_bytes) {
        // Queue as many transactions as possible
        while (queued - completed < NUM_BUFFERS &&
               queued * BUFFER_SIZE < total_bytes) {
            int idx = queued % NUM_BUFFERS;
            size_t chunk = BUFFER_SIZE;
            if (offset + chunk > total_bytes) {
                chunk = total_bytes - offset;
                transactions[idx].length = chunk * 8;
            }
            memcpy(tx_buffers[idx], source + offset, chunk);
            spi_device_queue_trans(spi_dev, &transactions[idx], portMAX_DELAY);
            offset += chunk;
            queued++;
        }

        // Wait for oldest transaction to complete
        spi_transaction_t* completed_trans;
        spi_device_get_trans_result(spi_dev, &completed_trans, portMAX_DELAY);
        completed++;
    }
}
```

The key insight is the pipeline depth: by keeping `NUM_BUFFERS` transactions in flight simultaneously, the DMA engine never idles waiting for the CPU to submit the next transfer. The CPU can fill the next buffer while DMA transmits the current one.

### DMA Memory Alignment Requirements

DMA buffers must reside in DMA-capable memory (internal SRAM, not PSRAM or flash). On the ESP32, this means:

- Buffers must be 32-bit aligned (4-byte boundary)
- Buffers must be in DRAM, not IRAM
- Buffers allocated with `heap_caps_malloc(size, MALLOC_CAP_DMA)` are guaranteed correct
- Stack-allocated buffers work if they happen to be in DRAM (which they normally are)
- `DRAM_ATTR` forces placement in DRAM when used with global/static variables

```cpp
// Correct: DMA-capable allocation
uint8_t* dma_buf = (uint8_t*)heap_caps_malloc(4096, MALLOC_CAP_DMA);

// Correct: Static DMA buffer
static DRAM_ATTR uint8_t static_dma_buf[4096];

// WRONG: PSRAM buffer (not DMA-accessible on base ESP32)
uint8_t* psram_buf = (uint8_t*)heap_caps_malloc(4096, MALLOC_CAP_SPIRAM);
// This will cause silent data corruption or DMA errors
```

<Callout type="warning" title="PSRAM and SPI DMA Do Not Mix on ESP32">
  The base ESP32 (non-S2/S3) cannot use PSRAM-allocated buffers for SPI DMA transfers. PSRAM is accessed through SPI0/SPI1, creating a bus conflict with the SPI DMA engine. Attempting to DMA from PSRAM results in corrupted data or bus errors. On the ESP32-S2 and ESP32-S3, this restriction is partially lifted due to improved bus architecture, but even there, throughput from PSRAM is lower than from internal SRAM.
</Callout>

### Why Efficiency Drops at Higher Clocks

The efficiency degradation at higher clock rates comes from three sources:

1. **Fixed per-transaction overhead**: CS assertion/deassertion, DMA setup, and interrupt handling take a fixed number of CPU cycles (approximately 5-8 microseconds per transaction). At 1 MHz, a 4096-byte transfer takes 32.8 ms, so 8 microseconds is 0.02% overhead. At 80 MHz, the same transfer takes 0.41 ms, so 8 microseconds is 2% overhead.

2. **Inter-transaction gaps**: Between consecutive DMA transactions, there is a gap where CS is deasserted and the next descriptor chain is loaded. This gap is approximately 1-3 microseconds regardless of clock speed, becoming proportionally larger at higher speeds.

3. **DMA descriptor fetch stalls**: The DMA controller must read the next descriptor from memory. If the bus is busy (e.g., CPU accessing SRAM simultaneously), the descriptor fetch is delayed by one or more bus arbitration cycles.

A single 14-byte burst read of an IMU's accelerometer, gyroscope, and temperature registers is approximately 7x more efficient than seven individual 2-byte reads at 40 MHz.
</Callout>

## SPI Slave Mode

The ESP32 can also operate as an SPI slave, useful for inter-processor communication or receiving data from an external master. Slave mode has its own set of throughput considerations.

```cpp
#include "driver/spi_slave.h"

#define SLAVE_BUF_SIZE 4096

static DRAM_ATTR uint8_t slave_rx_buf[SLAVE_BUF_SIZE];
static DRAM_ATTR uint8_t slave_tx_buf[SLAVE_BUF_SIZE];

void spi_slave_init(void) {
    spi_bus_config_t bus_cfg = {
        .mosi_io_num = 13,
        .miso_io_num = 12,
        .sclk_io_num = 14,
        .quadwp_io_num = -1,
        .quadhd_io_num = -1,
    };

    spi_slave_interface_config_t slave_cfg = {
        .spics_io_num = 15,
        .flags = 0,
        .queue_size = 3,
        .mode = 0,
        .post_setup_cb = NULL,
        .post_trans_cb = NULL,
    };

    spi_slave_initialize(HSPI_HOST, &bus_cfg, &slave_cfg, SPI_DMA_CH_AUTO);
}

// Queue a slave transaction (non-blocking)
void spi_slave_queue_rx(void) {
    spi_slave_transaction_t trans = {
        .length = SLAVE_BUF_SIZE * 8,
        .tx_buffer = slave_tx_buf,
        .rx_buffer = slave_rx_buf,
    };

    spi_slave_queue_trans(HSPI_HOST, &trans, portMAX_DELAY);
}
```

## Display Driver Optimization: SPI LCD

One of the most common high-throughput SPI applications on ESP32 is driving SPI-connected LCD displays (ILI9341, ST7789, ST7735, etc.). Display updates require pushing large framebuffers at the highest possible rate.

### Display SPI Configuration for Maximum Fill Rate

```cpp
#include "driver/spi_master.h"

// Optimized display SPI configuration
void display_spi_init(spi_device_handle_t* dev) {
    spi_bus_config_t bus_cfg = {
        .mosi_io_num = 13,          // IOMUX pin for HSPI
        .miso_io_num = -1,          // Display is write-only (no MISO needed)
        .sclk_io_num = 14,          // IOMUX pin for HSPI
        .quadwp_io_num = -1,
        .quadhd_io_num = -1,
        .max_transfer_sz = 240 * 320 * 2,  // Full frame (153,600 bytes for 240x320 RGB565)
    };

    spi_bus_initialize(HSPI_HOST, &bus_cfg, SPI_DMA_CH_AUTO);

    spi_device_interface_config_t dev_cfg = {
        .clock_speed_hz = 40 * 1000 * 1000,  // 40 MHz (safe for most displays)
        .mode = 0,
        .spics_io_num = 15,
        .queue_size = 7,
        .flags = SPI_DEVICE_HALFDUPLEX | SPI_DEVICE_NO_DUMMY,
        .pre_cb = display_pre_transfer_callback,
    };

    spi_bus_add_device(HSPI_HOST, &dev_cfg, dev);
}

// Pre-transfer callback: toggle D/C pin for command vs data
static void display_pre_transfer_callback(spi_transaction_t* t) {
    int dc = (int)t->user;
    gpio_set_level(PIN_DC, dc);
}

// Send a full framebuffer to the display
void display_send_framebuffer(spi_device_handle_t dev,
                               const uint16_t* fb, size_t pixel_count) {
    spi_transaction_t trans = {
        .length = pixel_count * 16,  // 16 bits per pixel (RGB565)
        .tx_buffer = fb,
        .user = (void*)1,  // D/C = 1 (data mode)
    };
    spi_device_transmit(dev, &trans);
}
```

### Double-Buffered Display Pipeline

For smooth animation, use double-buffering: render into one framebuffer while DMA transmits the other:

```cpp
#define FB_WIDTH  240
#define FB_HEIGHT 320
#define FB_SIZE   (FB_WIDTH * FB_HEIGHT * 2)  // RGB565

static DRAM_ATTR uint16_t framebuffer_a[FB_WIDTH * FB_HEIGHT];
static DRAM_ATTR uint16_t framebuffer_b[FB_WIDTH * FB_HEIGHT];

static uint16_t* front_buffer = framebuffer_a;  // Being displayed (DMA)
static uint16_t* back_buffer  = framebuffer_b;  // Being rendered (CPU)

void display_swap_buffers(spi_device_handle_t dev) {
    // Wait for previous DMA transfer to complete
    spi_transaction_t* completed;
    spi_device_get_trans_result(dev, &completed, portMAX_DELAY);

    // Swap buffers
    uint16_t* temp = front_buffer;
    front_buffer = back_buffer;
    back_buffer = temp;

    // Queue new front buffer for DMA transfer
    spi_transaction_t trans = {
        .length = FB_SIZE * 8,
        .tx_buffer = front_buffer,
        .user = (void*)1,
    };
    spi_device_queue_trans(dev, &trans, portMAX_DELAY);

    // CPU can now render into back_buffer while DMA sends front_buffer
}
```

### When to Choose Which Interface

**Use SPI when:**
- Throughput is critical (displays, high-speed sensors, inter-MCU communication)
- You have pins available for per-device chip select
- The device count is small (1-3 devices)
- You need DMA for CPU offloading

**Use I2C when:**
- Pin count is constrained (only 2 GPIOs available for the bus)
- Many low-speed devices share the bus (accelerometer, temperature, humidity, pressure, RTC, EEPROM)
- Throughput requirements are under 50 KB/s
- Hot-plug capability is needed (I2C is more robust to dynamic device attachment)

**Use UART when:**
- Communicating with a host PC or debug console
- Point-to-point links over longer distances
- The peripheral only supports UART (GPS modules, cellular modems, some Bluetooth modules)
- You need asynchronous communication (no shared clock)

## Measuring Throughput Correctly

Accurate throughput measurement requires careful methodology. The two approaches -- software timing and logic analyzer -- give different views:

```cpp
#include "esp_timer.h"
#include "driver/spi_master.h"

void benchmark_spi_throughput(spi_device_handle_t dev, size_t burst_bytes,
                               int iterations) {
    uint8_t* buf = (uint8_t*)heap_caps_malloc(burst_bytes, MALLOC_CAP_DMA);
    memset(buf, 0xAA, burst_bytes);

    // Warm-up: 10 transfers to stabilize caches and DMA state
    for (int i = 0; i < 10; i++) {
        spi_transaction_t t = { .length = burst_bytes * 8, .tx_buffer = buf };
        spi_device_transmit(dev, &t);
    }

    // Timed benchmark
    int64_t t0 = esp_timer_get_time();

    for (int i = 0; i < iterations; i++) {
        spi_transaction_t t = { .length = burst_bytes * 8, .tx_buffer = buf };
        spi_device_transmit(dev, &t);
    }

    int64_t dt = esp_timer_get_time() - t0;

    double total_bytes = (double)iterations * burst_bytes;
    double seconds = dt / 1e6;
    double mbps = total_bytes / seconds / (1024.0 * 1024.0);

    printf("Burst: %u bytes, Iterations: %d\n", burst_bytes, iterations);
    printf("Total: %.1f KB in %.3f ms\n", total_bytes / 1024.0, dt / 1000.0);
    printf("Throughput: %.2f MB/s\n", mbps);
    printf("Per-transaction: %.1f us\n", (double)dt / iterations);

    heap_caps_free(buf);
}
```

<Callout type="tip" title="Logic Analyzer Validation">
  Software timing measures end-to-end throughput including all software overhead. A logic analyzer on SCLK and CS measures actual bus utilization. The difference between the two reveals software overhead. For example, if the logic analyzer shows 92% bus utilization but software measures 77% efficiency, the remaining 15% is software overhead (interrupt handling, DMA setup, task scheduling). Both measurements are valuable: software timing tells you the throughput your application actually gets, while the logic analyzer tells you how much headroom the bus has.
</Callout>

## Optimization Checklist

To summarize the critical optimization points:

1. **Use IOMUX pins** for clock rates above 26 MHz. This is the single largest performance lever.
2. **Enable DMA** for all transfers larger than 32 bytes. CPU-driven transfers waste cycles in busy-wait loops.
3. **Maximize burst size** to at least 256 bytes, preferably 1024+ bytes. Per-transaction overhead dominates small transfers.
4. **Pipeline transactions** using `spi_device_queue_trans()` with multiple buffers. Keep the DMA engine fed.
5. **Use half-duplex mode** for unidirectional transfers (displays, DACs, flash writes) to avoid the 4092-byte DMA size limit.
6. **Place critical code in IRAM** if you are pushing maximum throughput alongside WiFi or heavy flash access.
7. **Use DMA-capable memory** (`MALLOC_CAP_DMA` or `DRAM_ATTR`). PSRAM buffers silently corrupt data.
8. **Verify with a logic analyzer**. Software benchmarks miss bus-level issues like clock glitches, signal integrity problems, and CS timing violations.
9. **Profile the full system**. WiFi, Bluetooth, flash reads, and PSRAM access all compete for internal bus bandwidth and can reduce SPI throughput by 20-30%.

## Conclusion

The ESP32 SPI peripheral is capable of 6.7 MB/s measured throughput at 80 MHz with IOMUX routing, or 3.85 MB/s at the more commonly used 40 MHz. The gap between theoretical and measured throughput (23-33% at the highest speeds) comes from per-transaction overhead, DMA descriptor fetch latency, and internal bus contention. For applications like display driving, these numbers translate to 24-40 fps for a 240x320 RGB565 display, which is adequate for most embedded UIs. The key optimization levers are IOMUX pin selection, DMA with pipelined transactions, adequate burst sizing, and minimizing bus contention with flash and WiFi. With these techniques applied correctly, SPI throughput on the ESP32 approaches the silicon's fundamental limits.
