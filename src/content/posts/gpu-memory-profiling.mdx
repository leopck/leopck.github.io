GPU out-of-memory (OOM) errors are the most common production issue in LLM serving. The challenge isn't that memory is insufficient -- it's that memory is being wasted by fragmentation, leaks, or hidden overhead. Profiling reveals where memory actually goes, often uncovering 20-40% waste that can be reclaimed.

The PyTorch caching allocator and fragmentation together consume ~13 GB (16% of GPU memory). Understanding and minimizing this overhead directly increases serving capacity.

## Tool 1: torch.cuda.memory_stats

The most useful built-in profiling tool:

```python
import torch

def print_memory_summary():
    stats = torch.cuda.memory_stats()
    allocated = stats['allocated_bytes.all.current'] / 1e9
    reserved = stats['reserved_bytes.all.current'] / 1e9
    fragmentation = reserved - allocated

    print(f"Allocated: {allocated:.2f} GB (actual tensors)")
    print(f"Reserved:  {reserved:.2f} GB (PyTorch allocator holds)")
    print(f"Fragmented: {fragmentation:.2f} GB ({fragmentation/reserved*100:.1f}% waste)")
    print(f"Peak: {stats['allocated_bytes.all.peak'] / 1e9:.2f} GB")
    print(f"# Allocations: {stats['allocation.all.current']}")

# Call periodically during serving
print_memory_summary()
```

<Callout type="tip" title="The Reserved vs Allocated Gap">
  If `reserved_bytes` is 20%+ higher than `allocated_bytes`, you have fragmentation. PyTorch's caching allocator holds freed blocks for reuse, but if allocation patterns vary widely (different sequence lengths, different batch sizes), blocks can't be reused efficiently. The fix: use `torch.cuda.empty_cache()` periodically, or better, pre-allocate fixed-size pools (which vLLM does via PagedAttention).
</Callout>

## Profiling Workflow

The systematic approach to memory profiling:

```python
# Step 1: Baseline measurement
torch.cuda.reset_peak_memory_stats()
baseline_allocated = torch.cuda.memory_allocated()

# Step 2: Run workload
output = model.generate(input_ids, max_new_tokens=100)

# Step 3: Check peak and current
peak = torch.cuda.max_memory_allocated()
current = torch.cuda.memory_allocated()
print(f"Peak: {peak/1e9:.2f} GB, Current: {current/1e9:.2f} GB")
print(f"Transient: {(peak-current)/1e9:.2f} GB (freed after generation)")

# Step 4: Check for leaks across requests
for i in range(10):
    output = model.generate(input_ids, max_new_tokens=100)
    del output
    torch.cuda.empty_cache()
    print(f"Request {i}: {torch.cuda.memory_allocated()/1e9:.2f} GB allocated")
    # If this number grows, you have a memory leak
```

<Callout type="warning" title="Memory Leaks Are Subtle">
  Python's garbage collector may not immediately free CUDA tensors that have circular references or are held by closures/callbacks. Use `torch.cuda.memory_allocated()` after explicit `del` + `torch.cuda.empty_cache()` to detect slow leaks. A growth of even 1 MB per request adds up to OOM after thousands of requests in production.
</Callout>

## Conclusion

GPU memory profiling is essential for production LLM serving. The key tools are `torch.cuda.memory_stats()` for quick checks, `torch.cuda.memory._snapshot()` for detailed analysis, and Nsight Systems for timeline correlation. The most common issue is KV cache waste (fixed by PagedAttention), followed by allocator fragmentation (fixed by periodic cache clearing or pre-allocated pools). A systematic profiling workflow -- measure baseline, run workload, check peak vs steady-state, test for leaks across requests -- catches issues before they cause production OOMs.
