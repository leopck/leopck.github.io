GPU out-of-memory (OOM) errors are the single most common production failure mode in LLM serving infrastructure. They are also the most frustrating -- an OOM can strike minutes, hours, or days after deployment, often in the middle of a traffic spike, and the default error message tells you almost nothing about where the memory went. The challenge is not that GPUs lack memory. Modern accelerators ship with 40 GB, 80 GB, or even 192 GB of HBM. The challenge is that memory is being wasted by fragmentation, leaked by subtle reference cycles, or consumed by overhead that is invisible without proper instrumentation.

Profiling reveals where memory actually goes. In our experience across dozens of production deployments, profiling consistently uncovers 20-40% memory waste that can be reclaimed -- waste that translates directly into higher batch sizes, longer supported sequence lengths, and more concurrent users per GPU. This post is the comprehensive guide to finding and fixing that waste.

## Why Profiling GPU Memory Matters

Before diving into tools, it is worth understanding why GPU memory profiling is fundamentally different from CPU memory profiling and why it demands specialized attention.

**OOM is the number one deployment killer.** In a survey of production LLM incidents at multiple organizations, OOM accounted for more than 60% of unplanned serving downtime. Unlike CPU OOM, which the OS can mitigate with swap or the OOM killer, GPU OOM is immediate and fatal to the serving process. There is no swap space on the GPU. When you exhaust HBM, the CUDA runtime throws an error and the entire inference request (and often the entire serving process) fails.

**Memory fragmentation is insidious.** The PyTorch caching allocator holds freed blocks for reuse. Over time, as allocation patterns shift -- different sequence lengths, varying batch sizes, model switching -- the allocator ends up with a patchwork of small free blocks that cannot satisfy large allocation requests even though total free memory is sufficient. This is the GPU equivalent of filesystem fragmentation, and it can waste 5-15% of GPU memory silently.

**KV cache growth is the dominant variable.** For autoregressive LLMs, the key-value cache grows linearly with both batch size and sequence length. A single request generating 4096 tokens on a 70B parameter model can consume 2-4 GB of KV cache alone. Multiply by a batch of 32 concurrent requests and the KV cache dwarfs the model weights themselves. Understanding and managing this growth requires continuous measurement.

**Activation peaks are transient but deadly.** During the forward pass, intermediate activations can spike memory usage 2-4x above steady state. These peaks last milliseconds but are enough to trigger OOM. Profiling reveals the exact layer and operation where the peak occurs, enabling targeted optimization through activation checkpointing or operation fusion.

The PyTorch caching allocator and fragmentation together consume approximately 13 GB -- 16% of the GPU's total memory. On a machine costing $2-3 per hour, that waste represents thousands of dollars per year in lost capacity. Understanding and minimizing this overhead directly increases serving throughput.

<Callout type="warning" title="Memory Accounting Is Non-Trivial">
  The numbers reported by `nvidia-smi` do not match `torch.cuda.memory_allocated()` which does not match what your model actually needs. `nvidia-smi` reports total GPU memory consumed by the process, including CUDA context, driver buffers, and the caching allocator's reserved pool. `torch.cuda.memory_allocated()` reports only the memory actively used by tensors. The gap between these numbers -- often 2-8 GB -- is where the waste hides. Profiling closes this gap.
</Callout>

## Nsight Compute Deep Dive

NVIDIA Nsight Compute is the definitive tool for understanding what happens inside individual CUDA kernels. While it is primarily a compute profiler, its memory analysis capabilities are essential for understanding how kernels interact with GPU memory subsystems.

### Speed-of-Light Analysis

The Speed-of-Light (SOL) section in Nsight Compute shows how close a kernel is to the theoretical maximum performance of the GPU, broken into compute utilization and memory utilization.

```bash
# Profile a specific kernel with full metrics
ncu --set full \
    --kernel-name "volta_fp16_s884gemm" \
    --launch-count 5 \
    -o gemm_profile \
    python inference.py
```

The SOL report yields two critical percentages:

- **Compute (SM) Throughput**: What fraction of the GPU's peak FLOPS the kernel achieves. For matrix multiplications in LLM inference, well-optimized kernels reach 70-85% on A100.
- **Memory (HBM) Throughput**: What fraction of peak memory bandwidth the kernel consumes. Memory-bound kernels like layer normalization, softmax, and element-wise operations often hit 80-90%.

### Memory Chart and Roofline View

The Memory Chart in Nsight Compute visualizes the data flow through the GPU memory hierarchy: from HBM through L2 cache, L1/shared memory, and into registers. For each level, it shows achieved bandwidth versus peak bandwidth.

```bash
# Collect memory hierarchy metrics
ncu --metrics \
    l1tex__t_bytes_pipe_lsu_mem_global_op_ld.sum,\
    l1tex__t_bytes_pipe_lsu_mem_global_op_st.sum,\
    lts__t_bytes.sum,\
    dram__bytes.sum,\
    sm__warps_active.avg.pct_of_peak_sustained_active \
    --kernel-name "attention_kernel" \
    python inference.py
```

The **Roofline View** plots each kernel on a chart where the x-axis is arithmetic intensity (FLOP/byte) and the y-axis is achieved performance (GFLOP/s). The roofline itself is formed by two lines: the memory bandwidth ceiling (diagonal) and the compute ceiling (horizontal). Kernels below the roofline are underperforming and the position tells you whether to optimize for compute or memory.

For LLM inference, the roofline is particularly revealing:

- **Prefill phase**: Large matrix multiplications with high arithmetic intensity. These kernels should be near the compute ceiling. If they fall below, suspect suboptimal tile sizes or failure to use TensorCore paths.
- **Decode phase**: Small batch GEMV operations with low arithmetic intensity. These are inherently memory-bandwidth-bound. The roofline confirms that no amount of compute optimization will help -- you need to reduce memory traffic (quantization, KV cache compression).

<Callout type="tip" title="Roofline for Decode: The Fundamental Limit">
  During autoregressive decode with batch size 1, the GEMV operation has arithmetic intensity of approximately $2 \times N / (2 \times N \times 2) = 0.5$ FLOP/byte for FP16 weights. On an A100 with 2 TB/s memory bandwidth and 312 TFLOP/s compute, the memory bandwidth ceiling at 0.5 FLOP/byte is $2000 \times 0.5 = 1000$ GFLOP/s -- only 0.3% of peak compute. This is why decode is memory-bound and why quantization (which reduces bytes per weight) has such dramatic speedup effects.
</Callout>

### Source-Level Metrics

Nsight Compute can map performance metrics back to specific source lines in CUDA kernels. This is invaluable for custom kernel development:

```bash
# Source-level profiling requires debug info
ncu --set source \
    --import-source yes \
    --source-folders /path/to/kernel/source \
    --kernel-name "custom_attention" \
    python inference.py
```

Source-level metrics show, for each line of CUDA code, how many cycles were spent, how many memory transactions were generated, and what the achieved occupancy was. This makes it possible to identify specific lines causing excessive memory traffic or pipeline stalls.

### Walking Through a Real Profiling Session

Let us walk through profiling a FlashAttention kernel to understand memory behavior:

```bash
# Step 1: Quick overview with default metrics
ncu --target-processes all \
    --set default \
    -o flash_attn_overview \
    python -c "
import torch
from flash_attn import flash_attn_func

q = torch.randn(2, 32, 2048, 128, dtype=torch.float16, device='cuda')
k = torch.randn(2, 32, 2048, 128, dtype=torch.float16, device='cuda')
v = torch.randn(2, 32, 2048, 128, dtype=torch.float16, device='cuda')
for _ in range(5):
    out = flash_attn_func(q, k, v)
torch.cuda.synchronize()
"
```

Opening the report in the Nsight Compute GUI reveals:

1. **Launch Statistics**: Grid size, block size, registers per thread, shared memory per block. FlashAttention typically uses 128 threads per block, 226 registers per thread, and 100+ KB of shared memory -- nearly maxing out A100's 164 KB shared memory capacity.
2. **SOL Analysis**: FlashAttention achieves 60-75% compute SOL and 40-55% memory SOL, confirming it is a balanced (neither purely compute-bound nor purely memory-bound) kernel.
3. **Memory Workload**: The key metric is L2 cache hit rate. FlashAttention's tiled algorithm is designed to keep data in SRAM, so L2 hit rates should be above 80%. If they drop below 60%, the tile size configuration may be wrong for the sequence length.
4. **Occupancy**: Achieved occupancy is typically 50-75% for FlashAttention due to high register and shared memory usage. This is acceptable because the kernel is designed to maximize data reuse per thread rather than maximize parallelism.

```bash
# Step 2: Deep dive into memory behavior
ncu --set full \
    --kernel-name "flash_fwd_kernel" \
    --launch-skip 2 --launch-count 1 \
    -o flash_attn_memory_deep \
    python flash_attn_bench.py
```

The full metric set reveals HBM read/write volumes. For a forward pass with sequence length $S$, head dimension $D$, and $H$ heads, the expected HBM reads are approximately $O(S^2 \cdot D / M)$ where $M$ is SRAM size per block -- compared to $O(S^2 \cdot D)$ for standard attention. Nsight Compute confirms this IO reduction empirically.

## Nsight Systems Timeline

While Nsight Compute examines individual kernels in isolation, Nsight Systems provides the timeline view -- how kernels, memory copies, CPU operations, and CUDA API calls interleave over time. For LLM inference profiling, the timeline is often more valuable than per-kernel metrics because serving performance depends on how well operations overlap.

### Capturing a Timeline

```bash
# Comprehensive timeline capture for LLM serving
nsys profile \
    -t cuda,nvtx,osrt,cudnn,cublas \
    --cuda-memory-usage=true \
    --cuda-graph-trace=node \
    --gpu-metrics-device=all \
    --gpu-metrics-frequency=10000 \
    -d 30 \
    -o vllm_timeline \
    python -m vllm.entrypoints.openai.api_server \
        --model meta-llama/Llama-2-13b-hf \
        --max-model-len 4096
```

The flags deserve explanation:

- `-t cuda,nvtx,osrt,cudnn,cublas`: Trace CUDA runtime, NVTX annotations (used by PyTorch and vLLM for marking regions), OS runtime calls, cuDNN, and cuBLAS.
- `--cuda-memory-usage=true`: Track GPU memory allocations and deallocations on the timeline.
- `--gpu-metrics-device=all`: Collect GPU-level metrics (SM utilization, memory bandwidth, PCIe throughput) at high frequency.
- `-d 30`: Capture 30 seconds of execution -- enough to see multiple inference iterations.

### Reading the Timeline

The Nsight Systems GUI presents several rows:

**CUDA HW row**: Shows actual GPU kernel execution on each SM. Gaps between kernels indicate the GPU is idle -- waiting for CPU to issue work. In well-optimized serving, gaps should be minimal during active inference.

**CUDA API row**: Shows host-side CUDA API calls (cudaLaunchKernel, cudaMemcpyAsync, cudaMalloc). Long API calls indicate driver overhead or synchronization.

**Memory row**: When `--cuda-memory-usage` is enabled, shows allocation and deallocation events. Spikes in this row correlate with KV cache expansion or activation allocation.

**NVTX row**: Shows annotated regions. PyTorch marks forward/backward passes, and vLLM marks scheduler steps, attention computation, and sampling. These annotations are essential for correlating GPU activity with logical operations.

### Identifying Gaps and Serialization

The most actionable finding in Nsight Systems is GPU idle time. Here is how to quantify it:

```bash
# Export timeline statistics
nsys stats vllm_timeline.nsys-rep \
    --report cuda_gpu_kern_sum \
    --format csv \
    -o kernel_summary

nsys stats vllm_timeline.nsys-rep \
    --report cuda_api_sum \
    --format csv \
    -o api_summary
```

The kernel summary report shows total GPU kernel time versus wall clock time. If total kernel time is 20 seconds but wall clock is 30 seconds, the GPU was idle 33% of the time. The API summary reveals which CUDA API calls consumed the most host time -- often `cudaLaunchKernel` overhead dominates when launching thousands of small kernels.

<Callout type="perf" title="CUDA Graphs Eliminate Launch Overhead">
  In autoregressive decode, the same sequence of kernels executes repeatedly with different data. CUDA Graphs capture this sequence once and replay it with a single launch, reducing per-step CPU overhead from 200-500 us to 10-20 us. Nsight Systems makes the improvement visually obvious: the GPU idle gaps between kernels disappear. vLLM and SGLang both support CUDA Graphs for decode steps.
</Callout>

## torch.profiler: PyTorch-Native Profiling

The PyTorch profiler provides the bridge between Python-level code and GPU execution. Unlike Nsight tools, which operate at the CUDA level, `torch.profiler` understands PyTorch operations -- it can tell you that a particular `nn.Linear` layer consumed a specific amount of memory and time.

### Basic Profiling Setup

```python
import torch
from torch.profiler import profile, record_function, ProfilerActivity

model = load_model()  # Your LLM
input_ids = torch.randint(0, 32000, (1, 512), device='cuda')

with profile(
    activities=[ProfilerActivity.CPU, ProfilerActivity.CUDA],
    record_shapes=True,
    profile_memory=True,
    with_stack=True,
    with_flops=True,
    schedule=torch.profiler.schedule(
        wait=1,    # Skip first iteration (warmup)
        warmup=1,  # Profile but discard (JIT compilation)
        active=3,  # Actually record these iterations
        repeat=1
    ),
    on_trace_ready=torch.profiler.tensorboard_trace_handler('./profiler_logs')
) as prof:
    for step in range(5):
        with record_function("inference_step"):
            with record_function("forward_pass"):
                output = model(input_ids)
            with record_function("sampling"):
                next_token = torch.argmax(output.logits[:, -1, :], dim=-1)
        prof.step()

# Print summary sorted by GPU memory usage
print(prof.key_averages().table(
    sort_by="self_cuda_memory_usage",
    row_limit=20
))
```

### Memory Timeline

The memory timeline is the most valuable feature for GPU memory profiling. It shows memory allocation and deallocation over time, broken down by operator:

```python
with profile(
    activities=[ProfilerActivity.CPU, ProfilerActivity.CUDA],
    profile_memory=True,
    record_shapes=True,
    with_stack=True,
) as prof:
    output = model.generate(input_ids, max_new_tokens=256)

# Export for Chrome trace viewer
prof.export_chrome_trace("memory_timeline.json")

# Export memory timeline specifically
# Open in TensorBoard: tensorboard --logdir=./profiler_logs
```

The Chrome trace (viewable at `chrome://tracing`) shows:

1. **Allocation waterfall**: Each tensor allocation appears as a block, with height proportional to size. You can see the KV cache growing step by step during autoregressive generation.
2. **Peak identification**: The highest point in the memory waterfall is the peak -- the moment most likely to trigger OOM. The stack trace at the peak tells you exactly which operation caused it.
3. **Deallocation patterns**: Freed tensors appear as negative blocks. Delayed deallocation (tensors freed long after they were last used) indicates opportunities for earlier memory release.

### CPU/GPU Correlation

The torch.profiler timeline shows both CPU and GPU events on the same time axis, with arrows connecting CPU launch calls to GPU kernel execution. This reveals:

- **Launch latency**: Time between CPU issuing `cudaLaunchKernel` and the GPU starting execution. Typically 5-15 us, but can spike to 100+ us under contention.
- **CPU-GPU overlap**: Whether CPU is doing useful work while the GPU executes. In well-pipelined inference, the CPU should be preparing the next batch while the GPU processes the current one.
- **Synchronization points**: Any `torch.cuda.synchronize()` or implicit sync (like `.item()` or `.cpu()`) shows as a CPU stall waiting for the GPU. These are performance killers in serving.

```python
# Common sync point that kills performance
# BAD: implicit sync
logits = model(input_ids)
next_token = logits[:, -1, :].argmax(dim=-1).item()  # .item() forces sync!

# GOOD: batch operations on GPU
logits = model(input_ids)
next_token = logits[:, -1, :].argmax(dim=-1)  # stays on GPU
# Only sync when absolutely necessary (e.g., returning to user)
```

<Callout type="warning" title="The .item() Trap">
  Calling `.item()`, `.cpu()`, `.numpy()`, or `print()` on a CUDA tensor forces a full device synchronization. In a decode loop generating 1000 tokens, a single `.item()` per step adds 5-15 ms of synchronization latency per step -- totaling 5-15 seconds of pure waste. The torch.profiler timeline makes these sync points clearly visible as CPU idle periods waiting for GPU completion.
</Callout>

## Memory Debugging: The Deep Dive

When the high-level profiling tools point to a memory problem, you need lower-level debugging tools to identify the root cause. PyTorch provides several APIs for this.

### torch.cuda.memory_stats()

This is the most comprehensive single function for understanding GPU memory state:

```python
import torch

def print_memory_report():
    stats = torch.cuda.memory_stats()

    # Current state
    allocated = stats['allocated_bytes.all.current'] / 1e9
    reserved = stats['reserved_bytes.all.current'] / 1e9
    active = stats['active_bytes.all.current'] / 1e9

    # Peak state
    peak_allocated = stats['allocated_bytes.all.peak'] / 1e9
    peak_reserved = stats['reserved_bytes.all.peak'] / 1e9

    # Fragmentation indicators
    num_allocs = stats['allocation.all.current']
    num_segments = stats['segment.all.current']
    inactive = stats['inactive_split_bytes.all.current'] / 1e9

    # OOM tracking
    num_ooms = stats.get('num_ooms', 0)
    num_retries = stats.get('num_alloc_retries', 0)

    print("=== GPU Memory Report ===")
    print(f"Allocated:     {allocated:.2f} GB (tensors in use)")
    print(f"Active:        {active:.2f} GB (blocks in use)")
    print(f"Reserved:      {reserved:.2f} GB (held by allocator)")
    print(f"Fragmented:    {reserved - allocated:.2f} GB "
          f"({(reserved - allocated)/reserved*100:.1f}% waste)")
    print(f"Inactive split: {inactive:.2f} GB (split blocks, unusable)")
    print(f"Peak alloc:    {peak_allocated:.2f} GB")
    print(f"Peak reserved: {peak_reserved:.2f} GB")
    print(f"# Allocations: {num_allocs}")
    print(f"# Segments:    {num_segments}")
    print(f"# OOMs:        {num_ooms}")
    print(f"# Retries:     {num_retries}")

    # Health checks
    if reserved > 0 and (reserved - allocated) / reserved > 0.2:
        print("WARNING: High fragmentation! Consider torch.cuda.empty_cache()")
    if num_retries > 0:
        print(f"WARNING: {num_retries} allocation retries -- memory pressure!")
    if inactive > 1.0:
        print(f"WARNING: {inactive:.1f} GB in inactive splits -- severe fragmentation")

print_memory_report()
```

### max_memory_allocated vs max_memory_reserved

These two functions are commonly confused:

```python
# Reset tracking
torch.cuda.reset_peak_memory_stats()

# Run workload
output = model.generate(input_ids, max_new_tokens=512)

# What tensors needed at peak
peak_alloc = torch.cuda.max_memory_allocated() / 1e9
# What the allocator reserved from CUDA at peak
peak_reserved = torch.cuda.max_memory_reserved() / 1e9
# What nvidia-smi shows (approximately)
total_used = torch.cuda.mem_get_info()[1] / 1e9 - torch.cuda.mem_get_info()[0] / 1e9

print(f"Peak tensor usage:     {peak_alloc:.2f} GB")
print(f"Peak allocator reserve:{peak_reserved:.2f} GB")
print(f"nvidia-smi shows:      {total_used:.2f} GB")
print(f"Allocator overhead:    {peak_reserved - peak_alloc:.2f} GB")
print(f"CUDA context overhead: {total_used - peak_reserved:.2f} GB")
```

The relationship is always: `nvidia-smi >= max_memory_reserved >= max_memory_allocated`. The gap between `reserved` and `allocated` is the caching allocator's overhead plus fragmentation. The gap between `nvidia-smi` and `reserved` is the CUDA context, driver buffers, and any non-PyTorch CUDA allocations.

<Callout type="tip" title="The Reserved vs Allocated Gap">
  If `reserved_bytes` is 20% or more higher than `allocated_bytes`, you have fragmentation. PyTorch's caching allocator holds freed blocks for reuse, but if allocation patterns vary widely (different sequence lengths, different batch sizes, model switching), blocks cannot be reused efficiently. The fixes, in order of effectiveness: (1) use `torch.cuda.empty_cache()` between requests; (2) pre-allocate fixed-size pools as vLLM does with PagedAttention; (3) configure `PYTORCH_CUDA_ALLOC_CONF=max_split_size_mb:128` to limit block splitting; (4) use `expandable_segments:True` to allow the allocator to return memory to CUDA.
</Callout>

### Memory Snapshots

For the most detailed view of every allocation in GPU memory, use memory snapshots:

```python
# Enable recording BEFORE running the workload
torch.cuda.memory._record_memory_history(
    max_entries=100000,
    stacks='all'  # Capture Python + C++ stacks
)

# Run workload
for i in range(10):
    output = model.generate(input_ids, max_new_tokens=100)
    del output
    torch.cuda.empty_cache()

# Capture snapshot
snapshot = torch.cuda.memory._snapshot()

# Save for visualization
from pickle import dump
with open("memory_snapshot.pickle", "wb") as f:
    dump(snapshot, f)

# Disable recording
torch.cuda.memory._record_memory_history(enabled=None)
```

Upload the pickle file to [pytorch.org/memory_viz](https://pytorch.org/memory_viz) for an interactive visualization that shows:

- **Allocation timeline**: Every allocation and deallocation plotted over time, with size on the y-axis. The "waterfall" view immediately reveals leaks (allocations that never get freed) and fragmentation (many small gaps between allocations).
- **Active memory plot**: Total active memory over time, showing peaks and valleys. Peaks during each inference step should be roughly equal -- growing peaks indicate a leak.
- **Stack traces**: Click any allocation to see the full Python and C++ stack trace. This identifies which line of code created the tensor.
- **Category breakdown**: Allocations are categorized by type (parameter, gradient, activation, temporary). For inference, you should see no gradients -- if they appear, something is wrong.

### Detecting Fragmentation

Fragmentation manifests as a growing gap between `reserved` and `allocated` memory over time. Here is a systematic detection approach:

```python
def monitor_fragmentation(model, input_fn, num_iterations=100):
    """Run repeated inference and track fragmentation growth."""
    history = []

    for i in range(num_iterations):
        inputs = input_fn()  # Generate varied inputs
        output = model.generate(**inputs)
        del output

        stats = torch.cuda.memory_stats()
        allocated = stats['allocated_bytes.all.current']
        reserved = stats['reserved_bytes.all.current']
        inactive_split = stats['inactive_split_bytes.all.current']
        num_segments = stats['segment.all.current']

        history.append({
            'step': i,
            'allocated_gb': allocated / 1e9,
            'reserved_gb': reserved / 1e9,
            'frag_gb': (reserved - allocated) / 1e9,
            'frag_pct': (reserved - allocated) / reserved * 100 if reserved > 0 else 0,
            'inactive_split_gb': inactive_split / 1e9,
            'num_segments': num_segments,
        })

        if i % 10 == 0:
            print(f"Step {i}: alloc={history[-1]['allocated_gb']:.2f} GB, "
                  f"frag={history[-1]['frag_pct']:.1f}%, "
                  f"segments={num_segments}")

    # Check for fragmentation growth
    early_frag = sum(h['frag_pct'] for h in history[:10]) / 10
    late_frag = sum(h['frag_pct'] for h in history[-10:]) / 10

    if late_frag > early_frag * 1.5:
        print(f"FRAGMENTATION GROWING: {early_frag:.1f}% -> {late_frag:.1f}%")
        print("Consider: empty_cache(), fixed pools, or max_split_size_mb")
    else:
        print(f"Fragmentation stable: {early_frag:.1f}% -> {late_frag:.1f}%")

    return history
```

The key indicator is `inactive_split_bytes`. When the caching allocator splits a large block to satisfy a small allocation, the remaining piece becomes an "inactive split." If these accumulate, it means the allocator is creating many small fragments that cannot be merged back. This is the most direct measure of fragmentation.

## OOM Debugging Methodology

When an OOM error occurs, follow this systematic methodology to diagnose and fix it.

### Step 1: Capture the Error Context

```python
import torch
import traceback
import json
from datetime import datetime

def oom_safe_inference(model, inputs, **kwargs):
    """Wrapper that captures detailed info on OOM."""
    # Record state before inference
    torch.cuda.reset_peak_memory_stats()
    pre_alloc = torch.cuda.memory_allocated()
    pre_reserved = torch.cuda.memory_reserved()

    try:
        return model.generate(**inputs, **kwargs)
    except torch.cuda.OutOfMemoryError as e:

        report = {
            'timestamp': datetime.now().isoformat(),
```

### Step 3: Calculate Memory Budget

Before reaching for optimizations, calculate what your workload actually needs:

```python
def calculate_memory_budget(
    model_params_billions: float,
    precision_bytes: int,  # 2 for FP16, 1 for INT8, 0.5 for INT4
    num_layers: int,
    num_kv_heads: int,
    head_dim: int,
    max_batch_size: int,
    max_seq_len: int,
    gpu_memory_gb: float = 80.0,
):
    """Calculate if workload fits in GPU memory."""
    # Model weights
    weights_gb = model_params_billions * precision_bytes

    # KV cache: 2 (K+V) x layers x kv_heads x head_dim x precision x batch x seq
    kv_per_token_bytes = 2 * num_layers * num_kv_heads * head_dim * 2  # FP16 KV
    kv_cache_gb = (kv_per_token_bytes * max_batch_size * max_seq_len) / 1e9

    # Activation peak (rough estimate: ~2x single layer's activations)
    # For transformer: batch x seq x hidden_dim x 4 (intermediate) x 2 bytes
    hidden_dim = num_kv_heads * head_dim * (model_params_billions / 7)  # rough
    activation_gb = (max_batch_size * max_seq_len * hidden_dim * 4 * 2) / 1e9 * 2

    # Fixed overhead
    cuda_context_gb = 1.5
    allocator_overhead_pct = 0.10  # 10% for caching allocator

    total_needed = weights_gb + kv_cache_gb + activation_gb + cuda_context_gb
    total_with_overhead = total_needed * (1 + allocator_overhead_pct)

    print(f"=== Memory Budget ===")
    print(f"Weights:      {weights_gb:.1f} GB")
    print(f"KV Cache:     {kv_cache_gb:.1f} GB (batch={max_batch_size}, seq={max_seq_len})")
    print(f"Activations:  {activation_gb:.1f} GB (peak estimate)")
    print(f"CUDA context: {cuda_context_gb:.1f} GB")
    print(f"Subtotal:     {total_needed:.1f} GB")
    print(f"With 10% allocator overhead: {total_with_overhead:.1f} GB")
    print(f"GPU capacity: {gpu_memory_gb:.1f} GB")
    print(f"Headroom:     {gpu_memory_gb - total_with_overhead:.1f} GB")

    if total_with_overhead > gpu_memory_gb:
        print(f"WILL NOT FIT! Need to reduce by {total_with_overhead - gpu_memory_gb:.1f} GB")

    return total_with_overhead
```

### Step 5: Gradient Checkpointing Deep Dive

For models where activation memory dominates (common during fine-tuning, but also relevant for very long sequences during inference), gradient checkpointing trades compute for memory:

```python
# Without checkpointing: O(L) activation memory (L = num layers)
# With checkpointing: O(sqrt(L)) activation memory, ~30% more compute

from transformers import AutoModelForCausalLM

model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-2-7b-hf",
    torch_dtype=torch.float16,
    device_map="auto",
)

# Enable gradient checkpointing
model.gradient_checkpointing_enable()

# For inference-only with very long sequences, selective checkpointing
# saves memory without full recomputation cost:
model.gradient_checkpointing_enable(
    gradient_checkpointing_kwargs={"use_reentrant": False}
)
```

The memory savings from gradient checkpointing can be calculated: for a model with $L$ layers and per-layer activation size $A$, standard forward pass requires $L \times A$ memory for activations. With checkpointing every $k$ layers, memory drops to $(L/k + k) \times A$, minimized at $k = \sqrt{L}$ giving $2\sqrt{L} \times A$. For a 32-layer model, this is a reduction from $32A$ to approximately $11.3A$ -- nearly 3x savings.

### Step 6: Detect and Fix Memory Leaks

Memory leaks in LLM serving are subtle. They often involve Python reference cycles that keep CUDA tensors alive:

```python
import gc
import torch

def detect_leak(model, input_fn, num_requests=50, warmup=5):
    """Detect memory leaks by tracking allocation growth across requests."""

    # Warmup
    for _ in range(warmup):
        output = model.generate(**input_fn())
        del output

    torch.cuda.empty_cache()
    gc.collect()
    baseline = torch.cuda.memory_allocated()

    allocations = []
    for i in range(num_requests):
        output = model.generate(**input_fn())
        del output
        torch.cuda.empty_cache()
        gc.collect()

        current = torch.cuda.memory_allocated()
        allocations.append(current)

        if i % 10 == 0:
            growth = (current - baseline) / 1e6
            print(f"Request {i}: {growth:+.2f} MB from baseline")

    # Linear regression to detect trend
    import numpy as np
    x = np.arange(len(allocations))
    slope, intercept = np.polyfit(x, allocations, 1)

    leak_rate_mb = slope / 1e6
    print(f"\nLeak rate: {leak_rate_mb:.4f} MB/request")

    if leak_rate_mb > 0.01:  # More than 10 KB per request
        print(f"LEAK DETECTED: {leak_rate_mb:.4f} MB/request")
        print(f"At this rate, OOM in ~{(torch.cuda.mem_get_info()[0]/1e6) / leak_rate_mb:.0f} requests")

        # Dump tensors to find the leak
        print("\nLargest CUDA tensors:")
        for obj in gc.get_objects():
            try:
                if torch.is_tensor(obj) and obj.is_cuda:
                    print(f"  {obj.shape} {obj.dtype} {obj.device} "
                          f"({obj.element_size() * obj.nelement() / 1e6:.2f} MB)")
            except:
                pass
    else:
        print("No leak detected.")

    return leak_rate_mb
```

<Callout type="warning" title="Memory Leaks Are Subtle">
  Python's garbage collector may not immediately free CUDA tensors that have circular references or are held by closures, callbacks, or hooks. Common leak sources in LLM serving: (1) registered forward hooks that capture output tensors; (2) logging callbacks that store tensor values; (3) KV cache entries that are not properly evicted when requests complete; (4) exception handlers that capture the traceback, which holds references to local tensors. A growth of even 1 MB per request leads to OOM after 80,000 requests on an 80 GB GPU -- roughly 2-3 days at moderate load.
</Callout>

### Prometheus Metrics for GPU Memory

```python
from prometheus_client import Gauge, Histogram, Counter, start_http_server
import torch
import threading
import time

# Define metrics
gpu_memory_allocated = Gauge(
    'gpu_memory_allocated_bytes',
    'GPU memory allocated by tensors',
    ['gpu_id']
)
gpu_memory_reserved = Gauge(
    'gpu_memory_reserved_bytes',
    'GPU memory reserved by caching allocator',
    ['gpu_id']
)
gpu_memory_fragmentation = Gauge(
    'gpu_memory_fragmentation_ratio',
    'Ratio of fragmented to reserved memory',
    ['gpu_id']
)
gpu_memory_peak = Gauge(
    'gpu_memory_peak_allocated_bytes',
    'Peak GPU memory allocated since last reset',
    ['gpu_id']
)
gpu_oom_total = Counter(
    'gpu_oom_errors_total',
    'Total GPU OOM errors',
    ['gpu_id']
)
gpu_alloc_retries = Counter(
    'gpu_alloc_retries_total',
    'GPU memory allocation retries',
    ['gpu_id']
)
gpu_memory_utilization = Gauge(
    'gpu_memory_utilization_ratio',
    'GPU memory utilization (allocated / total)',
    ['gpu_id']
)

def collect_gpu_metrics():
    """Background thread collecting GPU memory metrics."""
    last_oom_count = {}
    last_retry_count = {}

    while True:
        for gpu_id in range(torch.cuda.device_count()):
            with torch.cuda.device(gpu_id):
                stats = torch.cuda.memory_stats()
                total_mem = torch.cuda.get_device_properties(gpu_id).total_mem

                allocated = stats['allocated_bytes.all.current']
                reserved = stats['reserved_bytes.all.current']
                peak = stats['allocated_bytes.all.peak']

                gpu_memory_allocated.labels(gpu_id=gpu_id).set(allocated)
                gpu_memory_reserved.labels(gpu_id=gpu_id).set(reserved)
                gpu_memory_peak.labels(gpu_id=gpu_id).set(peak)
                gpu_memory_utilization.labels(gpu_id=gpu_id).set(
                    allocated / total_mem
                )

                if reserved > 0:
                    frag = (reserved - allocated) / reserved
                    gpu_memory_fragmentation.labels(gpu_id=gpu_id).set(frag)

                # Track OOM counter increments
                current_ooms = stats.get('num_ooms', 0)
                prev_ooms = last_oom_count.get(gpu_id, 0)
                if current_ooms > prev_ooms:
                    gpu_oom_total.labels(gpu_id=gpu_id).inc(
                        current_ooms - prev_ooms
                    )
                last_oom_count[gpu_id] = current_ooms

                current_retries = stats.get('num_alloc_retries', 0)
                prev_retries = last_retry_count.get(gpu_id, 0)
                if current_retries > prev_retries:
                    gpu_alloc_retries.labels(gpu_id=gpu_id).inc(
                        current_retries - prev_retries
                    )
                last_retry_count[gpu_id] = current_retries

        time.sleep(5)  # Collect every 5 seconds

# Start metrics server and collector
start_http_server(9090)
collector_thread = threading.Thread(target=collect_gpu_metrics, daemon=True)
collector_thread.start()
```

## Putting It All Together: A Complete Profiling Workflow

Here is the systematic workflow we use for every new LLM deployment:

**Phase 1: Pre-deployment estimation.** Calculate the memory budget using the formula above. Determine if the model fits with the desired batch size and sequence length. If not, decide on quantization strategy before deployment.

**Phase 2: Startup profiling.** Deploy the model and immediately capture memory baselines after warmup. Use `torch.cuda.memory_stats()` to record allocated, reserved, and fragmentation numbers. Compare against the pre-deployment estimate -- deviations indicate unexpected overhead.

**Phase 3: Load testing with memory tracking.** Send realistic traffic patterns while monitoring GPU memory. Vary sequence lengths and batch sizes. Record peak memory for each configuration. This establishes the operational envelope -- the maximum batch size and sequence length that fit without OOM at the target memory utilization.

**Phase 4: Stability testing.** Run for 24+ hours with production-like traffic. Monitor for memory leaks (growing `allocated_bytes`), fragmentation growth (growing gap between `reserved` and `allocated`), and allocation retries. Any upward trend is a bug that must be fixed before production.

**Phase 5: Production monitoring.** Deploy Prometheus metrics, Grafana dashboards, and alerting rules. Set alerts for memory utilization thresholds, fragmentation, OOMs, and allocation retries. Review dashboards weekly to catch gradual degradation.

```python
# Phase 2: Startup profiling script
import torch
import json
from datetime import datetime

def startup_profile(model, sample_inputs, output_file="startup_profile.json"):
    """Capture memory baseline after model load and warmup."""
    report = {}

    # After model load
    torch.cuda.synchronize()
    report['after_load'] = {
        'allocated_gb': torch.cuda.memory_allocated() / 1e9,
        'reserved_gb': torch.cuda.memory_reserved() / 1e9,
    }

    # Warmup (3 iterations to trigger all lazy initialization)
    torch.cuda.reset_peak_memory_stats()
    for i in range(3):
        with torch.no_grad():
            output = model.generate(**sample_inputs)
        del output
    torch.cuda.empty_cache()

    report['after_warmup'] = {
        'allocated_gb': torch.cuda.memory_allocated() / 1e9,
        'reserved_gb': torch.cuda.memory_reserved() / 1e9,
        'peak_allocated_gb': torch.cuda.max_memory_allocated() / 1e9,
        'peak_reserved_gb': torch.cuda.max_memory_reserved() / 1e9,
    }

    stats = torch.cuda.memory_stats()
    report['allocator_stats'] = {
        'num_segments': stats['segment.all.current'],
        'num_allocations': stats['allocation.all.current'],
        'inactive_split_gb': stats['inactive_split_bytes.all.current'] / 1e9,
        'num_alloc_retries': stats.get('num_alloc_retries', 0),
    }

    total_gpu = torch.cuda.get_device_properties(0).total_mem / 1e9
    report['capacity'] = {
        'total_gpu_gb': total_gpu,
        'available_gb': total_gpu - report['after_warmup']['peak_reserved_gb'],
        'utilization_pct': report['after_warmup']['peak_reserved_gb'] / total_gpu * 100,
    }

    report['timestamp'] = datetime.now().isoformat()
    report['gpu_name'] = torch.cuda.get_device_name(0)

    with open(output_file, 'w') as f:
        json.dump(report, f, indent=2)

    print(json.dumps(report, indent=2))
    return report
```

## Conclusion

GPU memory profiling is not optional for production LLM serving -- it is as essential as load testing or monitoring. The tools form a layered stack: `torch.cuda.memory_stats()` for quick health checks, `torch.cuda.memory._snapshot()` for detailed allocation forensics, `torch.profiler` for operation-level memory attribution, Nsight Compute for kernel-level memory behavior, and Nsight Systems for timeline-based analysis of when and where memory events occur.

The most common issues -- KV cache over-allocation (fixed by PagedAttention), caching allocator fragmentation (fixed by `expandable_segments` and `empty_cache()`), and activation peaks (fixed by gradient checkpointing or operator fusion) -- are all diagnosable with these tools and fixable without fundamental architecture changes.

The methodology matters as much as the tools: calculate the memory budget before deployment, profile at startup, load-test with memory tracking, verify stability over 24+ hours, and monitor continuously in production with Prometheus and Grafana. An OOM that is caught by an alert at 90% utilization is a tuning task. An OOM that strikes at 3 AM during peak traffic is an incident. The difference is profiling.
