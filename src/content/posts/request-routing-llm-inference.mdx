Request routing is a solved problem for stateless web services. Nginx round-robins across backends, and life is good. But LLM inference is not a stateless web service. Every GPU accumulates KV cache state that represents seconds of prefill computation. Routing a request to the wrong GPU means either transferring gigabytes of cache over the network or recomputing everything from scratch. The routing decision -- which GPU handles which request -- directly determines whether you waste 30% of your cluster's compute or not.

This post covers the full landscape of LLM request routing: why traditional algorithms fail, how KV cache awareness transforms routing economics, disaggregated prefill-decode architectures, multi-model and LoRA-aware routing, autoscaling signals unique to LLM workloads, SLO-aware scheduling with priority and preemption, and production multi-tier architectures. By the end, you will understand when simple routing suffices and when you need the full complexity.

## Why Routing Is Hard for LLMs

Traditional web service routing operates on a simple assumption: all backends are interchangeable. Any request can go to any server with roughly equal cost. This assumption breaks catastrophically for LLM inference.

### The Statefulness Problem

Every LLM inference request creates state on the GPU it runs on. During the prefill phase, the model processes the input prompt and generates a KV cache -- a set of key-value tensors for every layer of the transformer. For a 70B parameter model like Llama 2 70B with a 4096-token prompt, the KV cache consumes:

$$\text{KV cache size} = 2 \times n_{layers} \times n_{heads} \times d_{head} \times S \times \text{precision bytes}$$

For Llama 70B with 80 layers, 64 KV heads (using GQA), $d_{head} = 128$, a 4096-token prompt at FP16:

$$2 \times 80 \times 64 \times 128 \times 4096 \times 2 = 10.7 \text{ GB}$$

That is 10.7 GB of state that exists only on the GPU that processed the prefill. If a follow-up request (e.g., a multi-turn conversation) arrives and gets routed to a different GPU, you have two options:

1. **Transfer the KV cache**: Move 10.7 GB over the network. Even on 100 Gbps InfiniBand, this takes $\sim$860 ms -- almost a full second of pure data movement.
2. **Recompute from scratch**: Re-run prefill on the new GPU. For a 4096-token prompt on Llama 70B, prefill takes 1.5-3 seconds on an A100. You have thrown away the original computation entirely.

Neither option is acceptable at scale. Session affinity -- routing follow-up requests to the same GPU -- is not a nice-to-have; it is an economic necessity.

<Callout type="warning" title="The Hidden Cost of Mis-routing">
  A single mis-routed request for a long context (32K+ tokens) can waste 5-10 seconds of GPU time on recomputation. At $2/hour/GPU, wasting 10% of requests this way costs roughly $0.20/hour/GPU in pure waste. Across a 1000-GPU cluster, that is $200/hour or $1.7M/year.
</Callout>

### Variable Request Costs

Unlike web requests that take roughly uniform time to process, LLM requests have cost variability spanning two orders of magnitude:

```python
# Request cost varies wildly
requests = [
    {"prompt_len": 50, "max_tokens": 10},      # ~30ms total
    {"prompt_len": 4096, "max_tokens": 2048},   # ~15,000ms total
    {"prompt_len": 128, "max_tokens": 50},      # ~200ms total
]
# Range: 30ms to 15,000ms -- a 500x spread
```

The total cost of a request is approximately:

$$T_{total} = T_{prefill}(S_{in}) + T_{decode}(S_{out})$$

where $T_{prefill}$ scales roughly linearly with input length (for moderate sequences) and $T_{decode}$ scales linearly with output length. A router that treats all requests as equal -- like round-robin -- will inevitably create hotspots where one GPU gets multiple expensive requests while others sit idle.

### The Batching Dimension

Modern LLM serving engines (vLLM, TensorRT-LLM, SGLang) use continuous batching: new requests join an in-flight batch dynamically. The throughput of a GPU depends on its current batch composition. A GPU running 20 short decode steps can absorb a new request with minimal impact. A GPU in the middle of a long prefill cannot. The router needs to understand not just how many requests each GPU has, but what phase each request is in and how the batch dynamics will evolve.

## Load Balancing Algorithms: From Simple to Cache-Aware

Let us walk through the progression of routing algorithms, from the simplest to the most sophisticated, understanding what each one gets right and where it falls short.

### Round-Robin: Simple, Predictable, Wrong

Round-robin assigns requests to GPUs in circular order: GPU 0, GPU 1, GPU 2, ..., GPU $N-1$, GPU 0, GPU 1, and so on. It requires zero state in the router, is trivially load-balanced in terms of request count, and completely ignores everything that matters for LLM workloads.

```python
class RoundRobinRouter:
    def __init__(self, replicas: list[str]):
        self.replicas = replicas
        self.index = 0

    def route(self, request: dict) -> str:
        replica = self.replicas[self.index % len(self.replicas)]
        self.index += 1
        return replica
```

**Why it fails**: Consider 4 GPUs receiving requests with these prompt lengths: [4096, 50, 4096, 50, 4096, 50, 4096, 50]. Round-robin sends them as: GPU 0 gets [4096, 50], GPU 1 gets [4096, 50], GPU 2 gets [4096, 50], GPU 3 gets [4096, 50]. This happens to work. But real traffic is not so conveniently distributed. Burst patterns, correlated request sizes from the same user session, and variable output lengths create persistent imbalances.

**When it works**: Single-model serving with homogeneous workloads (all requests have similar prompt/output lengths), or when you truly do not care about tail latency. Some internal batch processing pipelines fit this profile.

### Least-Connections: Better but Cache-Blind

Least-connections routing sends each new request to the GPU with the fewest active (in-flight) requests. It adapts to variable request duration: if GPU 2 has a slow request, it accumulates fewer connections and stops receiving new work until the slow request completes.

```python
import threading

class LeastConnectionsRouter:
    def __init__(self, replicas: list[str]):
        self.replicas = replicas
        self.connections = {r: 0 for r in replicas}
        self.lock = threading.Lock()

    def route(self, request: dict) -> str:
        with self.lock:
            replica = min(self.replicas, key=lambda r: self.connections[r])
            self.connections[replica] += 1
            return replica

    def release(self, replica: str):
        with self.lock:
            self.connections[replica] -= 1
```

**What it gets right**: It naturally load-balances based on actual GPU availability. If one GPU is slow (thermal throttling, PCIe contention, etc.), it accumulates connections and the router steers traffic away.

**What it misses**: A GPU with 10 in-flight requests generating 20-token responses is far less loaded than a GPU with 2 in-flight requests generating 2000-token responses. Connection count is a proxy for load, but a poor one when request costs vary by 100x. And it has zero awareness of KV cache state.

### Estimated-Completion-Time Routing

Instead of counting connections, estimate how long each GPU's current workload will take to complete, and route to the GPU with the earliest estimated completion time. This requires a cost model for request processing.

```python
import time
import threading

class EstimatedCompletionTimeRouter:
    def __init__(self, replicas: list[str]):
        self.replicas = replicas
        self.estimated_completion = {r: 0.0 for r in replicas}
        self.lock = threading.Lock()

    def estimate_cost(self, request: dict) -> float:
        """Estimate request processing time in milliseconds."""
        prompt_len = request.get('prompt_len', 100)
        max_tokens = request.get('max_tokens', 100)

        # Empirical model calibrated per-GPU
        # Prefill: ~0.5ms per prompt token (varies with batch size)
        prefill_ms = prompt_len * 0.5
        # Decode: ~15ms per output token (with continuous batching)
        decode_ms = max_tokens * 15

        return prefill_ms + decode_ms

    def route(self, request: dict) -> str:
        cost = self.estimate_cost(request)
        now = time.monotonic() * 1000  # ms

        with self.lock:
            # Find GPU with earliest estimated completion
            best = min(self.replicas,
                       key=lambda r: max(self.estimated_completion[r], now))
            self.estimated_completion[best] = (
                max(self.estimated_completion[best], now) + cost
            )
            return best

    def complete(self, replica: str, actual_ms: float, estimated_ms: float):
        with self.lock:
            # Correct the estimate based on actual completion
            self.estimated_completion[replica] -= estimated_ms
            self.estimated_completion[replica] = max(
                self.estimated_completion[replica],
                time.monotonic() * 1000
            )
```

This is meaningfully better than least-connections. But notice the cost model is static -- it does not account for batch size effects (decode is faster per-token at higher batch sizes due to better GPU utilization), and it still ignores KV cache locality entirely.

<Callout type="info" title="Cost Model Calibration">
  The per-token prefill and decode costs depend heavily on model size, GPU type, batch size, and sequence length. Production systems typically calibrate these models online using exponentially-weighted moving averages of observed latencies, updated per-GPU.
</Callout>

### Prefix-Aware Routing: Cache Locality as a First-Class Concern

Here is where LLM-specific routing begins. The key insight: if GPU A has already processed requests with a particular system prompt, it has that system prompt's KV cache resident in GPU memory. Routing a new request with the same system prompt to GPU A allows the engine to skip prefill for the cached prefix, saving potentially seconds of computation.

Consider a deployment serving a customer-facing chatbot. Every request starts with the same 2000-token system prompt defining the assistant's persona, tools, and behavioral constraints. Without prefix-aware routing, every request on every GPU recomputes this 2000-token prefill. With prefix-aware routing, each GPU computes it once, and subsequent requests reuse the cached KV.

```python
import hashlib
import threading

class PrefixAwareRouter:
    def __init__(self, replicas: list[str]):
        self.replicas = replicas
        self.prefix_map: dict[str, str] = {}  # prefix_hash -> replica
        self.replica_load = {r: 0.0 for r in replicas}
        self.lock = threading.Lock()

    def _hash_prefix(self, prompt: str, prefix_len: int = 1024) -> str:
        """Hash the system prompt / common prefix."""
        prefix = prompt[:prefix_len]
        return hashlib.sha256(prefix.encode()).hexdigest()[:16]

    def route(self, request: dict) -> str:
        prompt = request.get('prompt', '')
        prefix_hash = self._hash_prefix(prompt)

        with self.lock:
            # Check if we have a cached prefix on some GPU
            if prefix_hash in self.prefix_map:
                candidate = self.prefix_map[prefix_hash]
                # Only use cache-affine GPU if it is not overloaded
                avg_load = sum(self.replica_load.values()) / len(self.replicas)
                if self.replica_load[candidate] < avg_load * 1.5:
                    self.replica_load[candidate] += 1
                    return candidate

            # No cache hit or cache-affine GPU is overloaded
            # Fall back to least-loaded GPU
            best = min(self.replicas, key=lambda r: self.replica_load[r])
            self.prefix_map[prefix_hash] = best
            self.replica_load[best] += 1
            return best

    def release(self, replica: str):
        with self.lock:
            self.replica_load[replica] = max(0, self.replica_load[replica] - 1)
```

The critical design decision is the **overload threshold**. If you always route to the cache-affine GPU regardless of load, you create hotspots: one popular system prompt sends all its traffic to a single GPU. The 1.5x average load threshold provides a balance -- prefer cache locality, but not at the cost of severe imbalance.

## KV Cache-Aware Routing: The Full Picture

Prefix-aware routing is a specific case of a broader principle: route requests to GPUs that already hold relevant KV cache state. Let us formalize this.

### The Routing Cost Function

For a request $r$ routed to GPU $g$, the total cost is:

$$C(r, g) = T_{prefill\_miss}(r, g) + T_{prefill\_hit}(r, g) + T_{decode}(r, g) + T_{queue}(g)$$

where:
- $T_{prefill\_miss}(r, g)$ is the time to compute prefill for prompt tokens **not** in GPU $g$'s cache
- $T_{prefill\_hit}(r, g)$ is the time to load cached KV from GPU memory (near-zero with PagedAttention)
- $T_{decode}(r, g)$ is the estimated decode time (roughly the same across GPUs)
- $T_{queue}(g)$ is the time spent waiting in GPU $g$'s queue

The optimal routing decision minimizes this cost:

$$g^* = \arg\min_g C(r, g)$$

In practice, computing $C(r, g)$ exactly is impossible because $T_{decode}$ depends on how many tokens the model actually generates (unknown until generation completes). But the prefill cost is estimable, and the queue depth is observable.

### Radix Tree-Based Cache Tracking

SGLang introduced a radix tree (also called a trie) to track which token prefixes are cached on each GPU. Each path in the tree represents a token sequence, and the tree structure enables efficient longest-prefix matching.

```python
class RadixNode:
    def __init__(self):
        self.children: dict[int, 'RadixNode'] = {}  # token_id -> child
        self.cached_on: set[str] = set()  # which GPUs have this prefix
        self.depth: int = 0

class CacheAwareRouter:
    def __init__(self, replicas: list[str]):
        self.replicas = replicas
        self.root = RadixNode()
        self.lock = threading.Lock()

    def find_best_gpu(self, token_ids: list[int]) -> tuple[str, int]:
        """Find GPU with longest cached prefix for given token sequence.
        Returns (gpu_id, cached_prefix_length)."""
        with self.lock:
            best_gpu = None
            best_match_len = 0

            node = self.root
            current_depth = 0

            for token in token_ids:
                if token not in node.children:
                    break
                node = node.children[token]
                current_depth += 1

                # Check which GPUs have cache at this depth
                for gpu in node.cached_on:
                    if current_depth > best_match_len:
                        best_match_len = current_depth
                        best_gpu = gpu

            return best_gpu, best_match_len

    def route(self, request: dict) -> str:
        token_ids = request.get('token_ids', [])
        best_gpu, match_len = self.find_best_gpu(token_ids)

        if best_gpu is not None and match_len > 128:
            # Significant cache hit -- worth the affinity
            return best_gpu

        # Fall back to estimated-completion-time routing
        return self._ect_route(request)
```

The threshold of 128 tokens is important. A cache hit on only 10 tokens saves negligible computation -- the overhead of the routing decision itself may exceed the savings. But a cache hit on 2000 tokens (a typical system prompt) saves over a second of prefill time.

### Cache-Aware Routing with Eviction Awareness

KV caches are not permanent. GPU memory is finite, and as new requests arrive, old cache entries get evicted. A router that sends a request to GPU A because it "has the prefix cached" may arrive to find the cache has been evicted in the meantime. Robust implementations require the serving engine to report cache state back to the router.

```python
class CacheStateReport:
    """Periodic report from serving engine to router."""
    gpu_id: str
    cached_prefixes: list[tuple[str, int]]  # (prefix_hash, token_count)
    gpu_memory_used: float  # fraction 0-1
    active_requests: int
    estimated_queue_time_ms: float
```

The router maintains a cache state map that is refreshed every 100-500ms. Staleness is acceptable because cache eviction is not instantaneous -- there is a grace period where the cache is likely still present even if the router's view is slightly outdated.

## Disaggregated Routing: Separating Prefill and Decode

Disaggregated serving splits the two phases of LLM inference -- prefill and decode -- onto separate GPU pools. This creates a fundamentally different routing problem: instead of routing a request to a single GPU, the router must orchestrate a two-step pipeline.

### Why Disaggregate?

Prefill and decode have opposite computational profiles:

- **Prefill** is compute-bound. It processes the entire prompt in one forward pass, generating large GEMMs that saturate GPU compute units. An A100 achieves 60-80% of peak FLOPS during prefill.
- **Decode** is memory-bandwidth-bound. It generates one token at a time, reading the entire model weights for a single token's worth of computation. An A100 achieves only 5-15% of peak FLOPS during decode, bottlenecked by HBM bandwidth.

Running both on the same GPU creates interference: prefill requests steal compute from decode requests, causing decode latency spikes (directly visible to users as stuttering output). Decode requests occupy GPU memory with their KV caches, reducing the batch size available for prefill.

### The Disaggregated Router

A disaggregated router makes two sequential decisions:

1. **Prefill routing**: Which GPU in the prefill pool handles the prompt processing?
2. **Decode routing**: After prefill completes, which GPU in the decode pool handles token generation?

Between these two decisions, the KV cache must be transferred from the prefill GPU to the decode GPU.

```python
class DisaggregatedRouter:
    def __init__(self, prefill_pool: list[str], decode_pool: list[str]):
        self.prefill_pool = prefill_pool
        self.decode_pool = decode_pool
        self.prefill_load = {g: 0.0 for g in prefill_pool}
        self.decode_load = {g: 0.0 for g in decode_pool}
        self.decode_cache_map: dict[str, str] = {}  # session -> decode GPU

    def route_prefill(self, request: dict) -> str:
        """Route to prefill pool, optimizing for compute throughput."""
        prompt_len = request.get('prompt_len', 100)
        cost = prompt_len * 0.5  # ms estimate

        # Prefill GPUs: maximize throughput, batch large prefills together
        best = min(self.prefill_pool,
                   key=lambda g: self.prefill_load[g])
        self.prefill_load[best] += cost
        return best

    def route_decode(self, request: dict, kv_cache_size_gb: float) -> str:
        """Route to decode pool, optimizing for memory and latency."""
        session_id = request.get('session_id', '')

        # Session affinity: reuse decode GPU for multi-turn conversations
        if session_id in self.decode_cache_map:
            gpu = self.decode_cache_map[session_id]
            if self._has_memory(gpu, kv_cache_size_gb):
                return gpu

        # Find decode GPU with most available memory
        best = min(self.decode_pool,
                   key=lambda g: self.decode_load[g])
        self.decode_cache_map[session_id] = best
        self.decode_load[best] += kv_cache_size_gb
        return best
```

### Optimization Criteria Differ Per Pool

The prefill pool and decode pool have different optimization objectives, and the router must reflect this:

**Prefill pool routing priorities:**
1. Maximize compute utilization (batch large prefills together)
2. Prefix cache locality (avoid redundant prefill)
3. Minimize queue depth (avoid prefill latency spikes)

**Decode pool routing priorities:**
1. Maximize memory efficiency (pack more concurrent decodes per GPU)
2. Session affinity (avoid KV cache transfers for multi-turn)
3. Minimize KV cache transfer time (choose decode GPUs with fast interconnect to the prefill GPU)

<Callout type="info" title="KV Cache Transfer Is the Bottleneck">
  In disaggregated serving, the KV cache must be transferred from the prefill GPU to the decode GPU after prefill completes. For Llama 70B with a 4096-token prompt, this is $\sim$10.7 GB. On NVLink (600 GB/s), transfer takes $\sim$18ms. On 100 Gbps Ethernet, it takes $\sim$860ms. The interconnect topology directly constrains which prefill-decode GPU pairs are viable.
</Callout>

### Mooncake's Prediction-Based Routing

Moonshot AI's Mooncake system predicts the output length of each request before routing, using a lightweight classifier trained on historical data. This prediction enables smarter decode GPU selection: short-output requests (which will free their KV cache quickly) can be packed more densely, while long-output requests need more memory headroom.

$$\hat{S}_{out} = f_{predict}(\text{prompt features})$$

The predicted output length $\hat{S}_{out}$ feeds into the decode routing decision:

$$\text{memory\_needed} = \text{KV cache per token} \times (S_{in} + \hat{S}_{out})$$

This prediction does not need to be precise. Even a coarse classification into short (fewer than 100 tokens), medium (100-500), and long (greater than 500) significantly improves packing efficiency.

## Multi-Model Routing: LoRA Affinity and Model Multiplexing

Production LLM platforms rarely serve a single model. They serve hundreds or thousands of fine-tuned variants, typically using LoRA (Low-Rank Adaptation) adapters loaded on top of a base model. Routing in this regime adds a new dimension: model affinity.

### The LoRA Loading Problem

A LoRA adapter is small (typically 10-100 MB) compared to the base model (tens of GB), but loading it is not free. Swapping a LoRA adapter requires:

1. Loading adapter weights from host memory to GPU memory
2. Merging or indexing the adapter for the serving engine's kernel
3. Potentially evicting another adapter if GPU memory is full

On an A100, loading a 50 MB LoRA adapter takes 1-5ms from host memory, or 50-500ms from disk. If the router sends requests for 100 different LoRA adapters to the same GPU in round-robin fashion, the GPU spends more time swapping adapters than running inference.

### S-LoRA: Grouping Requests by Adapter

S-LoRA (Sheng et al., 2023) demonstrates that grouping requests by adapter and routing them to GPUs that already have that adapter loaded dramatically improves throughput. The routing strategy is:

```python
class LoRAAwareRouter:
    def __init__(self, replicas: list[str], max_adapters_per_gpu: int = 8):
        self.replicas = replicas
        self.max_adapters = max_adapters_per_gpu
        # Track which adapters are loaded on each GPU
        self.gpu_adapters: dict[str, list[str]] = {r: [] for r in replicas}
        self.gpu_load = {r: 0 for r in replicas}
        self.lock = threading.Lock()

    def route(self, request: dict) -> str:
        adapter_id = request.get('adapter_id', 'base')

        with self.lock:
            # Priority 1: GPU that already has this adapter loaded
            for gpu in self.replicas:
                if adapter_id in self.gpu_adapters[gpu]:
                    if self.gpu_load[gpu] < self._capacity(gpu):
                        self.gpu_load[gpu] += 1
                        return gpu

            # Priority 2: GPU with room for another adapter
            for gpu in sorted(self.replicas,
                              key=lambda g: self.gpu_load[g]):
                if len(self.gpu_adapters[gpu]) < self.max_adapters:
                    self.gpu_adapters[gpu].append(adapter_id)
                    self.gpu_load[gpu] += 1
                    return gpu

            # Priority 3: Evict least-used adapter, load new one
            best_gpu = min(self.replicas,
                           key=lambda g: self.gpu_load[g])
            self._evict_lru_adapter(best_gpu)
            self.gpu_adapters[best_gpu].append(adapter_id)
            self.gpu_load[best_gpu] += 1
            return best_gpu
```

### Multi-Model Routing Across Different Base Models

When serving entirely different base models (not just LoRA variants), routing becomes a capacity planning problem. Each base model requires a dedicated GPU pool (you cannot serve Llama 70B and Mistral 7B on the same GPU simultaneously in most serving engines). The router must:

1. Identify which model the request targets
2. Route to the correct model-specific GPU pool
3. Within that pool, apply cache-aware or load-based routing

```python
class MultiModelRouter:
    def __init__(self, model_pools: dict[str, list[str]]):
        # model_name -> list of GPU endpoints
        self.model_pools = model_pools
        self.pool_routers = {
            model: PrefixAwareRouter(gpus)
            for model, gpus in model_pools.items()
        }

    def route(self, request: dict) -> str:
        model = request.get('model', 'default')
        if model not in self.pool_routers:
            raise ValueError(f"Unknown model: {model}")
        return self.pool_routers[model].route(request)
```

## Autoscaling Signals for LLM Workloads

Autoscaling -- dynamically adding or removing GPUs based on demand -- requires the right signals. LLM workloads have unique characteristics that make traditional autoscaling signals (CPU utilization, request count) misleading.

### Signal 1: Queue Depth (Primary)

Queue depth -- the number of requests waiting to be processed -- is the most direct signal of capacity pressure. When queue depth exceeds a threshold, you need more GPUs.

$$\text{scale\_up if } \frac{\sum_g Q(g)}{N_{gpus}} > Q_{threshold}$$

A typical threshold is 5-10 queued requests per GPU. The challenge is distinguishing between transient bursts (which resolve in seconds) and sustained overload (which requires scaling). A common approach is to require the signal to persist for 60-120 seconds before triggering scale-up.

### Signal 2: P99 Latency (User-Facing)

P99 latency directly reflects user experience. If P99 time-to-first-token exceeds the SLO (e.g., 2 seconds), the system is overloaded regardless of what other metrics say.

$$\text{scale\_up if } P99_{TTFT} > SLO_{TTFT} \text{ for } t > t_{threshold}$$

The problem with P99 latency as a scaling signal is lag: by the time P99 degrades, users are already experiencing poor service. It works better as a validation signal (confirming that queue-depth-based scaling is working) than as a primary trigger.

### Signal 3: GPU Memory Utilization

For LLM serving, GPU memory is often the binding constraint, not compute. The KV cache for active requests plus the model weights must fit in GPU HBM. When GPU memory utilization exceeds 85-90%, the serving engine cannot accept new requests without evicting cached state.

$$\text{scale\_up if } \max_g \frac{M_{used}(g)}{M_{total}(g)} > 0.85 \text{ for } t > 30s$$

### Signal 4: Tokens Per Second Per GPU

Throughput measured in tokens/second/GPU captures the efficiency of the serving engine. A drop in per-GPU throughput indicates either hardware degradation or workload changes (e.g., a shift to longer sequences that reduce batch efficiency).

$$\text{tokens/sec/GPU} = \frac{\sum_{requests} S_{out}}{N_{gpus} \times \Delta t}$$

### The Scaling Lag Problem

The critical challenge with GPU autoscaling is latency. Adding a GPU to a serving cluster is not like adding a web server:

1. **GPU provisioning**: 1-5 minutes on cloud providers (if capacity is available)
2. **Model loading**: Loading a 70B model from storage to GPU takes 30-120 seconds
3. **Warmup**: The first few requests may be slower as caches populate and JIT compilation completes
4. **Joining the pool**: The new GPU must register with the router and start receiving traffic

Total time from scaling signal to serving capacity: 3-10 minutes. This means your autoscaling must be predictive, not purely reactive. Common strategies:

- **Scheduled scaling**: Pre-scale for known traffic patterns (business hours, product launches)
- **Headroom buffer**: Maintain 15-20% excess capacity at all times to absorb bursts during the scaling window
- **Predictive models**: Use historical traffic patterns and time-series forecasting to predict demand 10-15 minutes ahead

<Callout type="warning" title="Scale-Down Is Even Harder">
  Scaling down requires draining active requests from the GPU being removed. An LLM request generating a 2000-token response may take 30+ seconds. Aggressive scale-down causes request failures. Best practice: mark the GPU as "draining" (no new requests), wait for all active requests to complete (with a timeout), then remove it. This drain period can take 1-5 minutes.
</Callout>

### Composite Scaling Signal

In practice, no single signal is sufficient. Production systems use a weighted composite:

```python
class AutoscalingPolicy:
    def should_scale_up(self, metrics: dict) -> bool:
        score = 0.0

        # Queue depth (primary signal, weight 0.4)
        avg_queue = metrics['avg_queue_depth']
        if avg_queue > 5:
            score += 0.4 * min(avg_queue / 10, 1.0)

        # P99 TTFT vs SLO (weight 0.3)
        ttft_ratio = metrics['p99_ttft_ms'] / metrics['slo_ttft_ms']
        if ttft_ratio > 0.8:
            score += 0.3 * min((ttft_ratio - 0.8) / 0.4, 1.0)

        # GPU memory pressure (weight 0.2)
        max_mem = metrics['max_gpu_memory_util']
        if max_mem > 0.80:
            score += 0.2 * min((max_mem - 0.80) / 0.15, 1.0)

        # Throughput degradation (weight 0.1)
        tps_ratio = metrics['current_tps'] / metrics['baseline_tps']
        if tps_ratio < 0.85:
            score += 0.1 * min((0.85 - tps_ratio) / 0.3, 1.0)

        return score > 0.5  # Scale up when composite score exceeds threshold
```

## SLO-Aware Scheduling and Priority Routing

Not all requests are created equal. A production LLM platform serves diverse clients with different latency requirements and willingness to pay. SLO-aware scheduling routes and prioritizes requests based on their service-level objectives.

### Priority Tiers

A typical tier structure:

| Tier | TTFT SLO | TPS SLO | Preemptible | Use Case |
|------|----------|---------|-------------|----------|
| P0 (Critical) | fewer than 500ms | greater than 30 tok/s | No | Real-time user-facing chat |
| P1 (Standard) | fewer than 2s | greater than 15 tok/s | Reluctantly | API customers |
| P2 (Batch) | fewer than 30s | greater than 5 tok/s | Yes | Background processing |
| P3 (Best-effort) | No SLO | Any | Yes | Internal analytics, evals |

### Priority-Aware Routing

The router must consider request priority when making routing decisions:

```python
class PriorityRouter:
    def __init__(self, replicas: list[str],
                 dedicated_p0: list[str],
                 shared_pool: list[str]):
        self.dedicated_p0 = dedicated_p0  # Reserved for P0 requests
        self.shared_pool = shared_pool     # P1-P3 requests
        self.all_replicas = replicas

    def route(self, request: dict) -> str:
        priority = request.get('priority', 2)

        if priority == 0:
            # P0: Use dedicated pool (guaranteed capacity)
            return self._least_loaded(self.dedicated_p0)

        if priority == 1:
            # P1: Use shared pool, or overflow to P0 if shared is full
            gpu = self._least_loaded(self.shared_pool)
            if self._queue_depth(gpu) > 3:
                # Overflow to dedicated pool
                return self._least_loaded(self.dedicated_p0)
            return gpu

        # P2/P3: Shared pool only, may be preempted
        return self._least_loaded(self.shared_pool)
```

### Preemption: Evicting Low-Priority Requests

When a P0 request arrives and all GPUs are at capacity, the system must preempt a lower-priority request to make room. Preemption in LLM serving is expensive: the preempted request's KV cache is discarded, and the request must be re-queued and restarted from scratch (or from a checkpoint).

The preemption decision considers:

1. **Priority difference**: Only preempt requests at least 2 tiers below the incoming request
2. **Progress**: Prefer preempting requests that have generated fewer tokens (less wasted work)
3. **Remaining time**: Prefer preempting requests with the most remaining work (frees the GPU for longest)

$$\text{preemption\_score}(r) = \frac{\text{priority\_diff} \times (S_{max} - S_{generated})}{\text{tokens\_generated} + 1}$$

The request with the highest preemption score gets evicted.

```python
class PreemptionManager:
    def find_preemption_candidate(self, gpu: str, incoming_priority: int
                                  ) -> dict | None:
        active = self.get_active_requests(gpu)
        candidates = []

        for req in active:
            priority_diff = req['priority'] - incoming_priority
            if priority_diff < 2:
                continue  # Only preempt if 2+ tiers lower

            tokens_remaining = req['max_tokens'] - req['tokens_generated']
            score = (priority_diff * tokens_remaining
                     / (req['tokens_generated'] + 1))
            candidates.append((score, req))

        if not candidates:
            return None

        candidates.sort(reverse=True)
        return candidates[0][1]
```

<Callout type="perf" title="Preemption Cost">
  Preempting a request that has generated 500 out of 2000 tokens wastes $\sim$7.5 seconds of GPU time (500 decode steps at $\sim$15ms each). If preemption happens more than 5% of the time, the system is fundamentally under-provisioned and needs more capacity, not better preemption logic.
</Callout>

### SLO-Aware Admission Control

Sometimes the best routing decision is to reject the request entirely. If the system cannot meet a request's SLO, it is better to return a 429 (rate limit) immediately than to accept the request and violate the SLO after consuming GPU time.

```python
class AdmissionController:
    def admit(self, request: dict) -> bool:
        priority = request.get('priority', 2)
        slo_ttft_ms = self.slo_targets[priority]['ttft_ms']

        # Estimate wait time based on current queue state
        estimated_wait = self._estimate_wait_time(request)

        if estimated_wait > slo_ttft_ms * 0.8:
            # Will likely miss SLO -- reject early
            return False

        return True
```

## Production Architectures: Multi-Tier Routing

Production LLM serving stacks are not a single router in front of GPUs. They are multi-tier systems with distinct responsibilities at each layer.

### The Three-Tier Architecture

```
                         ┌─────────────────┐
                         │   API Gateway    │  Rate limiting, auth, model resolution
                         └────────┬────────┘
                                  │
                         ┌────────▼────────┐
                         │  Global Router   │  Model pool selection, priority routing
                         └────────┬────────┘
                                  │
              ┌───────────────────┼───────────────────┐
              │                   │                   │
     ┌────────▼────────┐ ┌───────▼────────┐ ┌────────▼────────┐
     │  Pool Scheduler  │ │ Pool Scheduler  │ │ Pool Scheduler  │
     │  (Llama 70B)     │ │ (Mistral 7B)   │ │ (Code Model)    │
     └────────┬────────┘ └───────┬────────┘ └────────┬────────┘
              │                   │                   │
        ┌─────┼─────┐      ┌─────┼─────┐      ┌─────┼─────┐
        │     │     │      │     │     │      │     │     │
       GPU   GPU   GPU    GPU   GPU   GPU    GPU   GPU   GPU
```

**Tier 1: API Gateway**
- Rate limiting per API key
- Authentication and authorization
- Model name resolution (e.g., "gpt-4" maps to a specific model pool)
- Request validation (prompt length, max tokens within limits)

**Tier 2: Global Router**
- Routes to the correct model-specific GPU pool
- Priority-based routing (P0 to dedicated pools, P1-P3 to shared)
- Cross-pool overflow (when one pool is saturated, route overflow to another region/cluster)
- Admission control (reject requests that will violate SLO)

**Tier 3: Pool Scheduler**
- KV cache-aware routing within a model pool
- Prefix affinity tracking
- Health monitoring of individual GPUs
- Autoscaling decisions for the pool

### Health Checking and Failover

GPU health checking for LLM serving goes beyond simple HTTP liveness probes. A GPU can be "alive" but degraded in ways that matter:

- **Memory leak**: KV cache is not being freed properly, reducing effective capacity over time
- **Thermal throttling**: GPU temperature exceeds limits, reducing clock speed by 20-40%
- **ECC errors**: Memory errors that cause silent corruption or CUDA crashes
- **Inference correctness**: The model produces garbage output (rare but catastrophic)

```python
class GPUHealthChecker:
    def __init__(self, gpu_id: str):
        self.gpu_id = gpu_id
        self.checks = {
            'liveness': self._check_liveness,      # Can respond at all?
            'latency': self._check_latency,         # Within expected range?
            'memory': self._check_memory_health,    # Memory utilization sane?
            'correctness': self._check_correctness,  # Outputs make sense?
        }

    def _check_latency(self) -> bool:
        """Send a canary request and verify response time."""
        start = time.monotonic()
        response = self._send_canary_request()
        elapsed_ms = (time.monotonic() - start) * 1000

        # Canary should complete in <200ms; >500ms indicates degradation
        return elapsed_ms < 500

    def _check_memory_health(self) -> bool:
        """Verify GPU memory is not leaking."""
        stats = self._get_gpu_stats()
        # If memory usage is >95% with fewer than 5 active requests,
        # something is wrong
        if stats['memory_util'] > 0.95 and stats['active_requests'] < 5:
            return False
        return True

    def _check_correctness(self) -> bool:
        """Periodically verify output quality with known-answer canaries."""
        response = self._send_canary_request(
            prompt="What is 2+2?",
            expected_contains="4"
        )
        return "4" in response
```

### Graceful Degradation

When capacity is insufficient to serve all traffic at full quality, the system should degrade gracefully rather than fail catastrophically. Graceful degradation strategies, ordered from least to most aggressive:

1. **Increase batching**: Accept higher per-request latency in exchange for higher throughput
2. **Reduce max output tokens**: Cap generation length for low-priority requests
3. **Shed best-effort traffic**: Stop accepting P3 requests
4. **Preempt low-priority requests**: Free GPU capacity by evicting P2-P3 work
5. **Enable quantized fallback**: Route to a quantized (INT8/INT4) replica that uses less memory per request
6. **Cross-region overflow**: Send requests to a different data center with available capacity

```python
class DegradationController:
    def __init__(self):
        self.level = 0  # 0 = normal, 5 = maximum degradation

    def evaluate(self, metrics: dict):
        p99_ratio = metrics['p99_ttft_ms'] / metrics['slo_ttft_ms']
        queue_ratio = metrics['avg_queue_depth'] / metrics['max_queue_depth']

        if p99_ratio > 1.5 or queue_ratio > 0.9:
            self.level = min(self.level + 1, 5)
        elif p99_ratio < 0.5 and queue_ratio < 0.3:
            self.level = max(self.level - 1, 0)

    def apply(self, request: dict) -> dict:
        if self.level >= 1:
            # Increase batching tolerance
            request['max_batch_wait_ms'] = 100

        if self.level >= 2 and request.get('priority', 2) >= 2:
            # Cap output for low-priority
            request['max_tokens'] = min(request.get('max_tokens', 2048), 512)

        if self.level >= 3 and request.get('priority', 2) >= 3:
            # Reject best-effort
            raise ServiceUnavailable("System under load, try again later")

        if self.level >= 4 and request.get('priority', 2) >= 2:
            # Reject batch requests too
            raise ServiceUnavailable("System under load, try again later")

        return request
```

## When Simple Routing Works

Not every deployment needs a cache-aware, priority-tiered, disaggregated routing stack. Simple routing is perfectly adequate in several common scenarios:

### Single Model, Homogeneous Workloads

If you are serving one model with requests that all look similar (same prompt length range, same output length range), round-robin or least-connections works fine. The variance in request cost is low enough that imbalances resolve quickly. Example: a document summarization service where all inputs are 1000-2000 tokens and outputs are 100-200 tokens.

### Small Scale (Fewer Than 8 GPUs)

With fewer than 8 GPUs, the routing decision space is small enough that even suboptimal routing does not cause catastrophic imbalances. The overhead of a sophisticated routing layer (state tracking, cache maps, health checks) may exceed the benefit. Start with least-connections and add complexity only when metrics show it is needed.

### No Shared Prefixes

If every request has a unique prompt (no shared system prompts, no multi-turn conversations), prefix-aware routing provides zero benefit. The cache hit rate will be near zero regardless of routing strategy. This is common in one-shot summarization or translation workloads.

### Latency-Insensitive Batch Processing

If requests are processed in large batches with no real-time latency requirements, simple routing with large batch sizes maximizes throughput. Queue depth does not matter when the entire batch is submitted at once and results are collected asynchronously.

<Callout type="info" title="Start Simple, Measure, Then Optimize">
  The best routing strategy is the simplest one that meets your SLOs. Implement least-connections first. Add prefix-aware routing when you measure cache hit rates below 50% for workloads with shared prefixes. Add priority routing when you have multiple tiers of traffic. Add disaggregation when prefill-decode interference is measurable. Each layer of complexity must justify itself with concrete metric improvements.
</Callout>

## Putting It All Together: A Decision Framework

Here is a practical decision tree for choosing your routing strategy:

1. **Do all requests target the same model?**
   - No: You need multi-model routing with separate pools per model. Add LoRA-affine routing if using adapters.
   - Yes: Continue.

2. **Do requests share common prefixes (system prompts, multi-turn)?**
   - Yes, and prefix is greater than 500 tokens: Prefix-aware routing is high-value. Implement it.
   - No, or prefix is fewer than 100 tokens: Skip prefix-aware routing.

3. **Do you have multiple priority tiers?**
   - Yes: Implement priority routing with dedicated P0 capacity and preemption for P2+.
   - No: Single-tier routing.

4. **Is prefill-decode interference measurable?**
   - Yes (decode latency spikes correlate with prefill arrivals): Consider disaggregated serving.
   - No: Colocated serving is fine.

5. **What scale?**
   - Fewer than 8 GPUs: Least-connections with optional prefix awareness.
   - 8-64 GPUs: Full cache-aware routing, health checking, basic autoscaling.
   - 64+ GPUs: Multi-tier architecture, composite autoscaling, graceful degradation.

## Conclusion

LLM request routing is fundamentally different from traditional web service load balancing because of one thing: state. The KV cache ties a request's history to a specific GPU, and ignoring this relationship wastes enormous amounts of compute. The progression from round-robin through cache-aware routing to full disaggregated, priority-aware, multi-tier architectures reflects the growing sophistication required as scale and workload diversity increase.

The key principles:

1. **Cache locality dominates**: Prefix-aware routing alone provides 30-50% throughput improvement for workloads with shared system prompts.
2. **Different phases need different GPUs**: Disaggregated prefill-decode routing eliminates the fundamental interference between compute-bound and memory-bound work.
3. **Adapter affinity matters for multi-model**: Grouping LoRA requests by adapter reduces swap overhead by 10x.
4. **Autoscaling must be predictive**: The 3-10 minute lag between scaling signal and GPU availability demands forecasting, not just reaction.
5. **Priority routing protects user experience**: Graceful degradation under load preserves P0 latency at the cost of best-effort traffic.
6. **Start simple**: Least-connections is a fine starting point. Add complexity only when metrics justify it.

The routing layer is invisible to end users when it works well, and catastrophically visible when it does not. Getting it right is one of the highest-leverage investments in any production LLM serving stack.
