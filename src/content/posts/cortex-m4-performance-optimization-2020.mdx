The Cortex-M4 is marketed as a "high-performance embedded processor," but achieving that performance requires understanding the microarchitecture: how the pipeline processes instructions, how flash memory wait states throttle execution, how the STM32F4 ART (Adaptive Real-Time) accelerator mitigates those wait states, how Tightly Coupled Memory eliminates them entirely, and how different compiler optimization levels affect the generated code. Without this understanding, you might run a 168 MHz STM32F4 at the effective throughput of a 30 MHz device because flash wait states dominate your execution time.

This post is a deep technical treatment of Cortex-M4 performance, from the pipeline microarchitecture through to real CoreMark scores measured at every optimization level and memory configuration.

## The Cortex-M4 Pipeline

The Cortex-M4 uses a 3-stage pipeline: Fetch, Decode, Execute. This means up to three instructions can be in-flight simultaneously, with the goal of executing one instruction per clock cycle (IPC = 1) in the steady state.

In the ideal case, the pipeline is full and one instruction completes every cycle. But several factors break this ideal:

**Flash wait states.** At 168 MHz, internal flash requires 5 wait states. Without the ART accelerator, every instruction fetch takes 6 cycles instead of 1, reducing effective throughput by approximately 6x.

**Branch penalties.** When a branch is taken, the pipeline must be flushed because instructions fetched after the branch (in the Fetch stage) are from the wrong path. The branch penalty on Cortex-M4 is 1-3 cycles depending on the branch type and whether the target is in the prefetch buffer.

**Multi-cycle instructions.** Some instructions inherently take more than one cycle in the Execute stage: SDIV/UDIV (2-12 cycles depending on operand values), multi-register load/store (1 cycle per register), and certain long-multiply instructions.

**Data dependencies.** When instruction B needs the result of instruction A, and A's result is not yet available, the pipeline stalls for one cycle (a "data hazard"). The Cortex-M4 has limited forwarding to mitigate this, but it is not as aggressive as desktop processors.

<Callout type="info" title="Cortex-M4 Is Not Cortex-A">
  Desktop and mobile ARM processors (Cortex-A series) have deep pipelines (8-15 stages), branch target buffers, return address stacks, and sophisticated dynamic predictors. Cortex-M4 has none of these. Its 3-stage pipeline means branch penalties are small (1-3 cycles), so the lack of prediction is less costly than it would be on a deeper pipeline. But it also means tight loops with conditional branches at the end will always pay a small penalty on the back-edge branch.
</Callout>

The implication for optimization: minimize branches in hot loops. Loop unrolling reduces the number of back-edge branches. Conditional execution (IT blocks in Thumb-2) can convert short conditional branches into predicated instructions that do not cause pipeline flushes.

## Flash Wait States and the ART Accelerator

### The Flash Speed Problem

STM32F4 internal flash has an access time of approximately 33 ns. At 168 MHz (5.95 ns per cycle), the flash cannot respond in one cycle. The number of wait states is configured in FLASH_ACR:

$$\text{Wait states} = \lceil \frac{f_{HCLK}}{f_{flash\_max}} \rceil - 1$$

At 168 MHz with 3.3V supply: 5 wait states. This means each flash access takes 6 cycles.

Without any acceleration, instruction throughput would drop to approximately:

$$IPC_{effective} = \frac{1}{1 + WS} = \frac{1}{6} \approx 0.167$$

That is an effective execution rate of $168 \times 0.167 = 28$ MHz. This is catastrophic.

### ART Accelerator Architecture

The ART (Adaptive Real-Time) accelerator consists of three components that work together to hide flash latency:

<Benchmark
  title="ART Accelerator Components"
  columns={["Component", "Size", "Function", "Effect"]}
  rows={[
    { values: ["Prefetch Buffer", "128-bit wide", "Reads entire flash line (4 Thumb-2 instructions) per access", "1 flash read serves 4 sequential instructions"], highlight: true },
    { values: ["Instruction Cache", "64 lines x 128 bits = 1 KB", "Caches recently fetched instruction lines", "Loops and hot code execute at 0 wait states"], highlight: true },
    { values: ["Data Cache", "8 lines x 128 bits = 128 bytes", "Caches constant data reads from flash", "Table lookups and const arrays faster"], highlight: false },
  ]}
/>

**Prefetch buffer:** Flash memory on STM32F4 is organized in 128-bit (16-byte) lines. Even though you requested one 32-bit instruction, the flash reads the entire 128-bit line in one access (6 cycles). The prefetch buffer stores this line and can serve the next 3 sequential instructions without another flash access. If instructions are executed sequentially (no branches), this effectively makes flash access 4x faster.

**Instruction cache:** The 64-line, 128-bit-wide I-cache stores recently accessed flash lines. When the CPU executes a loop body, the first iteration incurs flash wait states, but subsequent iterations hit the I-cache and execute at zero wait states. The cache uses a direct-mapped or set-associative replacement policy (implementation-specific).

Combined, the prefetch buffer and I-cache achieve near-zero-wait-state performance for typical firmware: most code is either sequential (prefetch buffer handles it) or in loops (I-cache handles it). The remaining penalty is on first-access cold misses and unpredictable branch targets.

```c
                 FLASH_ACR_ICEN         |  // Instruction cache enable

}
```

## Tightly Coupled Memory (TCM)

On STM32F4 devices, there is no formal TCM (that appears on STM32F7/H7), but there is CCM (Core-Coupled Memory) -- 64 KB of SRAM at 0x10000000 accessible only via the CPU D-bus with zero wait states. While not instruction-executable on STM32F4 (the CCM is data-only), it is useful for stack, variables, and lookup tables.

On STM32F7 and STM32H7, true TCM is available:

### Executing from ITCM (STM32H7)

On STM32H7 at 480 MHz, flash requires many wait states even with the ART accelerator. For the most critical code, copying it to ITCM (Instruction Tightly Coupled Memory) guarantees zero-wait-state execution:

```c
// Linker script section for ITCM-resident functions
// Place in .ld file:
// .itcm_text : {
//     *(.itcm_text)
// } > ITCM AT > FLASH

// Mark a function for ITCM placement
__attribute__((section(".itcm_text")))
void time_critical_isr(void) {
    // This function executes from ITCM: zero wait states
    // Guaranteed deterministic timing
    process_motor_control();
}

// Copy ITCM code from flash to ITCM at startup
// (Usually done in the startup file / SystemInit)
extern uint32_t _sitcm_text, _eitcm_text, _itcm_text_load;
void copy_to_itcm(void) {
    uint32_t *src = &_itcm_text_load;
    uint32_t *dst = &_sitcm_text;
    while (dst < &_eitcm_text) {
        *dst++ = *src++;
    }
}
```

### DTCM for Zero-Wait-State Data

Similarly, DTCM provides zero-wait-state data access. Place frequently accessed variables, DSP state buffers, and lookup tables in DTCM:

```c
// Place variable in DTCM
__attribute__((section(".dtcm_data")))
static q15_t fir_state[128];

// Place lookup table in DTCM
__attribute__((section(".dtcm_data")))
static const int16_t sine_table[256] = { /* ... */ };
```

### IT Blocks: Eliminating Short Branches

The Thumb-2 IT (If-Then) instruction enables conditional execution of up to 4 following instructions. This converts short conditional branches into predicated instructions, avoiding pipeline flush:

```c
// C code with branch
if (x > 0) {
    a = b + c;
} else {
    a = b - c;
}

// Compiler generates (with -O2):
// CMP   R0, #0
// ITE   GT           ; If-Then-Else: next instruction if GT, one after if not
// ADDGT R3, R1, R2   ; a = b + c (if x > 0)
// SUBLE R3, R1, R2   ; a = b - c (if x <= 0)
// No branch penalty!
```

```c
// Avoid division in hot loops
// BAD: 12 cycles per iteration (worst case)
for (int i = 0; i < n; i++) {
    result[i] = data[i] / divisor;
}

// GOOD: Precompute reciprocal, use multiply
// 1 cycle per iteration
uint32_t recip = 0xFFFFFFFF / divisor;  // Precompute once
for (int i = 0; i < n; i++) {
    result[i] = (uint32_t)(((uint64_t)data[i] * recip) >> 32);
}
```

### What -O2 Does on Cortex-M4

- **Inlining**: Small functions are inlined, eliminating function call overhead (BL + push/pop).
- **Register allocation**: Variables are kept in registers instead of being spilled to stack.
- **Loop optimization**: Simple loops are unrolled 2-4x, reducing branch overhead.
- **Instruction selection**: Uses Cortex-M4-specific instructions (SMLAD, QADD, barrel shifter combined with ALU ops).
- **Dead code elimination**: Unreachable code is removed.
- **Constant folding**: Compile-time constant expressions are evaluated at compile time.
- **Tail call optimization**: Recursive tail calls are converted to loops.
- **Strength reduction**: Multiplications by constants are converted to shifts and adds when faster.

<Callout type="tip" title="Start with -O2, Profile, Then Decide">
  -O2 is the best default for STM32F4. It provides strong optimization without excessive code bloat. Switch to -Os only if you are running out of flash or if profiling shows that I-cache misses are a bottleneck. ### GCC Flags for Cortex-M4

```makefile
# Recommended GCC flags for STM32F4 (Cortex-M4F with FPU)
CFLAGS  = -mcpu=cortex-m4        # Target CPU
CFLAGS += -mthumb                # Thumb-2 instruction set
CFLAGS += -mfloat-abi=hard       # Hardware float calling convention
CFLAGS += -mfpu=fpv4-sp-d16      # Single-precision FPU
CFLAGS += -O2                    # Optimization level
CFLAGS += -ffunction-sections    # Each function in its own section
CFLAGS += -fdata-sections        # Each data item in its own section
CFLAGS += -flto                  # Link-Time Optimization

LDFLAGS  = -Wl,--gc-sections     # Remove unused sections (with -ffunction-sections)
LDFLAGS += -flto                 # LTO at link time
```

**Link-Time Optimization (LTO)** deserves special mention. With LTO, the compiler defers final code generation to the link stage, allowing it to inline functions across translation units, eliminate dead code globally, and perform interprocedural optimization. On a typical STM32 firmware, LTO reduces code size by 5-15% and improves speed by 2-8%.

### Key Observations

**1. ART accelerator recovers most of the flash penalty.** Without ART, the score drops from 210 to 142 (32% loss). ART recovers performance to within 8% of zero-wait-state SRAM execution.

**2. CoreMark/MHz is constant across frequencies.** At 168 MHz, 84 MHz, and 16 MHz, the CoreMark/MHz ratio is consistently 1.25. This confirms that the ART accelerator scales linearly and flash wait states are properly compensated.

**3. -O2 vs. -O3 difference is small.** Only 4% improvement from -O3, because CoreMark's hot loops already fit in the I-cache at -O2. The -O3 unrolling does not help much and slightly increases code size.

**4. LTO provides meaningful improvement.** 7% gain over -O2 without LTO, nearly matching SRAM execution. LTO enables cross-file inlining that the non-LTO compiler cannot do.

**5. -O0 is catastrophic.** 3.4x slower than -O2. Never ship -O0 code.

### Profiling with DWT

Always profile before optimizing. The DWT cycle counter is free and always available:

```c
void profile_function(void) {
    // Enable DWT
    CoreDebug->DEMCR |= CoreDebug_DEMCR_TRCENA_Msk;
    DWT->CYCCNT = 0;
    DWT->CTRL |= DWT_CTRL_CYCCNTENA_Msk;

    // Measure
    volatile uint32_t start = DWT->CYCCNT;

    // --- Code under test ---
    my_function();
    // -----------------------

    volatile uint32_t cycles = DWT->CYCCNT - start;

    // cycles now contains the exact cycle count
    // Run multiple times and record min/max/avg
}
```

  Engineers routinely guess wrong about where their code spends time. Profile with DWT->CYCCNT first, identify the actual bottleneck, then optimize that specific function. I have seen engineers spend days optimizing a function that consumes 2% of CPU time while ignoring one that consumes 40%.
</Callout>

Understanding the memory map is essential for placing code and data in the right locations:

**Performance placement strategy for STM32F4:**
- **Hot loop code**: Keep in flash, rely on ART I-cache. Copy to SRAM1 only if measured improvement justifies the SRAM usage.
- **Stack**: Place in CCM (0x10000000) for zero-wait-state access and to free SRAM1 for data buffers.
- **DMA buffers**: Must be in SRAM1 or SRAM2 (not CCM).
- **DSP state buffers**: Place in CCM if not DMA-involved, otherwise SRAM1.
- **Lookup tables**: Place in CCM (data access is zero WS) or leave in flash (ART D-cache covers small tables).

```c
// Linker script: place stack in CCM
_estack = 0x10010000;  /* Top of CCM (64 KB) */

// Or in code: place specific data in CCM
__attribute__((section(".ccmram")))
static uint32_t dsp_scratch[4096];
```

## Conclusion

Cortex-M4 performance optimization is a layered discipline. The foundation is hardware configuration: enable the ART accelerator (instruction cache + prefetch + data cache) and configure the correct flash wait states. This single step recovers 30-50% of the performance lost to flash latency. On top of that, compiler optimization (-O2 with LTO) provides another 200-400% over debug builds, bringing the MCU to its rated 1.25 CoreMark/MHz.

Beyond the basics, understanding the pipeline's 1-3 cycle branch penalty motivates loop unrolling and IT-block usage. Understanding instruction timing (1-cycle MAC vs. 2-12 cycle divide) motivates replacing divisions with reciprocal multiplies. Understanding the I-cache size (1 KB on STM32F4) motivates keeping hot loops compact and preferring -Os over -O3 when cache pressure is the bottleneck. And on STM32H7, TCM placement eliminates both the average latency and the variance of flash/cache access, providing deterministic zero-wait-state execution for the most critical code paths.

The CoreMark results validate the approach: a properly configured STM32F407 at 168 MHz achieves 210-228 CoreMark (depending on LTO and memory placement), matching ARM's reference score. The gap between "properly configured" (210) and "misconfigured" (62 at -O0, or 142 without ART) is a factor of 1.5-3.4x. That is the difference between a system that meets its real-time deadlines and one that does not.
