Pruning removes unnecessary weights from neural networks to reduce compute and memory. The promise is appealing: remove 90% of weights, get 10x speedup. The reality is more nuanced -- most pruning methods don't deliver real speedup on GPUs because sparse computation is hard to accelerate on hardware designed for dense matrix math.

<Callout type="warning" title="The Pruning Speedup Myth">
  Unstructured pruning (the most common in research papers) almost never delivers real GPU speedup. Setting 90% of weights to zero doesn't help if the GPU still performs dense matrix multiplication -- zeros are multiplied just as fast as non-zeros on tensor cores. Only structured and semi-structured pruning translate to actual wall-clock improvements.
</Callout>

## Unstructured Pruning: High Sparsity, No Speedup

Unstructured pruning (magnitude pruning, lottery ticket, etc.) zeros out individual weights anywhere in the matrix:

```python
# Simple magnitude pruning -- zeros out smallest weights
def magnitude_prune(weight, sparsity=0.9):
    threshold = torch.quantile(weight.abs(), sparsity)
    mask = weight.abs() >= threshold
    return weight * mask  # 90% of values are now zero
```

The resulting sparse matrix has random zero patterns. GPUs can't skip these zeros in standard GEMM -- they still allocate the full matrix and perform all multiply-accumulates. Libraries like cuSPARSE can exploit sparsity, but their overhead typically exceeds the compute savings until sparsity exceeds 95%.

## Structured Pruning: Real Speedup

Structured pruning removes entire channels, attention heads, or layers -- producing a genuinely smaller dense model:

```python
# Structured channel pruning -- remove least important output channels
def structured_prune_conv(conv_layer, channels_to_keep):
    """Remove entire output channels from a conv layer."""
    new_weight = conv_layer.weight[channels_to_keep]  # Smaller tensor
    new_bias = conv_layer.bias[channels_to_keep] if conv_layer.bias is not None else None

    # Create new, smaller conv layer
    new_conv = nn.Conv2d(
        conv_layer.in_channels,
        len(channels_to_keep),  # Fewer output channels
        conv_layer.kernel_size,
        stride=conv_layer.stride,
        padding=conv_layer.padding,
    )
    new_conv.weight.data = new_weight
    if new_bias is not None:
        new_conv.bias.data = new_bias
    return new_conv
```

Because the pruned model is dense (just smaller), standard GEMM works perfectly -- no sparse libraries needed:

## Semi-Structured (2:4) Sparsity: The Hardware Sweet Spot

NVIDIA's Ampere (A100) and Hopper (H100) tensor cores natively support 2:4 sparsity: exactly 2 out of every 4 consecutive elements must be zero. The hardware skips the zero computations, delivering ~2x speedup:

<Callout type="perf" title="The Production Choice">
  Semi-structured 2:4 sparsity on Ampere+ GPUs is currently the most practical pruning approach for production: ~1.8x real speedup with minimal accuracy loss, native hardware support, and no custom sparse kernels needed. Apply it with `torch.ao.pruning` or NVIDIA's ASP (Automatic SParsity) tool.

## Conclusion

Pruning's real-world value depends entirely on whether it translates to actual speedup on your target hardware. Unstructured pruning reduces parameter count on paper but delivers negligible GPU speedup. Structured pruning (channels, heads, layers) produces genuinely smaller dense models with proportional speedup. Semi-structured 2:4 sparsity on Ampere+ is the current sweet spot: ~1.8x real speedup with minimal accuracy loss and native hardware support. For maximum impact, combine structured pruning with quantization -- the speedups multiply.
