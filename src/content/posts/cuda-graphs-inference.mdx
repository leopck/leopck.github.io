Every CUDA kernel launch involves CPU-side overhead: driver validation, parameter setup, and GPU command buffer submission. Individually, each launch costs ~5-10 µs — negligible for large kernels. But an LLM decode step launches 200-400 tiny kernels (normalization, activation, attention scoring). At 5 µs each, launch overhead alone consumes 1-2 ms per decode step — often 15-30% of total latency.

CUDA Graphs capture an entire sequence of kernel launches once, then replay the sequence with a single GPU-side submission. This eliminates all per-kernel CPU overhead.

## How CUDA Graphs Work

CUDA Graphs have three phases: **capture**, **instantiate**, and **replay**.

```python
import torch

# Phase 1: Capture — record the kernel sequence
static_input = torch.randn(1, 4096, device='cuda')  # Placeholder
graph = torch.cuda.CUDAGraph()

with torch.cuda.graph(graph):
    # All CUDA operations here are recorded, not executed
    static_output = model.decode_step(static_input)

# Phase 2: Instantiate — happens automatically in PyTorch

# Phase 3: Replay — execute the entire captured sequence
# Just copy new data into the static input buffer and replay
for token_input in token_stream:
    static_input.copy_(token_input)  # Update input in-place
    graph.replay()  # Single GPU submission — no per-kernel overhead
    result = static_output.clone()   # Read output
```

<Callout type="warning" title="The Static Shape Constraint">
  CUDA Graphs capture a fixed computation sequence. Tensor shapes, control flow, and memory allocations cannot change between replays. For LLM decode, this means you need separate graphs for each batch size and (sometimes) different sequence length buckets. This is the primary engineering complexity.
</Callout>

The pattern is clear: CUDA Graphs help most when launch overhead is a large fraction of total time — small batch, many kernels. At large batch sizes, individual kernels take longer (diluting the fixed launch overhead), and the benefit shrinks.

<Callout type="tip" title="The Warmup Cost">
  CUDA Graph capture triggers compilation of all kernels in the graph. For LLM decode, this takes 5-15 seconds per batch size during server startup. Pre-warm at deployment time, not during the first request. vLLM does this automatically with `--enforce-eager=False` (default).
</Callout>

## When CUDA Graphs Don't Help

<Benchmark
  title="CUDA Graph Limitations"
  columns={["Limitation", "Impact", "Workaround"]}

    { values: ["Capture overhead", "5-15 sec per graph variant", "Warmup at startup"], highlight: false },
    { values: ["Memory overhead", "Each graph caches GPU memory", "Limit graph variants (8-12 typical)"], highlight: false },
  ]}

The prefill phase is typically NOT graphed because sequence lengths vary widely (requires too many graph variants). Only the decode phase — where each step processes exactly 1 token per request — benefits from CUDA Graphs.

## Conclusion

CUDA Graphs eliminate kernel launch overhead by replaying pre-captured kernel sequences with a single GPU submission. For LLM decode at small batch sizes, this saves 20-30% latency. The trade-off is engineering complexity: fixed shapes require padding and multiple graph variants. vLLM, TensorRT-LLM, and most production serving frameworks use CUDA Graphs for decode by default. The rule of thumb: if your workload launches >100 kernels per inference step and your batch size is small (<32), CUDA Graphs will deliver meaningful speedup.
