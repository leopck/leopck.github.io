The AI accelerator market is no longer NVIDIA-only. AMD MI300X offers 192GB HBM3 at 5.3 TB/s. Intel Gaudi3 claims competitive training throughput at lower cost. Google TPU v5p dominates internal workloads. Yet NVIDIA's market share remains above 80%. Understanding WHY requires looking beyond raw hardware specs at the software ecosystem that makes or breaks an accelerator.

On paper, AMD MI300X has the best memory bandwidth AND capacity. But raw specs don't determine real-world performance — software optimization does.

## Intel Gaudi: The Graph Compiler Approach

Gaudi takes a fundamentally different approach from CUDA. Instead of writing kernels, you define a computation graph and let the compiler optimize it.

**Architecture**: Gaudi has two compute engines — the **TPC (Tensor Processor Core)** for general operations and the **MME (Matrix Math Engine)** for GEMMs. The graph compiler schedules operations across both, overlapping TPC work with MME matrix multiplies.

**Programming model**: No kernel-level control. You express models in PyTorch, the Habana bridge captures the graph, SynapseAI compiles and optimizes it. This is similar to XLA's approach — higher abstraction, less control.

**Strength**: The graph compiler can optimize across operations automatically — fusion, scheduling, memory allocation happen without programmer intervention. For standard transformer architectures, this works well.

**Weakness**: Custom operations, non-standard architectures, and debugging are much harder. If the compiler doesn't optimize your pattern well, you have limited recourse. There's no equivalent of writing a custom CUDA kernel.

## AMD MI300X: HIP Translation

AMD's ROCm/HIP approach translates CUDA code to run on AMD hardware:

```cpp
// CUDA code:
cudaMalloc(&d_ptr, size);
kernel<<<grid, block>>>(d_ptr);

// HIP equivalent (often auto-converted):
hipMalloc(&d_ptr, size);
kernel<<<grid, block>>>(d_ptr);  // Same syntax
```

**hipify**: AMD provides tools to automatically convert CUDA code to HIP. For straightforward kernels, this works. For complex code using advanced CUDA features (tensor cores, cooperative groups, TMA), manual porting is required.

**FlashAttention on ROCm**: The community has ported FlashAttention to ROCm, but performance is typically 70-85% of the NVIDIA equivalent because AMD's matrix cores have different tile sizes and scheduling characteristics.

**Strength**: 192GB HBM3 is genuinely useful — it fits larger models without tensor parallelism. Llama 3 70B in FP16 (140GB) fits on a single MI300X.

**Weakness**: Library ecosystem lags NVIDIA by 6-18 months. When NVIDIA releases a new cuDNN optimization, it takes time for ROCm to catch up.

## Google TPU: XLA-First

TPU v5p uses the XLA compiler exclusively. All computation is expressed as HLO (High-Level Operations) and compiled to TPU instructions.

**Strength**: For JAX/Flax workloads, TPUs are highly optimized. Google trains Gemini on TPU pods with excellent scaling.

**Weakness**: PyTorch support is secondary. torch_xla exists but introduces compilation overhead and doesn't support all PyTorch operations. The ecosystem is JAX-centric.

**ICI (Inter-Chip Interconnect)**: TPU pods have 4.8 Tb/s all-to-all connectivity — vastly superior to InfiniBand for distributed training. This is TPU's killer feature for very large training runs.

The software moat has three layers:

1. **Libraries**: cuDNN, cuBLAS, NCCL, TensorRT, Triton — each with years of optimization. Alternatives exist but with fewer features and less tuning.

2. **Community**: When a new technique appears (FlashAttention, PagedAttention, speculative decoding), the first implementation is always CUDA. Ports come months later.

3. **Tooling**: nsight compute, nsight systems, cuda-gdb, compute sanitizer — NVIDIA's profiling and debugging tools are vastly superior to alternatives.

<Callout type="warning" title="The Portability Trap">
Code that runs well on H100 often runs poorly on MI300X or Gaudi, even after porting. Tile sizes, memory access patterns, and scheduling strategies are hardware-specific. Real optimization requires understanding each architecture, not just translating syntax.
</Callout>

For cost-sensitive workloads at scale, AMD and Gaudi are increasingly viable — especially for standard transformer training where the software gap matters less.
