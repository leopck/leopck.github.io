Standard attention scales as O(N^2) in sequence length, making 32K+ contexts expensive even with FlashAttention. Sparse attention replaces the full NxN attention matrix with a structured pattern that attends to only O(NsqrtN) or O(N log N) positions, trading some attention capacity for dramatic compute reduction. The question: when does the quality trade-off justify the savings?

<Callout type="info" title="Sparse Attention vs FlashAttention">
  FlashAttention computes **exact** full attention with O(N) memory instead of O(N^2), but still does O(N^2) computation. Sparse attention reduces both memory AND computation but **approximates** the attention pattern. They're complementary: you can apply FlashAttention to the sparse pattern's non-zero blocks for maximum efficiency.

## Implementation Patterns

Sparse attention requires modified attention masks and kernels:

```python
def sliding_window_attention(Q, K, V, window_size=4096):
    """Sliding window attention -- each token attends to W previous tokens."""
    seq_len = Q.shape[1]
    # Create band diagonal mask
    mask = torch.ones(seq_len, seq_len, dtype=torch.bool)
    mask = torch.triu(mask, diagonal=-window_size)  # Keep lower band
    mask = torch.tril(mask)  # Causal

    scores = Q @ K.transpose(-1, -2) / math.sqrt(Q.shape[-1])
    scores.masked_fill_(~mask, float('-inf'))
    attn = F.softmax(scores, dim=-1)
    return attn @ V
```

For production, this naive implementation is inefficient because it still allocates the full NxN matrix. Efficient implementations use block-sparse kernels that only compute the non-zero blocks:

## Conclusion

Sparse attention reduces the O(N^2) compute and memory cost of attention to O(NxW) or O(NsqrtN), enabling practical long-context inference. Sliding window attention (Mistral-style, W=4096) is the production standard: it provides 8x compute/memory reduction at 32K context with only 0.1-0.3 perplexity loss, and layer stacking extends the effective receptive field far beyond the window size. For implementation, FlashAttention-2's built-in window support is the recommended path -- it combines sparse pattern efficiency with FlashAttention's optimized memory access. Full dense attention remains optimal below 8K tokens where the quadratic cost is manageable.
