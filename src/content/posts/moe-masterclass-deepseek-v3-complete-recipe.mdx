## 2. Multi-head Latent Attention (MLA)

### The KV Cache Problem

Standard multi-head attention stores $K$ and $V$ projections for every token in the sequence. With GQA-8 (Llama 3 style), the KV cache per token per layer is:

$$\text{KV}_{\text{GQA}} = 2 \times n_{\text{kv\_heads}} \times d_{\text{head}} \times \text{dtype\_size}$$

For Llama 3.1 405B: $2 \times 8 \times 128 \times 2 = 4096$ bytes per token per layer. Across 126 layers at 128K context: $4096 \times 126 \times 128000 \approx 63$ GB.

### MLA: Compress KV into a Latent

MLA compresses the KV representation into a low-rank latent vector $c_t$ of dimension $d_c = 512$, then reconstructs $K$ and $V$ on the fly during attention.

```python
import torch
import torch.nn as nn
import math

class MultiHeadLatentAttention(nn.Module):
    """
    DeepSeek V3 Multi-head Latent Attention (MLA).
    Stores compressed KV cache instead of full K, V tensors.
    """
    def __init__(
        self,
        d_model=7168,
        num_heads=128,
        kv_lora_rank=512,      # Compressed KV dimension
        q_lora_rank=1536,      # Compressed Q dimension
        qk_rope_dim=64,        # Dimensions for RoPE
        qk_nope_dim=128,       # Dimensions without RoPE
    ):
        super().__init__()
        self.d_model = d_model
        self.num_heads = num_heads
        self.kv_lora_rank = kv_lora_rank
        self.q_lora_rank = q_lora_rank
        self.qk_rope_dim = qk_rope_dim
        self.qk_nope_dim = qk_nope_dim
        self.head_dim = qk_nope_dim + qk_rope_dim  # 192

        # KV compression
        # Step 1: Project input to low-rank KV latent
        self.kv_down = nn.Linear(d_model, kv_lora_rank, bias=False)
        # Step 2: Reconstruct K (no-RoPE part) from latent
        self.k_nope_up = nn.Linear(kv_lora_rank, num_heads * qk_nope_dim, bias=False)
        # Step 3: Reconstruct V from latent
        self.v_up = nn.Linear(kv_lora_rank, num_heads * qk_nope_dim, bias=False)
        # Step 4: K (RoPE part) - separate, NOT compressed
        self.k_rope_proj = nn.Linear(d_model, qk_rope_dim, bias=False)

        # Q compression (also low-rank for efficiency)
        self.q_down = nn.Linear(d_model, q_lora_rank, bias=False)
        self.q_nope_up = nn.Linear(q_lora_rank, num_heads * qk_nope_dim, bias=False)
        self.q_rope_up = nn.Linear(q_lora_rank, num_heads * qk_rope_dim, bias=False)

        # Output projection
        self.o_proj = nn.Linear(num_heads * qk_nope_dim, d_model, bias=False)

    def forward(self, hidden_states, kv_cache=None, position_ids=None):
        B, T, D = hidden_states.shape

        # === KV Compression ===
        # Compress to latent: [B, T, d_c=512]
        kv_latent = self.kv_down(hidden_states)

        # Reconstruct K (no-RoPE): [B, T, H * qk_nope_dim]
        k_nope = self.k_nope_up(kv_latent)
        k_nope = k_nope.view(B, T, self.num_heads, self.qk_nope_dim)

        # K (RoPE part): [B, T, qk_rope_dim] - from original input, not latent
        k_rope = self.k_rope_proj(hidden_states)
        k_rope = k_rope.unsqueeze(2).expand(-1, -1, self.num_heads, -1)
        # Apply RoPE to k_rope here (omitted for clarity)

        # Reconstruct V: [B, T, H * qk_nope_dim]
        v = self.v_up(kv_latent)
        v = v.view(B, T, self.num_heads, self.qk_nope_dim)

        # === Q Compression ===
        q_latent = self.q_down(hidden_states)
        q_nope = self.q_nope_up(q_latent)
        q_nope = q_nope.view(B, T, self.num_heads, self.qk_nope_dim)

        q_rope = self.q_rope_up(q_latent)
        q_rope = q_rope.view(B, T, self.num_heads, self.qk_rope_dim)
        # Apply RoPE to q_rope here

        # Concatenate RoPE and non-RoPE parts
        q = torch.cat([q_nope, q_rope], dim=-1)  # [B, T, H, head_dim]
        k = torch.cat([k_nope, k_rope], dim=-1)  # [B, T, H, head_dim]

        # === KV Cache ===
        # Key insight: cache kv_latent (512 dims) + k_rope (64 dims)
        # instead of full K, V (128 * 128 * 2 = 32768 dims)
        if kv_cache is not None:
            # Cache only: kv_latent [B, T, 512] + k_rope [B, T, 64]
            # Total: 576 values per token (vs 32768 for standard GQA-8)
            pass

        # === Attention Computation ===
        q = q.transpose(1, 2)  # [B, H, T, head_dim]
        k = k.transpose(1, 2)  # [B, H, T, head_dim]
        v = v.transpose(1, 2)  # [B, H, T, qk_nope_dim]

        scale = 1.0 / math.sqrt(self.qk_nope_dim + self.qk_rope_dim)
        attn_weights = torch.matmul(q, k.transpose(-2, -1)) * scale
        attn_weights = torch.softmax(attn_weights, dim=-1)

        attn_output = torch.matmul(attn_weights, v)
        attn_output = attn_output.transpose(1, 2).reshape(B, T, -1)

        output = self.o_proj(attn_output)
        return output
```

### KV Cache Savings

The cache stores only:
- `kv_latent`: $d_c = 512$ values per token per layer
- `k_rope`: $d_{\text{rope}} = 64$ values per token per layer

Total per token per layer: $576 \times 2 = 1152$ bytes (FP16). Compare to standard GQA-8 Llama 3 style: $2 \times 8 \times 128 \times 2 = 4096$ bytes.

<Callout type="perf" title="MLA: Quality Without KV Cost">
The key insight of MLA is that the KV cache does not need to store the full-rank K and V representations. A compressed latent of rank 512 captures sufficient information to reconstruct attention-quality K and V. The decompression (multiplying by the up-projection matrices) happens on the fly during attention, trading compute for memory. Since attention at inference is memory-bound (loading KV cache from HBM), the extra compute from decompression is effectively free.

## 3. Fine-Grained MoE with Group-Limited Routing

### 256 Experts, 8 Active

DeepSeek V3 uses 256 routed experts plus 1 shared expert. For each token:
1. The shared expert always activates (baseline computation).
2. The router selects 8 of 256 routed experts via group-limited gating.
3. The outputs are combined with sigmoid-normalized weights.

```python
class DeepSeekV3MoELayer(nn.Module):
    def __init__(self, config):
        super().__init__()
        self.num_routed = config.num_routed_experts   # 256
        self.num_shared = config.num_shared_experts    # 1
        self.top_k = config.top_k                      # 8
        self.num_groups = config.num_groups            # 8
        self.top_k_group = config.top_k_group          # 4
        self.experts_per_group = self.num_routed // self.num_groups  # 32

        # Shared expert (always runs)
        self.shared_expert = SwiGLUFFN(config.d_model, config.shared_expert_d_ff)

        # Routed experts
        self.experts = nn.ModuleList([
            SwiGLUFFN(config.d_model, config.expert_d_ff)
            for _ in range(self.num_routed)
        ])

        # Router
        self.gate = nn.Linear(config.d_model, self.num_routed, bias=False)

        # Aux-free bias for load balancing
        self.expert_bias = nn.Parameter(
            torch.zeros(self.num_routed), requires_grad=False
        )

    def forward(self, x):
        # Shared expert (always)
        shared_out = self.shared_expert(x)

        # Routing
        logits = self.gate(x) + self.expert_bias  # [T, 256]

        # Group-limited selection
        grouped = logits.view(-1, self.num_groups, self.experts_per_group)
        group_top2, _ = grouped.topk(2, dim=-1)
        group_scores = group_top2.mean(dim=-1)  # [T, 8]

        _, top_groups = group_scores.topk(self.top_k_group, dim=-1)  # [T, 4]
        group_mask = torch.zeros_like(group_scores, dtype=torch.bool)
        group_mask.scatter_(1, top_groups, True)

        expert_mask = group_mask.unsqueeze(-1).expand(
            -1, -1, self.experts_per_group
        ).reshape(-1, self.num_routed)

        masked_logits = logits.masked_fill(~expert_mask, float('-inf'))

        # Sigmoid routing (not softmax)
        probs = torch.sigmoid(masked_logits)
        top_k_probs, top_k_idx = probs.topk(self.top_k, dim=-1)

        # Normalize
        weights = top_k_probs / (top_k_probs.sum(dim=-1, keepdim=True) + 1e-6)

        # Expert computation
        routed_out = torch.zeros_like(x)
        for k in range(self.top_k):
            for e in range(self.num_routed):
                mask = (top_k_idx[:, k] == e)
                if mask.any():
                    out = self.experts[e](x[mask])
                    routed_out[mask] += weights[mask, k:k+1] * out

        return shared_out + routed_out
```

## 4. FP8 Mixed-Precision Training

### The Compute Advantage

FP8 (8-bit floating point) provides double the throughput of BF16 on NVIDIA H100 GPUs: 1978 TFLOPS (FP8) vs 989 TFLOPS (BF16). DeepSeek V3 was the first frontier model trained end-to-end with FP8 for expert computation.

```python
class FP8ExpertFFN(nn.Module):
    """
    SwiGLU FFN with FP8 compute.
    Weights stored in FP8, computation in FP8, accumulation in FP32.
    """
    def __init__(self, d_model, d_ff):
        super().__init__()
        # Weights stored in FP8 E4M3 format
        self.w1 = nn.Linear(d_model, d_ff, bias=False)
        self.w2 = nn.Linear(d_ff, d_model, bias=False)
        self.w3 = nn.Linear(d_model, d_ff, bias=False)

    def forward(self, x):
        # Dynamic quantization to FP8
        x_fp8 = self._to_fp8(x)

        # Gate: SiLU(W1 * x)
        gate = torch.nn.functional.silu(
            self._fp8_matmul(x_fp8, self.w1.weight)
        )

        # Up: W3 * x
        up = self._fp8_matmul(x_fp8, self.w3.weight)

        # Combine and down-project
        hidden = gate * up
        hidden_fp8 = self._to_fp8(hidden)
        output = self._fp8_matmul(hidden_fp8, self.w2.weight)

        return output

    def _to_fp8(self, tensor):
        """Dynamic quantization to FP8 E4M3 with per-tile scaling."""
        # Tile size: 128 elements (DeepSeek's choice)
        tile_size = 128
        shape = tensor.shape
        flat = tensor.reshape(-1, tile_size)

        # Per-tile absmax scaling
        absmax = flat.abs().max(dim=-1, keepdim=True).values
        scale = absmax / 448.0  # E4M3 max value
        quantized = (flat / (scale + 1e-12)).clamp(-448, 448)

        # In practice, use torch.float8_e4m3fn dtype
        return quantized.to(torch.float8_e4m3fn), scale

    def _fp8_matmul(self, x_fp8, weight):
        """FP8 matrix multiply with FP32 accumulation."""
        x_data, x_scale = x_fp8
        w_data = weight.to(torch.float8_e4m3fn)
        w_scale = weight.abs().max() / 448.0

        # Compute in FP8, accumulate in FP32
        result = torch._scaled_mm(
            x_data, w_data.t(),
            scale_a=x_scale, scale_b=w_scale,
            out_dtype=torch.bfloat16,
        )
        return result
```

### Per-Tile Quantization

DeepSeek V3 uses a fine-grained quantization strategy: instead of per-tensor or per-channel scaling, they use per-tile scaling with tiles of 128 elements. This provides better precision for outlier-heavy distributions.

```python
def compare_quantization_granularity(tensor, granularities):
    """
    Compare quantization error at different granularities.
    """
    results = {}
    for name, tile_size in granularities:
        flat = tensor.reshape(-1)
        num_tiles = flat.numel() // tile_size
        tiles = flat[:num_tiles * tile_size].view(num_tiles, tile_size)

        # Per-tile quantize-dequantize
        absmax = tiles.abs().max(dim=-1, keepdim=True).values
        scale = absmax / 448.0
        quantized = (tiles / (scale + 1e-12)).clamp(-448, 448).round()
        dequantized = quantized * scale

        # Relative error
        error = (tiles - dequantized).abs() / (tiles.abs() + 1e-12)
        results[name] = {
            "mean_relative_error": error.mean().item(),
            "max_relative_error": error.max().item(),
            "p99_relative_error": error.quantile(0.99).item(),
        }

    return results

# Example granularities
granularities = [
    ("per-tensor", tensor.numel()),
    ("per-channel", tensor.shape[-1]),
    ("per-tile-128", 128),
    ("per-tile-32", 32),
]
```

### What Stays in BF16

Not everything is FP8. DeepSeek V3 keeps these components in BF16 for stability:

- Attention computation (Q, K, V projections and softmax)
- Router/gating computation
- Layer norms (RMSNorm)
- Embedding layers
- Master weights (for optimizer state)
- Loss computation

Only the expert FFN computation (the most compute-intensive part, ~60% of total FLOPs) runs in FP8.

<Callout type="info" title="FP8 Training Stability">
FP8 training requires careful handling of dynamic ranges. DeepSeek V3 uses delayed scaling: the quantization scale for each tile is computed from the previous iteration's activations, not the current one. This avoids the overhead of computing absmax in the current forward pass while keeping the scale approximately correct. The scale is updated every few steps via exponential moving average.
</Callout>

## 5. DualPipe: Zero-Bubble Pipeline Parallelism

### The Pipeline Bubble Problem

Standard pipeline parallelism (PP) divides the model into stages across GPUs. During the forward pass, GPU $i$ must wait for GPU $i-1$ to finish before it can start. This creates a "bubble" where GPUs are idle.

For $P$ pipeline stages and microbatch size $M$:

$$\text{Bubble fraction} = \frac{P - 1}{P + M - 1}$$

With $P = 8$ stages and $M = 1$: bubble = 87.5%. Even with $M = 8$: bubble = 46.7%.

### DualPipe: Overlap Forward and Backward

DualPipe sends two streams of microbatches through the pipeline simultaneously — one forward and one backward — so that each GPU is always doing useful work.

```python
class DualPipeScheduler:
    """
    DualPipe scheduling: interleave forward and backward passes
    to eliminate pipeline bubbles.
    """
    def __init__(self, num_stages, num_microbatches):
        self.P = num_stages
        self.M = num_microbatches

    def generate_schedule(self):
        """
        Generate the DualPipe schedule.
        Each cell is (stage, microbatch, direction).
        """
        schedule = []  # List of (time_slot, stage, microbatch, "fwd"/"bwd")

        # Phase 1: Warmup - fill the pipeline with forward passes
        for t in range(self.P):
            for s in range(t + 1):
                schedule.append((t, s, t - s, "fwd"))

        # Phase 2: Steady state - interleave forward and backward
        for m in range(self.M - self.P):
            t = self.P + m
            for s in range(self.P):
                # Forward pass for new microbatch
                schedule.append((t, s, self.P + m - s, "fwd"))
                # Backward pass for completed microbatch
                schedule.append((t, self.P - 1 - s, m + s, "bwd"))

        # Phase 3: Drain - complete remaining backward passes
        for t in range(self.M - self.P, self.M):
            for s in range(self.M - t):
                schedule.append((self.P + t, self.P - 1 - s, t + s, "bwd"))

        return schedule

    def compute_bubble_fraction(self):
        """Compute the theoretical bubble fraction."""
        # Standard PP
        standard_bubble = (self.P - 1) / (self.P + self.M - 1)

        # DualPipe: bubbles only at warmup and drain
        # Approximately P forward-only slots + P backward-only slots
        dualpipe_bubble = (self.P - 1) / (2 * self.M + self.P - 1)

        return {
            "standard_pp_bubble": standard_bubble,
            "dualpipe_bubble": dualpipe_bubble,
            "improvement": standard_bubble / dualpipe_bubble,
        }
```

### How Communication Overlaps

The key innovation: while GPU $i$ runs the forward pass for microbatch $m+1$, it simultaneously runs the backward pass for microbatch $m$, and the all-to-all communication for MoE layers overlaps with both.

```python
def dualpipe_step(stage, fwd_microbatch, bwd_microbatch, moe_layer):
    """
    A single DualPipe step for one stage.
    Three activities overlap:
    1. Forward computation for fwd_microbatch
    2. Backward computation for bwd_microbatch
    3. All-to-all communication for MoE experts
    """
    # Start all-to-all dispatch for forward (async)
    fwd_dispatch_handle = start_async_all_to_all(
        fwd_microbatch.tokens, fwd_microbatch.expert_assignments
    )

    # While all-to-all runs, do backward computation
    # (backward uses experts already on this GPU from previous step)
    bwd_grads = compute_backward(stage, bwd_microbatch)

    # Wait for forward dispatch
    fwd_expert_inputs = fwd_dispatch_handle.wait()

    # Start backward all-to-all (async)
    bwd_dispatch_handle = start_async_all_to_all(
        bwd_grads, bwd_microbatch.expert_assignments
    )

    # While backward all-to-all runs, do forward expert computation
    fwd_expert_outputs = compute_experts(stage, fwd_expert_inputs)

    # Wait for backward dispatch
    bwd_dispatch_handle.wait()

    return fwd_expert_outputs
```

## 6. Multi-Token Prediction (MTP)

### The Idea

Standard language models predict one token at a time: given tokens $1, \ldots, t$, predict token $t+1$. MTP predicts tokens $t+1$ and $t+2$ (and potentially further) simultaneously from the same hidden state.

```python
class MultiTokenPredictionHead(nn.Module):
    """
    DeepSeek V3 Multi-Token Prediction.
    Predicts the next token AND the token after that.
    """
    def __init__(self, d_model, vocab_size, mtp_depth=1):
        super().__init__()
        self.mtp_depth = mtp_depth

        # Standard next-token prediction head
        self.lm_head = nn.Linear(d_model, vocab_size, bias=False)

        # Additional prediction heads for future tokens
        # Each head has its own projection + shared embedding
        self.mtp_projections = nn.ModuleList([
            nn.Sequential(
                nn.Linear(d_model, d_model, bias=False),
                nn.RMSNorm(d_model),
            )
            for _ in range(mtp_depth)
        ])

        # Embedding layer for feeding predicted tokens back
        self.token_embedding = nn.Embedding(vocab_size, d_model)

    def forward(self, hidden_states, input_ids=None):
        """
        hidden_states: [B, T, d_model] - final transformer output
        input_ids: [B, T] - for teacher forcing during training
        """
        B, T, D = hidden_states.shape

        # Standard next-token logits
        logits_t1 = self.lm_head(hidden_states)  # [B, T, V]

        mtp_logits = [logits_t1]

        # Predict future tokens
        current_hidden = hidden_states
        for depth in range(self.mtp_depth):
            # Get embedding of the next token (teacher forcing during training)
            if input_ids is not None and depth == 0:
                # During training: use ground truth token t+1 to help predict t+2
                next_token_embed = self.token_embedding(input_ids[:, 1:])
                # Pad to match sequence length
                next_token_embed = torch.nn.functional.pad(
                    next_token_embed, (0, 0, 0, 1), value=0
                )
            else:
                # During inference: use argmax of predicted logits
                next_token_id = mtp_logits[-1].argmax(dim=-1)
                next_token_embed = self.token_embedding(next_token_id)

            # Combine current hidden state with next token embedding
            combined = current_hidden + next_token_embed

            # Project to predict token t+2 (or further)
            projected = self.mtp_projections[depth](combined)
            future_logits = self.lm_head(projected)
            mtp_logits.append(future_logits)

            current_hidden = projected

        return mtp_logits  # List of [B, T, V], length = 1 + mtp_depth
```

### Training Loss

The MTP training loss is a weighted sum of cross-entropy losses for each predicted token:

```python
def mtp_loss(mtp_logits, target_ids, mtp_weight=0.3):
    """
    Compute Multi-Token Prediction loss.
    mtp_logits: list of [B, T, V] for each prediction depth
    target_ids: [B, T] ground truth token IDs
    """
    total_loss = 0
    num_valid = 0

    for depth, logits in enumerate(mtp_logits):
        # Target for depth d is the token at position t+d+1
        if depth + 1 < target_ids.shape[1]:
            shifted_targets = target_ids[:, depth + 1:]
            shifted_logits = logits[:, :shifted_targets.shape[1]]

            loss = torch.nn.functional.cross_entropy(
                shifted_logits.reshape(-1, shifted_logits.shape[-1]),
                shifted_targets.reshape(-1),
                reduction='mean',
            )

            # Weight decreases with depth
            weight = 1.0 if depth == 0 else mtp_weight
            total_loss += weight * loss
            num_valid += 1

    return total_loss / max(num_valid, 1)
```

### Dual Purpose: Quality + Speculative Decoding

MTP serves two roles in DeepSeek V3:

1. **Training quality**: The additional prediction targets provide a richer learning signal. The model must understand not just the next token but the token after that, which requires deeper understanding of the context.

2. **Inference speed via speculative decoding**: At inference, the MTP head can draft candidate future tokens that are then verified in parallel by the main model.

```python
class MTPSpeculativeDecoder:
    """
    Use MTP head for speculative decoding.
    The MTP head drafts 1 additional token.
    The main model verifies both in one forward pass.
    """
    def __init__(self, model):
        self.model = model

    def generate_step(self, input_ids, kv_cache):
        # Step 1: Forward pass to get token t+1 and speculative t+2
        with torch.no_grad():
            outputs = self.model(input_ids, kv_cache=kv_cache)
            mtp_logits = outputs.mtp_logits

        # Token t+1 (high confidence)
        token_t1 = mtp_logits[0][:, -1:].argmax(dim=-1)

        # Token t+2 (speculative, from MTP head)
        token_t2_draft = mtp_logits[1][:, -1:].argmax(dim=-1)

        # Step 2: Verify both tokens with one forward pass
        verify_input = torch.cat([token_t1, token_t2_draft], dim=-1)
        with torch.no_grad():
            verify_output = self.model(verify_input, kv_cache=kv_cache)

        # Check if t+2 draft matches verification
        verified_t2 = verify_output.mtp_logits[0][:, 0:1].argmax(dim=-1)

        if (verified_t2 == token_t2_draft).all():
            # Both tokens accepted
            return torch.cat([token_t1, token_t2_draft], dim=-1), 2
        else:
            # Only t+1 accepted
            return token_t1, 1
```

## 7. Auxiliary-Loss-Free Load Balancing

### The Innovation

DeepSeek V3 eliminates the load-balancing auxiliary loss entirely. Instead, it adds a per-expert bias term to routing logits, updated by an exponential moving average of expert utilization.

```python
class AuxFreeMoEBalancer:
    """
    DeepSeek V3's auxiliary-loss-free load balancing.
    Uses per-expert bias terms updated via EMA.
    """
    def __init__(self, num_experts, ema_decay=0.999, bias_scale=0.1):
        self.num_experts = num_experts
        self.ema_decay = ema_decay
        self.bias_scale = bias_scale

        # EMA of expert utilization
        self.ema_count = torch.ones(num_experts) / num_experts

        # Bias added to routing logits
        self.bias = torch.zeros(num_experts)

    def update(self, expert_counts, total_tokens):
        """
        Update bias based on observed utilization.
        Called once per training step.
        """
        # Current step utilization
        current = expert_counts.float() / total_tokens

        # EMA update
        self.ema_count = (
            self.ema_decay * self.ema_count +
            (1 - self.ema_decay) * current
        )

        # Bias: push tokens toward under-utilized experts
        target = 1.0 / self.num_experts
        deviation = self.ema_count - target

        # Update bias: negative deviation (under-utilized) gets positive bias
        self.bias = -self.bias_scale * deviation

    def get_bias(self):
        """Return current bias to add to routing logits."""
        return self.bias
```

### Why This Is Better Than Auxiliary Loss

The auxiliary loss creates a tension: the load-balancing gradient pulls the router toward uniform routing, while the language modeling gradient pulls it toward routing tokens to the best experts. This tension degrades the quality of the routing.

With bias-based balancing, there is no gradient conflict. The router gradients come purely from the language modeling loss. The bias terms steer tokens toward under-utilized experts at the logit level, without affecting the gradient computation.

## 11. Reproducing the Architecture

For those building on DeepSeek V3's design, here are the critical implementation details:

```python
class DeepSeekV3Layer(nn.Module):
    """Complete DeepSeek V3 transformer layer."""
    def __init__(self, config, layer_idx):
        super().__init__()
        self.layer_idx = layer_idx

        # Attention: MLA for all layers
        self.attention_norm = RMSNorm(config.d_model)
        self.attention = MultiHeadLatentAttention(
            d_model=config.d_model,
            num_heads=config.num_attention_heads,
            kv_lora_rank=config.kv_lora_rank,
            q_lora_rank=config.q_lora_rank,
            qk_rope_dim=config.qk_rope_dim,
            qk_nope_dim=config.qk_nope_dim,
        )

        # FFN: Dense for first 3 layers, MoE for rest
        self.ffn_norm = RMSNorm(config.d_model)
        if layer_idx < config.num_dense_layers:
            self.ffn = SwiGLUFFN(config.d_model, config.expert_d_ff * 8)
        else:
            self.ffn = DeepSeekV3MoELayer(config)

        self.is_moe = layer_idx >= config.num_dense_layers

    def forward(self, x, kv_cache=None, position_ids=None):
        # Pre-norm attention
        residual = x
        x = self.attention_norm(x)
        x = self.attention(x, kv_cache=kv_cache, position_ids=position_ids)
        x = residual + x

        # Pre-norm FFN (MoE or dense)
        residual = x
        x = self.ffn_norm(x)

        if self.is_moe:
            x = self.ffn(x)  # Returns combined shared + routed output
        else:
            x = self.ffn(x)

        x = residual + x
        return x
```

## 12. Summary

DeepSeek V3 is a masterclass in systems-level optimization applied to LLM training. Each innovation targets a specific bottleneck:

- **MoE (256 experts)**: Reduces active compute per token by 18x relative to total parameters.
- **MLA**: Reduces KV cache by 4x vs GQA-8, enabling longer contexts and larger batches.
- **FP8 training**: Doubles expert computation throughput on H100/H800 hardware.
- **DualPipe**: Eliminates pipeline bubbles, increasing GPU utilization from ~60% to ~97%.
- **MTP**: Provides richer training signal (+2.5% quality) and enables 1.4x faster inference via speculative decoding.
- **Aux-free balancing**: Eliminates the quality cost of load-balancing auxiliary losses.

The result: frontier-quality at 18x lower training cost. The innovations are modular — each can be adopted independently — but their combination is what makes the economics transformative.
