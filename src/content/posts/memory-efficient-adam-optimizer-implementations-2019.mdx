Adam is the default optimizer for deep learning, but it's memory-hungry: for every model parameter, Adam stores two additional FP32 tensors (momentum and variance), tripling the memory per parameter. For a 7B model in mixed precision, Adam's optimizer states alone consume 56 GB — more than any single GPU can hold. This is the primary motivation for ZeRO, 8-bit Adam, and alternative optimizers.

With mixed precision Adam, a 7B model needs ~98 GB just for weights + optimizer states — before counting gradients (14 GB) or activations. This exceeds the 80 GB of an A100.

<Callout type="perf" title="Where the Memory Goes">
  For a 7B model with Adam in mixed precision: FP16 weights (14 GB) + FP32 master weights (28 GB) + FP32 momentum (28 GB) + FP32 variance (28 GB) + FP16 gradients (14 GB) = **112 GB**. The optimizer states (56 GB) are the single largest component and the primary optimization target.
</Callout>

## Solution 1: 8-bit Adam (bitsandbytes)

Quantize Adam's momentum and variance from FP32 to INT8 using dynamic block-wise quantization:

    { values: ["Training loss (converged)", "1.82", "1.83", "+0.5% — negligible"], highlight: false },
    { values: ["Step time overhead", "0%", "+3%", "Quant/dequant cost"], highlight: false },
    { values: ["Perplexity (Llama-7B fine-tune)", "5.68", "5.71", "+0.03"], highlight: false },
  ]}
/>

8-bit Adam quantizes each block of 2048 values independently, maintaining a per-block scale factor. The 3% step time overhead is negligible compared to the 4× memory savings.

```python
import bitsandbytes as bnb

# Drop-in replacement — same API as torch.optim.Adam
optimizer = bnb.optim.Adam8bit(
    model.parameters(),
    lr=1e-4,
    betas=(0.9, 0.999),
    # Block size for dynamic quantization (default 2048)
)
# Everything else is identical to standard training
```

## Solution 2: Adafactor (Memory-Efficient Alternative)

Adafactor (Shazeer & Stern, 2018) factorizes the second moment matrix into row and column statistics, reducing memory from O(mn) to O(m+n) for weight matrices:

<Benchmark
  title="Adafactor vs Adam Memory"
  columns={["Optimizer", "Memory for [4096×11008] weight", "Total for 7B Model", "Quality"]}
  rows={[
    { values: ["Adam", "2 × 4096 × 11008 × 4 = 344 MB", "56 GB", "Baseline"], highlight: false },
    { values: ["Adafactor (factored)", "(4096 + 11008) × 4 = 60 KB", "~4 GB", "Comparable for LLMs"], highlight: true },
    { values: ["Adafactor (unfactored, 1D)", "Same as Adam for biases", "~5 GB", "—"], highlight: false },
  ]}
  notes="Adafactor factorizes 2D weight matrices but stores full state for 1D parameters (biases, norms)."
/>

For large weight matrices (which dominate LLMs), Adafactor uses **5700× less memory** per parameter. The catch: Adafactor can be less stable than Adam and may require careful learning rate tuning.

## Solution 3: ZeRO Optimizer State Partitioning

ZeRO-1 (DeepSpeed) partitions optimizer states across N GPUs so each GPU stores only 1/N of the states:

ZeRO-1 is the standard approach for multi-GPU training — it adds no communication overhead (same as standard data parallelism) while distributing optimizer states.

<Callout type="tip" title="The Decision Tree">
  **Multi-GPU available?** → ZeRO-1 (free, no quality loss). **Single GPU, fine-tuning?** → 8-bit Adam (drop-in, 4× savings). **Single GPU, pretraining?** → Adafactor (biggest savings, needs LR tuning). **Extreme memory constraint?** → LOMO or GaLore (recent, experimental).
</Callout>

## Conclusion

Adam's optimizer states are the dominant memory consumer in LLM training — 2× the model weight memory. The solution depends on your setup: ZeRO-1 for multi-GPU (zero overhead, zero quality loss), 8-bit Adam for single-GPU (4× savings, drop-in replacement), Adafactor for extreme memory constraints (10×+ savings, needs tuning). These compose multiplicatively — 8-bit Adam + ZeRO-1 on 4 GPUs reduces per-GPU optimizer memory from 56 GB to 3.5 GB, enabling 7B model training on consumer hardware.
