Mixed precision training runs forward and backward passes in FP16 while maintaining a master copy of weights in FP32. The result: near-2× training speedup on tensor core GPUs with negligible accuracy loss — one of the highest-impact optimizations available.

## Why FP16 Is Faster

Two compounding effects:

**Tensor core utilization.** NVIDIA tensor cores (V100+) perform FP16 matrix multiply-accumulate at 2-8× the throughput of FP32 CUDA cores. V100: 125 TFLOPS FP16 vs 15.7 TFLOPS FP32 (8× ratio). A100: 312 TFLOPS FP16 vs 19.5 TFLOPS FP32 (16× ratio for structured sparsity).

**Memory bandwidth.** FP16 tensors are half the size of FP32. This means 2× less data to move through the memory hierarchy — critical because most training operations are memory-bound.

### Why FP32 Master Weights?

FP16 has only 10 bits of mantissa (vs 23 for FP32). When a large weight (e.g., 10.0) receives a tiny gradient update (e.g., 1e-5), the addition `10.0 + 0.00001 = 10.0` in FP16 — the update is lost entirely. FP32 master weights accumulate these small updates correctly, then cast back to FP16 for the next forward pass.

### Why Loss Scaling?

FP16's dynamic range is 5.96e-8 to 65504. Many gradient values fall below 5.96e-8 and become zero in FP16 ("gradient underflow"). Loss scaling multiplies the loss by a large factor (e.g., 1024) before the backward pass, shifting gradients into representable range. After the backward pass, gradients are unscaled by dividing by the same factor.

```python
# The core mixed precision training loop
scaler = torch.cuda.amp.GradScaler()  # Manages dynamic loss scaling

for batch in dataloader:
    optimizer.zero_grad()

    with torch.cuda.amp.autocast():     # FP16 forward + backward
        output = model(batch)
        loss = criterion(output, target)

    scaler.scale(loss).backward()        # Scaled backward pass
    scaler.step(optimizer)               # Unscale gradients, check for inf/nan, step
    scaler.update()                      # Adjust scale factor
```

<Callout type="info" title="Dynamic Loss Scaling">
  PyTorch's `GradScaler` uses dynamic scaling: it starts with a large scale factor (e.g., 2^16) and halves it whenever gradients overflow (inf/nan). If N consecutive steps succeed without overflow, it doubles the scale. This auto-tunes to the largest safe scale for your model.
</Callout>

PyTorch's `autocast` handles this automatically — it maintains a list of FP16-safe and FP32-required operations and casts accordingly. You don't need to manage this manually.

<Callout type="warning" title="Hardware Requirement">
  Mixed precision provides meaningful speedup only on GPUs with tensor cores: V100, A100, H100, RTX 3090/4090, and Intel Gaudi. On older GPUs (P100, GTX 1080), FP16 computation uses regular CUDA cores with no throughput advantage — you get memory savings but not compute speedup.
</Callout>

## BF16: The Modern Alternative

BFloat16 (BF16) keeps FP32's exponent range (8 bits) while reducing mantissa to 7 bits. This eliminates most gradient underflow issues, making loss scaling unnecessary for most models:

On A100/H100/Gaudi hardware, BF16 is the default choice — same throughput as FP16 with fewer numerical issues.

## Conclusion

Mixed precision training delivers 1.8-2× speedup with under 0.1% accuracy loss on standard benchmarks. The implementation is straightforward with PyTorch's `autocast` + `GradScaler`. The three essential components — FP16 compute, FP32 master weights, and loss scaling — work together to maintain numerical stability while exploiting tensor core throughput. For new projects on A100+ hardware, prefer BF16 for its wider dynamic range and simpler workflow.
