WiFi is the dominant power consumer on any ESP32-based IoT device. A single WiFi transmission burst can consume more energy than hours of deep sleep. Understanding the WiFi power state machine at a granular level -- knowing exactly where each milliamp goes during scanning, connecting, transmitting, and receiving -- is the difference between a device that dies in a week and one that runs for a year on batteries. This guide dissects the ESP32 WiFi subsystem's power behavior with real oscilloscope measurements, explores every knob available for optimization, and compares WiFi against BLE for common IoT data collection patterns.

## The WiFi Power State Machine

The ESP32 WiFi subsystem is not simply "on" or "off." It transitions through a series of power states, each with a distinct current signature and duration. Understanding this state machine is essential for predicting and optimizing power consumption.

## DTIM Interval Optimization

DTIM (Delivery Traffic Indication Message) is a mechanism defined in the IEEE 802.11 standard that allows stations to sleep between beacon intervals. The access point buffers frames for sleeping stations and announces buffered data in the DTIM beacon. The ESP32 can sleep between DTIM intervals, waking only to check if the AP has buffered data.

### How DTIM Works

The access point sends beacons at a fixed interval (typically 100 ms). The DTIM interval is a multiple of the beacon interval -- DTIM 1 means every beacon is a DTIM beacon, DTIM 3 means every 3rd beacon is a DTIM beacon.

When the ESP32 is in modem sleep with `WIFI_PS_MIN_MODEM`, it wakes at every DTIM beacon to check the Traffic Indication Map (TIM). If no data is buffered, it immediately returns to sleep. If data is buffered, it stays awake to receive it.

### The DTIM Energy Cost Model

Each DTIM wake event costs energy, even if there is no data to receive. The radio must power up, tune to the channel, receive and decode the beacon, check the TIM, and power down again. This costs approximately:

$E_{wake} = I_{rx} \times t_{wake} = 95\ \text{mA} \times 2\ \text{ms} = 0.19\ \text{mAs}$

The average power during modem sleep is then:

$I_{avg} = \frac{E_{wake}}{T_{DTIM}} + I_{leakage}$

For DTIM 3 with 100 ms beacon interval ($T_{DTIM} = 300$ ms):

$I_{avg} = \frac{0.19\ \text{mAs}}{300\ \text{ms}} + 5\ \text{mA} \approx 5.6\ \text{mA}$

Wait -- that does not match the 18 mA measured value. The discrepancy comes from the ESP32's internal state maintenance between wakes: the WiFi task, RTOS tick, and clock recovery add a significant baseline. In practice, the CPU must remain powered (in modem sleep, the CPU keeps running) which adds approximately 12-15 mA on its own.

This is why **light sleep with WiFi** provides dramatically lower power than modem sleep alone. In light sleep, the CPU is also suspended:

```cpp
#include "esp_pm.h"
#include "esp_wifi.h"

// Configure light sleep with WiFi -- the most power-efficient
// way to maintain a WiFi connection
void setup_light_sleep_wifi() {
    // Initialize and connect WiFi first
    wifi_init_sta();

    // Enable modem sleep (required base for light sleep + WiFi)
    esp_wifi_set_ps(WIFI_PS_MIN_MODEM);

    // Enable automatic light sleep via power management
    esp_pm_config_esp32_t pm_config = {
        .max_freq_mhz = 240,
        .min_freq_mhz = 80,
        .light_sleep_enable = true
    };
    ESP_ERROR_CHECK(esp_pm_configure(&pm_config));

    // Now the CPU automatically enters light sleep when idle.
    // It wakes at each DTIM interval to check for WiFi data,
    // processes any incoming data, then sleeps again.
    // Average current: 1-3 mA (vs 18-25 mA with modem sleep alone)
}
```

<Callout type="info" title="Light Sleep + WiFi Compatibility">
  Light sleep with WiFi works well for applications that primarily receive data (push notifications, MQTT subscriptions). For high-throughput applications or those requiring consistent low-latency responses, modem sleep (`WIFI_PS_MIN_MODEM`) without light sleep is more appropriate. Light sleep adds approximately 1-5 ms of additional wake latency when incoming data is detected.
</Callout>

## Listen Interval Configuration

The listen interval tells the AP how many beacon intervals the station may sleep through before waking. This is distinct from the DTIM interval:

- **DTIM interval**: AP-side setting controlling how often the AP sends DTIM beacons
- **Listen interval**: Station-side setting controlling how many beacon intervals the station skips

With `WIFI_PS_MAX_MODEM`, the ESP32 uses the listen interval to skip DTIM beacons entirely:

```cpp
#include "esp_wifi.h"

void configure_listen_interval() {
    // Set listen interval to 10 beacon intervals
    // If beacon interval = 100 ms, station wakes every 1000 ms
    wifi_config_t wifi_config = {};
    esp_wifi_get_config(WIFI_IF_STA, &wifi_config);
    wifi_config.sta.listen_interval = 10;
    esp_wifi_set_config(WIFI_IF_STA, &wifi_config);

    // Enable max modem sleep to use listen interval
    esp_wifi_set_ps(WIFI_PS_MAX_MODEM);
}
```

## Static IP vs DHCP: The Hidden Power Cost

DHCP negotiation is one of the most variable and often unexpectedly expensive phases of the WiFi connection cycle. A DHCP exchange requires four packets (Discover, Offer, Request, Acknowledge), each requiring the radio to be active. With network congestion or slow DHCP servers, this can take 200 ms to several seconds.

The combination of cached WiFi parameters and static IP eliminates both the scan phase and the DHCP phase, reducing connection energy by 10-20x compared to a cold start:

```cpp
#include "esp_wifi.h"
#include "esp_netif.h"

RTC_DATA_ATTR static uint8_t cached_channel = 0;
RTC_DATA_ATTR static uint8_t cached_bssid[6] = {0};
RTC_DATA_ATTR static bool cache_valid = false;

void fast_wifi_connect() {
    // Configure static IP -- skip DHCP entirely
    esp_netif_t *sta_netif = esp_netif_create_default_wifi_sta();
    esp_netif_dhcpc_stop(sta_netif);

    esp_netif_ip_info_t ip_info = {
        .ip = { .addr = ESP_IP4TOADDR(192, 168, 1, 100) },
        .gw = { .addr = ESP_IP4TOADDR(192, 168, 1, 1) },
        .netmask = { .addr = ESP_IP4TOADDR(255, 255, 255, 0) },
    };
    esp_netif_set_ip_info(sta_netif, &ip_info);

    // Configure DNS
    esp_netif_dns_info_t dns_info = {
        .ip = { .u_addr.ip4 = { .addr = ESP_IP4TOADDR(192, 168, 1, 1) } },
    };
    esp_netif_set_dns_info(sta_netif, ESP_NETIF_DNS_MAIN, &dns_info);

    // Configure WiFi with cached parameters
    wifi_config_t wifi_config = {
        .sta = {
            .ssid = "MyNetwork",
            .password = "MyPassword",
        },
    };

    if (cache_valid) {
        memcpy(wifi_config.sta.bssid, cached_bssid, 6);
        wifi_config.sta.bssid_set = true;
        wifi_config.sta.channel = cached_channel;
        // Skips scan phase: connects in ~200 ms instead of 3-5 seconds
    }

    esp_wifi_set_config(WIFI_IF_STA, &wifi_config);
    esp_wifi_connect();
}

// Call this in the WiFi connected event handler
void on_wifi_connected(wifi_event_sta_connected_t *event) {
    memcpy(cached_bssid, event->bssid, 6);
    cached_channel = event->channel;
    cache_valid = true;
}
```

<Callout type="tip" title="Cache Invalidation Strategy">
  Cached WiFi parameters become invalid when the AP changes channel (e.g., due to DFS radar detection or admin reconfiguration). Implement a fallback: if connection with cached parameters fails within 2 seconds, clear the cache and perform a full scan. This adds a one-time penalty but prevents persistent connection failures.
</Callout>

```cpp
// Robust connection with cache fallback
void connect_with_fallback() {
    if (cache_valid) {
        // Try cached parameters first (fast path)
        fast_wifi_connect();

        // Wait up to 2 seconds for connection
        int retries = 20;
        while (!is_wifi_connected() && retries-- > 0) {
            vTaskDelay(pdMS_TO_TICKS(100));
        }

        if (is_wifi_connected()) {
            return;  // Fast path succeeded
        }

        // Cache miss -- invalidate and fall through to full scan
        cache_valid = false;
        esp_wifi_disconnect();
    }

    // Full scan path (slow but reliable)
    wifi_config_t wifi_config = {
        .sta = {
            .ssid = "MyNetwork",
            .password = "MyPassword",
        },
    };
    esp_wifi_set_config(WIFI_IF_STA, &wifi_config);
    esp_wifi_connect();
    // DHCP or static IP depending on configuration
}
```

## WiFi TX/RX Power Peaks: Understanding the Oscilloscope Traces

The WiFi radio produces sharp current spikes during transmission and reception that differ dramatically from the average current figures commonly cited. Understanding these peaks is important for power supply design and decoupling.

### Transmission Current Profile

During 802.11 frame transmission, the current profile follows this pattern:

1. **Preamble and header**: The radio ramps to full TX power over approximately 2 us. Current rises from idle (~50 mA) to TX level (140-220 mA depending on output power setting).

2. **Data payload**: Current remains at TX level for the frame duration. At 54 Mbps (802.11g), a 1500-byte frame takes approximately 250 us. At 1 Mbps (802.11b), the same frame takes approximately 12 ms.

3. **Inter-frame spacing**: Between frames, current drops to RX level (~95 mA) as the radio listens for ACKs. SIFS (Short Inter-Frame Spacing) is 10 us.

4. **ACK reception**: If the AP acknowledges, the radio receives the ACK frame at ~95 mA for approximately 40 us.

### When WiFi Wins

WiFi is more energy-efficient than BLE when:

1. **Large data transfers**: Above approximately 5-10 KB per transfer, WiFi's higher data rate amortizes the connection overhead. Transferring 100 KB over WiFi costs roughly the same energy as transferring 10 KB over BLE.

2. **Direct cloud connectivity**: If data must reach the internet (cloud server, MQTT broker), WiFi provides direct access. BLE requires a gateway device (phone, hub), adding infrastructure complexity.

3. **High-throughput continuous streaming**: Camera streams, audio, or high-rate sensor data (multiple kHz) are impractical over BLE.

### When BLE Wins

BLE is more energy-efficient when:

1. **Small, infrequent data packets**: For a sensor sending 20-100 bytes every few minutes, BLE uses 10-100x less energy per transfer.

2. **Advertising-only (beacon mode)**: BLE advertising requires no connection at all, consuming approximately 0.1-0.5 mAs per advertising event. Perfect for presence detection, asset tracking, and broadcast sensors.

3. **Always-connected low-data applications**: BLE can maintain a connection at approximately 10-50 uA average (with long connection intervals), while WiFi modem sleep requires 15-25 mA.

```cpp
#include "esp_bt.h"
#include "esp_gap_ble_api.h"
#include "esp_gatts_api.h"

// BLE advertising for sensor broadcast
// No connection needed -- receivers just scan for advertisements
void setup_ble_beacon(float temperature, float humidity) {
    // Encode sensor data into advertising payload
    uint8_t adv_data[8];
    memcpy(&adv_data[0], &temperature, 4);
    memcpy(&adv_data[4], &humidity, 4);

    esp_ble_gap_config_adv_data_raw(adv_data, sizeof(adv_data));

    esp_ble_adv_params_t adv_params = {
        .adv_int_min = 0x0100,  // 160 ms interval
        .adv_int_max = 0x0100,
        .adv_type = ADV_TYPE_NONCONN_IND,  // Non-connectable
        .channel_map = ADV_CHNL_ALL,
    };
    esp_ble_gap_start_advertising(&adv_params);

    // Advertise for 1 second, then stop
    vTaskDelay(pdMS_TO_TICKS(1000));
    esp_ble_gap_stop_advertising();

    // Energy cost: ~25 mA * 1 s = 25 mAs
    // Compare: WiFi transfer = 40-750 mAs
}
```

## Batch Transmission Strategies

Batching multiple sensor readings into a single WiFi transmission is one of the highest-impact optimizations. The energy cost of establishing a WiFi connection is largely fixed regardless of how much data you send, so sending more data per connection amortizes that overhead.

The diminishing returns above batch size 50-100 are because the connection overhead is fully amortized, and the marginal cost per reading is dominated by the TX energy for the additional payload bytes.

### Optimal Batch Size Calculation

The optimal batch size depends on your RTC memory budget, acceptable data latency, and the risk of data loss if the device loses power before transmitting:

$N_{optimal} = \min\left(\frac{RTC_{available}}{S_{reading}}, \frac{T_{max\_latency}}{T_{sample}}, N_{risk\_tolerance}\right)$

Where:
- $RTC_{available}$ is available RTC memory in bytes (typically 4-6 KB after WiFi cache and boot state)
- $S_{reading}$ is bytes per sensor reading (2 bytes for uint16_t fixed-point)
- $T_{max\_latency}$ is maximum acceptable data delay
- $T_{sample}$ is sampling interval
- $N_{risk\_tolerance}$ is maximum readings you can afford to lose

For a typical sensor sampling every 60 seconds with 2-byte readings and 6 KB available RTC memory:

- Memory limit: $6144 / 2 = 3072$ readings
- Latency limit (1 hour max): $3600 / 60 = 60$ readings
- Risk tolerance: 60 readings (1 hour of data)

**Optimal batch size: 60 readings** (limited by latency requirement)

```cpp
#define BATCH_SIZE 60       // Transmit every hour
#define SAMPLE_INTERVAL_US (60ULL * 1000000ULL)  // 60 seconds

RTC_DATA_ATTR static uint16_t readings[BATCH_SIZE];  // 120 bytes
RTC_DATA_ATTR static uint8_t read_index = 0;

void app_main() {
    // Read sensor
    readings[read_index++] = read_sensor_fixed_point();

    if (read_index >= BATCH_SIZE) {
        // Transmit batch
        wifi_connect_cached();
        http_post_batch(readings, BATCH_SIZE);
        wifi_disconnect();
        read_index = 0;
    }

    // Sleep
    esp_sleep_enable_timer_wakeup(SAMPLE_INTERVAL_US);
    esp_deep_sleep_start();
}
```

The direct connect approach (no scan) combined with cached BSSID is always preferred for battery devices. The scan is only necessary on first boot or when the cache becomes invalid:

```cpp
// Optimized scan: only when cache is invalid
// Use async scan to allow timeout control
void scan_if_needed() {
    if (cache_valid) {
        // Skip scan entirely -- connect directly
        direct_connect(cached_channel, cached_bssid);
        return;
    }

    // Need to scan -- use async scan with timeout
    wifi_scan_config_t scan_config = {
        .ssid = (uint8_t *)"MyNetwork",  // Target SSID scan (faster)
        .bssid = NULL,
        .channel = 0,      // All channels
        .show_hidden = false,
        .scan_type = WIFI_SCAN_TYPE_ACTIVE,
        .scan_time = {
            .active = {
                .min = 50,   // 50 ms min per channel
                .max = 120,  // 120 ms max per channel
            },
        },
    };

    esp_wifi_scan_start(&scan_config, false);  // Non-blocking

    // Wait with timeout
    EventBits_t bits = xEventGroupWaitBits(
        wifi_event_group, SCAN_DONE_BIT,
        pdTRUE, pdFALSE, pdMS_TO_TICKS(5000));

    if (bits & SCAN_DONE_BIT) {
        uint16_t ap_count = 0;
        esp_wifi_scan_get_ap_num(&ap_count);
        if (ap_count > 0) {
            wifi_ap_record_t ap_info;
            uint16_t count = 1;
            esp_wifi_scan_get_ap_records(&count, &ap_info);
            // Cache and connect
            cached_channel = ap_info.primary;
            memcpy(cached_bssid, ap_info.bssid, 6);
            cache_valid = true;
            direct_connect(cached_channel, cached_bssid);
        }
    }

    esp_wifi_scan_stop();
    // Critical: free scan memory
    esp_wifi_clear_ap_list();
}
```

## Summary

WiFi power optimization on the ESP32 is not a single tweak but a systematic approach across multiple layers:

1. **Minimize radio-on time**: Cache WiFi parameters, use static IP, batch data, and reduce TX power. Every millisecond of radio-on time costs approximately 0.1-0.2 mAs.

2. **Maximize sleep time**: Deep sleep between transmissions for duty-cycled devices. Light sleep + WiFi for connected devices that need to receive data.

3. **Choose the right protocol**: BLE for small, frequent payloads. WiFi for bulk data and direct cloud connectivity. CoAP/MQTT over raw TCP when HTTP overhead is significant.

4. **Configure DTIM and listen interval**: Work with your access point settings. DTIM 3+ with MAX_MODEM sleep can halve the modem sleep current.

5. **Measure, do not guess**: Use a shunt resistor and oscilloscope to verify your power profile. The difference between a well-optimized and poorly-optimized WiFi implementation is easily 10x in energy consumption.
