SIMD (Single Instruction, Multiple Data) processes multiple data elements per clock cycle using wide registers. On modern x86 with AVX-512, a single instruction operates on 16 floats simultaneously — a theoretical 16× speedup over scalar code. In practice, achievable gains range from 2× to 8× depending on the workload. Understanding when and how to vectorize is a core performance engineering skill.

## Real-World Speedups

Theoretical maximum is `register_width / element_width` (e.g., 8× for AVX2 with floats). Actual speedups are lower due to memory bandwidth limits, data dependencies, and instruction overhead:

Simple arithmetic on contiguous arrays achieves 5-7× speedup. Operations with irregular access patterns or branches see much less benefit. Pointer chasing (linked lists, trees) cannot be vectorized at all.

<Callout type="perf" title="The 3 Requirements for Vectorization">
  SIMD works when data is: (1) **contiguous in memory** — arrays, not linked lists; (2) **independent** — no loop-carried dependencies; (3) **uniform** — same operation on every element, no per-element branching. If any of these are missing, vectorization is limited.
</Callout>

## Auto-Vectorization vs Intrinsics

Modern compilers auto-vectorize simple loops. Here's what works and what doesn't:

```c
// Auto-vectorizes perfectly: contiguous, independent, uniform
void scale_array(float *out, const float *in, float factor, int n) {
    for (int i = 0; i < n; i++)
        out[i] = in[i] * factor;
}
// gcc -O3 -mavx2 generates vmulps (8-wide multiply)

// Does NOT auto-vectorize: loop-carried dependency
void prefix_sum(float *out, const float *in, int n) {
    out[0] = in[0];
    for (int i = 1; i < n; i++)
        out[i] = out[i-1] + in[i];  // Depends on previous iteration
}
```

When auto-vectorization fails, use intrinsics:

```c
#include <immintrin.h>

// AVX2 dot product: processes 8 floats per iteration
float dot_product_avx2(const float *a, const float *b, int n) {
    __m256 sum = _mm256_setzero_ps();
    for (int i = 0; i < n; i += 8) {
        __m256 va = _mm256_load_ps(&a[i]);
        __m256 vb = _mm256_load_ps(&b[i]);
        sum = _mm256_fmadd_ps(va, vb, sum);  // FMA: sum += a * b
    }
    // Horizontal sum of 8 elements
    __m128 hi = _mm256_extractf128_ps(sum, 1);
    __m128 lo = _mm256_castps256_ps128(sum);
    __m128 s = _mm_add_ps(hi, lo);
    s = _mm_hadd_ps(s, s);
    s = _mm_hadd_ps(s, s);
    return _mm_cvtss_f32(s);
}
```

Auto-vectorization gets 80-90% of intrinsics performance for simple operations. For complex patterns (FIR filters, matrix operations, custom reductions), intrinsics can deliver 2× more than auto-vec.

## ARM NEON for Embedded

On ARM Cortex-M4/M7 (and Cortex-A series), NEON provides 128-bit SIMD:

```c
#include <arm_neon.h>

// NEON FIR filter for audio processing
void fir_neon(const float *input, float *output, const float *coeff,
              int n, int taps) {
    for (int i = 0; i < n; i += 4) {
        float32x4_t acc = vdupq_n_f32(0.0f);
        for (int t = 0; t < taps; t += 4) {
            float32x4_t in_vec = vld1q_f32(&input[i + t]);
            float32x4_t co_vec = vld1q_f32(&coeff[t]);
            acc = vmlaq_f32(acc, in_vec, co_vec);  // Multiply-accumulate
        }
        vst1q_f32(&output[i], acc);
    }
}
```

<Callout type="tip" title="ESP32 Note">
  The ESP32's Xtensa LX6 cores don't have NEON, but the ESP32-S3's LX7 cores include a vector extension for 128-bit SIMD operations. For audio DSP on ESP32 (original), use the hardware MAC unit via `dsps_dotprod_f32_ae32` from ESP-DSP.
</Callout>

## Memory Alignment

SIMD loads require aligned memory for peak performance. Misaligned loads have a 10-20% penalty on x86 and may fault on some ARM implementations:

```c
// Aligned allocation (required for best SIMD performance)
float *aligned_buf = (float*)aligned_alloc(32, n * sizeof(float));  // 32-byte for AVX2

// Or with posix_memalign
float *buf;
posix_memalign((void**)&buf, 32, n * sizeof(float));
```

## Conclusion

SIMD delivers 2-7× speedup for data-parallel operations on contiguous arrays. Start with compiler auto-vectorization (`-O3 -mavx2`), verify with `gcc -fopt-info-vec-optimized`, and drop to intrinsics for complex patterns where auto-vec falls short. The prerequisites are straightforward: contiguous data, independent iterations, and uniform operations. If your hot loop meets these criteria and isn't already vectorized, you're leaving significant performance on the table.
