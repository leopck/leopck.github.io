## Introduction

The Transformer's dominance in NLP brought a fundamental architectural bottleneck into focus: **positional encoding**. Traditional absolute embeddings tied a model's understanding of sequence order to a fixed context length. Train on 512 tokens, and the model has no learned representation for token 513.

Two approaches emerged to solve this: **ALiBi** (Attention with Linear Biases, Press et al. 2021) and **RoPE** (Rotary Position Embeddings, Su et al. 2021). Both eliminate the fixed-length constraint, but they take fundamentally different engineering paths — with real consequences for inference performance.

## The Problem: Absolute Embedding Limitations

Traditional transformers add a fixed vector to input embeddings at each position. This creates three engineering problems:

## ALiBi: The Additive Solution

ALiBi sidesteps learned position vectors entirely. Instead of encoding position at the input layer, it injects positional information directly into attention scores by penalizing token distance with a linear bias.

The attention score between query at position *i* and key at position *j* becomes:

**score(i, j) = q_i · k_j - m × |i - j|**

where *m* is a per-head slope (fixed, not learned). Each attention head gets a different slope, creating a multi-resolution view of distance — some heads attend nearby, others attend far.

<Callout type="perf" title="Engineering Advantage">
  Because ALiBi uses a constant penalty based on distance, the model never encounters a "new" position at inference time. It only sees relative distances it already understands from training, enabling extrapolation to much longer sequences without fine-tuning.
</Callout>

## RoPE: The Geometric Solution

Rotary Position Embeddings take a more mathematically elegant approach. Rather than adding a bias to attention scores, RoPE **rotates** the query and key vectors in a complex plane before computing their dot product.

<Theorem type="Definition" title="The Rotation Property">
  RoPE ensures that the dot product of two rotated vectors depends only on their relative distance, not their absolute positions. This is achieved by applying a rotation matrix parameterized by position index and frequency, giving each dimension pair a different "wavelength" of positional sensitivity.
</Theorem>


The key difference: absolute PE modifies the input once, while RoPE modifies Q and K at every attention layer. This means RoPE has slightly higher per-layer compute cost but provides stronger positional signal throughout the network.

At short sequences (512 tokens), all three approaches perform similarly — the positional encoding is a negligible fraction of total compute. At 2048 tokens, ALiBi's quadratic bias matrix starts consuming meaningful memory bandwidth, while RoPE's element-wise rotations scale more gracefully.

## Hardware Considerations

For those working on custom kernels or specialized hardware:

**ALiBi** is easier to implement in standard softmax kernels — it's just an addition to the attention logits. However, the bias matrix grows quadratically with sequence length, making it memory-bandwidth-bound at extreme context lengths (32K+).

**RoPE** requires trigonometric operations (sin/cos precomputation) which add compute cost, but the element-wise rotation has excellent L1/L2 cache locality. On hardware with fast FMA units (NVIDIA Ampere+, Intel Gaudi), the trig overhead is negligible compared to the memory savings.

<Callout type="info" title="Why Modern LLMs Chose RoPE">
  The LLaMA, Mistral, and Qwen model families all adopted RoPE. The combination of minimal memory overhead, strong extrapolation (especially with NTK-aware or YaRN scaling), and compatibility with FlashAttention's fused kernels made it the clear winner for decoder-only architectures deployed at scale.
</Callout>

## Conclusion

For production systems today, **RoPE is the standard choice** — it offers the best balance of memory efficiency, extrapolation capability, and kernel compatibility. ALiBi remains interesting for its simplicity and zero-parameter design, but the ecosystem has moved decisively toward rotation-based embeddings.

The choice matters most at long context lengths (4K+ tokens) where positional encoding overhead becomes measurable. At shorter sequences, both approaches perform nearly identically, and your optimization effort is better spent on attention computation itself.
